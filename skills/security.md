# skills/security.md

## 目的（WHAT）

OgoRoulette において、以下を守る：

- 不正操作を防ぐ
- ユーザーデータを守る
- 安全に認証・認可する
- 信頼できるアプリにする

---

## なぜ重要か（WHY）

このアプリは：

- 奢り（お金）が関わる
- ユーザー間の関係がある
- 不正が疑われやすい

そのため：

- 「安全であること」
- 「安全に見えること」

両方が必要

---

## 基本原則（HOW）

### 1. 入力はすべて信用しない

対象：

- フォーム入力
- URLパラメータ
- クエリ
- APIリクエスト

対応：

- バリデーションする
- 型チェックする
- 想定外値を拒否する

---

### 2. 認証と認可は別物

認証（Authentication）

- ログインしているか

認可（Authorization）

- その操作をして良いか

例：

- ログインしていても
- 他人のグループは操作不可

---

### 3. フロントは信用しない

- フロント制御だけに依存しない
- サーバー側で必ずチェックする

---

### 4. 最小権限の原則

- 必要な権限だけ与える
- 不要なアクセスを許可しない

---

## 認証（WHAT）

### 使用方針

- Supabase Auth（Googleログイン）

---

### 実装ルール（HOW）

- トークンはサーバーで検証する
- セッションの有効性を確認する
- 未ログインは必ず拒否する

---

### 禁止

- フロントだけでログイン判定
- トークン未検証
- 仮ログイン状態での処理

---

## 認可（WHAT）

### 必須チェック

- グループ所属確認
- 操作対象の所有権確認

---

### 例

- グループ作成 → OK
- グループ参加者のみ操作可能
- 他人のデータ → NG

---

### 実装ルール（HOW）

- userIdで必ず照合する
- groupIdと紐付ける
- DBレベルで制約も意識する

---

## データ保護（WHAT）

### 実装ルール（HOW）

## APIセキュリティ（HOW）

## OgoRoulette 特有: ゲストセッション認可

### 問題

ゲストはログインしていないため、`roomId` だけで POST /api/sessions を呼べる。
roomId を知っている第三者がセッション（結果・金額）を保存できてしまう。

### 対策: roomId の存在・ステータス検証

POST /api/sessions でゲスト保存時は必ず以下を確認する：

```ts
if (roomId) {
  const room = await prisma.room.findUnique({ where: { id: roomId } })
  if (!room)                      return 404 // 存在しない
  if (room.status === "EXPIRED")  return 403 // 期限切れ
  if (room.status === "COMPLETED") return 409 // 重複保存防止
}
```

これで：
- 無効な roomId での保存を防ぐ
- COMPLETED ルームへの2重保存を防ぐ
- EXPIRED ルームへの保存を防ぐ

### 将来対応: guestToken

より強固にするには：
- ルーム作成時に `crypto.randomUUID()` を生成し DB + localStorage に保存
- セッション保存時に guestToken を照合する
- ただし DBスキーマ変更が必要（prisma migration）

## 禁止事項

- 検証なし実装
