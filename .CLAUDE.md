## Project: OgoRoulette

OgoRoulette is a social roulette app designed to make group payment decisions fun, engaging, and shareable.

The goal is to deliver a **fast, intuitive, and memorable experience** where users can:
- Create a room
- Add members
- Spin a roulette
- Decide who pays
- Share the result

---

## Core Product Principles

- Speed over complexity
- Intuitive UX (no explanation needed)
- Mobile-first design
- One flow = one goal completion
- Avoid friction at all costs
- Always think: "Can first-time users finish this in under 30 seconds?"

---

## Development Priorities

1. Core flow must never break
   - Room creation
   - Member input
   - Roulette execution
   - Result display

2. Errors must be visible and understandable
3. No silent failures
4. No broken states after API calls
5. Avoid 500 errors at all costs

---

## Critical Flows

### Guest Flow
- Enter → Create Room → Add Members → Spin → Result

### Logged-in Flow (future)
- Login → Create Room → Save Groups → Reuse Members

---

## Architecture Principles

- Frontend: Next.js (App Router)
- Backend: API Routes / Supabase
- State: SWR / client state
- DB: Prisma + Supabase
- Auth: Supabase Auth (Google login planned)

---

## Environment Variables Rules

- Never assume env exists
- Always validate before use
- Fail gracefully, never crash

Example:
- GUEST_HOST_SECRET must be checked before usage
- If missing → return clear error, not 500

---

## Error Handling Rules

- Never throw raw errors to user
- Always return structured error responses
- Always log server errors with clear context

Bad:
- 500 with no message

Good:
- "Guest room creation is temporarily unavailable"

---

## UX Rules

- Buttons must always have visible feedback
- Loading state is mandatory
- Disable double submit
- Show clear success / failure states

---

## API Rules

- No silent failure
- Always return JSON
- Include success / error states explicitly

## Development Workflow

### Before Implementation
- Clarify UX flow
- Validate necessity (avoid overbuild)

### During Implementation
- Keep components small
- Avoid deep nesting
- Handle loading/error states first

### After Implementation
- Run /review
- Run /qa
- Confirm no console errors
- Confirm no 500 errors

---

## Bug Fix Rules

- Always identify root cause (not symptoms)
- Never patch blindly
- Add logs if unclear
- Reproduce before fixing

---

## Deployment Rules

Before deploying:

- No 500 errors
- No hydration errors
- No console errors
- Core flow works 100%
- Env variables configured

Use:
- /qa
- /review
- /investigate (if needed)

---

## Known Risk Areas

- Guest room creation (env dependent)
- Hydration mismatch
- State inconsistency (SWR vs local)
- API error handling
- Group persistence logic

---

## Absolute Anti-Patterns

- Ignoring errors
- Hardcoding env-dependent logic
- Silent API failures
- UI with no feedback
- Fixing without reproduction

---

## Final Rule

Always ask:

"Would a first-time user succeed without thinking?"

If not, it is a bug.
