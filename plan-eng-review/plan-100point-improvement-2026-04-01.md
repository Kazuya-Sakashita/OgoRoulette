## 2. 100点状態の定義

各軸の「完成形」を定義する。

### Reliability (95)
- iPhoneを含む全主要端末で、全フローが初回から壊れなく動く
- デプロイブロッカーゼロ。ルームに参加すると全員の名前が正しく表示される
- 録画失敗はgraceful degradationで静止画にフォールバックし、ユーザーに不明確なエラーが見えない

### Speed (88)
- ランディング→ルーレットスタートまで3タップ以内
- スピン後、結果確定から「シェアできる状態」まで2秒以内
- ページ遷移はすべて体感100ms以下（Prefetch + SSR活用）

### UX (92)
- 非ログインでも「何ができるか」が30秒以内に分かる（デモスピン）
- ルームへの参加フローが直感的（QRスキャン失敗なし、コード入力補完）
- モバイルでのタッチ操作が全てのボタンで確実に機能

### Emotion (90)
- 当選確定の瞬間に「叫びたくなる」演出（near-miss、サウンド、フラッシュ）
- 「誰が奢るか」という緊張感がカウントダウンで最大化される
- 当選者の名前がルーレットから飛び出すような視覚的クライマックス

### Share+Growth (85)
- 感情ピーク（当選確定の瞬間）に1タップでシェアできる
- iOS含む全プラットフォームで動画または高品質GIFをシェアできる
- シェアされた投稿が「見た人も使いたくなる」OGP・動画品質になっている

### Trust+Safety (90)
- 公開名（displayName）が全画面・全フローで一貫して使われる（本名流出ゼロ）
- OGP・シェアテキストにも本名が絶対に含まれない
- ルームの参加コードが予測不可能、招待URLが短期有効

## 4. 項目別の改善ポイント

### Reliability（信頼性）

**現状の問題点**
- iPhone QRスキャン: `getUserMedia` の失敗ハンドリングが不十分。エラー時に手動入力へのフォールバックがない
- join API (`/api/rooms/[code]/join`): `user.user_metadata.name || user.user_metadata.full_name` を直接 nickname に使用しており、getDisplayName() をバイパスしている
- 録画失敗: MediaRecorder の ondataavailable が空の場合のフォールバックは実装済みだが、ユーザーへの通知がない

**具体的な修正箇所**
- `app/scan/page.tsx` or `components/qr-scanner.tsx`: camera init エラー時に手動コード入力へ自動切替 + 「カメラが使えません。コードを入力してください」のフレンドリーメッセージ
- `app/api/rooms/[code]/join/route.ts`: nickname の生成を `getDisplayName(user)` に統一
- `lib/use-video-recorder.ts`: 録画失敗時に `onRecordingFailed` コールバックを発火し、UIに「動画なしでシェア」への切替を提示

### Speed（速度体感）

**現状の問題点**
- landing→play まで4ステップ（landing → login → home → room → play）
- スピン後の動画エンコードに数秒かかり、シェアシートが開くまでブロックされる
- ルーレットのスピン時間 8.3 秒は設計通りだが、「待ち」の演出が足りない

**具体的な改善箇所**
- 非ログインユーザー向けに landing → home → spin を2タップで到達できるようにする（ゲストフロー短縮）
- `use-video-recorder.ts`: 録画とシェアシート表示を非同期分離。録画完了を待たずにシェアシートを開き、録画完了後にボタンをunlockする
- スピン中カウントダウンの最後3秒に「あと3, 2, 1...」のテキストアニメーションを追加して体感時間を短縮

### UX（使いやすさ）

**現状の問題点**
- ランディングページがログインボタンのみで「何ができるか」が分からない（コンバージョン最大の障壁）
- 非ログイン状態でのルーム作成は名前入力が必須だが、ゲストとしての制限が説明されていない
- モバイルの削除ボタンはQAレポート修正済みだが、長押し→削除のようなより自然なジェスチャーへの移行を検討

**具体的な改善箇所**
- `app/page.tsx`: ランディングページにデモルーレット（3人固定メンバー、1クリックでスピン体験）を追加 (issue-096)
- 非ログインユーザーのルーム作成後のロビーに「アカウント登録でメンバー保存・履歴が使えます」バナーを表示
- `app/room/[code]/page.tsx`: Supabase Realtimeによるリアルタイム参加者表示 + 参加トースト (issue-100)

### Emotion（感情設計）

**現状の問題点**
- near-miss演出がない（最後の数秒で「惜しい...」という緊張感が生まれない）
- 当選確定の瞬間のビジュアルが紙吹雪のみで弱い（issue-092）
- BGM/サウンドエフェクトはあるが、当選時の音がファンファーレ3音で地味

**具体的な改善箇所**
- `components/roulette-wheel.tsx`: スピン終了2マス前に意図的に「一瞬止まりかける」デセレーションを追加し、near-miss感を演出 (issue-098)
- WinnerCard Phase A: confetti量を現状の3倍に、当選者名のスケールアニメーションを0.5→1.2倍に
- `lib/spin-sound.ts`: 当選音をファンファーレから「ドラムロール→ドン！」のパターンに変更し、スピン終了前からサウンドで盛り上げる

### Share+Growth（拡散力）

**現状の問題点**
- iOSでは動画シェアが原理的に不可能（captureStream未対応）→ PNG静止画のみ。「動画で盛り上がり」が伝わらない
- 当選確定からシェアまで3ステップ。感情が冷める
- OGPとXシェアのビジュアルが未最適化。シェアされた投稿が広がりにくい

**具体的な改善箇所**
- `components/winner-card.tsx`: Phase A（当選確定全画面）に「今すぐシェア（画像）」ボタンを配置。動画完了を待たない即シェアパスを提供 (issue-093)
- iOS向け: `canvas` フレームを gifshot または gif.js で GIF化してシェア (issue-105)。30fps × 3秒 = 90フレームで十分。ファイルサイズはpalette最適化で3MB以内に収める
- OGP: 当選者名・ルーム名をダイナミックに含む `@vercel/og` での画像生成。Xカード表示で目立つ黒背景×紫グラデーションデザイン (issue-012, issue-097)

### Trust+Safety（安心感）

**現状の問題点**
- 参加時の本名流出 (G-02) が最も深刻。全員の画面に本名が映る状況が今でも発生している
- シェア時にも本名が含まれる可能性がある（shareテキスト生成ロジック要確認）
- ルームの招待コードはランダムだが、URLで公開されるため推測・スクレイピングのリスクがある

**具体的な改善箇所**
- join APIの修正 (G-02と同じ): `getDisplayName(user)` を全API・全ページで一貫して使用
- `lib/share-service.ts`: シェアテキスト生成時に `winner.nickname` ではなく `winner.displayName || winner.nickname` を優先
- 中期的に: ルーム招待コードに有効期限の表示をUIに追加し、ユーザーが「古いリンクは使えない」ことを認識できるようにする

## 9. 最終提案

**OgoRouletteが「62点→90点」に到達するための本質的な変化は2つ。**

**第一に、信頼の確立。**

本名が画面に映るプライバシー問題（issue-090）とiPhoneでQRが使えない問題（issue-089）は、今この瞬間も起きている。これが直っていない限り、プロダクトをSNSや口コミで広めることができない。「バグのあるプロダクトを人に勧める人はいない」という原則通り、このフェーズで最初にやることはここだ。工数の合計は1時間以内。やらない理由がない。

**第二に、感情の連鎖設計。**

OgoRouletteの本質的な価値は「誰が奢るか決まる瞬間の盛り上がり」だ。その瞬間が今、孤立している。盛り上がりが画面の中に閉じていて、外に出ていかない。WinnerCardに即シェアボタンを置き、near-miss演出で緊張を高め、OGPで見た人も「これ何？」と思わせる連鎖を作る。これがShare+Growthスコアを45→85に引き上げる唯一の道だ。

**行動優先順位をひとことで言うと:**

> 壊れているものを直す（Phase 1） → 盛り上がりを外に出す（Phase 2） → 来てもらえる仕組みを作る（Phase 3）

この3フェーズを2週間で完走すれば、総合スコアは87〜90に届く。残りの10点は体験の磨きと分析による最適化で積み上げていく。
