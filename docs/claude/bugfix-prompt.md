# bugfix-prompt.md

skills/testing.md を必ず参照し、bugfix は再発防止テストまで含めて対応してください。

---

# 目的（WHAT）

OgoRoulette の不具合を、
その場しのぎではなく「原因特定・安全な修正・再発防止」まで含めて解決する。

---

# 背景（WHY）

このアプリでは、以下の不具合が信頼性を大きく下げる：

- 金額計算ミス
- 一部奢り表示漏れ
- ルーレット結果と表示のズレ
- 多重実行
- 画面遷移の不自然さ
- 過剰な再取得
- 状態不整合

そのため、バグ修正は「直ったように見える」ではなく、
「原因が説明できる」「再発しにくい」状態を目指す。

---

# 基本方針（HOW）

バグ修正では以下を必ず守る：

1. 原因が分かるまで修正しない
2. 仮説だけでコードを書かない
3. 最小差分で直す
4. 副作用を確認する
5. 必要なら再発防止テストを追加する
6. test-prompt.md の観点を活用する

---

# 必須プロセス

必ず以下の順で進める：

1. WHAT：何が起きているか
2. 再現：どうすれば起きるか
3. 原因：なぜ起きるか
4. HOW：どう直すか
5. 検証：直ったことをどう確認するか
6. 再発防止：どのテストを追加するか

---

# 対応してほしいこと

## 1. WHAT（現象整理）

- 何が起きているか
- 期待動作との違い
- どの画面 / ロジック / 状態で発生しているか

---

## 2. 再現手順

- 再現手順を具体的に書く
- 必要な前提条件を書く
- 毎回起きるか、条件付きかを書く

---

## 3. 原因特定（最重要）

以下を明確にする：

- 根本原因
- どのコードが問題か
- なぜそのコードで不具合が起きるか
- 他に原因候補がある場合は優先順位付きで整理する

推測で終わらず、ロジックとして説明すること。

---

## 4. 修正方針

以下を整理する：

- どこを直すか
- なぜその修正で直るか
- 影響範囲
- 最小差分で済むか
- refactor を混ぜるべきでないか

---

## 5. 修正コード

- 必要最小限の差分で提示する
- 関係ない変更を含めない
- 必要なら責務分離する
- UIでなく pure function 側で直すべきならそうする

---

## 6. 検証方法

以下を提示する：

### 正常系
- 期待通り動くか

### 異常系
- 想定外入力で安全に失敗するか

### 境界値
- 0件 / 1件 / 空値 / 最大値 / 最小値 など

### UI確認
- 表示崩れがないか
- 表示とロジックが一致するか

---

## 7. 再発防止テスト（必須）

skills/testing.md と test-prompt.md の観点を使って、
再発防止のために必要なテストを提案・追加すること。

必ず整理する：

- どの種類のテストか
  - Unit Test
  - Integration Test
  - UI Test

- なぜそのテストが必要か
- 何を保証するか
- 今回のバグをどう防ぐか

# 出力形式

## WHAT
- 何が起きているか

## 再現手順
- どうすれば起きるか

## 原因
- 根本原因
- 問題箇所

## 修正方針
- 何をどう直すか

## 修正コード
- 最小差分

## 検証
- 正常系
- 異常系
- 境界値
- UI確認

## 再発防止テスト
- 追加すべきテスト
- テストコード
- 何を保証するか

---

# OgoRoulette 特有の重点チェック

## 金額計算
- 合計金額と支払い合計が一致するか
- 一部奢りで金額欠けがないか
- 端数処理が正しいか

## ルーレット
- 当選者表示と内部結果が一致するか
- 同じ抽選を不正に何度も回せないか

## 結果表示
- 本人の支払い金額が見えるか
- 全員分の支払い表が見えるか
- オーナー / メンバーの必要情報が揃うか

## ルーティング
- 行き止まり画面になっていないか
- ホームへ戻れるか
- back依存になっていないか

## 状態管理
- 多重実行が起きないか
- 過剰fetchが起きないか
- loading / success / error が破綻しないか

---

# 禁止事項

- 原因不明のまま修正する
- とりあえず if を足す
- バグ修正に無関係なリファクタを混ぜる
- 再現できないまま終わる
- テストなしで「直った」と判断する
- 手動確認だけで済ませる
- 表示だけ直してロジックを放置する

---

# 良い修正の条件

- 原因が説明できる
- 修正が最小
- 副作用が少ない
- 表示とロジックの両方が直る
- 再発防止テストがある
- 将来の変更にも耐えやすい

---

# 使用するskills

- skills/testing.md（必須）
- skills/frontend.md（UI起因なら）
- skills/roulette.md（結果 / 金額 / 抽選起因なら）
- skills/state.md（状態不整合 / 過剰fetchなら）
- skills/routing.md（遷移起因なら）
- skills/security.md（認証 / 認可 / 入力起因なら）

---

# 最終指示

あなたは単なるデバッガではありません。
原因を特定し、安全に修正し、再発防止テストまで含めて品質を上げるエンジニアとして振る舞ってください。
