# Claude プロンプト運用ガイド（OgoRoulette）

本ディレクトリは、Claude Code を用いた開発のためのプロンプトを管理する。

目的は以下：

- 開発品質の安定化
- 実装スピード向上
- 判断のブレ防止
- チームでの再現性確保

---

# ディレクトリ構成

/docs/claude/
  ├── initial-prompt.md   # 初期指示（最初に必ず渡す）
  ├── review-prompt.md    # コードレビュー用
  ├── bugfix-prompt.md    # バグ修正用
  ├── refactor-prompt.md  # リファクタリング用
  ├── test-prompt.md      # テスト追加・検証用
  └── README.md           # 本ドキュメント

---

# 基本原則

## Less is more
- プロンプトは短く保つ
- 必要な情報のみ渡す

## Progressive Disclosure
- 必要な情報だけ追加する
- 最初から全部渡さない

## WHAT / WHY / HOW
- WHAT：何をするか
- WHY：なぜやるか
- HOW：どうやるか

## 検証必須
- 実装だけで終わらない
- 必ず確認方法を持つ

# 推奨ワークフロー

## 開発フロー

1. initial-prompt
2. 実装
3. review-prompt
4. 修正
5. 必要に応じて refactor-prompt

---

## バグ対応フロー

1. bugfix-prompt
2. 原因特定
3. 修正
4. 検証
5. review-prompt（最終確認）

---

# ベストプラクティス

## 1. 小さく回す
- 一気にやらない
- 小さい単位で依頼する

## 2. 情報は具体的に

良い例：
エラー内容：
再現手順：
該当コード：

悪い例：
動きません

## 3. 1プロンプト1目的
- 実装とリファクタを混ぜない
- バグ修正と設計変更を混ぜない

## 4. 差分を意識
- 必要最小限の変更にする
- 無関係な変更を避ける

---

# よくあるアンチパターン

- 全部まとめて依頼する
- 原因不明のまま修正させる
- プロンプトを使い分けない
- 検証しない
- 「なんとなく」で修正する

# 最後に

Claudeは魔法ではない。
「良い指示」が「良いコード」を生む。

- 明確に伝える
- 小さく進める
- 検証する

これを徹底すること。
