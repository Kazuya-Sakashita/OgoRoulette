# ルーティング設計の崩壊と再構築

## 問題の始まり

「戻る」は簡単に見える。でも実際には難しい。

OgoRouletteでは最初、こういうルーティングだった：

```
ホーム → ルーム → プレイ → (結果モーダル)
         ↑
         プレイの「戻る」ボタンの向き先
```

プレイ画面の戻るボタンが `/room/${code}` に向いていた。

直感的には「プレイの前はルームだったから」という理由だ。正しいように見える。

---

## 何が起きたか

### パターン1：普通の完了後

```
ルーレット完了 → 結果モーダル表示
     ↓
「閉じる」を押す（モーダルが消える）
     ↓
プレイ画面に戻る（でも何も操作できない）
     ↓
「どうすればいいの？」
```

プレイ完了後のプレイ画面には「もう回せない」状態のホイールだけが残る。
次の行動が何もない。行き止まり。

### パターン2：ブラウザバックを使われたとき

```
プレイ完了 → ← ← ← と戻る
     ↓
ルーム画面
     ↓
また ← と戻る
     ↓
ホーム
```

2回バックが必要。
飲み会中にスマホを渡されて「次の人に渡して」という状況で2回タップは多い。

---

## なぜこうなったのか

**「物理的なフロー」と「ユーザーの行動フロー」を混同していた。**

物理フロー（画面の遷移順）：
```
ホーム → ルーム → プレイ → 結果
```

ユーザーの行動フロー（何をしたいか）：
```
プレイ完了 → 家に帰る（ホームへ戻る）
           → 次の幹事を決める（新しい抽選）
```

ユーザーはプレイが終わったらルーム画面に戻りたくない。
ルームに戻っても何もない。用が済んでいる。

---

## 修正の方針

`skills/routing.md` に整理した原則：

> **一時画面（play）の戻るボタンはルームを経由せずホームへ直接向ける**
> 「play画面はルームへの中継点ではなく、完結した一時画面である」

具体的な変更：

```tsx
// Before
<Link href={`/room/${code}`}>
  <ArrowLeft className="w-5 h-5" />
</Link>

// After
<Link href="/">
  <ArrowLeft className="w-5 h-5" />
</Link>
```

そしてルーム名をリンクにした：

```tsx
// ルーム名をタップするとルームに戻れる（代替導線）
<Link href={`/room/${code}`} className="flex-1 min-w-0">
  <h1 className="text-lg font-bold text-foreground truncate">
    {room.name || "ルーレット"}
  </h1>
</Link>
```

これで：
- 戻るボタン → ホーム（1タップで終わる）
- ルーム名タップ → ルーム（戻りたい人用）

の2つの導線を両立した。

---

## 結果画面（COMPLETED状態）の整備

ルーレット完了後、画面に何も残らない問題も直した。

```tsx
{isCompleted ? (
  <div className="flex flex-col items-center gap-3 w-full max-w-[280px]">
    <Button onClick={showResult}>
      結果を見る
    </Button>
    <Button asChild>
      <Link href="/">ホームへ戻る</Link>
    </Button>
    {isHost && (
      <Button asChild>
        <Link href="/room/create">新しい抽選を作る</Link>
      </Button>
    )}
  </div>
) : ...}
```

「結果を見る」「ホームへ戻る」「新しい抽選を作る」が並ぶ。
行き止まりがなくなった。

---

## WinnerCard（結果モーダル）の導線

モーダル内にも同じ問題があった。

```tsx
// Before
// 「閉じる」ボタンだけ → 閉じたら何も残らない

// After
<Button asChild>
  <Link href="/room/create" onClick={onClose}>新しい抽選を作る</Link>
</Button>
<Button asChild>
  <Link href="/" onClick={onClose}>ホームへ戻る</Link>
</Button>
```

モーダルを閉じてもプレイ画面のCOMPLETED状態があるので、
どこからでも次の行動に進めるようになった。

## ルーティング設計で学んだこと

### 1. 画面の「役割」を先に決める

| 画面 | 役割 |
|------|------|
| ホーム | 拠点 |
| ルーム | 準備・招待 |
| プレイ | 一時画面 |
| 結果(モーダル) | 情報表示 |

「一時画面」からの戻り先は「直近の拠点」ではなく「ユーザーが次に行きたい場所」。

### 2. `router.back()` を信用しない

```tsx
// これはやらない
router.back()

// これをやる
<Link href="/">ホームへ戻る</Link>
```

理由：
- 外部リンクから来たユーザーにはbackが機能しない
- リロードするとhistoryが壊れる
- 行き先が不明確

### 3. 行き止まり画面を作らない

ユーザーが次に何をすべきか分からない状態を作らない。
特に「重要な結果を表示した後」は必ず次の導線を用意する。

---

## skills/routing.md への反映

この経験は `skills/routing.md` に以下として追記した：

```markdown
### play 画面（ルーレット実行画面）

play 画面は「一時画面」として扱う。

- 戻るボタン（←）は `/room/${code}` ではなく `/` へ向ける
- ルーム画面へ戻りたい場合はルーム名リンク（タイトル部）で対応する

理由：
- play 完了後にルーム画面を経由することにユーザーメリットがない
- 「何も考えずにホームへ戻れる」を最優先にする
```

コード修正だけでなく、ルールとして文書化することで次のセッションでも再現できる。
