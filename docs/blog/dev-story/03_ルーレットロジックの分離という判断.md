# ルーレットロジックの分離という判断

## 「誰が当選するか」と「ルーレットが止まる位置」は別の問題

最初の実装では、当選者の決定はルーレットのアニメーション完了後に行っていた。

```
アニメーション実行
     ↓
ホイールが止まる
     ↓
止まった位置から当選セグメントを逆算
     ↓
当選者を決定
```

一見合理的に見える。でもこれは問題だった。

---

## 問題：アニメーションとロジックが密結合

**問題1：テストできない**

「誰が当選するか」をテストしようとすると、アニメーションを実行しなければならない。
Framer Motionの `animate` を含むコードは単体テストが非常に難しい。

**問題2：ランダム性が見えない**

```tsx
// この中のどこで当選者が決まっているのか分からない
const anim = animate(rotation, targetRotation, {
  duration: 4 + Math.random(),
  ease: [0.2, 0.8, 0.2, 1],
  onComplete: () => {
    // ここで初めて当選者が計算される
    const winner = calculateWinnerFromAngle(rotation.get())
  }
})
```

ランダム性の発生源がアニメーションのコールバックに埋まっている。

**問題3：「演出と結果がズレる可能性」**

`skills/roulette.md` にはこう書いてある：

> 演出と結果がズレている → 不信感の原因

もしアニメーション中に状態が変わってズレが起きたら、
「当選者と止まった位置が違う」という最悪のケースになる。

---

## 解決策：先に当選者を決める

```
当選者を先に決定（ランダム）
     ↓
その当選者のセグメントに止まるよう角度を計算
     ↓
アニメーション実行
```

コードで見ると：

```tsx
// play/page.tsx: ボタンを押した時点で当選者を決定
const handleSpin = () => {
  if (!isSpinning && !isCompleted && participants.length >= 2) {
    // ① まず当選者のインデックスを決める
    const idx = Math.floor(Math.random() * participants.length)
    setPendingWinnerIndex(idx)
    setIsSpinning(true)
    setWinner(null)
  }
}
```

```tsx
// roulette-wheel.tsx: targetWinnerIndex を受け取って角度を計算
const resolvedIdx = (targetWinnerIndexRef.current !== undefined)
  ? ((targetWinnerIndexRef.current % segments) + segments) % segments
  : Math.floor(Math.random() * segments)  // フォールバック（テスト用）

// ポインタは上(0°)。セグメント0は-90°から始まる
const targetEffectiveAngle = resolvedIdx * segmentAngle + segmentAngle / 2
const targetNormalized = (360 - targetEffectiveAngle % 360 + 360) % 360

// 現在位置から目標まで必ず前方向に回転
let angleDiff = targetNormalized - currentNormalized
if (angleDiff <= 0) angleDiff += 360

const targetRotation = currentRotation + (360 * minSpins) + angleDiff
```

## 角度計算の詳細

ルーレットホイールの座標系：

```
         ↑ ポインタ（0°）
         │
    270° ┼ 90°
         │
         ↓ 180°
```

セグメント0は -90°（上）から始まる。
N人参加の場合、各セグメントは `360/N` 度。

当選者インデックスが `i` の場合：
- そのセグメントの中心角 = `i * (360/N) + (360/N)/2`
- ポインタに来るべき回転量（正規化）= `(360 - セグメント中心角 + 360) % 360`

さらに `minSpins`（最低5回転）を加えて、なめらかに止まるよう計算する。

## この分離で得られたもの

### 1. テスタビリティ

当選者の決定ロジックは純粋な計算になった：

```ts
// テスト可能
const idx = Math.floor(Math.random() * participants.length)
// → 0以上、participants.length未満の整数になること
// → indexが範囲内であること
// → 毎回異なる値になること（確率的に）
```

### 2. アニメーションの独立性

ルーレットコンポーネントは「どこに止まるか」だけを担当する。
「誰が当選か」は外から決める。

```tsx
<RouletteWheel
  isSpinning={isSpinning}
  participants={participants}
  targetWinnerIndex={pendingWinnerIndex}  // 外から決める
  onSpinComplete={handleSpinComplete}
/>
```

コンポーネントの責務が明確になった。

### 3. セッション保存の信頼性

先に当選者が決まっているので、
「アニメーション完了後にAPIを叩いたら当選者が違った」が起きない。

```tsx
const handleSpinComplete = (winnerName: string, winnerIndex: number) => {
  // winnerName は pendingWinnerIndex から決まった名前のはず
  // アニメーションが担保している
  fetch("/api/sessions", {
    body: JSON.stringify({ winnerName, ... })
  })
}
```

## 金額計算の純粋関数化

同じ思想で、金額計算も純粋関数に切り出した。

```ts
// lib/bill-calculator.ts
export function calculateBillSplit(
  totalBill: number,
  treatAmount: number,
  participantCount: number
): BillSplit {
  const bill = Math.max(0, Math.floor(totalBill))
  const treat = Math.min(Math.max(0, Math.floor(treatAmount)), bill)

  const isActive = bill > 0
  if (!isActive || participantCount < 2) {
    return { remainingAmount: 0, splitAmount: 0, isActive }
  }

  const remainingAmount = bill - treat
  const nonWinnerCount = participantCount - 1
  const splitAmount = nonWinnerCount > 0
    ? Math.ceil(remainingAmount / nonWinnerCount)
    : 0

  return { remainingAmount, splitAmount, isActive }
}
```

UIに依存しない。引数を渡せば結果が返る。
20のテストケースで守られている。

---

## 学び

> **「ランダム選択」と「それを見せるアニメーション」は別の責務。分離しないとテストも修正も難しくなる。**

> **金額・当選者といった「信頼が必要な計算」は純粋関数に寄せる。UIのstate変化に引きずられない。**

これは OgoRoulette 特有の話ではなく、
「表示ロジックと業務ロジックの分離」という普遍的な原則だった。
