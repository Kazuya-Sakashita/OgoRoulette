# 本名がUIに出続けた理由：名前フィールドの責務設計

## 概要

OgoRoulette に表示名（公開名）機能を実装した後も、
いくつかの画面でユーザーの本名やプロバイダ由来名が表示され続けていた。

「名前」という単純に見えるフィールドが、実は複数の責務を持ち、
画面ごとに参照元がバラバラになっていたのが原因だった。

---

## 背景

OgoRoulette はソーシャル前提のアプリだ。

- 「誰が奢る？」をルーレットで決める
- 結果画面をスクリーンショットして共有する
- 他人と一緒に使う

このため、UI に表示される名前は「公開してよい名前」だけにすべきだった。
しかし Google や LINE でログインすると、アカウント名がそのまま表示名として使われる。
本名でアカウントを作っているユーザーは、意図せず本名が画面に出ることになる。

対策として「公開名（display_name）」機能を実装した。
ユーザーが自分で好きな名前を設定でき、SNS シェアやルーム表示にはその名前が使われる。

実装後、シェア機能と結果画面では正しく公開名が使われていた。
しかし問題が残っていた。

---

## 問題1：home画面の挨拶

home 画面の挨拶文に本名が表示されていた。

```tsx
// app/home/page.tsx
ようこそ、<span>{user.user_metadata?.full_name || user.email?.split('@')[0]}</span> さん
```

`user.user_metadata.full_name` は Supabase Auth が OAuth プロバイダから取得した名前をそのまま保持する。
Google アカウントであればフルネーム（「田中 一郎」など）が入る。

同じファイルの中に、すでに以下が存在していた。

```ts
// line 31 — 既にインポート済み
import { getDisplayName } from "@/lib/display-name"

// line 44 — profile state 取得済み
const [profile, setProfile] = useState<{...} | null>(null)

// line 186 — ルーム作成時には正しく使用済み
const ownerName = profile ? getDisplayName(profile) : ""
```

**同じファイルの中で、ルーム作成時は公開名を使っているのに、挨拶だけが取り残されていた。**

---

## 問題2：ルーム参加時の nickname

ルームに参加するとき、メンバーの表示名（nickname）が本名で保存されていた。

```ts
// app/api/rooms/join/route.ts
const resolvedNickname =
  user.user_metadata?.name ||
  user.user_metadata?.full_name ||  // ← 本名
  user.user_metadata?.display_name ||
  "LINEユーザー"

// nickname として DB に保存
return tx.roomMember.create({
  data: {
    nickname: resolvedNickname,  // ← 本名がそのまま保存される
    ...
  }
})
```

ルーム**作成**時は正しく実装されていた。

```ts
// app/api/rooms/route.ts（作成時）
const resolvedNickname = getDisplayName({ id: profile.id, displayName: profile.displayName })
```

**作成と参加で、同じ `nickname` フィールドへの書き込みロジックが別々になっていた。**

---

## 名前関連フィールドの整理

コードベースに存在する名前関連フィールドを整理すると以下になる。

| フィールド | 場所 | 役割 | 公開表示 |
|---|---|---|---|
| `user.user_metadata.full_name` | Supabase Auth | OAuthプロバイダ由来名 | ❌ |
| `user.user_metadata.name` | Supabase Auth | 同上（プロバイダによって異なる）| ❌ |
| `user.email` | Supabase Auth | メールアドレス | ❌ |
| `Profile.name` | DB | プロバイダ由来名（内部保持用）| ❌ |
| `Profile.displayName` | DB | ユーザーが設定した公開名 | ✅ |
| `RoomMember.nickname` | DB | ルーム参加時の表示名スナップショット | ⚠️ 設定次第 |
| `getDisplayName(profile)` | lib | displayName ∥ "ユーザー"+id末尾4文字 | ✅ |

`lib/display-name.ts` には以下の設計方針が書かれていた。

```ts
/**
 * 設計方針:
 *   - provider_name（Profile.name）は外部公開しない
 *   - display_name が設定されていれば使う
 *   - 未設定の場合は "ユーザー" + id末尾4文字 を返す（本名が出ない）
 */
export function getDisplayName(profile: DisplayNameSource): string {
  if (profile.displayName?.trim()) return profile.displayName.trim()
  return "ユーザー" + profile.id.slice(-4)
}
```

この設計方針はあったが、参照箇所が守っていなかった。

## 修正

### home 画面の挨拶

```tsx
// Before（危険）
ようこそ、<span>{user.user_metadata?.full_name || user.email?.split('@')[0]}</span> さん

// After（安全）
{profile?.displayName
  ? <>こんにちは、<span>{profile.displayName}</span> さん 👋</>
  : "今日もルーレット回しますか？"}
```

`displayName` が設定されていれば名前入りの挨拶を、未設定なら名前を出さない。
`full_name` も `email` も JSX から完全に消えた。

### ルーム参加 API

```ts
// Before（危険）
const resolvedNickname =
  user.user_metadata?.name ||
  user.user_metadata?.full_name ||
  user.user_metadata?.display_name ||
  "LINEユーザー"

// After（安全）
const providerName =
  user.user_metadata?.name || user.user_metadata?.full_name || "ユーザー"

const profile = await prisma.profile.upsert({
  where: { id: user.id },
  update: {},
  create: { id: user.id, email: user.email, name: providerName, ... },
  select: { id: true, displayName: true }
})

const resolvedNickname = getDisplayName({ id: profile.id, displayName: profile.displayName })
```

`getDisplayName()` を通すことで、`displayName` が設定されていればそれを使い、
未設定なら "ユーザーXXXX" という匿名表示になる。

### 既存データへの対応（既に DB に本名が保存されているユーザー）

過去に参加した既存ユーザーの `nickname` には本名が保存されたままだ。
これをルーム取得 API のレスポンス時点で上書きすることにした。

```ts
// app/api/rooms/[code]/route.ts
const sanitizedMembers = room.members.map((m) => ({
  ...m,
  nickname: m.profileId && m.profile
    ? getDisplayName({ id: m.profile.id, displayName: m.profile.displayName })
    : m.nickname,  // ゲストはそのまま
}))

return NextResponse.json({ ...room, members: sanitizedMembers })
```

DB に保存されているデータを書き換えず、レスポンスを加工する方法を選んだ。
データ移行のリスクなしに既存ユーザーにも即座に効く。

### 表示層の二重防護

ルーム画面の表示コンポーネントでも `profile.name` フォールバックを除去した。

```tsx
// Before
{member.nickname || member.profile?.name || "ゲスト"}

// After
{member.nickname || (member.profile ? getDisplayName(member.profile) : "ゲスト")}
```

API 側で修正しても、表示層が `profile.name` を参照し続ければ意味がない。
守りを二重にした。

---

## なぜこうなったのか

### 責務が曖昧だった

「名前」という概念に以下の異なる目的が混在していた。

- **認証プロバイダが返した名前**（身元確認・プロフィール初期化用）
- **DB に保存するプロフィール名**（内部管理用）
- **画面に表示する名前**（UI表示用）
- **SNS シェアで使う名前**（公開用）

これらを `name` というフィールド名で扱っていたため、どこで何を使うべきかが曖昧になった。

### 機能追加の順番

公開名（`displayName`）機能は後から追加された。
最初は `user_metadata.full_name` を使うことが「普通」で、それが各所に広まっていた。
後から「これは使ってはいけない」というルールができても、
古いコードは自動的には直されない。

### 設計方針はあったが守られていなかった

`lib/display-name.ts` には「provider_name は外部公開しない」と書いてあった。
しかし `join/route.ts` も `home/page.tsx` もそのコメントを読んでいなかった。

コードのルールをコメントやドキュメントで管理するには限界がある。
強制力のある仕組み（lint ルール、型制約）にしなければ迷子のコードが生まれ続ける。

---

## 学び

### 1. 「名前」は1つではない

認証用・内部管理用・公開表示用・シェア用は、それぞれ責務が異なる。
同じ「名前」という概念でも、フィールド名・型・使用可否のルールを分けて設計すべきだ。

### 2. 後から追加した安全な代替手段が、古いコードに浸透しない

`getDisplayName()` という安全な関数を作っても、
既存の `user_metadata.full_name` 参照は残り続ける。
追加した時点で一括検索して全置換するか、古い参照を型レベルで封じる必要がある。

### 3. API レスポンス加工でデータ移行コストなしに修正できる

DB の過去データを書き換えるマイグレーションはリスクが高い。
GET レスポンスを返すときにアプリ層で加工すれば、既存データをそのままにして表示だけ直せる。
「一度保存したデータは変えない、出力時に安全化する」というアプローチは応用が広い。

### 4. 画面が増えるほど名前参照元の不整合が起きやすい

home・room・play・result・history・profile・share の各画面が、
それぞれ独自に「名前をどこから取るか」を決めていた。
共通の `getDisplayName()` を経由するルールを徹底しなければ、
新しい画面を追加するたびに同じ問題が繰り返される。

---

## まとめ

| 問題 | 原因 | 修正 |
|---|---|---|
| home 挨拶に本名表示 | `user_metadata.full_name` を直接参照 | `profile.displayName` に変更 |
| ルーム参加で本名が nickname として保存 | join route が `user_metadata` を使用 | `getDisplayName()` を経由するよう変更 |
| 既存ユーザーの nickname が本名のまま | 過去データに本名が保存済み | API レスポンス加工で上書き |
| 表示層に `profile.name` フォールバック | UI が provider 由来名を直接参照 | `getDisplayName()` に統一 |

名前フィールドの責務を設計当初から明確にしておけば、この問題は起きなかった。
