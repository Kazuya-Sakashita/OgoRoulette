# 修正したら壊れた：モバイルUXバグ3選とその原因

## 概要

OgoRoulette の開発中、「直した」つもりが「悪化した」という経験を3回した。いずれもモバイル固有の挙動を見落としたことが原因だ。

- iPhone のフリック入力が不安定 → 修正 → 入力不能になった
- シェアボタンを押すと Chrome に戻ったとき about:blank になる
- SPINボタンが数秒間押せない

それぞれ原因・修正・学びを記録する。

---

## 発生した問題

### iPhone入力バグ

`/scan` ページのルームID入力欄で、iPhone の日本語キーボード（ローマ字・フリック）を使うと同じ文字が二重入力される問題が起きていた。

```
ユーザーが「A」を入力
→ state に "A" が入る
→ toUpperCase() で "A" → IME が再処理 → "AA" になる
```

IME 対策として `onCompositionStart` / `onCompositionEnd` を追加し、composition 中は `onChange` を `return` でスキップした。

```tsx
onChange={(e) => {
  if (isComposing.current) return  // ← これが問題
  setInviteCode(e.target.value.toUpperCase().slice(0, 6))
}}
```

デプロイ後、報告が来た。**「入力できなくなった」。**

### シェア後 about:blank

X や LINE のシェアボタンを押すと、Chrome に戻ったとき `about:blank` が表示される問題が起きていた。

```tsx
window.open(
  `https://twitter.com/intent/tweet?...`,
  "_blank",
  "noopener,noreferrer"
)
```

ボタンを押すと新しいタブが開き、Twitter アプリが起動する。ユーザーが Chrome に戻ると、直前にフォーカスされていたタブは「新しく開いた空のタブ」になっていた。

### SPINボタン遅延

ルームオーナーがルームページを開いた直後、SPINボタンが数秒間 `disabled` になっていた。

オーナー判定に使う `isOwner` フラグが、認証の非同期完了を待っている間 `false` のままだったため、ボタンが押せない状態が続いた。

---

## 原因

### iPhone入力バグ

React の Controlled Input は `value` prop と DOM の `input.value` を常に同期する。`onChange` で `setState` を呼ばなかった場合、React は次のレンダーで `input.value` を強制的に state の値（古い値）に戻す。

```
1. compositionStart → isComposing = true
2. DOM: "a" が表示される
3. onChange → return（スキップ）→ setState 呼ばれず
4. React: state は "" のまま → input.value を "" にリセット
5. IME バッファが破壊される
6. compositionEnd → e.currentTarget.value = ""
7. 結果: 何も入力されなかった
```

`onChange` をスキップすることで IME バッファを守ろうとしたが、React が DOM を強制リセットして IME を殺す——という逆効果を生んだ。

### シェア後 about:blank

`window.open(url, "_blank", "noopener,noreferrer")` はモバイル Chrome で新しいタブを開く。Twitter や LINE のアプリリンクが起動すると、新しいタブは `about:blank` のまま残留する。

`noopener` によって元のページからそのタブを閉じる手段も封じられているため、孤立した `about:blank` タブが残り続ける。ユーザーが Chrome に戻ると、最後にアクティブだったタブ（`about:blank`）が表示される。

### SPINボタン遅延

`isOwner` の判定が Supabase の `getUser()` 完了後に行われており、ページ読み込み直後の数百ミリ秒〜数秒は `false` だった。この間、ボタンは `disabled` のまま表示される。ネットワークが遅い環境では数秒以上かかることもある。

---

## 修正内容

### iPhone入力バグ

`onChange` で完全に `return` するのをやめ、composition 中は sanitize なしで raw 値を state に入れるよう変更した。これにより React と DOM が同期し、IME バッファが生き続ける。sanitize は `onCompositionEnd` に委譲する。

```tsx
onChange={(e) => {
  if (isComposing.current) {
    // composition 中は raw 値で更新し React と DOM を同期させる
    setInviteCode(e.target.value.slice(0, 10))
    return
  }
  setInviteCode(e.target.value.replace(/[^a-zA-Z0-9]/g, "").toUpperCase().slice(0, 6))
  setError(null)
}}

onCompositionEnd={(e) => {
  isComposing.current = false
  const val = e.currentTarget.value
    .replace(/[^a-zA-Z0-9]/g, "")
    .toUpperCase()
    .slice(0, 6)
  setInviteCode(val)
}}
```

### シェア後 about:blank

`window.open(..., "_blank")` を `window.location.href =` に変更した。同一タブで遷移することで、ユーザーがアプリを閉じて Chrome に戻ると、ブラウザ履歴を通じて元のページへ戻れる。

```tsx
// Before
window.open(`https://twitter.com/intent/tweet?...`, "_blank", "noopener,noreferrer")

// After
window.location.href = `https://twitter.com/intent/tweet?...`
```

計6箇所（`shareToX`、`shareToLine`、各ページのインライン実装）を統一した。

### SPINボタン遅延

認証状態のロード中は `isOwner` を `undefined` で扱い、ロード完了まで「オーナー判定中」として UI をローディング状態に維持した。確定した `false` とロード中の `false` を区別することで、誤った `disabled` 表示を防いだ。

## 学び

### 1. React Controlled Input は onChange をスキップすると DOM をリセットする

`setState` を呼ばない `onChange` ハンドラは「何もしない」ではなく「React に state 変化なしと伝える」ことを意味する。次のレンダーで `input.value` が state に強制同期され、ユーザーの入力が消える。

IME 対応で composition 中の処理を制御したい場合、`setState` 自体は呼びつつ sanitize のタイミングを `onCompositionEnd` に後ろ倒しにするのが正しいアプローチだ。

### 2. window.open(_blank) はモバイルで about:blank タブを残す

デスクトップでは `_blank` による新タブは自然な動作だが、モバイルでは外部アプリに遷移した後の「戻り先」が空のタブになる。SNS シェアのように「ボタンを押す → アプリが開く → ブラウザに戻る」という流れでは `window.location.href` を使うほうが安全だ。

### 3. 非同期ロード中の false と確定した false は別物

認証や権限チェックが非同期で完了する場合、「まだわからない」状態と「権限なし」状態を区別しないと、ロード中に誤った表示や制限が発生する。`boolean` ではなく `boolean | undefined` や `"loading"` 状態を持たせるパターンが有効だ。

### 4. 修正は「症状の改善」ではなく「仕組みの理解」から始める

フリック入力バグの最初の修正は「composition 中の処理をスキップすれば解決する」という仮説で実装した。しかし React Controlled Input がどう動くかを理解していなかったため、悪化した。技術的な仕組みを確認してから修正に入ることが、リグレッション防止の基本だ。

---

## 再発防止

- Controlled Input に関わる変更は「setState を呼ばない onChange」が存在しないか確認する
- モバイル向けの `window.open` は `_blank` を使わず `location.href` に統一するルールを設ける
- 認証・権限フラグは `loading` 状態を明示的に持つ設計にする
- IME 対応は iOS Safari と Chrome の composition イベント発火順序の違いを考慮して設計する（iOS は onChange → compositionEnd、Chrome は compositionEnd → onChange）
- 「直した」と思ったらモバイル実機で確認してからデプロイする

---

## まとめ

| バグ | 原因 | 修正 |
|------|------|------|
| iPhone フリック入力不能 | Controlled Input が IME バッファをリセット | composition 中も setState を呼び DOM と同期 |
| シェア後 about:blank | `window.open(_blank)` が空タブを残す | `window.location.href` に変更 |
| SPINボタン遅延 | 認証ロード中の false を権限なしと誤判定 | loading 状態を分離して表示を制御 |

いずれも「モバイル実機で確認しなかった」「ブラウザの仕組みを確認せずに直した」という共通の根本原因がある。修正の速さより、仕組みを理解してから手を動かすことのほうが、結果的に速い。
