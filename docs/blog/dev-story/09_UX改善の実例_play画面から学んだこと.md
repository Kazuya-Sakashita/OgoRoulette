# UX改善の実例：play画面から学んだこと

## 「2ステップ問題」という名前をつけた

プレイ画面でルーレットが終わったあとの動線。
最初の設計では：

```
ルーレット完了
  → 結果モーダルを閉じる
  → プレイ画面（何もできない）
  → ← ボタンを押す
  → ルーム画面（COMPLETEDで何もできない）
  → ← ボタンを押す
  → ホーム
```

4アクションでホームに帰れる。

「2ステップ問題」と名付けた。
「2ステップ多い」「ユーザーが何も考えずにホームへ帰れない」という問題。

---

## なぜ「2ステップ問題」が起きたか

### 開発者視点で考えていた

「プレイの前にルームがある。だから戻るボタンはルームへ向ける」

これは**開発者が知っているフロー**から考えた判断。

ユーザーは「プレイ画面がどの画面の子か」を知らない。
知っているのは「ルーレットが終わった。次は何？」という状態だけ。

### 画面の「役割」を考えていなかった

- ホーム：起点・終点（拠点）
- ルーム：招待・準備（拠点）
- プレイ：ルーレットを回す（一時画面）

一時画面の戻り先は「直前の拠点」ではなく「ユーザーが次に行きたい場所」。

プレイが終わったら「用が済んだ」。ルームに戻る理由がない。

---

## 改善1：戻るボタンをホームへ向ける

```tsx
// Before
<Link href={`/room/${code}`}>
  <ArrowLeft className="w-5 h-5" />
</Link>

// After
<Link href="/">
  <ArrowLeft className="w-5 h-5" />
</Link>
```

ただしルームへ戻れなくなるのは困る。
「ルーム名」をリンクにすることで代替導線を用意した。

```tsx
<Link href={`/room/${code}`} className="flex-1 min-w-0">
  <h1 className="text-lg font-bold text-foreground truncate">
    {room.name || "ルーレット"}
  </h1>
  <p className="text-xs text-muted-foreground">
    {isHost ? "ホスト" : "参加中"} · {room._count.members}人
  </p>
</Link>
```

- 戻るボタン（←）→ ホーム
- ルーム名タップ → ルーム

2つの導線を1つのヘッダーに収めた。

---

## 改善2：COMPLETED状態の行き止まりをなくす

ルーレットが終わったプレイ画面には「SPIN」ボタンがなくなる。
最初は「回せない状態」だけが残り、ユーザーが何をすべきか分からなかった。

```tsx
// After: COMPLETED状態専用のアクションエリア
{isCompleted ? (
  <div className="flex flex-col items-center gap-3 w-full max-w-[280px]">
    <Button onClick={showResult}>
      結果を見る
    </Button>
    <Button asChild>
      <Link href="/">ホームへ戻る</Link>
    </Button>
    {isHost && (
      <Button asChild>
        <Link href="/room/create">新しい抽選を作る</Link>
      </Button>
    )}
  </div>
) : ...}
```

3つのアクションをはっきりと提示する：
1. 結果を再表示する（見逃した人用）
2. ホームへ帰る（全員向け）
3. 新しい抽選を作る（ホスト向け）

---

## 改善3：WinnerCardの次の行動

結果モーダル（WinnerCard）の最初のバージョン：

```tsx
// Before
<button onClick={onClose}>閉じる</button>
```

「閉じる」しかない。閉じたら行き止まりのプレイ画面に戻るだけ。

```tsx
// After
{isHost && roomCode && (
  <Button asChild>
    <Link href="/room/create" onClick={onClose}>
      <RotateCcw className="w-4 h-4 mr-2" />
      新しい抽選を作る
    </Link>
  </Button>
)}
<Button asChild>
  <Link href="/" onClick={onClose}>
    ホームへ戻る
  </Link>
</Button>
```

モーダルを閉じなくても次の行動に進める。

---

## 改善4：「ホームへ戻る」をすべてのユーザーに

最初のバージョン：ホストだけに「ホームへ戻る」ボタンを表示していた。

これは間違い。非ホスト（参加者）も帰りたい。

```tsx
// Before（ホストだけ）
{isHost && (
  <Button asChild>
    <Link href="/">ホームへ戻る</Link>
  </Button>
)}

// After（全員）
<Button asChild>
  <Link href="/">ホームへ戻る</Link>
</Button>
{isHost && (
  <Button asChild>
    <Link href="/room/create">新しい抽選を作る</Link>
  </Button>
)}
```

「ホームへ戻る」は全員共通。「新しい抽選を作る」はホストのみ。

## UXで気づいた「見えないユーザー」問題

開発中、自分はホストとして使っていた。

でも実際の使用では：
- ホスト（ルームを作った人）
- 非ホスト（QRで参加した人）
- ゲスト（ログインしていない）
- 再訪問（リロードした / 後から結果を確認したい）

これを全部確認していなかった。

非ホストの画面をきちんと確認すると：
- 「ホームへ戻る」がない
- 「閉じる」しかない
- 閉じたら次の行動が分からない

**教訓**：
> 開発者は「ホスト目線」で使いすぎる。参加者目線・ゲスト目線・再訪問目線で必ず確認する。

---

## UX改善の効果

| 状態 | Before | After |
|------|--------|-------|
| プレイ完了後のアクション数 | 4回タップ | 1回タップでホームへ |
| COMPLETED状態の次の行動 | 分からない | 3択が明示 |
| WinnerCard閉じた後 | 行き止まり | WinnerCard内からも遷移可能 |
| 非ホストの体験 | 孤立 | 全員に「ホームへ戻る」 |

---

## UX設計で繰り返し使えるパターン

### 1. 「完了後の次の行動」を必ず設計する

どんな画面でも「ここで何かが完了したとき、次に何を提示するか」を先に決める。

### 2. 「全ユーザーパターン」で設計する

```
チェックリスト：
□ ホストが使う場合
□ 非ホスト（メンバー）が使う場合
□ ゲスト（ログインなし）が使う場合
□ エラーが出た場合
□ 再訪問（リロード後）の場合
```

### 3. 「1タップで主要操作」の原則

ユーザーが「次に何をしたいか」を予測して、その行動を1タップで実現できる位置に置く。

飲み会という状況を考えると「スマホをいじれる時間が短い」。
2タップ・3タップは離脱につながる。

---

## 「ユーザーが何も考えずにホームへ戻れる」

この言葉がUXの核心だった。

「戻れる」ではなく「何も考えずに戻れる」。

考えさせない = 迷わない = 次の行動が自明。

プレイ画面でルーレットが終わったら：
- 画面のどこかに「ホームへ戻る」が見える
- 1タップで終わる
- 迷う余地がない

これを全画面の設計基準にした。
