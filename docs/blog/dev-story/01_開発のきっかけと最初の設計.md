# OgoRoulette 開発のきっかけと最初の設計

## なぜ作ったか

飲み会のたびに「誰が奢る？」という空気が生まれる。
じゃんけん、指差し、コイントス。どれも盛り上がるようで、実は微妙にシラける。

「ちゃんと楽しい体験を作れないか」と思ったのがきっかけだった。

機能は単純でいい。
**参加者を登録して、ルーレットを回して、奢る人が決まる。**

ただし、それだけでは成立しない。

- 疑われない（操作されていない）
- 盛り上がる（演出がある）
- 結果が分かりやすい（支払い金額が出る）

この3つが揃って初めて「使えるアプリ」になる。

---

## 最初に考えた設計

### 画面構成

```
ホーム
  └─ ルーム作成 / 参加
       └─ ルーム（QRコード・参加者確認）
            └─ プレイ（ルーレット回す）
                 └─ 結果（当選者・支払い金額）
```

シンプルに見える。でもここにいくつかの罠が隠れていた。

---

### 技術スタックの選択

| 要素 | 選択 | 理由 |
|------|------|------|
| フレームワーク | Next.js 16 App Router | SSR/CSRの使い分け、ファイルベースルーティング |
| DB | PostgreSQL (Supabase) | Prismaとの相性、リアルタイム対応 |
| 認証 | Supabase Auth (Google OAuth) | ゼロから作らない |
| アニメーション | Framer Motion | ルーレットの演出に必須 |
| スタイル | Tailwind CSS | 開発速度 |

---

### 最初の設計で正しかったこと

**1. ルーム概念を置いたこと**

参加者リストをURLで管理するか、ルームをDBに持つかを悩んだ。

ルームをDBに持つ設計にした。理由は2つ：
- 参加者がリアルタイムで確認できる
- 結果を後から見返せる

これは正しかった。

**2. 認証をゲストでも使えるようにしたこと**

Googleログインを強制すると離脱が増える。
「ゲストでもルーム作れる」設計にした。

ただしこれが後で大きなセキュリティ課題を生む（後述）。

---

### 最初の設計で間違えたこと

**1. 「結果画面」の出し方**

最初、結果はモーダル（WinnerCard）だけで表示していた。
ページ遷移なし。

これが「閉じたら消える」問題を引き起こした。
再度結果を見るためにリロードが必要になる。
参加者（非ホスト）が結果を見逃す。

**2. ホームへの導線がなかった**

プレイ画面には「戻る」ボタンがあった。
でもそれは `/room/${code}` に戻るボタンだった。

つまり：
```
プレイ終了 → ルーム画面 → ホーム
```

2ステップ必要だった。誰も気にしないように見えて、実はUXとして最悪だった（後述）。

**3. セッション保存の設計が甘かった**

ゲストホストがルーレットを回したとき、`roomId` だけを持って POST /api/sessions を叩く。

最初は「roomId があれば OK」にしていた。
でも `roomId` を知っている人なら誰でもセッションを重複保存できる穴があった。

---

## 設計の教訓

> シンプルな画面フローに見えても、「誰が」「何を」「どこに」送るかを最初に整理しないとセキュリティが崩れる。

> 「戻る」の先をちゃんと考えないと、UXが後で崩壊する。
