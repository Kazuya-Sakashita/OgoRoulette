## はじめに

「ログインするとおかしくなる」という報告は、わりと怖い。

`if (user) { ... }` という条件分岐が山ほどある中で、どこかで録画処理が止められているのかもしれない。もしくは、ログイン後のセッションが何かを上書きしているのかもしれない。

調査を始める前は「ログイン状態が絡んでいる」というだけで、直感的に「面倒そう」と感じていた。

でも結果は真逆だった。

**ログイン状態はまったく関係なかった。**

この記事は、「間違った仮説を立てて、コードで丁寧に潰して、本当の原因を見つける」プロセスの記録だ。

---

## 問題の報告

OgoRoulette（会食の奢りを決めるルーレットアプリ）に、こんな問題が報告された。

```
1. LINEでログインする
2. ルームを作成する
3. メンバーを招待してルーレットを回す
4. 当選者が表示される
5. しかし「🎬 動画でシェア」ボタンが表示されない
   → 動画が生成されていない
```

未ログイン状態でホームから直接ルーレットを回すと、ボタンは表示される。

ログイン後にルームを作ってから回すと、ボタンが出ない。

表面上の観察：**ログインしている場合だけ動画が保存されない。**

---

## 最初の仮説とその検証

最初に思い浮かぶ仮説はこうだ。

> 「`if (user)` のような条件分岐で、ログイン時に録画処理が止められているのではないか」

もしくは、

> 「ログイン時に何かの処理が割り込んで、録画の state が初期化されてしまっているのではないか」

この仮説をコードで検証するには、
「`user` の状態によって録画処理がどう変わるか」を調べればいい。

---

## コードを読んで仮説を潰していく

### 録画関連の state を追う

`home/page.tsx` には録画用の state が存在する。

```typescript
// lib/use-video-recorder.ts（カスタムフックに分離済み）
const [recordingPhase, setRecordingPhase] = useState<RecordingPhase>("idle")
const [recordedBlob, setRecordedBlob]     = useState<Blob | null>(null)
const [showShareSheet, setShowShareSheet] = useState(false)
```

これらは `useVideoRecorder()` というカスタムフックにまとめられており、
`user` の状態を受け取っていない。完全に独立している。

### 録画開始処理に `user` はあるか

`handleSpin`（スピン開始）の中を見る。

```typescript
const handleSpin = () => {
  if (isSpinning || participants.length < 2 || countdown !== null) return
  setWinner(null)
  resetRecording()
  setRecordingPhase("countdown")
  setCountdown(3)
  countdownTimersRef.current = [
    setTimeout(() => setCountdown(2), 1000),
    setTimeout(() => setCountdown(1), 2000),
    setTimeout(() => {
      setCountdown(null)
      setIsSpinning(true)
      startRecording()  // ← ここ
    }, 3000),
  ]
}
```

`startRecording()` の前に `if (user)` は一切ない。
**ログイン状態に関係なく録画は開始される。**

### 録画停止処理に `user` はあるか

`handleSpinComplete`（スピン完了時）を見る。

```typescript
const handleSpinComplete = useCallback((winnerName: string, winnerIndex: number) => {
  // ...
  stopRecordingAfterReveal()  // ← ここ

  // DB保存はログイン時のみ（録画とは別の処理）
  if (user) {
    fetch("/api/sessions", { method: "POST", ... }).catch(() => {})
  }
}, [user, ...])
```

`stopRecordingAfterReveal()` の前に `if (user)` はない。
**録画停止もログイン状態に関係なく動く。**

DB保存だけが `if (user)` でガードされているが、これは録画とは別の処理だ。

### WinnerCard への props に `user` はあるか

動画シェアボタンが表示されるのは `WinnerCard` の中で `videoBlob && onShareVideo` が真のとき。

```tsx
{winner && (
  <WinnerCard
    ...
    videoBlob={recordedBlob}
    onShareVideo={() => setShowShareSheet(true)}
  />
)}
```

`recordedBlob` の受け渡しに `user` チェックはない。

---

### 結論：`home/page.tsx` の録画処理はログインと無関係

| 処理 | ログイン必要か |
|------|:---:|
| 録画開始 | ❌ 不要 |
| 録画停止・Blob生成 | ❌ 不要 |
| 「動画でシェア」ボタン表示 | ❌ 不要 |
| ShareSheet 表示 | ❌ 不要 |
| DB（セッション）保存 | ✅ 必要 |

**最初の仮説「ログイン状態が録画を壊している」は誤りだった。**

では、なぜログイン後にルームを経由すると録画できないのか。

---

## 本当の原因

ここで視点を変えた。

「`home/page.tsx` では動く」「ログイン後にルームから回すと動かない」

この差分は「ファイル」だ。

ルームからルーレットを回す画面は `home/page.tsx` ではなく、
`app/room/[code]/play/page.tsx` だった。

そして `play/page.tsx` を開いて確認すると：

```tsx
// app/room/[code]/play/page.tsx（修正前）

// RecordingCanvas → 存在しない
// VideoRecorder   → 存在しない
// recordedBlob    → 存在しない
// recordingPhase  → 存在しない

{winner && (
  <WinnerCard
    winner={winner.name}
    winnerIndex={winner.index}
    onClose={...}
    // videoBlob     → 渡していない
    // onShareVideo  → 渡していない
  />
)}
```

**録画スタック全体が `play/page.tsx` に存在しなかった。**

録画機能は `home/page.tsx` にのみ実装されており、ルームモード用の `play/page.tsx` への移植が行われていなかったのだ。

「ログインすると壊れる」という現象の正体は：

- 未ログイン → `home/page.tsx` でルーレットを回す → 録画あり
- ログイン → ルームを作る → `play/page.tsx` でルーレットを回す → 録画なし

単に「使うページが違った」というだけだった。

ログイン状態はまったく関係なかった。

---

## 修正内容

修正は明快だった。`play/page.tsx` に録画スタックを追加する。

すでに `useVideoRecorder` というカスタムフックがあったので、
フックを呼び出して連携するだけで済んだ。

```typescript
// app/room/[code]/play/page.tsx（修正後）

// 1. フック呼び出し
const {
  recordingPhase,
  setRecordingPhase,
  recordedBlob,
  showShareSheet,
  setShowShareSheet,
  recordingCanvasRef,
  wheelRotationRef,
  startRecording,
  stopRecordingAfterReveal,
  reset: resetRecording,
} = useVideoRecorder()

// 2. ルーレット phase → 録画 phase を同期
useEffect(() => {
  if (phase === "preparing") setRecordingPhase("countdown")
  else if (phase === "spinning") startRecording()
}, [phase])

// 3. スピン完了時に録画停止
const handleSpinComplete = (winnerName: string, winnerIndex: number) => {
  stopRecordingAfterReveal()  // 追加
  // ...既存の処理...
}
```

JSX には `RecordingCanvas`・REC インジケーター・`ShareSheet` を追加し、
`WinnerCard` に `videoBlob` と `onShareVideo` を渡す。

```tsx
{/* 追加: オフスクリーンの録画キャンバス */}
<RecordingCanvas
  phase={recordingPhase}
  countdown={countdownValue}
  wheelRotationRef={wheelRotationRef}
  participants={participants}
  winnerIndex={winner?.index ?? null}
  winner={winner?.name ?? null}
  winnerColor={...}
  canvasRef={recordingCanvasRef}
/>

{/* 追加: REC インジケーター */}
{(recordingPhase === "countdown" || recordingPhase === "spinning" || recordingPhase === "reveal") && (
  <div className="fixed top-4 right-4 z-30 ... animate-pulse">
    ● REC
  </div>
)}

{/* 既存の WinnerCard に props 追加 */}
<WinnerCard
  ...
  videoBlob={recordedBlob}
  onShareVideo={() => setShowShareSheet(true)}
/>
```

もし `useVideoRecorder` フックが存在しなければ、
同じロジックを `play/page.tsx` にそのままコピーする必要があった。
フック化が先にあったことで、修正が数行で済んだ。

## 学び

### 1. 「ログインが原因では？」という仮説はすぐ疑え

認証まわりのバグは「ログインが悪さをしている」という仮説を立てやすい。
しかし実際には、単にページが違う、コンポーネントが違う、という原因のほうが多い。

コードを読んで仮説を検証する前に「ページが正しいか」を確認するクセをつけると、
調査時間を大幅に節約できる。

### 2. 「どのページでテストしたか」は重要な情報

「ログインすると壊れる」という報告の裏に
「ログイン後は別ページに誘導される」という導線の差があった。

バグ報告を受けたとき、**再現手順のページ遷移をそのままトレースする**ことが不可欠だ。

### 3. 機能を1ページに実装したとき、「他のページにも必要か」を考える習慣

今回の根本的な問題は「録画機能を `home/page.tsx` に実装したが、`play/page.tsx` への移植を忘れた」ことだ。

Next.js のような画面ベースのフレームワークでは、
同じコンテンツを異なるルートで扱うことがある。
機能追加のたびに「他のページでも同じことが起きるか」を確認する習慣が重要だ。

### 4. カスタムフックは「後から移植しやすい」という価値がある

もし録画ロジックが `home/page.tsx` に直書きされていたら、
移植には数十行のコピー&ペーストが必要だった。

`useVideoRecorder()` というフックに分離されていたおかげで、
`play/page.tsx` への追加はフック呼び出しと数行のグルーコードだけで完了した。

**「どこでも使えるように抽象化する」ことは、予期せぬ拡張に強い設計になる。**

## まとめ

「ログインすると録画できない」という現象は、
実際には「ルームモード用のページに録画スタックが存在しない」という問題だった。

認証状態は無関係で、録画処理のコードをすべて調べても `if (user)` による制限は一切見つからなかった。

正しい調査手順はこうだ：

1. **仮説を立てる**（「ログインが悪さしている？」）
2. **コードで仮説を検証する**（`user` による分岐を全部確認する）
3. **仮説が崩れたら視点を変える**（「そもそもどのページで動いているか」）
4. **本当の原因を特定する**（`play/page.tsx` に録画スタックがない）
5. **修正する**（フックを使って追加する）

「ログインが関係する」という思い込みを早めに捨てて、
コードを地道に読んだことで、正確な原因にたどり着けた。

1. 「ログインすると録画できない」は誤解だった：コードで証明した調査記録

5. `if (user)` を全部読んでも見つからなかった録画バグの正体
