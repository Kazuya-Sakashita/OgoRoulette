# プロンプト設計とコンテキスト圧迫問題

## コンテキスト圧迫とは何か

Claude Codeにはコンテキストウィンドウの上限がある。
長い会話を続けると、過去の情報が「圧縮」または「切り捨て」られる。

OgoRouletteの開発で実際に起きたこと：
- セッション前半で決めた設計判断を、後半でClaudeが知らないまま実装する
- 「さっきこう決めたのに」という状態になる
- 修正のやり取りで会話が長くなり、さらにコンテキストが圧迫される

---

## コンテキスト圧迫が起きたとき何が変わるか

### 判断の一貫性が失われる

```
セッション前半で決めた：
「戻るボタンはホームへ向ける」

セッション後半でClaudeが：
「エラー状態の戻るボタンは /room/${code} にしました」
```

セッション前半の決定が後半に引き継がれなかった。

### 禁止していたことが出てくる

```
CLAUDE.md で禁止していた：
「UIにロジックを直書きすること」

コンテキスト圧迫後のClaudeが：
JSXの中に計算ロジックを直書きしたコードを出してくる
```

CLAUDE.md が読まれているはずなのに、コンテキスト圧迫でその内容が薄れる。

---

## 対策1：CLAUDE.md の役割を明確にする

`CLAUDE.md` はClaude Codeが**毎セッション自動で読む**特殊ファイル。
ここに書いた内容はコンテキスト圧迫に強い（毎回注入されるから）。

だから：
- **コンテキスト圧迫しても消えてほしくないルール** → `CLAUDE.md` に書く
- **タスク固有の詳細** → `skills/` に書いて必要時に参照させる

```markdown
# CLAUDE.md の理想的な構造

## 必ず守ること（禁止事項含む）← ここは短く・明確に
## 参照すべきスキルファイル一覧
## リスク判断の基準
```

---

## 対策2：skills/ で情報を「外部化」する

コンテキストに乗せないことがコンテキスト節約になる。

```
悪い：
Claudeに直接「ルーティングのルールはXXXで、セキュリティはYYYで、
ルーレットはZZZで...」と全部伝える

良い：
「ルーティングはskills/routing.mdを参照して」と伝える
→ Claudeがファイルを読む（そのタスクに必要なときだけ）
```

こうすることで：
- コンテキストに乗るのは「参照命令」だけ（数行）
- 詳細はファイルから読む（無限には乗せられない）

---

## 対策3：プロンプトテンプレートで「毎回同じ構造」を作る

```
docs/claude/
  ├── initial-prompt.md   ← 新セッション開始時に貼る
  ├── bugfix-prompt.md    ← バグ修正時に貼る
  ├── refactor-prompt.md  ← リファクタ時に貼る
  ├── test-prompt.md      ← テスト追加時に貼る
  └── review-prompt.md    ← レビュー時に貼る
```

テンプレートを貼ることで「Claude に毎回同じ文脈を渡す」構造ができる。

コンテキスト圧迫が起きても、次のセッションで同じテンプレートを貼ればリセットできる。

---

## 対策4：会話を短く保つ

コンテキストが圧迫される根本原因は「会話が長すぎること」。

**長くなる原因**：
1. 大きすぎる依頼（「全体的に改善して」）
2. 修正のやり取りが増える（依頼が曖昧だから）
3. 前提の共有から始まる（毎回説明している）

**対策**：
1. 1依頼1タスク（小さく分ける）
2. 依頼時に WHAT/WHY/HOW を明確にする（修正往復を減らす）
3. 前提はファイルに書いて参照させる（口頭で繰り返さない）

---

## refactor-prompt.md の重複問題

実際に起きた失敗例：

`docs/claude/refactor-prompt.md` が482行になっていた。
原因を見ると、同じ内容が**2回書かれていた**（241行 × 2）。

おそらく以前のセッションでファイルを更新したとき、
Claudeが既存内容を読まずに全文を追記してしまった。

**発見方法**：`grep -c "^#" docs/claude/refactor-prompt.md` で見出し数を確認。
241行なのに見出しが倍あった。

**修正**：全文書き直し（241行の正しいバージョンに戻す）。

**学び**：
> 既存ファイルを更新する前に必ず `Read` ツールで現在の内容を確認させる。
> Claudeに「ファイルを書いて」と言うだけでは、上書きか追記かが不明確。

---

## initial-prompt.md の重複削除

`docs/claude/initial-prompt.md` には以下のセクションがあった：

```
## 7. ルーティング・UXルール
（20行）

## 8. ルーレット・金額ロジック
（10行）
```

しかし `CLAUDE.md` にも同じ内容があった。
`CLAUDE.md` は自動ロードされるので、`initial-prompt.md` に書く必要がない。

削除後：
- `initial-prompt.md` は30行短くなった
- `CLAUDE.md` の内容が「唯一の真実」になった
- 矛盾が生まれなくなった

**原則**：
> 同じ情報を2箇所に書かない。1箇所に書いて他から参照する。

## プロンプト設計で気づいたこと

### 「禁止事項」の書き方

```markdown
# 悪い禁止事項の書き方
「無駄なコードを書かないこと」

# 良い禁止事項の書き方
「UIコンポーネント内に金額計算を直書きしない。
 計算は lib/bill-calculator.ts のような純粋関数に切り出す。」
```

具体的であるほど効果が高い。

### 「なぜ」を書く

```markdown
# なぜを書かない
「router.back() は使用禁止」

# なぜを書く
「router.back() は使用禁止。
 理由：外部リンクから来たユーザーや、リロード後のユーザーでhistoryが保証されないため。
 代わりに明示的な Link href を使うこと。」
```

理由があると、Claude が「判断」できる。
例外が正当な場合に「この場合は理由が当てはまらないから OK」と判断できる。

### スキルの粒度

`skills/` のファイルは「1ファイル1テーマ」が良かった。

```
skills/routing.md    ← ルーティングだけ
skills/security.md   ← セキュリティだけ
skills/testing.md    ← テストだけ
```

「フロントエンド全般」のような大きなファイルは、
必要な情報を探すのにコンテキストを食いすぎる。

## 現在のコンテキスト管理戦略まとめ

| 情報の種類 | 置き場所 | 理由 |
|-----------|---------|------|
| 絶対守る原則・禁止事項 | CLAUDE.md | 毎セッション自動ロード |
| タスク別の詳細ルール | skills/*.md | 必要時のみ参照 |
| タスク別のプロンプト構造 | docs/claude/*.md | 毎セッション貼り付け |
| 一時的な決定事項 | 会話内のみ | セッション終了で消えてよい |
| 恒久的な決定事項 | skills/ か CLAUDE.md | 必ず記録する |

**最重要の教訓**：
> 「会話の中でしか言っていない重要なルール」は存在しないのと同じ。ファイルに書いて初めて次のセッションで使える。
