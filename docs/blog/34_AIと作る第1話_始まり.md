# AIと作る ── 第1話「始まり」

## なぜやろうと思ったか

飲み会のあと、帰り道に友人から言われた。

「今日の誰が奢るか決め、あんまり盛り上がらなかったな。」

言われてみれば確かにそうだった。
じゃんけん、指差し、アプリのルーレット。
どれも「公平に決まる」のに、なぜかシラけた空気が漂う。

考えてみると原因はシンプルだった。
**「決め方」に個性がない。**

コイントスは一瞬で終わる。結果が出ても「えっもう？」ってなる。
アプリのルーレットはスマホの画面を1人でぽちぽちするだけ。
周りは何が起きているかすら分からない。

盛り上がる理由がない。

---

## 「作れるかもしれない」と思った

そのころ Claude Code の話をよく聞くようになっていた。

「AIにコードを書かせる」というのは以前から知っていたが、
「AIと一緒にアプリを完成まで持っていく」というのは別の話だと思っていた。

ある記事を読んだ。
個人開発者が Claude Code で数週間でリリースまで漕ぎ着けた話だった。

読み終えて、2つのことを同時に思った。

ひとつ。**自分にも作れるかもしれない。**
もうひとつ。**本当に作れるか確かめたい。**

どちらが先だったかは分からない。
ただ、「確かめる」には実際に作るしかない。

---

## 作るものを決めた

「飲み会で誰が奢るか、ルーレットで決めるアプリ」

機能はシンプルでいい。

- 参加者を登録する
- ルーレットを回す
- 奢る人が決まる

ただし、これだけでは盛り上がらない。
盛り上げるには3つの要素が必要だと思った。

```
1. 疑われない（操作されていないと全員が信じられる）
2. 全員で見られる（スマホの小さい画面を1人でぽちぽちしない）
3. 結果が分かりやすい（誰が奢る、いくら払う）
```

特に「疑われない」は重要だった。
ルーレットを回した本人が当選者を決定できる構造では、どんな演出をしても信頼されない。
当選者はサーバーが決定する必要がある。

---

## Claude Code を使うと決めた理由

正直に言うと、技術的な確信はなかった。

Next.js は触ったことがある。
Supabase は名前を知っている程度。
Prisma はほぼ初見。
Framer Motion は完全に未知。

普通なら「知っている技術だけで作ろう」と縮小するか、
「勉強してから始めよう」と先送りする。

でも Claude Code があれば「知らなくても始めてみる」ができる気がした。

これが正しい判断だったかどうかは、実際に作ってみないと分からない。

---

## 最初の一歩

ターミナルを開いた。

```bash
npx create-next-app@latest ogoroulette
```

Claude Code を立ち上げ、最初の指示を打った。

「飲み会で誰が奢るか決めるアプリを作ります。まずルーム機能から作ってください。」

Claude が動き始めた。

「思ったより早い」というのが最初の感想だった。
ファイルが増えていく。コードが書かれていく。
自分がタイプしていないのに、アプリの骨格ができていく。

これが違和感の始まりでもあった。

*続く*
