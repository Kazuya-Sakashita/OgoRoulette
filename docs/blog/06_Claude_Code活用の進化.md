# Claude Code活用の進化

## 最初のClaude Codeの使い方

最初は「賢いコード補完ツール」として使っていた。

「このコンポーネント作って」「このバグ直して」

それだけ。

精度は高くない。同じ間違いを繰り返す。コンテキストが途切れると前の判断を忘れる。
「なんで前と同じことをするんだ」と何度も思った。

---

## 最初に直面した問題：「Claude は毎回初めて会う」

Claudeはセッションを跨いで記憶しない。

昨日「この画面の戻るボタンは/に向ける」と決めても、
今日のセッションでは知らない状態から始まる。

結果：同じ判断を何度も説明し直す羽目になる。

これを解決するために `CLAUDE.md` を書いた。

```markdown
# CLAUDE.md
このファイルはClaudeが最初に読む。プロジェクトの設計原則・禁止事項・判断基準を書く。
```

`CLAUDE.md` はClaude Codeが自動で読む特殊ファイル。
ここに書いたことはセッションをまたいで適用される。

---

## CLAUDE.mdの進化

### v1：とにかく書いた（長すぎ）

最初に書いた `CLAUDE.md` は400行以上あった。

- 全ての設計原則
- ルーティングのルール
- ルーレットの仕様
- セキュリティの注意
- 金額計算の仕様
- ...

全部書いた。

結果：Claude がどれを重視すべきか判断できなくなった。
「ルーティングの詳細」と「セキュリティの基本原則」が同じ深さで書いてあると、優先度が分からない。

### v2：Progressive Disclosure を導入

設計原則：**「最初に全部書かない、必要なときに渡す」**

```
CLAUDE.md
  → 基本原則のみ（50行以内）
  → 詳細は skills/ に分離

skills/routing.md     ← ルーティング詳細
skills/security.md    ← セキュリティ詳細
skills/roulette.md    ← ルーレット仕様
skills/testing.md     ← テスト方針
```

`CLAUDE.md` には「どのファイルをいつ参照するか」だけを書く：

```markdown
## スキル使用ルール

- UI変更 → skills/frontend.md + skills/ui-ux.md
- セキュリティが関係する → skills/security.md を必ず参照
- ルーレット・金額計算 → skills/roulette.md を必ず参照
```

**効果**：`CLAUDE.md` が軽くなり、Claudeが優先度を理解しやすくなった。

### v3：重複削除（現在）

さらに問題が起きた。

- `CLAUDE.md` にルーティングの詳細を書く
- `docs/claude/initial-prompt.md` にも同じルーティングの詳細を書く
- `skills/routing.md` にも書く

3箇所に同じことが書いてある状態になった。
→ どれかを更新すると他が古くなる
→ Claude が3箇所の矛盾に混乱する

**解決**：1箇所に書いて他から参照する

```markdown
# CLAUDE.md（最終版）

## ルーティング・UX（要点）
詳細は skills/routing.md を参照。
必須: 戻る操作依存NG / 結果画面に「ホームへ戻る」+「再実行」を置く / 行き止まり画面NG
```

要点だけ `CLAUDE.md` に書き、詳細は `skills/routing.md` に一元化。

## docs/claude/：プロンプトテンプレートの管理

実装タスクの種類によって最適なプロンプト構造が違う。

| タスク | プロンプトファイル |
|--------|----------------|
| 新機能実装 | initial-prompt.md |
| バグ修正 | bugfix-prompt.md |
| リファクタリング | refactor-prompt.md |
| テスト追加 | test-prompt.md |
| コードレビュー | review-prompt.md |

「バグを直してほしい」とだけ伝えるより、
`bugfix-prompt.md` の構造に従って「再現条件・エラー内容・該当コード」を揃えて渡す方が、
明確な修正が返ってくる。

**失敗例**：
```
ユーザー: 戻るボタンが変な動作してます
Claude: どのような動作ですか？
ユーザー: ルームに戻る
Claude: ホームに戻るように変えました
```
→ 「なんでホームに変えたの？なぜルームに戻るのが問題なの？」が伝わっていない

**改善例（bugfix-prompt.mdを使った場合）**：
```
再現条件: プレイ画面で←ボタンを押す
期待値: ホームへ戻る
実際の動作: /room/${code} へ戻る
問題: COMPLETEDルームに戻るだけで次の行動がない
該当コード: app/room/[code]/play/page.tsx:302
```
→ Claudeが一度で正確な修正を返す

## WHAT / WHY / HOW の枠組みを強制する

`CLAUDE.md` に書いた重要なルール：

```markdown
## 出力ルール

### 実装前
- WHAT：何をするか
- WHY：なぜ必要か
- HOW：どう実装するか
- リスク：低 / 中 / 高
- 確認要否
```

この枠組みを強制することで、Claudeが「なぜそうするか」を説明してから実装する。

効果：
- 意図と違う実装に気づきやすい
- 危険な変更を見落とさない
- リスク判断が共有される

## Claude Code で失敗した使い方

### 失敗1：大きすぎる依頼

```
「プレイ画面のUXを全部改善して」
```

→ Claude は何十もの変更を一度に出してくる。
→ レビューできない。何が変わったか追えない。
→ 取り消しが大変。

**改善**：小さく分けて一つずつ依頼する。

### 失敗2：コンテキストを共有しないで修正させる

```
「WinnerCardのこのボタンを直して」（コード渡さず）
```

→ Claude が既存コードを知らない状態で実装する
→ 実際のコードと合わない実装が返ってくる

**改善**：必ず関係するファイルを先に読ませてから指示する。

### 失敗3：「とりあえず動けばいい」実装を承認する

Claude が出したコードが「動く」けど「設計がよくない」ことがある。
急いでいると「まあいいか」と承認してしまう。

これが後で「同じ場所を何度も直す」につながった。

**改善**：`review-prompt.md` を毎回使う。実装後に必ずレビューを挟む。

## Claude Code で成功した使い方

### 1. 純粋関数のテスト生成

```
calculateBillSplit のテストケースを網羅的に書いて。
境界値（0円・1人・奢り超過・負の値）も含めること。
```

→ 20ケースのテストコードを即座に生成。
→ その中で「この期待値が間違っている」と気づくことができた（仕様の確認になった）。

### 2. skills/ ファイルの構造化

「ルーティングの設計原則をドキュメントとして整理して」

→ `skills/routing.md` の初版を生成。
→ その後の会話で「routing.mdを参照して」と伝えるだけで一貫した判断ができる。

### 3. セキュリティレビュー

「このAPIルートのセキュリティ問題を洗い出して」

→ race conditionの問題を指摘。
→ `prisma.$transaction` による修正案を提示。

Claudeは「考えていなかった角度」から問題を見つけるのが得意だった。

## 現在の使い方（安定版）

```
1. 新セッション開始 → CLAUDE.md が自動ロード
2. タスクの種類に応じたプロンプトテンプレートを選ぶ
3. 関係するコードを先に読ませる
4. WHAT/WHY/HOW の形式で依頼する
5. 実装前にリスク確認
6. 実装後にreview-prompt.mdでレビュー
7. テスト追加
```

この流れで「同じ問題を繰り返す」頻度が大幅に減った。
