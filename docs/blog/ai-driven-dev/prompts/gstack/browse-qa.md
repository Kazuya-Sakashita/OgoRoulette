# ブラウザ自動操作による実機QA

**レベル:** 中級  
**再利用性:** 高  
**効果:** ★★★★☆  
**タグ:** `テンプレ化`

---

## 目的

gstack browseを使ってヘッドレスブラウザを操作し、  
本番URLの動作確認・スクリーンショット取得・コンソールエラー確認を自動化する。

## 使用タイミング

- デプロイ直後の動作確認
- デザイン変更後のビジュアル確認
- レスポンシブ対応の確認

---

## プロンプト（基本確認セット）

```
gstack browseを使って以下を確認してください。

URL: {your-app-url}

確認項目:
1. ページが正常にロードされるか（200 OK）
2. コンソールエラーが出ていないか
3. 主要要素が表示されているか（スクリーンショット）
4. モバイル（375x812）・デスクトップ（1280x800）でのレスポンシブ確認

コマンド例:
$B goto {your-app-url}
$B wait --networkidle
$B console --errors
$B responsive /tmp/qa-screenshot
```

## プロンプト（特定要素の確認）

```
以下の要素が正しく表示・動作しているか確認してください。

確認対象: {コンポーネント名 or セレクタ}

1. 要素が表示されているか: $B is visible "{selector}"
2. 実際の表示: $B screenshot /tmp/element.png
3. 計算済みCSSの確認: $B css "{selector}" "{property}"
4. アクセシビリティツリーの確認: $B snapshot -i -s "{selector}"
```

## プロンプト（デプロイ後の変更確認）

```
直前のデプロイで以下の変更を入れました。
変更が正しく反映されているか確認してください。

変更内容: {変更の説明}
確認方法:
- $B js "(()=>{ ... })()"  で実際の値を取得
- スクリーンショットで目視確認

キャッシュがある場合はキャッシュバスター付きURLを使用:
$B goto "{url}?v=$(date +%s)"
```

---

## 工夫ポイント

- デプロイ後の確認は必ずキャッシュバスター付きURLで行う（CDNキャッシュ問題を回避）
- `$B console --errors` でJSエラーを最初に確認してから他の確認に進む
- `responsive` コマンドで3サイズ同時確認すると効率が良い

## 改善余地

- Playwright との組み合わせで E2Eテストとして記録しておくとより再現性が高い
- 複数ページを横断して確認するときは `newtab` でタブを使い分ける

## 実行結果

実際に確認できた内容:
- viewport の `maximum-scale=1` 削除がHTMLに反映されていることを確認
- Google Fonts の新しいURL（500ウェイト削除後）が反映されていることを確認
- muted-foreground が `rgba(249,250,251,0.65)` に更新されていることをJS実行で確認
