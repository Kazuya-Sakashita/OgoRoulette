# ルーム管理APIのデバッグと状態追跡

**レベル:** 中級  
**再利用性:** 中  
**効果:** ★★★☆☆

---

## 目的

Supabase + Next.js App RouterのAPIで、  
Realtime（WebSocket）とポーリングの併用時における状態同期問題を診断する。

## 使用タイミング

- 「一部のユーザーだけ状態が更新されない」問題が発生したとき
- Realtimeイベントの取りこぼしが疑われるとき
- IN_SESSION 中の状態管理を最適化したいとき

---

## プロンプト（状態同期問題の診断）

```
以下のコードで状態同期の問題が発生しています。

構成:
- Supabase Realtime（postgres_changes）でリアルタイム更新
- 10秒間隔のポーリングでfallback
- IN_SESSION 中のスピン状態を全クライアントに同期

問題の症状: {症状を具体的に}

分析してほしいこと:
1. Realtimeイベントが取りこぼされる条件
2. ポーリング間隔とアニメーション時間の不整合
3. 再接続時のリカバリー処理の有無

関連コード:
{コードの該当部分}

根本原因の特定と修正方針（優先度付き）を出力してください。
```

## プロンプト（IN_SESSION 中のポーリング最適化）

```
以下のポーリング処理を最適化してください。

現状:
- 常に10秒間隔でポーリング
- IN_SESSION 中（ルーレット回転中）も同じ間隔

問題:
- スピン後のアニメーション時間が5秒
- 10秒間隔では全員がアニメーション開始前に受信できないケースがある

修正方針:
- IN_SESSION 中は2秒間隔に短縮
- それ以外は10秒を維持

コード:
{既存のポーリング処理}

変更は最小限に。副作用（過剰なポーリング、メモリリーク）がないことを確認してください。
```

---

## 工夫ポイント

- 「副作用がないことを確認」と指示することで、AIがポーリング解除処理を忘れない
- 状態同期問題は「最悪ケースのタイムライン」を書かせると原因が見えやすい

## 実行結果

- ISSUE-146 として診断: ポーリング10秒 vs TOTAL_ANIM_MS 5000ms の不整合
- 修正方針: TOTAL_ANIM_MS を 8000ms に引き上げ + IN_SESSION 中ポーリング 2秒化
