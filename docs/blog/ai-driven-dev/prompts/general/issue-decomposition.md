# 評価結果をISSUEに分解して文書化する

**レベル:** 初級  
**再利用性:** 高  
**効果:** ★★★★★  
**タグ:** `テンプレ化` `おすすめ`

---

## 目的

Lighthouse・gstackなどの評価結果を、  
追跡可能なISSUEドキュメントとしてファイルに保存する。

## 使用タイミング

- 品質評価を実施した直後
- バグを発見したとき
- 技術的負債を記録したいとき

---

## プロンプト（評価結果→ISSUE一括生成）

```
以下の評価結果をもとに、ISSUEドキュメントを生成してください。

評価結果:
{Lighthouse / gstack / 実機確認の結果をここに貼る}

各ISSUEを以下のフォーマットで docs/issues/ に保存してください。
ファイル名: issue-{番号}-{kebab-case-title}.md

---
# ISSUE-{番号}: {タイトル}

## 概要
{1〜2文で問題を説明}

## 症状
{ユーザーが体験する問題}

## 根本原因
{技術的な原因を具体的に}

## 修正方針
{ファイル名・行数・変更内容まで具体的に}
### 応急（即日）
### 安全（推奨）
### 理想（根本解決）

## 影響範囲
{どのユーザー・機能に影響するか}

## ステータス
🔴 未修正

**優先度:** {P1/P2/P3}（理由）
---

優先度基準:
P1: セキュリティ / WCAG違反 / ユーザーがアプリを使えない
P2: スコア改善5点以上 / 主要UXへの影響
P3: 軽微な改善 / 内部品質
```

## プロンプト（単発ISSUE作成）

```
以下のバグをISSUEとして docs/issues/issue-{番号}-{タイトル}.md に保存してください。

バグ概要: {バグの説明}
再現手順: {手順}
期待動作: {本来の動作}
実際の動作: {バグの動作}
関連ファイル: {ファイル名:行番号}

上記に加えて「根本原因の分析」と「修正方針（3段階）」を含めてください。
```

---

## 工夫ポイント

- 「3段階の修正方針（応急/安全/理想）」を必須にすることで、後から修正の優先度を選べる
- `ステータス: 🔴 未修正` を固定することで、完了時に ✅ に変えるだけで追跡できる
- ファイル名を `issue-{番号}-{kebab-case}.md` に統一することでgrepで検索しやすい

## 改善余地

- ISSUEに「検証方法」セクションを追加すると、修正後の確認が楽になる
- ISSUE番号の採番を自動化する（現在は手動）

## 実行結果

このプロンプトで生成したISSUE:
- ISSUE-141〜150 の計10件を docs/issues/ に保存
- 各ISSUEが修正のインプットとして機能し、修正時間が短縮された
- ブログ記事の素材としても流用できた
