# ChunkLoadError 無限ループの診断と修正

**レベル:** 上級  
**再利用性:** 中  
**効果:** ★★★★★  
**タグ:** `おすすめ`

---

## 目的

本番環境で `ChunkLoadError: Loading chunk failed` が発生し、  
ページリロードのたびに繰り返される無限ループを診断・修正する。

## 使用タイミング

- `next/dynamic` を使った動的importを導入した後
- デプロイ後に「ページが開けない」報告があったとき
- error.tsx のリロードが終わらないとき

---

## プロンプト（状況診断）

```
本番環境で以下のエラーが発生しています。

エラー: ChunkLoadError: Loading chunk {chunk-hash}.js failed.
症状: ページをリロードするたびに同じエラーが繰り返される（無限ループ）
直前の変更: {変更内容を記載}

以下の観点で根本原因を分析してください：

1. なぜこのchunkが見つからないのか
2. なぜリロードで解決しないのか（キャッシュ関係を疑う）
3. error.tsx の reload 処理が無限ループになる条件
4. ブラウザキャッシュ・CDNキャッシュ・Service Workerの関与

ファイル: app/error.tsx の reload 処理を確認します。
```

## プロンプト（修正方針の決定）

```
ChunkLoadError の根本原因が特定できました。

原因: {診断結果}

修正方針を2案提示してください：

案A: 動的importをやめて静的importに戻す（根本解決）
案B: error.tsx の reload を安全なキャッシュバイパス版に変更（安全弁追加）

各案について：
- 実装コスト（行数・複雑さ）
- リスク
- 再発可能性

両案のトレードオフを整理した上で、推奨案を教えてください。
判断は私が行います。
```

## プロンプト（error.tsx の安全リロード実装）

```
app/error.tsx に以下の要件を満たすリロード処理を実装してください。

要件:
1. ChunkLoadError のときのみ自動リロードを実行する
2. ブラウザキャッシュをバイパスするリロードを使う
   （window.location.reload() ではなく location.href にランダムクエリを付与）
3. sessionStorage でリロード回数を管理する
4. 30秒以内に3回を超えたらリロードを止め、通常のエラー画面を表示する
5. sessionStorage が使えない環境でも壊れない（try/catch で囲む）

同じ修正を app/room/[code]/error.tsx にも適用してください。
```

---

## 工夫ポイント

- 「なぜリロードで解決しないか」を最初に問うことで、`window.location.reload()` の問題（キャッシュを再利用する）に気づける
- 修正案の判断をAIに委ねず「判断は私が行います」と明示することで、AIが「あなたに決めてほしい」と答えるようになる
- sessionStorage のガードを要件に明示しないと、AIがシンプルなカウンタを使って無限ループを防げない実装を出すことがある

## 改善余地

- Service Worker が原因の場合はこのプロンプトでは対応できない。SW のキャッシュ戦略も合わせて確認する
- Sentryなどのエラートラッキングがあれば、エラーログを渡すと診断精度が上がる

## 実行結果

- 根本原因: `next/dynamic` でchunk分割 → デプロイ時にハッシュ変更 → 旧HTMLが新chunkを参照できない → `error.tsx` の `window.location.reload()` がキャッシュから旧HTMLを取得 → 無限ループ
- 修正: 静的importに戻す + sessionStorageガード付きキャッシュバイパスリロード
- 解決時間: 約40分（診断15分 + 実装20分 + 確認5分）
