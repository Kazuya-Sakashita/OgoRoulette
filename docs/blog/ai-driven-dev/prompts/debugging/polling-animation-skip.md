# ポーリング遅延によるアニメーションスキップの診断

**レベル:** 上級  
**再利用性:** 中  
**効果:** ★★★★☆

---

## 目的

Realtimeイベントの受信が失敗し、ポーリングfallbackに頼ったクライアントが  
アニメーションをスキップして結果画面が突然表示されるバグを診断する。

## 使用タイミング

- 「一部のユーザーだけアニメーションが表示されない」という報告があったとき
- Realtime（WebSocket）とポーリングを併用している実装でタイミング問題が疑われるとき

---

## プロンプト（タイムライン分析）

```
以下のコードとバグ症状から根本原因を分析してください。

症状:
- 3名参加のルーレットで、Realtime受信が失敗したメンバーだけアニメーションをスキップする
- スキップしたメンバーには結果画面が直接表示される

関連コード:
- TOTAL_ANIM_MS = {値}（スキップ判定の閾値）
- ポーリング間隔 = {値}ms
- spinStartedAt = スピン時刻 + {COUNTDOWN_MS}ms

分析してほしいこと:
1. ポーリング到着タイミングの最悪ケースを計算する
2. elapsed が TOTAL_ANIM_MS を超える条件を求める
3. TOTAL_ANIM_MS の値はいくつにすれば100%カバーできるか

タイムライン形式で整理してください。
```

## プロンプト（修正方針の比較）

```
以下の3つの修正方針を比較してください。

案A（応急）: TOTAL_ANIM_MS を {new_value} に引き上げる
案B（安全）: IN_SESSION 中のポーリング間隔を {shorter_interval} に短縮する
案C（根本）: Realtime再接続時に即座に fetchRoom() を実行する

各案を以下の軸で評価:
- 実装コスト（小/中/大）
- 副作用リスク
- 再発可能性
- ユーザー体験への影響

推奨順を教えてください。
```

---

## 工夫ポイント

- 「最悪ケースを計算する」と指示することで、AIが具体的な数値で分析してくれる
- 修正案を3段階（応急/安全/根本）で用意してAIに比較させると、チームでの議論にもそのまま使える

## 実行結果

- 最悪ケース: T+10000ms到着 → elapsed = 7000ms > TOTAL_ANIM_MS(5000ms) → スキップ発動
- 推奨修正: TOTAL_ANIM_MS を 8000ms に引き上げ（案A）+ IN_SESSION 中ポーリング 2秒化（案B）
