# AI駆動開発 実践ガイド — 1人でプロダクトを動かすための開発メソッド

> このガイドは、1ヶ月のハッカソンでAI駆動開発を実践し、  
> OgoRoulette（https://ogo-roulette.vercel.app）を完成させた過程から抽出した再現可能な手法です。

---

## はじめに

「AI駆動開発」という言葉は広い。

ChatGPTにコードを聞く、Copilotで補完する — それも一種のAI活用だ。  
でも本稿で扱うのは、もっと踏み込んだやり方だ。

**AIを「コンサルタント・設計者・実装者」として機能させ、  
人間は「意思決定者・品質管理者」に徹する開発スタイル。**

このスタイルに移行すると、何が変わるか。

- 1人でもチーム開発に近いスピードが出る
- 「知らないこと」が開発のボトルネックでなくなる
- 品質評価が定量化される

ただし、使い方を間違えると信頼性を犠牲にする判断を誘発する。  
その失敗も含めて書く。

---

## 1. AI駆動開発とは何か

### 定義

AI駆動開発とは、**AIを開発プロセスの中核に据え、仮説・設計・実装・評価のループをAIと共に高速で回すこと**だ。

「AIが書いたコードを使う」ではない。  
「AIと一緒に問題を解く」ことを指す。

### 従来開発との違い

| 項目 | 従来開発 | AI駆動開発 |
|------|---------|----------|
| 情報収集 | Google、StackOverflow | AIに問題ごと渡す |
| 設計 | 自分で考える | AIに叩き台を作らせ、判断する |
| 実装 | 自分で書く | AIが書き、自分がレビューする |
| デバッグ | ログを読んで推測する | AIに状況を渡して分析させる |
| 品質評価 | 手動テスト・勘 | Lighthouse・gstackで定量評価 |
| ボトルネック | 「知らない」こと | 「何を判断するか」を誤ること |

### なぜ今重要か

AIの能力が上がり、「実装の壁」が低くなった。  
かつては「作り方を知っているか」が差になっていた。  
今は「何を作るべきか」と「それを正しく判断できるか」が差になる。

## 2. 開発フロー — 7ステップのループ

これがAI駆動開発の核心だ。  
一度だけ回すのではなく、継続的に繰り返すことで品質が積み上がる。

```
① 仮説 → ② 評価 → ③ 分解 → ④ 改善案 → ⑤ 実装 → ⑥ 検証 → ⑦ 再評価
               ↑_______________________________________________|
```

### ① 仮説を立てる

「このアプリは〜という課題があるはずだ」という仮説を言語化する。  
抽象的でもいい。「なんか遅い気がする」でも入り口になる。

### ② 評価する（gstack / Lighthouse / 実機）

仮説を定量化する。感覚ではなく数字にする。

- **Lighthouse**: Performance、Accessibility、SEO、Best Practices
- **gstack**: 多軸評価スコア、A11y、Core Web Vitals
- **実機**: 実際のデバイス・ネットワーク環境で動かす

```bash
# Lighthouseの実行例
npx lighthouse https://your-app.vercel.app \
  --output=json \
  --output-path=/tmp/lh-report.json \
  --only-categories=performance,accessibility,best-practices,seo
```

### ③ 問題を分解する

評価結果をAIに渡し、ISSUEとして分解させる。

**プロンプト例:**
```
Lighthouseの結果です。
Performance: 62, Accessibility: 86
主な指摘:
- FCP 5.8s（Google Fonts 466KBがブロッキング）
- maximum-scale=1でズーム禁止
- muted-foregroundのコントラスト不足

優先度をつけてISSUEとして整理してください。
```

AIが返すISSUEリストをそのままMarkdownファイルとして保存する。  
これが次のステップの入力になる。

### ④ 改善案を生成する（AI）

各ISSUEに対して、AIに修正方針を出させる。  
この段階では「何をどう変えるか」の設計だ。

```
ISSUE-147: viewport の maximum-scale=1 を削除

修正内容: app/layout.tsx の viewport エクスポートから
maximumScale と userScalable を削除する。

期待効果: Lighthouse A11y の meta-viewport 失敗を解消。
Accessibility スコア +2〜4点想定。
```

### ⑤ 実装する（AI + 人間のレビュー）

AIがコードを変更する。人間は必ず変更内容を確認する。

**レビューで確認すること:**
- ロジックが意図通りか
- 既存の動作を壊していないか
- セキュリティ上の問題がないか

AIの実装は速い。でも確認を省くと信頼性を失う。

### ⑥ 検証する（実機・staging）

ローカル or stagingで動作確認する。  
「動いた」だけでなく「壊れていないか」も確認する。

```bash
npm run build   # ビルドエラーがないか
npm run dev     # ローカルで動作確認
```

### ⑦ 再評価する

修正後にもう一度②の評価を実行する。  
スコアが改善されているか確認する。

**このループが鍵だ。**  
1回で完成させようとしない。何度も回すことで品質が上がる。

## 3. 実際の運用方法

### AIへの指示の出し方

**悪い例:**
```
ここのパフォーマンスを改善して
```

**良い例:**
```
app/layout.tsx の Google Fonts リンクがあります。
現在 Noto Sans JP を wght@400;500;700;900 で読み込んでいます。
Lighthouseで466KBのCSSが render-blocking と指摘されています。

修正方針: wght@400;700;900 に絞り、500を削除してください。
変更対象は layout.tsx:68 の href属性のみです。
```

**ポイント:**
- 具体的なファイルと行数を伝える
- 現状と問題を説明する
- 変更範囲を明示する

### ISSUEベース運用

すべての課題をISSUEファイルとして `docs/issues/` に保存する。

```
docs/issues/
  issue-147-remove-maximum-scale-viewport.md
  issue-148-google-fonts-performance.md
  issue-149-color-contrast-a11y.md
```

各ISSUEファイルには以下を含める:

```markdown
# ISSUE-xxx: タイトル

## 概要
## 症状
## 根本原因
## 修正方針
## 影響範囲
## ステータス（🔴未修正 / ✅完了）
```

このファイルがAIへの文脈になる。  
「ISSUE-148を見て修正してください」と伝えるだけで、AIがファイルを読んで実装する。

### 修正ループの回し方

```
ISSUEを確認 → AIに修正依頼 → レビュー → コミット → デプロイ → 再評価
```

1サイクルを細かく刻む。一度に大量に変えない。  
変更範囲が小さいほど、何が原因か特定しやすい。

## 4. gstackの役割

gstackは、AI駆動開発のワークフローを構造化するフレームワークだ。

### なぜ使ったか

素のAIに「評価して」と言っても、評価軸がブレる。  
gstackを使うと、評価・計画・実装・検証のフローが標準化される。

| gstack スキル | 役割 |
|-------------|------|
| `/investigate` | バグの根本原因を分析する |
| `/browse` | ブラウザを操作して実機確認する |
| EnterPlanMode | 実装前に設計を人間に確認させる |

### 何が変わったか

**計画フェーズが必須になった。**

gstackのPlanModeは、実装前にAIが設計を提示し、人間が承認するフローを強制する。  
これにより「実装してから設計の問題に気づく」ミスが減った。

## 5. 従来開発との違い

### 思考の違い

| 従来 | AI駆動 |
|------|-------|
| 「どう実装するか」から考える | 「何が問題か」から考える |
| 解決策を自分で生成する | AIに複数案を出させて選ぶ |
| 知識の範囲が上限になる | 知識の範囲を超えた提案が来る |

### スピードの違い

実感として、実装速度は3〜5倍になった。  
ただしレビュー時間は省けない。速くなるのは「書く」時間だ。

### 品質の違い

定量評価（Lighthouse、A11y）を毎回回すことで、  
「なんとなく良くなった」ではなく「スコアが何点上がった」になる。

これが品質管理の仕組みとして機能する。

## 6. 失敗と学び

### 失敗①: 最適化が信頼性を壊した

**起きたこと:**  
Lighthouse の「92KB JS削減できます」という推定を信じ、  
`RouletteWheel`を`next/dynamic`で動的ロードに変更した。  
本番デプロイ後にChunkLoadErrorが発生し、ページが開けなくなった。

**原因:**  
デプロイのたびにchunkハッシュが変わり、  
古いHTMLキャッシュが新しいchunkを参照できなくなった。  
`window.location.reload()`がキャッシュから古いHTMLを再取得し、無限ループになった。

**学び:**  
「スコアを上げる」と「ユーザーが使える」は別の問題。  
推定値（estimated savings）を追う前に、リスクを評価する。

**修正:**  
`next/dynamic`を廃止し静的importに戻した。  
あわせて`error.tsx`にsessionStorageでリロード回数を制限する安全弁を追加した。

### 失敗②: CDNキャッシュで評価が狂った

**起きたこと:**  
コードを修正してデプロイしたのに、Lighthouseが古いスコアを返し続けた。  
「修正が効いていない」と誤判断しそうになった。

**原因:**  
Vercel CDNが古いHTMLをキャッシュしていた。

**学び:**  
評価はキャッシュバスター付きURLで実行する。

```bash
npx lighthouse "https://your-app.vercel.app/?v=$(date +%s)" ...
```

## 7. 強みと弱み

### 向いているケース

- 1人または小チームでの高速プロトタイピング
- 品質基準（A11y、Performance）が明確に定義できる開発
- ISSUEが具体的に言語化できる問題
- 既存コードベースの改善・リファクタリング

### 向いていないケース

- 要件が曖昧なまま「とりあえず作る」フェーズ（AIも迷子になる）
- ドメイン知識が深く必要なコア業務ロジック（AIの提案を検証するコストが高い）
- セキュリティクリティカルな実装（AIの提案を盲信しない体制が必要）

## 8. 実践ステップ — 明日から始める手順

### Step 1: 評価から始める

まず現状を数字にする。

```bash
npx lighthouse https://your-app.com \
  --output=json \
  --output-path=/tmp/lh.json \
  --only-categories=performance,accessibility,best-practices,seo
```

### Step 2: ISSUEに分解する

評価結果をAIに渡し、ISSUEとして整理させる。  
各ISSUEをファイルとして保存する。

### Step 3: 優先度をつける

P1（即修正・影響大）とP2（後回し可）に分ける。  
全部を一度に直そうとしない。

### Step 4: 1ISSUEずつ修正する

- AIに修正案を出させる
- 変更内容を自分でレビューする
- ビルドが通ることを確認する
- コミットする（変更を小さく保つ）

### Step 5: 再評価する

修正後に再度Lighthouseを実行する。  
スコアが改善されたことを確認する。  
改善されていなければ原因を分析する。

### Step 6: ループを繰り返す

これを繰り返す。  
1サイクル 30〜60分を目安にする。

## まとめ

AI駆動開発の本質は、**「AIに何でも任せる」ではなく「AIと役割分担する」**ことだ。

| AIがやること | 人間がやること |
|------------|-------------|
| コードを書く | 何を作るか決める |
| 選択肢を提示する | どれを選ぶか判断する |
| バグの原因を分析する | 修正方針を承認する |
| スコアを測定する | 何点を目標にするか設定する |

AIが実装速度を上げてくれる。  
でも「速く間違った方向に進む」ことも起きる。

だから評価ループと人間のレビューを外せない。

**AI駆動開発は「楽になる開発」ではなく「密度が上がる開発」だ。**  
同じ時間でより多くのことを考え、より多くの判断を下すことになる。

それが楽しいかどうかは、人による。  
私は楽しかった。
