# V2はV1を壊すことなく作れ

**「V2を作るとき、V1を上書きした。それが間違いだった。」**

AIと一緒に作業していると、たまに取り返しのつかない方向に進みかけることがある。

今日はその話だ。

---

## AI駆動開発の教科書を作っていた

OgoRoulette を4ヶ月かけて開発したあと、その経験をまとめた「AI駆動開発の教科書」というドキュメントを作っていた。

ISSUE 駆動開発、AIへの文脈転移、プロの判断基準——
150以上のISSUEを積み重ねた実録を、再現可能なメソッドとして整理したものだ。

これが「V1」だった。

---

## V2 を作ることにした

開発を続けるうちに、評価フレームワークが進化した。

G-STACK だけじゃなく、HEART・AARRR・Kano モデルを組み合わせるようになった。
「誰も笑わなかった夜」という体験から、感情設計の重要性に気づいた。

これをV1の教科書に追加して、V2として更新しようと思った。

---

## 最初の失敗

AIに指示を出した。

「V1をV2と同様に更新してください。guide.html と README.md を今回の内容で更新する。」

AIは作業を始めた。

しばらくして、ファイルが変わっていた。

確認すると——

**README.md（V1）が V2 の内容で上書きされていた。**
**guide.html（V1）も同様に上書きされていた。**

V1が消えた。

---

## 何が起きたのか

AIは「V1をV2の内容で更新」という指示を、「V1に V2の内容を書き込む」と解釈した。

私が意図していたのは「V1をベースにしてV2を作る」だったが、
指示の言葉が曖昧だった。

「V1を上書きする」と「V1からV2を作る」は、人間にとっては全然違う意味に聞こえる。
でも言葉にすると、どちらも「V1を更新する」という表現になりうる。

AIは「更新」を「置き換え」と解釈した。

---

## やめてください

私は言った。

**「やめてください。V1が正しい内容です。V1をアップデートしてV2を作成してください。」**

これが大事な一言だった。

AIは git から V1 を復元した。

```bash
git restore docs/ai-driven-development/README.md
git restore docs/ai-driven-development/guide.html
git restore docs/ai-driven-development/pdf/ai-driven-development-guide.pdf
```

V1 が戻ってきた。

---

## 正しいV2の作り方

改めて指示を明確にした。

「V1（README.md、1227行、13章）を完全なベースとして使い、その上に新しい章を追加してください。V1の内容はすべて保持すること。」

今度は正しく動いた。

V2は:
- V1の全13章をそのまま保持
- 新しい第5章（評価フレームワーク四重奏）を追加
- 旧章5〜13を6〜14に繰り下げ
- 合計1500行超のドキュメントに

guide.html（PDF版）も同様に:
- V1の16ページをすべて保持
- 評価フレームワーク5ページを新規追加
- 合計31ページのPDFに

**V1を壊すことなく、V2が生まれた。**

---

## AIへの指示の粒度について

この失敗で改めてわかったことがある。

AIへの指示は、「何をするか」だけじゃなく「何をしないか」も伝えないといけない。

「V1を更新してV2を作る」は不完全な指示だ。

正しくは：
「V1を**完全にそのまま保持しながら**、新しい内容を追加してV2を作る。V1のファイルは変更しない。V2は新しいファイルとして作成する。」

これだけの言葉が必要だった。

普通の会話なら「V2を作る」で伝わる。
でも AIへの指示の場合、「V1を壊さない」という当たり前の前提が、明示しなければ通じないことがある。

---

## バージョン管理が救ってくれた

今回は git があったから助かった。

上書きされた直後に `git restore` で戻せた。

もしバージョン管理をしていなければ、V1は永遠に消えていた。

AI駆動開発でISSUE管理と git を徹底するようにしてきたことが、
まさかこういう形で役に立つとは思っていなかった。

**AIの失敗を救うのも、人間が整備したシステムだ。**

---

## 教訓

今日のことをまとめると、二つになる。

**① 「上書き」と「追加」は明示的に区別する**

「更新する」という言葉は曖昧だ。
「V2を新しいファイルに書く。V1のファイルには触らない。」と言う。

**② バージョン管理は AIの失敗に対するセーフネットだ**

AIは間違える。
でも git があれば、間違いは取り消せる。

ISSUE を細かく刻んで commit を積み重ねるのは、「AIと協働するためのリスク管理」でもある。

---

## V2が完成した

README-v2.md と guide-v2.html（31ページのPDF）が完成した。

V1の内容はすべてそこにある。
その上に、新しい知見が積み重なった。

V1は V1 として残っている。
V2は V2 として生まれた。

「上書き」ではなく「積み重ね」だ。

ソフトウェアのバージョン管理と、まったく同じ考え方だ。

---

## まとめ

- AIに「V1を更新してV2を作れ」と言ったら V1 が上書きされた
- `git restore` で V1 を復元した
- 指示を明確にして V2 を正しく再作成した
- 結果：V1の全内容を保持しつつ、31ページのV2が完成
- 学び：「何をするか」だけでなく「何をしないか」も AIに伝える

バージョンをまたぐ作業は、AIへの指示が特に繊細になる。

「更新」という言葉の裏に、「保持」という意図があるとき。
それを省略せずに言葉にする。

それだけで、V1が守られる。
