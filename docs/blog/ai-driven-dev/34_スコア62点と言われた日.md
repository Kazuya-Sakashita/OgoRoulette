# スコア62点と言われた日

**「動いている。それで十分じゃないか。」**

そう思っていた。

でも、AIはそう言わなかった。

---

## ハッカソン1ヶ月目の終盤

締め切りまであと数日というタイミングで、私は一つの問いを立てた。

「このアプリ、本当に完成していると言えるのか？」

見た目は整っている。
動作も問題ない。
QRコードでみんなが参加できて、ルーレットも回る。

でも「プロの目線で見たとき、どうなのか」という答えを、自分では出せなかった。

だから、AIに評価させることにした。

---

## 数字という残酷な現実

**Lighthouse。**

Googleが提供する、ウェブの品質測定ツール。

正直、怖かった。
「評価」という言葉には、常に怖さがある。
特に自分が作ったものを評価されるのは。

実行してみた。

```
Performance:    62/100
Accessibility:  86/100
Best Practices: 100/100
SEO:            100/100
```

Best PracticesとSEOは満点だった。

でもPerformanceは62点。
Accessibilityは86点。

---

## 「動いているのにダメ」の意味

Performance 62点。

これが何を意味するか。

- FCP（最初の描画）：**5.8秒**
- LCP（主要コンテンツの表示）：**7.3秒**

5.8秒。

スマートフォンで飲み会中にアプリを開いた人が、
5.8秒も白い画面を見続けるということだ。

「あ、壊れてる？」と思って閉じる時間として十分すぎる。

犯人はすぐ特定された。

**Google Fonts 466KB。**

日本語フォント（Noto Sans JP）を4ウェイト全部読み込んでいた。
CSS だけで 466KB。それが画面の描画をすべてブロックしていた。

「フォントがきれいに表示されるように」と思って追加したものが、
ユーザーの体験を5秒以上遅くしていた。

---

## Accessibility 86点の正体

もう一つの問題は、アクセシビリティだった。

86点。
悪くはない。でも良くもない。

AIが指摘した内容の中に、こんなものがあった。

**`maximum-scale=1, user-scalable=no`**

これはViewportの設定。
モバイルでダブルタップしてもズームできないようにする指定だ。

「ズームできないとUI崩れるから」という理由で設定していた。

でもこれは、**弱視のユーザーがブラウザのズーム機能を使えなくなる**ことを意味していた。

WCAG 2.1（ウェブアクセシビリティの国際基準）の違反。

1行の設定が、誰かのアクセスを奪っていた。
それを、自分は知らなかった。

---

## AIと一緒に直す

修正の指示はシンプルだった。

| ISSUE | 内容 | 優先度 |
|-------|------|--------|
| 147 | viewport の zoom 制限を削除 | P1（5分で直せる） |
| 148 | Noto Sans JP のウェイトを 500 削除 | P1 |
| 149 | muted テキストのコントラスト改善 | P2 |
| 150 | RouletteWheel を遅延読み込みに変更 | P2 |

AIがISSUEを分析し、修正コードを提案し、コミットまでサポートする。

普通なら半日かかるような調査と修正が、
1時間かからずに終わった。

---

## 数字が変わった瞬間

再デプロイして、Lighthouseを再実行した。

```
Accessibility: 86 → 92（+6点）
Google Fonts:  466KB → 90KB（-81%）
LCP:           7.3s → 6.5s（改善）
```

Accessibilityが6点上がった。

たった1行の設定（`maximum-scale=1`）を削除しただけで。
フォントのウェイトを一つ削除しただけで、転送量が81%減った。

「知っていれば5分で直せたこと」が、
知らないまま放置されていた。

---

## でも、問題は終わらなかった

**「成功して終わり」にならないのが、リアルな開発だ。**

RouletteWheelを遅延読み込みに変更した後、
本番環境で予期しないエラーが発生した。

```
ChunkLoadError: Loading chunk ab8635c6e7470436 failed.
```

しかも、ページをリロードするたびに同じエラーが出る。
リロード → エラー → リロード → エラー。

**無限ループ。**

`next/dynamic`でコンポーネントを分割したことで、
古いHTMLキャッシュが新しいchunkのURLを参照できなくなっていた。

パフォーマンスを改善しようとした修正が、
別の壊れ方を生み出した。

---

## 「最適化」が「信頼性」を壊す

ここで気づいたことがある。

Lighthouse の「92KB削減できます」という推定は、
あくまで推定だ。

その数字を追って動的ロードを導入した結果、
ユーザーがページを開けなくなった。

スコアより、**ユーザーが確実に使えること**の方が大事だった。

結論：**RouletteWheelは静的importに戻した。**

そして error.tsx に、ChunkLoadErrorが起きたときの安全なリロード処理を追加した。
無限ループに陥らないよう、sessionStorageでリロード回数を制限しながら。

---

## 1ヶ月で学んだこと

**「動いている」と「正しく動いている」は違う。**

この1ヶ月、機能を追加することに必死だった。
でも品質を測ることをしていなかった。

Lighthouseのスコアは、
「自分には見えていなかった問題」を数字にしてくれた。

466KBのフォント。
弱視のユーザーをブロックする1行の設定。
動的ロードが生んだ無限ループ。

どれも、意図して作った問題じゃない。
知らなかっただけだ。

**AIは「知らないこと」を教えてくれる。**

それがこのセッションで一番大きな気づきだった。

---

## まとめ

- Lighthouse Performance: 62点（FCP 5.8s）
- Google Fonts だけで 466KB の render-blocking
- viewport の1行がアクセシビリティ違反
- 修正後: Accessibility 86 → **92点**
- 動的import → ChunkLoadError → 静的importに戻す

「数字で評価する」ことを怖がっていた。

でも数字を見なければ、問題に気づけない。

AIがいると、評価も修正も速い。
でも「判断」は人間がしなければいけない。

それが、今日わかったことだ。

---

## 次回予告

Accessibilityは92点になった。
でもPerformanceはまだ62点のままだ。

FCP 5.8秒の根本原因は、Google Fontsがrendering をブロックしていること。

フォントのウェイトを削っても、
**「CSS ファイルそのものが blocking される」**という構造は変わっていない。

次回は「日本語フォントとの戦い」。

システムフォントへの切り替えか、
フォントの非同期読み込みか。

パフォーマンスを70点以上にするために、何を選ぶのか。
