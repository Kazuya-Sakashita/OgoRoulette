# 「AIに根本原因を探してくれと頼んだら、設計の話になった」
— gstack × AI駆動開発で障害解析はどう変わるか —

---

## また ChunkLoadError が出た

デプロイを終えてSlackに「リリース完了」と投稿した、その10分後だった。

「本番でエラーが出てる」という報告が届く。画面を開くと、全面エラー。ルーレットアプリのメイン機能が一切動かない。ログには `ChunkLoadError: Loading chunk X failed.` という見慣れたメッセージ。

「また、これか」

応急処置はわかっている。`error.tsx` にセッションストレージを使ったガードを実装してあって、エラーが出たらタイムスタンプ付きでリロードをかける。これで大抵は「直る」。ユーザーへの影響も最小限に抑えられる。

でも心の中ではわかっていた。**また次のデプロイで同じことが起きる**、と。

---

## 「直ったように見える」を生産し続けるサイクル

従来の障害対応はこういう流れだった。

1. エラーログを見る
2. 怪しいファイルを探す
3. 修正してデプロイ
4. しばらく様子を見る
5. 「大丈夫そう」と判断して次の機能開発へ

このサイクルの何が問題かというと、**「直った」ではなく「エラーが消えた」だけ**という状態を量産してしまうことだ。

ChunkLoadError は典型的な例で、`error.tsx` のリロード処理は確かに機能する。ユーザーは画面から追い出されることなく、アプリを使い続けられる。だから「対応済み」になる。でも根本原因には一切触れていないから、次のデプロイで必ず再発する。

「なぜ毎回再発するのか」という問いを本気で立てたことが、今回の出発点だった。

---

## gstack という「問いの整理ツール」

この問いに向き合うにあたって、**gstack** というフレームワークを使った。

gstack は Product / UX / Engineering / Risk の4軸で現状を評価するフレームワークだ。数値スコアをつけることで「今何が一番問題なのか」を構造化できる。感覚的な「なんとなくまずい」を、議論できる形に変換してくれる。

ChunkLoadError を gstack で評価するとこうなる。

| 軸 | 評価 | スコア |
|---|---|---|
| **Product/Experience** | 全画面エラー → 無限ループ。コア体験の完全な破壊 | Critical |
| **UX** | エラー画面は実装済みでガードは機能する。でも体感は最悪 | High |
| **Engineering** | 検出・ガードの設計は正しい。しかし原因を隠しているだけ | Medium |
| **Risk** | デプロイのたびに再発。新規ユーザーが最初のアクセスで遭遇する可能性あり | High |

この表を眺めて気づいたことがある。**Engineering の評価だけが Medium** なのだ。

エラーを検出してリロードする設計自体は間違っていない。問題は、その「正しい設計」が原因を隠してしまっているということ。つまり Engineering の問題というよりも、**設計のレイヤーで何かがおかしい**ということを、gstack が示していた。

「このバグはどこを直せばいいのか」という問いから、「この設計は本当に正しいのか」という問いへ。gstack を使ったことで、問いの質が変わった。

---

## AI へのシンプルな依頼

「根本原因を探してほしい」

AIへの依頼は、それだけだった。ChunkLoadError が再発し続けているという事実と、これまでの対応履歴を共有した。

人間がこの調査をやろうとすると、まず「怪しいと思うファイル」を開く。自分の場合なら `next.config.mjs` や `webpack.config.js` あたりから始めていただろう。ビルド設定に問題があると先入観があったから。

AIは違う動きをした。

---

## AIが「全部見る」

AIがまず開いたのは `public/sw.js` だった。

**AI:** 「Service Worker のファイルを確認します。キャッシュ戦略が ChunkLoadError に関与している可能性があります」

正直に言うと、この時点では「なんでそこ？」と思った。`sw.js` は PWA のお守りみたいなもので、「設定してあれば動く」という認識だった。問題が起きるとしたらビルド側だろう、という先入観があった。

AIは続けてファイルを読んでいった。

```
→ public/sw.js を確認中...
  CACHE_VERSION = 'v1' （固定値を発見）
  PRECACHE_URLS に HTML ページが含まれている（問題の予兆）

→ src/app/error.tsx を確認中...
  sessionStorage ガード → _r=timestamp でリロード（設計は正しい）

→ next.config.mjs を確認中...
  ビルド設定に特記事項なし

→ ビルド出力とチャンク命名規則を確認中...
```

数分後、AIが答えを返してきた。

---

## 「これは応急処置が原因を隠しています」

**AI:** 「ChunkLoadError の根本原因は Service Worker の設計にあります。具体的には以下の問題が連鎖しています」

その説明を読んだ瞬間、全部つながった。

```
新デプロイ
  → Vercel が旧チャンクを削除
  → SW が旧 HTML をキャッシュから提供し続ける
  → 旧 HTML が存在しない chunk URL をリクエスト
  → CDN が 404 を返す
  → ChunkLoadError が発生
  → error.tsx が _r=timestamp でリロード
  → SW がキャッシュミスし、ネットワークから新 HTML を取得
  → 一時的に「直る」
  → 次のデプロイでまた同じことが起きる
```

`CACHE_VERSION = 'v1'` が固定値だったこと。誰も気にしたことがなかった。`PRECACHE_URLS` に HTML ページが含まれていたこと。「キャッシュしてるんだろうな」くらいの認識で、何も問題視していなかった。

`sw.js` は意識の外にあった。PWA の機能として「あるもの」として扱われていて、「正しく動いているか」を問われることがなかった。

人間だと「怪しいと思ったファイル」しか見ない。AIは「全部見る」。その違いがここで効いた。

---

## 「どこを直すか」から「なぜそう設計したか」へ

AIの最終的な回答は、「このコードを直してください」ではなかった。

**AI:** 「この `sw.js` の設計は、Service Worker の使い方として根本的に誤っています。HTML ページをキャッシュするということは、アプリのエントリポイントをブラウザのネットワーク層よりも下の層で制御するということです。Next.js のようにデプロイのたびにチャンクハッシュが変わる構成では、これは必ず問題を起こします」

「根本的に誤っている」という言葉は重かった。応急処置が何度も効いていたから、「設計は正しい、実装の問題だ」という思い込みがあった。でもAIは設計レベルの問題を指摘していた。

提案されたのが **Tombstone SW**（墓標 Service Worker）という解決策だった。

```js
// 全キャッシュを削除し、fetch は一切キャッシュしない
self.addEventListener('install', () => {
  self.skipWaiting()
})

self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys()
      .then((keys) => Promise.all(keys.map((key) => caches.delete(key))))
      .then(() => self.clients.claim())
  )
})

// fetch ハンドラなし — すべてのリクエストはネットワーク直通
```

既存の Service Worker を「全部消す係」に置き換える。キャッシュをすべて削除して、新しいリクエストは何もキャッシュしない。PWA の機能としては退化するように見えるが、チャンクの整合性は常に保たれる。

「直す」のではなく「設計を変える」という選択肢。これがデバッグを超えた瞬間だった。

## AI駆動開発が変えた4つのこと

この経験を通じて、AI駆動のデバッグが何を変えるのかが具体的にわかった。

### 1. スピード

コードベースを横断して調査するのに、人間なら数時間かかる。どのファイルを見るべきか考える時間、実際に読む時間、仮説を立てて検証する時間。AIはこれを数分でやる。

### 2. 網羅性

人間には先入観がある。「ここじゃないだろう」という感覚で読まないファイルが出てくる。AIには先入観がない。コードベースに存在するものは全部読む。`sw.js` を読んだのは、AIに先入観がなかったからだ。

### 3. 構造化

「なんとなくここが怪しい」という感覚的な仮説ではなく、「このファイルの、この設定が、この理由で問題を起こしている」という根拠のある仮説を立てられる。人間もやれることだが、疲弊した状態では難しい。

### 4. 昇華

「どこを直すか」という問いが、「なぜそう設計したか」という問いに変わる。これが一番大きい。バグ修正を超えて、設計レビューになる。再発しないためには設計を変えるしかない、という結論に自然にたどり着く。

---

## AIの出力を鵜呑みにしない

ただし、重要な注意点がある。

AIの回答を読んで「なるほど、そういうことか」と膝を打った後、自分でも連鎖を追って検証した。AIが指摘したキャッシュの問題は、実際に Chrome DevTools の Application > Cache Storage を開いて確認した。旧HTMLが残っていることを自分の目で見た。

**AIはコードを読めるが、本番の挙動を知らない。**

Tombstone SW を適用した後、意図的に旧チャンクが残っている状態を作って ChunkLoadError が出なくなることを確認した。

「AIが言ったから正しい」ではなく、「AIの仮説を人間が検証する」というプロセスが不可欠だ。AIは探索のエージェントであって、判断の主体ではない。仮説を立てる速度と網羅性においてAIは人間を超えているが、その仮説が本番の文脈で正しいかどうかを判断するのは人間の役割だ。

役割分担を明確にすることで、AI駆動開発は機能する。

---

## 「見えていなかったところ」に気づく体験

今回最も印象的だったのは、`sw.js` が「意識の外にあった」という事実そのものだった。

コードは書いたのに、見ていなかった。問題が起きているとは思っていなかった。「PWAの機能」というラベルが貼られていたから、「設定済み」として扱っていた。

人間の認知には限界がある。注意を向けられるファイルの数に限界があるし、「怪しいと思わないファイル」は読まない。これは怠慢ではなく、効率のために必要な認知の絞り込みだ。でもその絞り込みが、根本原因を見えなくさせることがある。

AIは絞り込まない。先入観なく全体を見る。その視点を借りることで、「見えていなかったところ」に光が当たる。

これが、AI駆動デバッグが最も効く瞬間だと思う。

## 学びのまとめ

- **gstack** は「何が問題か」を整理するツールであり、問いの質を変える
- **AI** は「どこが問題か」を先入観なく探索するエージェントである
- 両方を組み合わせることで、「なぜ問題が起きているか」の根本に到達できる
- 再発しないバグ修正は「コードを直す」ではなく「設計を変える」ことから生まれる
- AIの仮説は必ず人間が検証する。判断の主体は常に人間にある

## 次に活かす視点

障害が起きたとき、最初に立てる問いを変えてみてほしい。

「どのコードが壊れているか」ではなく、「この問題はどの設計レイヤーで起きているか」。

gstack の4軸（Product/UX/Engineering/Risk）でスコアをつけてみると、「Engineering だけ Medium」のような非対称性が見えてくることがある。その非対称性こそが、設計の問題を示すサインだ。

そしてAIに依頼するとき、「このコードを直してください」ではなく「根本原因を探してください」と頼んでみてほしい。AIが開くファイルの順番を見ているだけで、自分の先入観に気づかされることがある。
