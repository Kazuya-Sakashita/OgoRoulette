# Lighthouse 評価から見えた OgoRoulette の課題と改善計画

**公開日:** 2026-04-03  
**カテゴリ:** 品質改善 / パフォーマンス / アクセシビリティ

---

## はじめに

OgoRoulette は「飲み会の奢りをルーレットで決める」パーティーゲームアプリ。ある程度の機能が揃ってきたタイミングで、**適切な評価指標で現状を計測**してみた。

感覚ではなく数字で課題を把握し、優先度をつけて改善する — そのプロセスをまとめる。

---

## 評価に使った指標

今回使ったのは 5 つの視点：

1. **Lighthouse / Core Web Vitals** — 技術品質の客観指標
2. **バンドル分析** — リソースの無駄遣いを可視化
3. **アクセシビリティ（A11y）** — 誰でも使えるかの確認
4. **AARRR ファネル** — プロダクトとしての成長指標
5. **UX ヒューリスティック** — Nielsen の 10 原則

## 最大の問題: Google Fonts 466KB

Core Web Vitals を見ると：

| 指標 | 値 | 判定 |
|---|---|---|
| FCP | 5.8s | 🔴 Poor |
| LCP | 7.3s | 🔴 Poor |
| TBT | 0ms | 🟢 Good |
| CLS | 0 | 🟢 Good |

FCP が 5.8 秒という数字に最初は驚いたが、原因はすぐ分かった。

**Google Fonts の CSS が 466KB** ある。Noto Sans JP を全ウェイト（400/500/700/900）でロードしており、そのサブセット CSS だけでページ全体のリソースの半分近くを占めていた。

実ブラウザで計測すると FCP は 764ms で全く問題ない。Lighthouse の「シミュレーテッド低速 4G」環境でのみ顕在化する問題だが、地方や海外から使うユーザーには実際に遅く感じる可能性がある。

```typescript
// 問題のある現状
const notoSansJP = Noto_Sans_JP({
  subsets: ["latin"],
  weight: ["400", "500", "700", "900"],  // 4ウェイト全部
})

// 修正方針: ウェイトを絞り display:swap を追加
const notoSansJP = Noto_Sans_JP({
  subsets: ["latin"],
  weight: ["400", "700", "900"],  // 500 を削除
  display: "swap",
})
```

## 見落としていたアクセシビリティ問題

Accessibility が 86/100 で止まっている原因は主に 3 つ：

### 1. `maximum-scale=1` の罠

```html
<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1">
```

これは**弱視ユーザーのズームを禁止する**。WCAG 2.1 の 1.4.4 (Resize text) 違反になる。

モバイルのダブルタップズームを防ぐために追加されがちだが、アクセシビリティ上は NG。削除するだけで Critical が 1 つ消える。

### 2. muted テキストのコントラスト不足

`text-muted-foreground` は dark モードの暗い背景に対してコントラスト比が 4.5:1 を下回っている。利用規約・プライバシーリンクも同様。

### 3. リンクの視覚的区別

本文中のリンクが周囲テキストとコントラスト差が小さく、リンクとして認識されにくい。

## バンドルの無駄

Lighthouse の指摘：
- **未使用 CSS: 119KB** 削減余地
- **未使用 JS: 92KB** 削減余地

CSS は Tailwind の purge が完全に機能していない可能性。JS は Next.js のチャンク分割で重いコンポーネント（RouletteWheel、録画機能）を遅延ロードすることで改善できる。


**Retention が弱点**。スピン後にアプリを閉じたユーザーが戻ってくる動線がない。ログイン促進はあるが、「次回の飲み会でもう一度使いたい」というトリガーが不足している。

## 学んだこと

**感覚で「速い」と思っていても、計測すると全然違う**。

実ブラウザで 764ms の FCP が Lighthouse では 5.8s になる。ターゲットユーザー（飲み会中のスマホ）は必ずしも高速回線ではない。Google Fonts の重さは「快適な環境では気にならない」が、そうでない環境では致命的になりうる。

また `maximum-scale=1` は自分では「ダブルタップ防止の常套句」として何も考えずに入れていた。5分で直せる問題が WCAG 違反になっていた。

定期的に Lighthouse を回して数字で見ることの価値を改めて感じた。
