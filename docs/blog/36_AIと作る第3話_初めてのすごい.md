# AIと作る ── 第3話「初めての『すごい』」

## CLAUDE.md を書いた翌日

CLAUDE.md を書いた翌朝、いつものようにターミナルを開いた。

違いはすぐ分かった。

「戻るボタンは `/home` に」と説明しなくていい。「頼んだ箇所以外は変えないで」と念押ししなくていい。最初から分かっているように振る舞ってくる。

コストが下がっていた。

ただ、この時点ではまだ「仕組みが整ったな」という感覚だった。「革命だ」とは思っていなかった。

## リアルタイム同期の実装をお願いした日

アプリの核心機能のひとつが「全員で同じルーレットを見る」だった。

部屋を作った人がスピンボタンを押すと、参加者全員の画面でルーレットが回り始める。
そして同じ結果が、全員の画面に同時に表示される。

これを実現するには Supabase Realtime が必要だった。

問題は、自分が Supabase Realtime を使ったことがないということだ。

ドキュメントを読んでみた。「チャンネル」「ブロードキャスト」「プレゼンス」。
用語は分かるが、どう組み合わせるのかが見えない。

「やってみるか」と Claude に投げた。

---

## 5分後に出てきたもの

```typescript
// Supabase Realtime チャンネル購読
const channel = supabase
  .channel(`room:${code}`)
  .on('broadcast', { event: 'spin-start' }, ({ payload }) => {
    setIsSpinning(true)
    setTargetAngle(payload.angle)
  })
  .on('broadcast', { event: 'spin-result' }, ({ payload }) => {
    setWinner(payload.winner)
    setIsSpinning(false)
  })
  .subscribe()
```

動いた。

5分かかっていない。自分で実装しようとしたら、半日は悩んでいたと思う。

ただ「動いた」という感動ではなかった。もっと別の何かだった。

## 「ここまで考えてくれているのか」

驚いたのはコードの中身だった。

最初のバージョンを見せてもらったとき、エラーハンドリングが入っていた。

```typescript
.subscribe((status, err) => {
  if (status === 'CHANNEL_ERROR') {
    console.error('[Realtime] channel error:', err)
    // フォールバック: ポーリングに切り替える
    startPollingFallback()
  }
  if (status === 'TIMED_OUT') {
    channel.subscribe()  // 再接続を試みる
  }
})
```

「リアルタイム接続が切れた場合、ポーリングにフォールバックする」

頼んでいなかった。でも必要だった。

WebSocket は切れる。特にスマホのバックグラウンドでは頻繁に切れる。
フォールバックがなければ、接続が切れた瞬間にルーレットが止まって全員が取り残される。

「ここまで考えてくれているのか」と思った。

これが最初の「すごい」だった。

## 「頼んでいないことをする」との違い

第2話で「頼んでいないことをされる」と書いた。
これは「同じことでは？」という疑問が湧くかもしれない。

違う。

第2話の「頼んでいないこと」は「エラーメッセージを変えた」「スタイルを修正した」という**要求外の変更**だった。
今回の「フォールバック」は、**実装すると約束したこと（リアルタイム同期）を成立させるために必要なもの**だった。

前者はノイズ。後者は判断。

Claude が「フォールバックなしでリアルタイム同期を実装する」という指示を文字通りに解釈せず、「フォールバックがないと本番で死ぬ」という判断を持ってきた。

CLAUDE.md に「頼んだ箇所以外は変更しない」と書いてあったが、Claude はそれを破った。
でも正しく破った。

## 「この人は分かっている」という感覚

飲み会でルーレットを実際に動かした日、友人から言われた。

「操作できない感じがするな。」

ルーレットを回した人間が結果を操作できない構造に見える、という意味だ。
それが「疑われない」の核心だった。

設計の思想が伝わっていた。

「この人は分かっている」と思う相手に仕事を頼むとき、細かく指示しなくても良いものが出てくる。
Claude がその感覚に近づきつつあった。

もちろん Claude は「人」ではない。覚えてもいない。
ただ、仕事の仕方として「分かっている」という感触があった。

---

## 初めての「すごい」の正体

しばらく経って、「すごい」の正体が分かった気がした。

「速い」ではない。
「知識がある」でもない。

**「自分が言語化できていないことを、実装に落としてくれる」**

リアルタイム同期をお願いしたとき、「切れたときどうする？」とは言っていない。
でも Claude はそれを考えて実装してきた。

自分の頭の中にあった「こうなってほしい」という曖昧なイメージが、コードになった。
しかも自分が考えきれていなかった部分まで。

それが初めての「すごい」だった。
