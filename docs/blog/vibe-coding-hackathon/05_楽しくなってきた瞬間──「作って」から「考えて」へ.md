# Vibe Codingハッカソンに参加して、AIと一緒にアプリを作ってみた話
## 第5話：楽しくなってきた瞬間──「作って」から「考えて」へ

---

## 何かが変わったのはいつだったか

正確な日は分かりません。

「作業が楽しい」と気づいたのは、ある夜の帰り道でした。その日の朝バグを直して、午後に機能を追加して、夜にデプロイした。

電車の中でスマホでアプリを開いて、友人にリンクを送りました。「試してみて」と。

2分後に返信が来ました。「これ本物のルーレットっぽい。」

思わず笑いました。

---

## 会話が変わっていた

開発序盤の会話はこうでした。

```
自分：「ルーム機能を作ってください」
Claude：[コードを出す]
自分：「ありがとう」
```

この時点の会話はこうなっていました。

```
自分：「参加者がリロードしたとき、Realtimeのチャンネルが切れることがある。
どういうケースが考えられる？」

Claude：「3つ考えられます。
① WebSocketがタイムアウト
② ネットワーク切断後の再接続
③ ページがバックグラウンドに移動したとき」

自分：「ケース③が一番厄介そう。どう対処する？」
→ 議論して、理解できる実装方針を固めてからコードを書いてもらう
```

**「作って」ではなく「考えて」になっていました。**

これが会話スタイルの最大の変化です。

## 楽しさの正体：「自分だけでは辿り着けなかった場所」

Lighthouseを開きました。

```
Performance:      91
Accessibility:    92
Best Practices:  100
SEO:              91
```

最初に計測したときは62点でした。

技術的なことだけではありません。デザインも変わりました。「スマホで見たときに、普通のアプリじゃないみたい」と言われるようになりました。

Framer Motion でルーレットのアニメーションを実装した日、スマホで動かして「きれい」と思いました。自分が直接書いたアニメーションではありませんでした。でも自分が「こういうふうに動いてほしい」と伝えたものでした。

**「自分が作れなかったものが、自分のアイデアのまま形になった」**

これが楽しさの正体だと気づきました。

## 友達とオンラインで試した日

リリースして最初の「本番」は、友達とのビデオ通話でした。

リンクを共有して、それぞれのスマホで開いてもらいました。「ランチ代、誰が払うか決めよう」と言ってスピンを押しました。

「なにこれ、ちゃんとしてる。」  
「操作できないようになってるの？」  
「あ、ルーレット回ってる。みんなの画面で同時に動いてる！」

画面越しにルーレットが止まる瞬間を、みんなで待ちました。

止まった。

**「うわー！」**

そのリアクションは、じゃんけんでも指差しでも出てこないものでした。画面越しでも盛り上がれると分かりました。

---

自分はお酒を飲まないので、飲み会の場には合いませんが、**ランチや食事会・会社の集まりで使えるはず**だと思っています。お酒が好きな人には飲み会で、自分のような人には食事の奢り決めで。どちらの場でもルーレットという「体験」は変わらない。

ビデオ通話が終わり、「今日うまくいった」と伝えたくなりましたが、Claude には今日のことは分かりません。そしてそれでいいと思えるようになっていました。

## AI駆動開発で学んだこと：5つの気づき

このシリーズを通じて気づいたことを整理します。

### ① CLAUDE.md（コンテキストファイル）が命

AIはセッションをまたいで記憶を持ちません。「プロジェクトのルール」「禁止事項」「設計の思想」をファイルに書いておくことで、毎回説明しなくて済みます。

```markdown
# OgoRoulette - 開発ルール

## 絶対に守ること
- router.back() は使用禁止。必ず明示的な Link href を使う
- 頼んだ箇所以外は変更しない
- 理解できないコードはマージしない
```

これはAIへの説明書ではなく、「自分がどう作りたいか」を言語化する作業でもあります。

### ② 「頼む」より「考える」方が良いものができる

「作って」という頼み方では、自分のコードではなくなります。「この設計どう思う？」「このケースが抜けていない？」という対話形式の方が、理解も深まり、より良い実装になります。

### ③ 理解できないコードは使わない

スピードのために理解を諦めると、バグを自分で直せなくなります。「なぜこう書いたか説明して」と聞く習慣が、コードのオーナーシップを守ります。

### ④ AIは「言語化できていないこと」を補完してくれる

最大の価値は処理速度ではなく、**自分が曖昧に思い描いている「こうなってほしい」をコードに変えてくれること**です。自分が考えきれていなかった部分まで。

### ⑤ 楽しさは「速く作れる」ではなく「辿り着けない場所へ行ける」から来る

最初は「速く作れる楽しさ」がありました。それを超えたのが「自分だけでは辿り着けなかった場所に辿り着く楽しさ」です。これはしばらく使い続けた後に来ました。

## まとめ：AI駆動開発の本質

> AIは「速く書けるツール」ではなく、  
> **「自分が言語化できていないことを、実装に落としてくれるパートナー」**。  
>
> 何を作るか・なぜ作るか・品質を判断するのは常に人間の仕事。

Vibe Codingハッカソンという場があったから、「確かめる」ためのアプリを完成まで持っていけました。

Claude Code は、使い始めればすぐに魔法のように動くわけではありません。一緒に仕事をする相手として扱うための、自分なりの方法を見つける必要がある。その過程が、開発という体験そのものでした。

このアプリは今も本番で動いています。よければ試してみてください。

**[OgoRoulette — おごりをルーレットで決めよう](https://ogo-roulette.vercel.app/)**
