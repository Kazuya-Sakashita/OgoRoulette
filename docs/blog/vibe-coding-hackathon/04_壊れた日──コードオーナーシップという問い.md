# Vibe Codingハッカソンに参加して、AIと一緒にアプリを作ってみた話
## 第4話：壊れた日──コードオーナーシップという問い

---

## ある日、本番だけ動かなくなった

その日は普通の朝でした。

コーヒーを淹れて、本番環境を確認しました。

「ルーレットの結果が表示されない。」

スピンボタンを押す。ルーレットが回る。止まる。画面は何も変わらない。当選者が出てこない。

ローカルでは動いている。本番だけ動かない。

---

## ログを読んだ

Vercel のログを開きました。エラーはない。APIは200を返している。データベースには書き込まれている。

「動いているのに動いていない。」

ブラウザのコンソールを開きました。

```
[OgoRoulette] handleSpinComplete fired
[OgoRoulette] pendingMemberWinner: null
```

`pendingMemberWinner` が `null` でした。

これはルーレットが止まったとき、サーバーから受け取るはずの「当選者情報」です。それが `null` のままだった。

---

## 原因：競合状態（race condition）

コードを追いかけました。

```typescript
// ルーレットが止まったとき
const handleSpinComplete = useCallback(() => {
  const winner = pendingMemberWinnerRef.current  // ← ここが null
  if (winner) {
    setWinner(winner)
  }
}, [])
```

```typescript
// サーバーから結果を受信したとき
channel.on('broadcast', { event: 'spin-result' }, ({ payload }) => {
  pendingMemberWinnerRef.current = payload.winner  // ← ここで設定されるはず
})
```

**順序の問題**でした。

ルーレットアニメーションが終わるのと、サーバーから結果が届くのが**競合**していました。

```
アニメーション完了 → handleSpinComplete 呼ばれる → pendingMemberWinner がまだ null → 結果が表示されない
```

ネットワーク遅延が大きいとき、アニメーションが先に終わり、当選者データがまだ届いていない状態が発生していました。ローカルでは遅延がないので再現しない。本番の遅延がある環境でだけ起きる。

---

## Claude に投げた

「こういう競合が起きている。どう直す？」と投げました。

Claude が提案してきたのはシンプルでした。「アニメーション完了時に winner がなければ、winner が来るまで待つ」

```typescript
const handleSpinComplete = useCallback(() => {
  const winner = pendingMemberWinnerRef.current
  if (winner) {
    // すでに結果が届いていたらすぐ表示
    setWinner(winner)
    pendingMemberWinnerRef.current = null
  } else {
    // まだ届いていなければ、届いたタイミングで表示する
    setWaitingForWinner(true)  // フラグを立てる
  }
}, [])

// 結果受信側
channel.on('broadcast', { event: 'spin-result' }, ({ payload }) => {
  pendingMemberWinnerRef.current = payload.winner
  if (waitingForWinnerRef.current) {
    // アニメーションがすでに終わっていたら即表示
    setWinner(payload.winner)
    setWaitingForWinner(false)
  }
})
```

どちらが先に来ても対応できる設計です。デプロイして動きました。

## でも、これで終わりじゃなかった

しばらくして、別のバグが出ました。「当選者が2回表示される」というものでした。

コードを読みました。修正のために入れたフラグ管理に漏れがありました。

直しました。するとまた別のバグが出ました。

「直してまた壊れる」サイクルが続きました。

---

## 「これは自分のコードなのか？」

この時期に思ったのは、「このコードを本当に理解しているのか？」という問いでした。

Claude が書いたコードを、自分がレビューして、「良さそう」と判断してデプロイしている。でも「良さそう」は「理解した」と同じではない。

バグが出たとき、すぐに原因が見えない。自分で書いたコードなら、書いたときの思考経路があるから「あそこかもしれない」と当たりがつく。Claude が書いたコードにはその経路がない。

デバッグのたびに初めて読むような感覚がありました。

**「壊れた日」の本当の意味は、バグではありませんでした。**

「自分がこのコードのオーナーではないかもしれない」という感覚が壊れた日でした。

---

## 方針を変えた

そこから少し変えました。

コードを出してもらったとき、**「なぜこう書いたのか説明して」と聞くようにしました。**

理解できないコードはマージしないことにしました。

複雑な実装は「まず方針だけ話して、実装は自分でやる」を選ぶことも増えました。

Claude がパートナーなら、パートナーの判断を丸ごと受け入れるのではなく、判断の根拠を共有して、自分が納得できるものを採用する。それが「コードのオーナーでいる」ということだと思いました。

---

## この時点で学んだこと

| 気づき | 内容 |
|-------|------|
| **競合状態はローカルで再現しない** | 本番の遅延がある環境でしか起きないバグがある |
| **「良さそう」≠「理解した」** | レビューして承認してもオーナーシップは自動では生まれない |
| **「なぜ」を聞く習慣** | 理解できないコードはマージしない。これがコードオーナーシップを守る |

> **ルール: 理解できないコードはマージしない**
