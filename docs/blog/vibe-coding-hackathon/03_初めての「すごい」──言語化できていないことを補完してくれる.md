# Vibe Codingハッカソンに参加して、AIと一緒にアプリを作ってみた話
## 第3話：初めての「すごい」──言語化できていないことを補完してくれる

---

## CLAUDE.md を書いた翌日

CLAUDE.md を書いた翌朝、いつものようにターミナルを開きました。

違いはすぐ分かりました。「戻るボタンは `/home` に」と説明しなくていい。「頼んだ箇所以外は変えないで」と念押ししなくていい。最初から分かっているように振る舞ってくる。

コストが下がっていました。

ただ、この時点ではまだ「仕組みが整ったな」という感覚でした。「革命だ」とは思っていませんでした。

---

## リアルタイム同期の実装をお願いした日

OgoRouletteの核心機能のひとつが「全員で同じルーレットを見る」でした。

部屋を作った人がスピンボタンを押すと、参加者全員の画面でルーレットが回り始める。そして同じ結果が、全員の画面に同時に表示される。

これを実現するには **Supabase Realtime** が必要でした。

問題は、自分が Supabase Realtime を使ったことがないということです。ドキュメントを読んでみました。「チャンネル」「ブロードキャスト」「プレゼンス」。用語は分かるが、どう組み合わせるのかが見えない。

「やってみるか」と Claude に投げました。

---

## 5分後に出てきたもの

```typescript
// Supabase Realtime チャンネル購読
const channel = supabase
  .channel(`room:${code}`)
  .on('broadcast', { event: 'spin-start' }, ({ payload }) => {
    setIsSpinning(true)
    setTargetAngle(payload.angle)
  })
  .on('broadcast', { event: 'spin-result' }, ({ payload }) => {
    setWinner(payload.winner)
    setIsSpinning(false)
  })
  .subscribe()
```

動きました。5分かかっていない。自分で実装しようとしたら、半日は悩んでいたと思います。

でも「動いた」という感動ではありませんでした。もっと別の何かでした。

---

## 「ここまで考えてくれているのか」

驚いたのはコードの中身でした。

最初のバージョンを見たとき、頼んでいないコードが入っていました。

```typescript
.subscribe((status, err) => {
  if (status === 'CHANNEL_ERROR') {
    console.error('[Realtime] channel error:', err)
    // フォールバック: ポーリングに切り替える
    startPollingFallback()
  }
  if (status === 'TIMED_OUT') {
    channel.subscribe()  // 再接続を試みる
  }
})
```

「リアルタイム接続が切れた場合、ポーリングにフォールバックする」処理です。

頼んでいなかった。でも**必要だった**。

WebSocket は切れます。特にスマホのバックグラウンドでは頻繁に切れる。フォールバックがなければ、接続が切れた瞬間にルーレットが止まって全員が取り残される。

「ここまで考えてくれているのか」と思いました。

これが最初の「すごい」でした。

---

## 「頼んでいないことをする」との違い

第2話で「頼んでいないことをされる」と書きました。「これは同じことでは？」という疑問が湧くかもしれません。

**違います。**

第2話の「頼んでいないこと」は「エラーメッセージを変えた」「スタイルを修正した」という**要求外の変更**でした。

今回の「フォールバック」は、**実装すると約束したこと（リアルタイム同期）を成立させるために必要なもの**でした。

前者はノイズ。後者は判断。

CLAUDE.md に「頼んだ箇所以外は変更しない」と書いてあっても、Claude はそれを破りました。でも**正しく破った**のです。

---

## 「すごい」の正体

しばらく経って、「すごい」の正体が分かった気がしました。

「速い」ではない。「知識がある」でもない。

> **「自分が言語化できていないことを、実装に落としてくれる」**

リアルタイム同期をお願いしたとき、「切れたときどうする？」とは言っていません。でも Claude はそれを考えて実装してきました。

自分の頭の中にあった「こうなってほしい」という曖昧なイメージが、コードになった。しかも自分が考えきれていなかった部分まで。

それが初めての「すごい」でした。

---

## 「この人は分かっている」という感覚

飲み会でルーレットを実際に動かした日、友人から言われました。

「操作できない感じがするな。」

ルーレットを回した人間が結果を操作できない構造に見える、という意味です。それがOgoRouletteの設計の核心でした。

設計の思想が伝わっていた。

「この人は分かっている」と思う相手に仕事を頼むとき、細かく指示しなくても良いものが出てきます。Claude がその感覚に近づいていました。

もちろん Claude は「人」ではない。覚えてもいない。ただ、仕事の仕方として「分かっている」という感触がありました。

---

## この時点で学んだこと

| 気づき | 内容 |
|-------|------|
| **AIの最大の価値** | 速さではなく「自分が言語化できていないことを実装に落とす」こと |
| **「頼んでいない変更」の2種類** | ノイズ（不要な変更）と判断（必要なもの）は区別できる |
| **コンテキストを渡すと変わる** | CLAUDE.md を書いたことで、AIが意図を超えた提案をするようになった |
