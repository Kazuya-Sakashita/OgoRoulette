# Vibe Codingハッカソンに参加して、AIと一緒にアプリを作ってみた話
## 第1話：ハッカソン参加のきっかけと OgoRoulette というアプリ

---

## はじめに

Vibe Codingハッカソンに参加し、**OgoRoulette（オゴルーレット）** というWebアプリをゼロからリリースしました。

「ランチや食事会・飲み会で誰が奢るかをルーレットで決める」アプリです。

このシリーズでは、アプリの機能よりも **Claude Code（AIコーディングツール）と一緒に開発するという体験が、自分にとってどういうものだったか** を書いていきます。

「AIを使えばすぐできる」でも「大したことない」でもなかった。その話です。

---

## なぜこのアプリを作ろうと思ったか

友人のグループチャットに、こんなメッセージが流れてきました。

「今日のランチ、誰が払うか決めようとしたんだけど、なんか盛り上がらなかったな。」

分かる気がしました。じゃんけん、指差し、アプリのルーレット。どれも「公平に決まる」のに、なぜかシラけた空気が漂う。

理由はシンプルでした。**「決め方」に演出がない。**

コイントスは一瞬で終わる。アプリのルーレットはスマホの画面を1人でぽちぽちするだけ。周りは何が起きているかすら分からない。盛り上がる理由がない。

「自分で作れるかもしれない」と思いました。

---

## 作ったもの：OgoRoulette

**本番URL**: https://ogo-roulette.vercel.app/

OgoRouletteが実現したかったのは3つです。

| 要件 | なぜ必要か | 解決策 |
|------|-----------|--------|
| **疑われない** | 操作できると思われると盛り上がらない | 当選者はサーバーが決定する設計 |
| **全員で見られる** | 1人の画面をみんなで覗き込む形では盛り上がらない | Supabase Realtimeで全員の画面が同期 |
| **結果がすぐ分かる** | 「で、いくら払うの？」をその場で解決 | 割り勘金額をリアルタイム計算 |

特に「疑われない」は重要な設計ポイントでした。ルーレットを回した本人が当選者を決定できる構造では、どんな演出をしても信頼されません。当選者はサーバーが決定する必要がある。

### 主な機能

- **ルーレット**: 参加者（最大8人）を登録してルーレットを回す。当選者がアニメーションで表示される
- **リアルタイム同期**: オーナーがスピンすると、全員の画面に同時にアニメーションが流れる
- **金額計算**: 合計金額と奢り金額を入力すると、割り勘金額を即計算
- **ルーム機能**: QRコードでメンバーが参加できる
- **ゲストモード**: ログイン不要で即試せる

### 使用技術

| カテゴリ | 技術 |
|---------|------|
| フレームワーク | Next.js 16（App Router） |
| 言語 | TypeScript |
| 認証 | Supabase Auth（Google / LINE / X） |
| リアルタイム同期 | Supabase Realtime（WebSocket） |
| アニメーション | Framer Motion |
| DB | PostgreSQL（Supabase）/ Prisma |
| デプロイ | Vercel |

Framer Motion は初めて触る技術でした。それでも選べたのは、Claude Codeがあったからです。

## なぜ Claude Code を使ったか

Vibe Codingハッカソンが、Claude Code を使って開発するというコンセプトのイベントでした。だから使うことは最初から決まっていました。

ただ、正直なところ「どこまで頼れるのか」は未知数でした。

個人開発で少し経験はありました。

- Next.js・Supabase・Prisma・PostgreSQL は個人開発で使ったことがある
- Framer Motion は完全に初めて

「ある程度知っている技術 + Claude Code」がどんな組み合わせになるのか。それを確かめる場でもありました。

ターミナルを開き、最初の指示を打ちました。

```
「飲み会で誰が奢るか決めるアプリを作ります。まずルーム機能から作ってください。」
```

Claude が動き始めました。ファイルが増えていく。コードが書かれていく。自分がタイプしていないのに、アプリの骨格ができていく。

「思ったより早い」というのが最初の感想でした。そして、これが違和感の始まりでもありました。
