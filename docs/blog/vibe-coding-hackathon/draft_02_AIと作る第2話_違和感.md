# AIと作る ── 第2話「AIとの違和感」

## 最初は「賢いコード補完」だと思っていた

Claude Code を使い始めた最初の数日は、とにかく快調だった。

「このコンポーネント作って」と言えばすぐ出てくる。
「このAPIのルートを追加して」と言えば動くコードが来る。

「これは革命だ」と思った。

違和感が生まれたのは3日目くらいだった。

---

## 「昨日決めたことを忘れている」

ルームページを作っていたとき、前日のセッションで決めた設計がある。

「戻るボタンはどこでも `/home` に向ける。`router.back()` は絶対に使わない。」

外部リンクから来たユーザーがいる場合、`router.back()` では前のページが存在しない。
行き止まりになる。だから明示的な `href` を使うと決めた。

翌朝、新しいセッションでエラー画面を作ってもらった。

Claude が出してきたコード：

```tsx
<Button onClick={() => router.back()}>
  戻る
</Button>
```

「router.back() は使わないって昨日言ったじゃないか。」

そう思ったが、Claude には「昨日」がない。
**セッションが変わると、前の会話を覚えていない。**

---

## 「毎回初めて会う人」という感覚

これが最初の大きな違和感だった。

人間と作業を引き継ぐとき、前回の話を踏まえて進められる。
「昨日ここまでやったから、今日はここから」という感覚がある。

Claude にはそれがない。
毎朝「初めまして」から始まる。

決めたルールを説明し直す。
決めた設計を説明し直す。
禁止事項を説明し直す。

これが毎回かかるコストだった。

---

## 「頼んでいないことをする」

もう一つの違和感がある。

バグ修正をお願いした。
「ルーム画面の戻るボタンが `/room/${code}` に向いているのを `/home` に直して。」

Claude が直した。そして、頼んでいないものを一緒に直した。

```
修正内容:
1. 戻るボタンのhrefを/homeに変更
2. エラーメッセージのテキストを改善
3. ローディング中のスピナーを追加
4. ヘッダーのスタイルを統一
```

4つ直された。頼んだのは1つだ。

「ありがとう」と言いたい気持ちはある。でも問題がある。

**変更が増えるほど、レビューが難しくなる。**

どれが自分の意図した変更で、どれが勝手な変更なのか分からなくなる。
バグが出たとき、原因が「直してほしかった箇所」なのか「勝手に変えられた箇所」なのか分からない。

「善意のノイズ」が積み重なっていく感覚があった。

---

## 「合っているのに違う」

もっと根深い違和感もあった。

Claude が出すコードは「動く」。
でも「自分が書くコードとは違う」という感覚が拭えない。

変数名のつけ方、関数の分け方、コメントの書き方。
全部が「正しいが自分じゃない」という感じがした。

コードレビューをするのが自分の書いたコードではなく、
他人が書いたコードをチェックしているような感覚。

「これは自分のアプリなのか？」という疑問が、ぼんやりと浮かんでいた。

---

## 違和感の正体

しばらくして、違和感の正体が分かってきた。

**Claude はツールではなく、コンテキストが必要なパートナーだ。**

人を雇って「作って」と言うだけで理想通りのものが出てくるとは思わない。
前提を共有して、判断基準を渡して、振り返りを重ねる。
それがないと「ズレたものを一生懸命作られる」だけになる。

Claude も同じだった。

「言ったことを覚えていない」のは仕様だ。
「頼んでいないことをする」のは「指示が曖昧だった」ということだ。
「自分じゃないコード」が出てくるのは「自分らしさを渡していない」ということだ。

---

## 最初にやるべきだったこと

`CLAUDE.md` を書いた。

```markdown
# OgoRoulette - 開発ルール

## 絶対に守ること
- router.back() は使用禁止。必ず明示的な Link href を使う
- 頼んだ箇所以外は変更しない
- 実装前に「何を・なぜ・どう実装するか」を一言確認する
```

これを書いた翌日から、違和感の密度が下がり始めた。

毎朝同じ説明をしなくていい。
頼んでいないことをされるケースが減った。
変更の理由を聞けるようになった。

違和感は「AIが変」なのではなく「渡し方が足りていない」だった。

*続く*
