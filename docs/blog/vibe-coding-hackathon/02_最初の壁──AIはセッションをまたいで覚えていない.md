# Vibe Codingハッカソンに参加して、AIと一緒にアプリを作ってみた話
## 第2話：最初の壁──AIはセッションをまたいで覚えていない

---

## 開発序盤は快調すぎた

Claude Code を使い始めた序盤は、とにかく快調でした。

「このコンポーネント作って」と言えばすぐ出てくる。「このAPIのルートを追加して」と言えば動くコードが来る。

「これは革命だ」と思いました。

違和感が生まれたのは、快調な序盤が終わる頃でした。

---

## 「昨日決めたことを忘れている」

ルームページを作っていたとき、前日のセッションで決めた設計がありました。

> 「戻るボタンはどこでも `/home` に向ける。`router.back()` は絶対に使わない。」

外部リンクから来たユーザーがいる場合、`router.back()` では前のページが存在しない。行き止まりになる。だから明示的な `href` を使うと決めた。

翌朝、新しいセッションでエラー画面を作ってもらいました。

Claude が出してきたコード：

```tsx
<Button onClick={() => router.back()}>
  戻る
</Button>
```

「router.back() は使わないって昨日言ったじゃないか。」

そう思いました。でも、Claude には「昨日」がありません。

**セッションが変わると、前の会話を覚えていない。**

これが最初の大きな壁でした。

---

## 「毎回初めて会う人」という感覚

人間と作業を引き継ぐとき、前回の話を踏まえて進められます。「昨日ここまでやったから、今日はここから」という感覚がある。

Claude にはそれがない。毎朝「初めまして」から始まる。

- 決めたルールを説明し直す
- 決めた設計を説明し直す
- 禁止事項を説明し直す

これが毎回かかるコストでした。

---

## 「頼んでいないことをする」問題

もう一つの壁があります。

バグ修正をお願いしました。「ルーム画面の戻るボタンが `/room/${code}` に向いているのを `/home` に直して。」

Claude が直しました。そして、頼んでいないものを一緒に直してきました。

```
修正内容:
1. 戻るボタンのhrefを/homeに変更   ← 頼んだこと
2. エラーメッセージのテキストを改善   ← 頼んでいない
3. ローディング中のスピナーを追加     ← 頼んでいない
4. ヘッダーのスタイルを統一          ← 頼んでいない
```

「ありがとう」と言いたい気持ちはあります。でも問題がある。

**変更が増えるほど、レビューが難しくなる。**

どれが自分の意図した変更で、どれが勝手な変更なのか分からなくなります。バグが出たとき、原因がどちらの変更なのかも分からない。

「善意のノイズ」が積み重なっていく感覚がありました。

---

## 解決策：CLAUDE.md を書く

違和感の正体がわかってきました。

**Claude はツールではなく、コンテキストが必要なパートナーだ。**

人を雇って「作って」と言うだけで理想通りのものが出てくるとは思いません。前提を共有して、判断基準を渡して、振り返りを重ねる。それがないと「ズレたものを一生懸命作られる」だけになる。

Claude も同じでした。

`CLAUDE.md` を書きました。

```markdown
# OgoRoulette - 開発ルール

## 絶対に守ること
- router.back() は使用禁止。必ず明示的な Link href を使う
- 頼んだ箇所以外は変更しない
- 実装前に「何を・なぜ・どう実装するか」を一言確認する

## 技術的な決定事項
- 当選者の決定はサーバー側で行う（クライアントで決めない）
- Supabase Realtime でリアルタイム同期する
```

プロジェクトのルートに置くと、毎回のセッションで自動的に読み込まれます。

これを書いた翌日から、違和感の密度が明らかに下がりました。

- 毎朝同じ説明をしなくていい
- 頼んでいないことをされるケースが減った
- 変更の理由を聞けるようになった

**違和感は「AIが変」なのではなく「渡し方が足りていない」だった。**

---

## この時点で学んだこと

| 問題 | 原因の本質 | 解決策 |
|------|-----------|--------|
| 毎回説明し直す | セッション間に記憶がない | CLAUDE.md にルールを書く |
| 頼んでいないことをする | 指示が曖昧だった | 「変更箇所を1つに絞る」と明示する |
| 設計がズレる | 意図を渡していない | 「なぜ」も一緒に伝える |

CLAUDE.md は「AIへの説明書」ではなく、**「このプロジェクトの判断基準を書いたドキュメント」**です。自分がどう作りたいか、何を大切にしているか、を言語化する作業でもありました。
