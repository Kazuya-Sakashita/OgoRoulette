# デプロイ直前に見つかった：知らないと事故るセキュリティ問題 3選

## 導入

「機能は全部動いている。テストも通った。あとはデプロイするだけ」。

そのタイミングでコードを改めて見直したら、**本番に出したら事故になっていた問題**が3つ見つかった。

いずれも「初心者がやりがちだが気づきにくい」種類の問題。
OgoRoulette での実例を元に説明する。

---

## 問題 1：OAuth のリダイレクト先を信頼してしまっていた（open redirect）

### 何が起きていたか

Google でログインした後のコールバック処理（`app/auth/callback/route.ts`）に、こんなコードがあった：

```typescript
const forwardedHost = request.headers.get("x-forwarded-host")

if (forwardedHost) {
  return NextResponse.redirect(`https://${forwardedHost}${next}`)
}
```

リクエストヘッダーの `x-forwarded-host` をそのまま信頼して、リダイレクト先の URL を組み立てていた。

### なぜ危険か

`x-forwarded-host` ヘッダーは、状況によっては攻撃者が偽造できる。
たとえば、攻撃者が：

```
GET /auth/callback?next=/home
x-forwarded-host: evil-phishing-site.com
```

というリクエストを作って、正規のログインページへのリンクとして共有したとする。
ユーザーが Google でログインを完了すると、`https://evil-phishing-site.com/home` にリダイレクトされる。

ログイン直後という「ユーザーが信頼している瞬間」に、フィッシングサイトへ誘導できてしまう。

### 修正方針

```typescript
// ❌ 修正前：ヘッダーをそのまま信頼
return NextResponse.redirect(`https://${forwardedHost}${next}`)

// ✅ 修正後：next パラメーターを相対パスに限定し、origin を固定する
const safeNext = next.startsWith("/") ? next : "/home"
const siteUrl = process.env.NEXT_PUBLIC_SITE_URL ?? origin
return NextResponse.redirect(`${siteUrl}${safeNext}`)
```

**チェックポイント：**
- リダイレクト先の URL を外部から制御できないか？
- `next` や `redirect` のようなクエリパラメーターを無検証で使っていないか？
- リダイレクト先が「自分のサイト内」であることを確認しているか？

## 問題 2：環境変数がなかったら DB にゴミデータが残る

### 何が起きていたか

ゲストユーザーがルームを作成する処理の流れ：

```typescript
// 1. ルームを DB に作成する
const room = await prisma.room.create({ data: { ... } })

// 2. ゲスト用のトークンを生成する
const hostToken = signGuestToken(hostMember.id, inviteCode)
//               ↑ GUEST_HOST_SECRET 環境変数が必要

// 3. トークンをクライアントに返す
return NextResponse.json({ ...room, hostToken })
```

`GUEST_HOST_SECRET` という環境変数がないと `signGuestToken` が throw する。
catch して 500 エラーを返していたが、**その時点でルームはすでに DB に作られている**。

### なぜ危険か

ユーザーから見ると「エラーが出た → もう一度やり直す」となる。
やり直すたびにルームレコードだけが DB に蓄積されていく（使えないゾンビルームの量産）。

環境変数の設定ミスは本番デプロイ時によく起きる。
開発環境では動いていたが、本番の環境変数を設定し忘れていた、というパターン。

```
本番デプロイ → 環境変数設定漏れ → ゲストがルーム作ろうとするたびにゾンビルーム増殖
→ DB 容量を圧迫 → サービス全体に影響
```

### 修正方針

**DB に書き込む前に、必要な設定が揃っているかチェックする（フェイルファスト）：**

```typescript
// ✅ リクエスト冒頭で環境変数を確認する
export async function POST(request: Request) {
  if (!process.env.GUEST_HOST_SECRET) {
    console.error("[rooms] GUEST_HOST_SECRET is not configured")
    return NextResponse.json(
      { error: "サーバー設定エラーが発生しました" },
      { status: 500 }
    )
  }

  // ここまで来たら設定が揃っている → DB 書き込みへ進む
  const room = await prisma.room.create({ ... })
  ...
}
```

**チェックポイント：**
- DB 書き込みの前に「その処理が最後まで完了できる条件」が揃っているか？
- 途中で失敗したとき、DB に「中途半端なデータ」が残らないか？
- 必須の環境変数を「使う直前」ではなく「リクエストの最初」で確認しているか？

## 問題 3：重要なエンドポイントにレートリミットがなかった

### 何が起きていたか

ルームへの参加 API（`POST /api/rooms/join`）にはレートリミットが実装されていた：

```typescript
// POST /api/rooms/join
const { allowed, resetAt } = checkRateLimit(ip, "join", 10, 60_000)
if (!allowed) {
  return NextResponse.json({ error: "..." }, { status: 429 })
}
```

しかし**ルーム作成 API（`POST /api/rooms`）とスピン API（`POST /api/rooms/[code]/spin`）にはなかった**。

### なぜ危険か

レートリミットのないエンドポイントは、連続リクエストで DB に過負荷をかけられる。

スピン API は特に重い処理をする：
1. DB でルームのステータスを確認（SELECT）
2. ステータスを `IN_SESSION` に更新（UPDATE のトランザクション）
3. ルーレットセッションを作成（INSERT）

これを 1秒間に 1000 回叩かれたら、DB コネクションが枯渇してサービス全体が落ちる。

```
悪意ある利用者（または bot）→ スピン API に連続リクエスト
→ Supabase の同時接続数制限に達する
→ 全ユーザーの DB 操作が失敗
→ サービス全体がダウン
```

### 修正方針

既存の `checkRateLimit` をそのまま流用するだけ：

```typescript
// POST /api/rooms（ルーム作成）
import { checkRateLimit, getClientIp } from "@/lib/rate-limit"

export async function POST(request: Request) {
  const ip = getClientIp(request.headers)
  const { allowed, resetAt } = checkRateLimit(ip, "room-create", 5, 60_000)
  if (!allowed) {
    return NextResponse.json(
      { error: "リクエストが多すぎます。しばらくしてからお試しください。" },
      { status: 429, headers: { "Retry-After": String(Math.ceil((resetAt - Date.now()) / 1000)) } }
    )
  }
  // 以降の処理へ
}
```

**チェックポイント：**
- 書き込み系の API（POST/PUT/DELETE）すべてにレートリミットがあるか？
- 「重い処理」をするエンドポイントほど、低めの制限を設けているか？
- 制限に引っかかったとき、ユーザーにわかりやすいメッセージを返しているか？

## 3問題に共通する考え方

今回見つかった3つの問題は、いずれも「作っているときは意識しにくい」種類のもの。

| 問題 | カテゴリ | 気づきにくい理由 |
|------|--------|----------------|
| open redirect | セキュリティ | 機能としては「動いている」 |
| ゾンビルーム | データ整合性 | 通常利用では発生しない |
| レートリミット欠落 | 可用性 | 普通に使う分には問題ない |

### デプロイ前のチェックリスト（最低限）

```
□ リダイレクト先を外部から制御できるコードはないか？
□ DB 書き込み前に「処理完了に必要な条件」を確認しているか？
□ 書き込み系 API にレートリミットはあるか？
□ 環境変数の設定漏れでサービスが静かに壊れないか？
□ エラーが起きたとき、DB に中途半端なデータが残らないか？
```

機能が動くことと、安全に公開できることは別の話。
「デプロイできる状態」を確認するのは、機能完成後に行う独立した作業として考えると良い。

---

## まとめ

| 問題 | 発見方法 | 対処 |
|------|--------|------|
| open redirect | コールバック処理のリダイレクト先を確認 | `next` パラメーターを相対パスに限定、origin を環境変数で固定 |
| ゾンビルーム | DB 書き込み後のエラーハンドリングを確認 | 必須環境変数をリクエスト冒頭でフェイルファスト確認 |
| レートリミット欠落 | 全 POST/PUT/DELETE エンドポイントを確認 | 既存の `checkRateLimit` を全書き込み系 API に適用 |

コードレビューやデプロイ前チェックで「セキュリティ・データ整合性・可用性」の観点を加えるだけで、
本番事故を大幅に減らせる。
