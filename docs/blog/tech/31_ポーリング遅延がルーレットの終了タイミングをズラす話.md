# ポーリング遅延がルーレットの終了タイミングをズラす話

## はじめに

「オーナーのルーレットが止まったのに、メンバーの画面ではまだ回ってる。」

これを聞いたとき、最初は「Supabase Realtime の設定ミスかな」と思った。
しかし調べてみると、原因は全く別のところにあった。

コードは意図通りに動いていた。
`spinStartedAt` はサーバーで決定され、全員に届いていた。
アニメーション時間も全員 4.5 秒固定だった。

それでも、数秒のズレが生じていた。

この記事は、その原因と修正の話だ。

---

## 前提: 同期の設計

このアプリのルーム機能では、ルーレットの当選者はサーバーが決定する。
不正防止のためだ。

スピン開始のタイムラインはこうなっている：

```
オーナーが「スピン」ボタンを押す
  ↓
POST /api/rooms/[code]/spin
  ↓
サーバーが当選者を決定
spinStartedAt = now + 3000ms  （3秒後にアニメーション開始）
  ↓
{ winnerIndex, spinStartedAt } を返す
```

3秒のバッファを設けた理由は、メンバーにもこの情報が届く時間を確保するためだ。

**オーナー**は API レスポンスで即座に `spinStartedAt` を受け取り、
3秒後にアニメーションを開始する。

**メンバー**は Supabase Realtime またはポーリングで `IN_SESSION` 状態を検知し、
`session.startedAt`（= `spinStartedAt`）を基準にアニメーション開始を計算する。

```typescript
// メンバー側のアニメーションスケジューリング（修正前）
const startMs = session.startedAt
  ? new Date(session.startedAt).getTime()
  : Date.now()
const delay = Math.max(0, startMs - Date.now())
setTimeout(() => setPhase("spinning"), delay)
```

理屈は正しい。`spinStartedAt` より前に受信できれば、ちょうど同時にアニメーションが始まる。

---

## 問題の核心: `Math.max(0, ...)` が隠していたもの

フォールバックポーリングの間隔は **10 秒**だ。

- `spinStartedAt` は 3 秒後に設定される
- メンバーの次のポーリングが最悪 10 秒後に来る場合、受信時点ですでに `spinStartedAt` は過去

このとき `startMs - Date.now()` は **負の値** になる。
`Math.max(0, 負の値) = 0` で、メンバーは受信後即座にアニメーションを開始する。

問題は「受信が遅れたこと」ではなく、「遅れたのにフルの 4.5 秒で動かしたこと」だ。

```
タイムライン（メンバーが 3 秒遅れて受信した場合）:

T+0:    オーナーがスピン → spinStartedAt = T+3000
T+3000: オーナーのアニメーション開始（4.5s）
T+6000: メンバーがポーリングで受信（spinStartedAt は 3 秒前）
T+6000: メンバーのアニメーション開始（フル 4.5s）← ここが問題

T+7500: オーナーのホイールが止まる  ✅
T+10500: メンバーのホイールが止まる ❌ 3 秒遅れ
```

**メンバーの「遅れた開始」を考慮せず、常にフル 4.5 秒を使っていた。**

Realtime が機能していても、WebSocket 接続の遅延や再接続が発生すれば同じことが起きる。

---

## もう一つのズレ: ランダムな回転数

実はもう一つ問題があった。

`minSpins`（何回転するか）がクライアントごとに独立したランダム値だった。

```typescript
// roulette-wheel.tsx（修正前）
const minSpins = 5 + Math.floor(Math.random() * 3) // 5 | 6 | 7
```

停止角度は同じでも、4.5 秒の中で「何回転するか」が違えば速度感が全く異なる。

- オーナー: 5 回転でゆったり減速
- メンバー: 7 回転で激しく回ってから急減速

同じ結果を見ているはずなのに、「違う演出を見ている感覚」になる。

> ※ この `minSpins` が整数であること自体は記事13で解決済みだが、
> 「全クライアントで同じ値にする」という課題は残っていた。

---

## 修正方針

### Fix 1: 経過時間に応じて duration を短縮する

メンバーがアニメーション開始直前に「どれだけ遅れているか」を計算し、
その分だけ `duration` を削る。

```
adjustedDuration = max(0.5, 4.5 - elapsed_seconds)
```

elapsed が 5 秒以上なら、アニメーション自体が終わっているはずなので、
アニメーションをスキップして直接結果表示に遷移する。

```
Before:
  オーナー: T+3000 開始、4.5s → T+7500 に終了
  メンバー: T+6000 開始、4.5s → T+10500 に終了  ← 3 秒ズレ

After:
  オーナー: T+3000 開始、4.5s → T+7500 に終了
  メンバー: T+6000 開始、1.5s → T+7500 に終了  ✅ 同時
```

### Fix 2: `spinStartedAt` を seed に `minSpins` を決定論化する

全クライアントが同じ `spinStartedAt`（Unix ms）を持っている。
これを seed として `minSpins` を導出すれば、全員が同じ値になる。

```typescript
// spinSeed = spinStartedAt ms
const minSpins = 5 + (Math.floor(spinSeed / 1000) % 3)
```

`spinStartedAt` は毎回異なる値なので、毎スピン 5・6・7 がランダムに選ばれる。
ただし **同じスピンでは全員が同じ値を得る**。

### `app/room/[code]/play/page.tsx`

メンバーの `scheduleSpin` を修正する。

```typescript
const scheduleSpin = (session: Session) => {
  // ... winner の取得 ...

  const startMs = session.startedAt
    ? new Date(session.startedAt).getTime()
    : Date.now()
  const delay = Math.max(0, startMs - Date.now())
  setSpinStartedAtMs(startMs)
  spinScheduledRef.current = true
  setPhase("preparing")

  const TOTAL_ANIM_MS = 5000  // 4500ms spin + 500ms bounce buffer
  setTimeout(() => {
    const elapsed = Math.max(0, Date.now() - startMs)

    // アニメーション終了済みならスキップして直接 result へ
    if (elapsed >= TOTAL_ANIM_MS) {
      const winnerData = pendingMemberWinnerRef.current
      if (winnerData) {
        setWinner(winnerData)
        setPhase("result")
        spinScheduledRef.current = false
      }
      return
    }

    setSpinElapsedMs(elapsed)    // ← 短縮 duration の計算に使う
    setPhase("spinning")
  }, delay)
}
```

`RouletteWheel` に props を渡す。

```tsx

  spinElapsedMs={spinElapsedMs}
  spinSeed={spinStartedAtMs ?? undefined}
  // ...既存 props

```

オーナーは常にフル duration なので `spinElapsedMs={0}` を設定するだけでいい。

## 設計の考え方

今回の修正を一言で表すなら：

**「同じ瞬間に終わること」を不変条件にする。**

従来の設計は「同じ瞬間に *始まること*」を目標にしていた。
サーバーで `spinStartedAt` を決め、全員がその時刻にアニメーションを開始する。

しかしネットワークの現実はその仮定を崩す。
メンバーは常に「ちょうど間に合う」とは限らない。

終了タイミングを合わせるアプローチは、開始の遅れを許容しながら最終結果だけを揃える。

```
旧設計: 「同時に始める」→ ネットワーク遅延に弱い
新設計: 「同時に終わる」→ 遅延を elapsed として吸収する
```

これは映像のフレーム同期に似た考え方だ。
ライブ配信で遅延が発生したとき、「最初から追いかける」より「現在時刻に合わせる」方が自然だ。

---

## solo play への影響

`spinElapsedMs` と `spinSeed` はどちらも optional props で、デフォルト値は `undefined`。

- solo play（ルーム外）では渡さない → `minSpins` はランダム、duration は 4.5 秒固定
- ルーム play では渡す → 全クライアントが同じ minSpins、duration は elapsed に応じて短縮

コンポーネントの動作が「渡された props に応じて変わる」設計なので、
既存の solo play には一切影響しない。

## チェックリスト: 多クライアント同期アニメーションの確認

```
□ アニメーションの「開始」だけでなく「終了」タイミングを揃えているか
□ ネットワーク遅延でイベント受信が遅れた場合の elapsed を考慮しているか
□ elapsed が duration を超えた場合（アニメーション終了済み）の処理があるか
□ ランダム値が「演出の見た目」だけでなく「timing」に影響していないか
□ solo と multi でコンポーネントの振る舞いが切り替わる仕組みになっているか
```

## まとめ

問題の本質はここだった：

```
メンバーは遅れて受信したのに、
遅れた分を考慮せずフルの 4.5 秒でアニメーションを動かしていた。
```

修正の核心はここだ：

```typescript
// アニメーション開始直前に elapsed を計算し、duration を削る
const elapsed = Math.max(0, Date.now() - spinStartedAt)
const duration = Math.max(0.5, 4.5 - elapsed / 1000)
```

`Math.max(0, ...)` は正しい処理に見える。
しかし「delay がゼロになった理由」を見落とすと、こういうバグが生まれる。

ポーリング遅延は「滅多に起きない例外」ではなく、
**常に発生しうる前提** として設計する必要がある。
