# 金額入力 UX の改善（事前入力 or 後入力対応）

## 概要

「割り勘金額の即計算」は OgoRoulette の重要な付加価値だが、現在の実装では金額をルーレットの前に入力する必要があり、初回ユーザーには「先に金額を入れるべき」が伝わりにくい。後入力対応 or 明確なヒントで価値体験の完走率を上げる。

## 背景

OgoRoulette の設計意図は「当選者が決まったら、すぐにいくら払うかも分かる」という体験。これを実現するには合計金額と奢り額を事前に入力する必要がある。

しかし:
- 金額入力フィールドはルーレット下部に配置
- 初回ユーザーはルーレットを先に回してしまう
- WinnerCard Phase B で「¥0（未入力）」の表示になる
- 「あ、先に入れればよかった」→ リスピンが必要

この流れが発生すると、割り勘計算という価値提案が機能しない。

## 現状の問題

1. 金額入力フィールドが「あとで入れても OK な補助的なもの」に見える
2. ルーレットと金額入力の視覚的な関係性が弱い
3. 金額を入れずに回すと WinnerCard Phase B の金額欄が空欄になる
4. Phase B になってから金額を入力する方法がない（一度閉じる必要がある）

## 目的

- 割り勘計算という価値提案を、より多くのユーザーが体験できるようにする
- 「金額を先に入れる」か「後で入れる」かに関わらず金額計算が完結できるようにする
- 入力忘れによる「体験が中途半端で終わる」ケースを減らす

## 対応内容

### 案 A: 後入力対応（推奨）

WinnerCard Phase B に「金額を後から入力できる」UI を追加:

```
当選者が決まりました！

合計金額: [ ¥_____ ] ← ここで入力できる
奢り金額: [ ¥_____ ]

→ 割り勘: ¥2,500 / 人
```

Phase B で金額フィールドを inline 編集可能にし、入力するとリアルタイムで計算結果が更新される。

実装:
- `totalBill` と `treatAmount` を WinnerCard の内部 state として持つ（外部から渡す初期値がある場合はそれを使用）
- `calculateBillSplit()` を WinnerCard 内でも呼び出す

### 案 B: 入力ヒントの強化（軽量案）

スピンボタン付近に「💰 先に金額を入れると割り勘が計算されます」というヒントを表示。
金額未入力のままスピンした場合、結果後に「金額を入れますか？」というトースト通知を出す。

### 案 C: スピン前に金額入力を促すモーダル

スピンボタンを押した際、金額が未入力なら「金額を入力しますか？」という選択肢を出す。
「入力する」→ 入力フォームに誘導。「このまま続ける」→ スキップして即スピン。

**推奨: 案 A（後入力対応）+ 案 B の軽量ヒント**

案 A を採用することで「先に入れなかったから体験できなかった」が解消される。
案 B のヒントで初回から正しい使い方を促す。

## 完了条件

- [ ] WinnerCard Phase B で金額をその場で入力・変更できる
- [ ] 入力値が変わると割り勘計算がリアルタイムで更新される
- [ ] スピンボタン付近に「先に金額を入れると計算されます」というヒントがある
- [ ] ホームページで金額入力済みの場合は Phase B に初期値として反映される

## 優先度

**Recommended** — 割り勘計算は OgoRoulette の差別化機能の1つ。使えないまま終わるユーザーを減らす。

## 期待効果

- UX スコア: 65 → 68（+3）
- 割り勘計算の利用率向上
- 「金額入れ忘れた」によるリスピン（体験のやり直し）の減少

## 備考

- 現在の計算ロジック: `lib/bill-calculator.ts` の `calculateBillSplit()`
- `home/page.tsx` と `play/page.tsx` の両方で使われているため、WinnerCard 自体に後入力機能を持たせるのが実装コスト最小
- 金額入力フィールドは `totalBill`・`treatAmount` の 2 フィールドのみ
- `hasBillInput` フラグ（totalBill > 0 && treatAmount > 0）で計算済みかどうかを判定している
