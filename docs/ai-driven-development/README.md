# 第1章: はじめに

## 1.1 AI駆動開発とは何か

AI駆動開発とは、**AIをペアプログラマーとして中心に据えた開発手法**だ。「AIにコードを書いてもらう」のとは根本的に違う。

AIは「速く書けるツール」ではなく、「自分が言語化できていないことを実装に落としてくれるパートナー」だ。AIは文脈を理解し、候補を出し、実装を提案する。しかし、**何を作るか・なぜ作るか・品質を判断するのは常に人間の仕事**だ。

AI駆動開発の本質は3つある。

1. **思考の言語化**: 頭の中の「なんかおかしい」をISSUEとして言語化する
2. **AIへの文脈転送**: 言語化した文脈をAIに正確に渡す
3. **人間による判断**: AIの提案を評価して採用・却下を決める

OgoRouletteでは、この3ステップを150回以上繰り返した。毎回のISSUEがドキュメントになり、そのドキュメントが次のISSUEを生む。**品質は積み上がる**のだ。

## 1.2 なぜ今必要か

2026年現在、個人開発の民主化が加速している。かつては「フルスタックエンジニア+デザイナー+PM」の3人チームで数ヶ月かかっていたアプリが、AIを使えば1人で数週間で作れる時代になった。

しかし、多くの個人開発者が「AIを使っているのに品質が上がらない」「AIに丸投げしたら意図しないものが出てきた」という壁に当たっている。

この壁の正体は **「AIの使い方の問題」ではなく「思考の言語化の問題」** だ。AIは賢い。しかし、曖昧な指示には曖昧な実装を返す。「なんかいい感じにして」では何も生まれない。

スピードが重要な時代において、この「思考の言語化→AIへの転送→判断」のサイクルを速く正確に回せる人が、圧倒的に有利だ。

## 1.3 従来開発との違い

| 観点 | 従来開発 | AI駆動開発 |
|---|---|---|
| 設計 | 人間が全部設計してから実装 | ISSUEを書いてAIと一緒に設計 |
| 実装速度 | 人間が一行ずつ書く | AIがドラフトを生成、人間がレビュー |
| デバッグ | スタックトレースを読んで手動調査 | ログ+コードをAIに渡して原因仮説を得る |
| ドキュメント | 後回しになりがち | ISSUEがドキュメントになる |
| 品質管理 | テスト・レビューのみ | gstack評価→数値化→改善ループ |
| ボトルネック | コーディング速度 | 思考の言語化速度 |
| チームサイズ | 複数人前提 | 1人でフルスタック対応可能 |
| 技術的負債 | 気づかず積み上がる | ISSUEで可視化・計画的に返済 |

従来開発では「コードを書く速さ」がボトルネックだった。AI駆動開発では「思考を言語化する速さ」がボトルネックになる。だからこそ、ISSUEの書き方が最も重要なスキルになる。

## 1.4 この教科書でできること

この教科書を読み終えたとき、あなたには以下が手に入る。

- **AI駆動開発の全体像**: 何をどの順序でやるか、なぜその順序なのかが理解できる
- **ISSUEの書き方**: 「思考の言語化」の技術。一生使えるスキルだ
- **テンプレート7種**: コピペで即使える。要件定義からLP生成まで
- **プロンプト20+種**: 場面別に選べる。試行錯誤なしで使える
- **ケーススタディ**: 実際のバグ・設計・最適化の事例。同じ壁にぶつかったときの地図になる
- **失敗パターン**: よくある罠を事前に知れる。回り道を省ける
- **バズ設計**: 作った後に広まる設計を最初から組み込む方法

# 第2章: 全体フロー（最重要）

## 2.1 AI駆動開発エンジン

AI駆動開発の核心は、以下のループだ。このループを理解せずにテンプレートやプロンプトだけ使っても、効果は半減する。

```
┌─────────────────────────────────────────────────────┐
│               AI駆動開発エンジン                    │
│                                                     │
│  ① 課題発見                                        │
│      「なんかおかしい」「ここが気になる」           │
│         ↓                                           │
│  ② ISSUE化（思考の言語化）                         │
│      背景・問題・原因・解決策・受け入れ条件を書く  │
│         ↓                                           │
│  ③ AI解決案生成                                    │
│      テンプレート + プロンプトでAIに渡す           │
│         ↓                                           │
│  ④ 実装（AI + 人間の判断）                         │
│      AIの提案を評価・修正・採用する                │
│         ↓                                           │
│  ⑤ 検証（gstack）                                  │
│      数値で品質を確認する                          │
│         ↓                                           │
│  ⑥ 再ISSUE化                                       │
│      検証で見つかった新課題をISSUEにする           │
│         ↓                                           │
│  ① に戻る（品質が積み上がる）                     │
└─────────────────────────────────────────────────────┘
```

OgoRouletteでは、このループを1日に複数回回した。1ループあたり30分〜2時間。150回回した結果が、現在のアプリだ。

## 2.2 各ステップの詳細解説

### ① 課題発見

「なんかおかしい」「ユーザーが戸惑いそう」「パフォーマンスが悪い気がする」という直感を大切にする。この直感を**その場でメモ**することが重要だ。後回しにすると消える。

OgoRouletteでは、Slackのメモチャンネルにひとこと投げておき、あとでISSUEにまとめた。「ルーレットが止まる瞬間の演出が弱い」「iOSで音が出ない」「ルーム参加時にローディングが長い」。これらがすべてISSUEになった。

### ② ISSUE化（思考の言語化）

課題を発見したら、すぐISSUEに落とす。ISSUEは「思考の結晶」だ。

良いISSUEには5つの要素がある。背景（なぜ今これを解決するのか）、問題（何が起きているか）、原因（なぜそうなっているか）、解決策（どう直すか）、受け入れ条件（どうなれば完了か）。この5要素がそろったISSUEは、AIへの指示書として完璧に機能する。

### ③ AI解決案生成

ISSUEが書けたら、テンプレートにはめてAIに渡す。テンプレートは「AIへの文脈転送の型」だ。適切な型で渡すと、AIの回答の質が劇的に上がる。

例えばデバッグのISSUEなら T-03（バグ調査テンプレート）を使う。UI改善なら T-02（UI生成テンプレート）を使う。「どのテンプレートを使うか」の判断が人間の重要な仕事だ。

### ④ 実装（AI + 人間の判断）

AIの提案は「叩き台」だ。そのまま使うのではなく、評価して採用・却下・修正を判断する。

**AIに任せる**: コードのドラフト生成、テスト作成、コメント追加、リファクタ候補の提案。
**人間が判断する**: アーキテクチャの最終決定、状態管理の設計、ユーザー体験の評価、採用・却下の決定。

この分担を崩すと、AIへの丸投げになる。丸投げは必ず失敗する。

### ⑤ 検証（gstack）

実装したら必ず検証する。gstackはQAテストを実行し、品質を数値化するツールだ。「なんとなく動いている」では品質は上がらない。数値で見ることで、改善の優先度が明確になる。

OgoRouletteでは、Lighthouseスコア・UX観点・視覚的一貫性・アクセシビリティを毎回チェックした。

### ⑥ 再ISSUE化

検証で見つかった問題は、すぐにISSUEにする。「あとで直す」はほぼ実現しない。ISSUEにすれば次のループで確実に処理される。

## 2.3 ループを速く回すコツ

**1ループ = 1〜2個の改善に絞ること**。これが最も重要なルールだ。

「一気にたくさん改善しよう」とすると、何が原因で何が改善したかわからなくなる。デバッグが複雑になり、AIへの指示も曖昧になる。小さく回す方が、結果として速い。

OgoRouletteで学んだ粒度感:

- 良い粒度: 「スピン後のアニメーション演出を追加する」
- 大きすぎる粒度: 「UXを全体的に改善する」
- 小さすぎる粒度: 「ボタンの色を#FF6B6Bから#FF5A5Aに変える」

「大きすぎる」は分割する。「小さすぎる」は関連するものとまとめる。1ISSUEが1セッションで完結するくらいの粒度が理想だ。

# 第3章: ISSUE駆動開発

## 3.1 ISSUEの役割

ISSUEは単なるTODOリストではない。**思考の結晶**だ。

「ルーレットのスピン中にルームを離れると状態がおかしくなる」という問題があったとき、ISSUEを書く過程で「なぜおかしくなるか」を考え、「どう直すか」を設計し、「どうなれば完了か」を定義する。この過程でAIへの指示書が完成する。

ISSUEには以下の4つの機能がある。

1. **思考の整理**: 書くことで問題が明確になる
2. **AIへの文脈転送**: 書いたISSUEをそのままAIに渡せる
3. **開発ログ**: 「なぜこの実装にしたか」が残る
4. **品質の証拠**: 150 ISSUEを消化した事実が品質保証になる

## 3.2 タスクとISSUEの違い

| 観点 | タスク | ISSUE |
|---|---|---|
| 粒度 | 大きい・あいまい | 一つの問題・一つの改善 |
| 背景 | 書かない | 「なぜ今やるか」を必ず書く |
| 原因分析 | しない | 可能な限り原因を書く |
| 受け入れ条件 | 「完了したら消す」 | 「どうなれば完了か」を明記 |
| AIへの使いやすさ | そのままでは使えない | AIへの指示書として完成している |
| 残り方 | 消える | ログとして残る |

タスク管理ツールの「ログイン機能を作る」という1行は、AIへの指示書にはならない。ISSUEは、AIが迷わずに動けるだけの情報を含む。

## 3.3 良いISSUEの条件

良いISSUEには5つの条件がある。

**条件1: 背景が書かれている**
「なぜ今これを解決するのか」が書かれている。「ユーザーから報告があった」「gstackでスコアが低かった」「本番でしか発生しない」など。

**条件2: 問題が具体的だ**
「おかしい」「遅い」ではなく、「iOS Safari 17.2でルーレットのスピンアニメーションが途中で止まる」のように具体的だ。

**条件3: 原因の仮説がある**
原因がわからなくても「おそらく〜だと思う」という仮説を書く。AIはこの仮説を検証してくれる。

**条件4: 解決策の方向性がある**
「〜の方向で修正したい」「〜の実装を試したい」という方向性を書く。完璧な解決策でなくていい。

**条件5: 受け入れ条件が明確だ**
「〜の状態になればこのISSUEは完了」が書かれている。これがないと実装が終わらない。

## 3.4 ISSUE記述フォーマット

```markdown
## ISSUE-XXX: [タイトル]

### 背景
[なぜ今これを解決するのか。ユーザーへの影響・発見のきっかけ・優先度の根拠]

### 問題
[何が起きているか。できるだけ具体的に。再現手順があれば書く]

### 原因
[なぜそうなっているか。わからない場合は仮説を書く]
- 仮説1: [〜ではないか]
- 仮説2: [〜の可能性もある]

### 解決策
[どう直すか。方向性だけでも書く]

### 受け入れ条件
- [ ] [条件1: 〜の状態になっている]
- [ ] [条件2: 〜が正常に動作する]
- [ ] [条件3: 回帰テストを通過している]

### 参考
- 関連ISSUE: #XXX
- 関連ファイル: `src/components/XXX.tsx`
```

## 3.5 NG例と改善例

OgoRouletteの実際のISSUEをベースにした比較だ。

**NG例（雑なISSUE）**

```
## ISSUE: ルーレット直す

ルーレットがバグる。早めに対応。
```

これをAIに渡しても、AIは何をすればいいかわからない。「どのバグか」「どんな状況で起きるか」「何が正しい状態か」が何もない。

**改善例（良いISSUE）**

```markdown
## ISSUE-143: ルーム内ルーレット同期ズレを修正

### 背景
本番環境（Vercel）でルームを使ったとき、ホストがルーレットを止めた後に
他の参加者の画面でルーレットが止まるタイミングがズレる問題が発生した。
ローカルでは再現しない。

### 問題
- ホストが停止操作をすると、ホスト画面では即座に当選者が表示される
- 他の参加者の画面では2〜5秒遅れて停止する
- まれに停止しないまま永遠に回り続けるケースがある

### 原因（仮説）
- Supabase Realtimeのブロードキャストに遅延が発生している可能性
- phase管理（spinning → stopping → stopped）のタイミングで競合状態が起きている可能性
- waitingForWinnerフラグが適切にリセットされていない可能性

### 解決策の方向性
- phase遷移に明示的なウェイト処理を追加する
- waitingForWinnerフラグをリセットするタイミングを見直す
- Supabaseのbroadcastイベントのハンドリングをデバウンスする

### 受け入れ条件
- [ ] 3人以上のルームで、全員の画面が同時に停止する
- [ ] 停止後5秒以内に全員の当選者表示が一致する
- [ ] 本番環境で10回連続テストして問題が起きない
```

差は明白だ。改善例のISSUEはそのままAIへの指示書になる。

## 3.6 ISSUEの粒度管理

**思考の単位で切る**のが基本原則だ。

「一つの問題を解決する」「一つの機能を追加する」「一つの最適化を行う」が理想の粒度だ。

大きすぎるISSUEの分割方法:

```
悪い例（大きすぎる）:
ISSUE-50: モバイル対応を全体的に改善する

分割後（良い粒度）:
ISSUE-51: スマホでのルーレット表示サイズを最適化
ISSUE-52: iOS Safariでの音声再生バグを修正
ISSUE-53: タッチ操作でのスワイプ誤反応を防止
ISSUE-54: 縦向き・横向きの切り替え時のレイアウト崩れを修正
```

ISSUEを分割する基準は「1セッションで完了できるか」だ。複数セッションまたがるISSUEは、途中で文脈が失われてAIへの引き継ぎが難しくなる。

# 第4章: OgoRoulette開発ストーリー

## 4.1 Day 0: アイデア誕生

「飲み会で誰が奢るか決めるのって、毎回揉めるよな」

このひとことが始まりだった。じゃんけんは「前も負けた」という不満が残る。声の大きい人が決めると不公平感がある。スマホのランダム機能を使っても、「本当にランダム？」という疑念が消えない。

「全員がリアルタイムで同じ画面を見て、同じルーレットを回せたら公平じゃないか」

このアイデアが生まれた瞬間に、T-01（要件定義テンプレート）を開いて、MVPの範囲を固め始めた。**アイデアが生まれたら即テンプレート**が習慣だ。

MVPとして絞った機能は3つだけだった。
1. 参加者のリストを作る
2. ルーレットを回す
3. 当選者を表示する

リアルタイムルーム、割り勘計算、WinnerCard、LPはすべてMVP後の機能として後回しにした。この判断が開発を速くした。

## 4.2 Week 1-2: MVP構築

技術選定で1日迷った。ReactかNext.jsか、Firebase かSupabaseか。

判断基準はシンプルだった。「将来リアルタイム機能を追加したい」→ Supabase Realtimeが有力。「SEOを気にしたい」→ Next.js。「TypeScriptは必須」→ 型安全を最初から担保する。

CLAUDE.mdを最初に書いた。これはAIとのセッションをまたいでも、一貫した判断基準を保つための「開発憲法」だ。

```markdown
# OgoRoulette — 開発ルール（CLAUDE.md）

## 絶対に守ること
- 頼んだ箇所以外は変更しない
- 実装前に「何を・なぜ・どう実装するか」を一言確認する
- 理解できないコードは書かない

## 禁止事項
- router.back() 禁止（Link href を使う）
```

# 第5章: ケーススタディ

## 5.1 ISSUE-001: Race Condition × AI診断

### 背景

Week 3でリアルタイムルームを実装した直後、「2人が同時にスタートボタンを押すとルーレットが変な動きをする」という問題が発見された。ローカルのシングルユーザーテストでは再現しなかった。

### 問題

ホストと参加者が同時に操作可能な状態で、以下の症状が出た。
- ルーレットが2回分回って止まらなくなる
- phase が `spinning` から `idle` に直接飛ぶ
- 当選者が表示されない

### 原因

Supabase Realtimeのブロードキャストが複数発火した際に、phase遷移のロジックが2回実行されていた。「1回目のstop」と「2回目のstop」が競合し、状態が矛盾した。

### 解決策: ロックトークンパターン

```typescript
// スピン開始時にlockTokenを生成
const lockToken = crypto.randomUUID();
await broadcastSpinStart({ lockToken });

// 停止処理はlockTokenが一致する場合のみ受け付ける
const handleSpinStop = (event: BroadcastEvent) => {
  if (event.lockToken !== currentLockToken) return; // 無視
  processSpinStop(event);
};
```

AIが提案したパターンの1つが「ロックトークンによる冪等性担保」だった。このパターンを採用して、race conditionを完全に解消した。

### AI活用のポイント

「Race conditionが起きている」という仮説をISSUEに書いて渡したところ、AIは「楽観的ロック」「悲観的ロック」「ロックトークン」の3パターンを提案した。それぞれのトレードオフを説明してもらい、リアルタイムアプリに最適なロックトークンを選択した。

### 学び

分散システムのrace conditionはローカルでは再現しない。「本番でしか起きないバグ」は、同期の問題を疑え。ロックトークンパターンを最初から設計に組み込んでおくべきだった。

## 5.3 ISSUE-143: ルーム同期ズレ修正

### 背景と問題

本番環境（Vercel）でのみ発生する同期ズレ。ホストが停止操作をすると、他の参加者の画面で停止が2〜5秒遅れる。最悪のケースでは停止しないまま永遠に回り続ける。

### 原因特定のプロセス

ローカルでは再現しないため、本番ログを仕込んで調査した。

```typescript
// デバッグ用ログ追加
console.log(`[SYNC] phase: ${phase}, lockToken: ${lockToken}, timestamp: ${Date.now()}`);
```

ログを分析すると、`stopping` フェーズのbroadcastイベントが参加者側で2回受信されていた。1回目でphaseを `stopped` にしようとするが、2回目のイベントが `stopping` に戻してしまう。

### 解決策

`waitingForWinner` フラグを導入して、停止処理の冪等性を担保した。

```typescript
// 停止処理中フラグ
const [waitingForWinner, setWaitingForWinner] = useState(false);

const handleStopEvent = (event: BroadcastEvent) => {
  if (waitingForWinner) return; // 処理中は無視
  setWaitingForWinner(true);
  
  // 停止処理...
  
  // 完了後にフラグをリセット
  setTimeout(() => setWaitingForWinner(false), 1000);
};
```

### タイミング問題の教訓

本番環境でのみ発生するバグは、ネットワーク遅延を仮定してコードを書く必要がある。「ローカルで動く」は「本番で動く」ではない。Supabase Realtimeのbroadcastは最低1回だが、最大複数回配信される可能性があることをコードに織り込む。

## 5.4 パフォーマンス最適化

### Lighthouse 62→95の改善ストーリー

| 指標 | Before | After |
|---|---|---|
| LCP (Largest Contentful Paint) | 4.2s | 1.1s |

**ボイラープレート生成**: フォームのバリデーション・APIルートの骨格・型定義のドラフト。これらを手書きしている時間は純粋な損失だ。

**テストの初期実装**: 「このコンポーネントのテストを書いて」でJestのテストが即座に生成される。カバレッジ100%近くのドラフトが5分で完成する。

**ドキュメント生成**: コードのコメント・JSDoc・READMEの初稿。AIは「コードを読んでドキュメントを書く」のが極めて得意だ。

**コンテンツ生成**: LPコピー・ブログ記事・SNS投稿文。「感情設計」と「事実」を渡せば、90%以上のコンテンツをAIが書ける。

**候補の列挙**: 「技術選定の候補を5つ出して」「このバグの原因仮説を3つ出して」。候補出しはAIが速い。選択は人間がやる。

## 8.2 人間が握る領域

以下は絶対に人間が主導する。AIに丸投げすると必ず後悔する。

**アーキテクチャの最終決定**: 「どの設計パターンを採用するか」はAIが提案しても、決定は人間がする。アーキテクチャの失敗は後工程で何倍も痛い。

**状態管理の設計**: 複雑な状態管理（特にリアルタイムアプリのphase管理）はAIが得意ではない。AIは局所的に正しいコードを書くが、全体の整合性を保つのは人間の責任だ。

**ユーザー体験の評価**: 「これは使いやすいか」「この演出は盛り上がるか」はAIには判断できない。人間が使って感じることが唯一の正解だ。

**ビジネス判断**: 「この機能をMVPに入れるか」「この問題の優先度は何か」は、ビジネスコンテキストを理解している人間がやる。

**セキュリティの最終確認**: AIはセキュリティホールを見落とすことがある。認証・認可・入力検証は人間が必ずレビューする。

## 8.4 AIとの対話スタイル

**指示の出し方**:

悪い: 「ルーレットを改善して」
良い: 「スピン停止後の当選者発表アニメーションに祝祭感を追加したい。Framer Motionを使って、停止時のバウンス + confetti + 当選者名の大きな表示を実装してほしい。既存の Roulette.tsx には触れない」

違いは「何を」「なぜ」「どう」「制約」の4要素が揃っているかどうかだ。

**フィードバックの与え方**:

AIの提案が気に入らなかった場合、「違う」と言うだけでは改善しない。「このアプローチはパフォーマンスに問題があるので別の方法を提案してほしい。具体的には〜の点が気になる」と理由と代替の方向性を伝える。

**確認のタイミング**:

実装が複雑なとき、「まず実装の方針を説明してから、コードを書いて」と先に確認を求める。コードが出てきてから「方向性が違う」と言うのは双方の損失だ。

```
良い確認プロセス:
① 「この問題の解決策として〇〇を考えているが、どう思うか?」
② AIが評価と代替案を出す
③ 「では〇〇の方針で実装して」
④ 実装が出てくる
⑤ 「ここは〜に変更して」
⑥ 完成
```

# 第9章: プロの実践原則

AI駆動開発を続ける中で体得した実践原則がある。詳細は [`principles.md`](principles.md) にまとめているが、ここでは代表的な5つを紹介する。

### 原則1: 「わからない」を「ISSUEになる前に消す」

技術的にわからないことがあったら、「後で調べよう」ではなく「今AIに聞く」。30秒で解決できることを1週間放置するのはプロではない。ただし、答えをコピペするのではなく、「なぜそうなるか」を理解してから進む。

### 原則2: 「動いた」で終わらない

「動いた」はゴールではない。「品質基準を満たして動いた」がゴールだ。動いたら必ずgstack評価を回す。数値で確認して、次のISSUEを作って初めて「このISSUEは完了」だ。

### 原則3: 「AIが書いたコードを自分のコードにする」

AIが生成したコードをコピペしただけでは、そのコードを「理解したコード」にならない。自分でコメントを書く・変数名を変える・テストを書く、のどれかをやると理解が深まる。将来バグが起きたとき、理解していないコードは直せない。

### 原則4: 「文脈を渡せない指示は指示ではない」

AIに「前に話した〇〇の続き」は通じない。セッションをまたぐと文脈がリセットされる。毎回のプロンプトに必要な文脈を含める習慣をつける。CLAUDE.mdはこの問題を解決するためのツールだ。

### 原則5: 「人間の判断をAIに委ねない」

「AIはこう言っているから正しい」という思考停止が最も危険だ。AIは間違える。AIは古い情報を持っている。AIはビジネスコンテキストを知らない。AIの提案は「叩き台」であり、「答え」ではない。最終判断は常に人間がする。

# 第10章: よくある失敗

## 10.1 ISSUEが雑

**症状**: 「〇〇を直す」という1行のISSUEが多い。バグが直らない・AIへの指示が通じない・実装が何を目指しているかわからない。

**原因**: 「ISSUEを書く」という行為の価値を理解していない。タスク管理ツールに慣れすぎて、1行で済ませる習慣がついている。

**処方箋**: 第3章のISSUEフォーマットを使う。特に「受け入れ条件」を必ず書く。受け入れ条件がないISSUEは永遠に完了しない。

---

## 10.2 AIに丸投げ

**症状**: AIが出したコードをそのままコミットする。しばらくすると動かないコードが増えて、デバッグが追いつかなくなる。

**原因**: 「AIが速く書いてくれる」の恩恵にフォーカスして、「AIが間違える」というリスクを忘れている。

**処方箋**: AIのコードは必ずレビューする。特に「状態管理」「認証・認可」「非同期処理」の3領域は入念に確認する。自分が理解できないコードはマージしない。

---

## 10.3 検証不足

**症状**: 「実装した→完了」で次に進む。気づいたらバグが積み上がっている。品質が上がっている実感がない。

**原因**: 検証を「別の作業」として捉えている。実装と検証はセットだという認識がない。

**処方箋**: 「ISSUEを閉じる前にgstack評価を必ず走らせる」をルールにする。数値が改善していない場合はISSUEを閉じない。これを徹底するだけで品質が上がる。

---

## 10.4 状態管理の放棄

**症状**: Propsのバケツリレーが深くなる・useEffectが乱立する・どこで状態が変わっているかわからなくなる。

**原因**: 「とりあえず動けばいい」で進め、状態管理の設計を後回しにし続けた結果。AIが局所的に正しいコードを書いても、全体の整合性が崩れる。

**処方箋**: 月1回、状態管理のリファクタISSUEを作る。「この状態はどこで管理するべきか」を設計から見直す。T-04（リファクタテンプレート）を使ってAIと一緒に整理する。

---

## 10.5 MVPを絞れない

**症状**: 作り始めると追加したい機能が次々出てきて、いつまでもリリースできない。完成のゴールが見えない。

**原因**: 「全部必要だ」という思い込み。「これはMVP後でいい」という判断ができていない。

**処方箋**: T-01（要件定義テンプレート）を開発開始前に必ず使う。「なければアプリが成立しないか？」という問いで機能を篩にかける。答えがNoなら全部MVP後だ。OgoRouletteはMVPの機能を3つに絞ったから2週間でリリースできた。

# 第11章: 成功パターン

## 11.1 小さく回す

1ループ1改善の徹底が、最も重要な成功パターンだ。

```
良いループ（小さい）:
ISSUEを1個書く → AIに渡す → 実装する → 検証する → 次のISSUEへ

悪いループ（大きすぎる）:
10個のISSUEをまとめて実装しようとする
→ AIへの指示が複雑になる
→ 何が原因でバグるかわからなくなる
→ 検証が複雑になる
→ 戻し方がわからなくなる
```

小さく回すことで、「改善した事実」が積み上がる。毎日1〜3個のISSUEを消化する習慣が、4ヶ月で150 ISSUEになった。

## 11.2 検証を必ず入れる

「実装 → 検証」をセットにすること。検証なしの実装は「やった気分」だけで品質は上がらない。

gstackによる数値化が有効な理由は「感覚ではなく数値で議論できる」からだ。「なんかUXが悪い」ではなく「happiness scoreが45点」という具体的な指標があると、改善の優先度が決めやすい。

検証のタイミング:
- 各ISSUEの実装完了後
- 大きな機能追加後
- リリース前
- 月次の品質確認

## 11.3 感情を設計に入れる

「機能が動く」と「使いたくなる」の間には大きな差がある。この差を埋めるのが感情設計だ。

**感情設計のプロセス**:

1. 「ユーザーがアプリを使い始めてから終わるまでの感情曲線」を書く
2. 「最も盛り上がってほしい瞬間」を特定する
3. その瞬間の演出を設計する
4. AIと実装する

OgoRouletteでの感情設計:
- ルーレットが止まる直前: ドキドキ（減速アニメーションで表現）
- 当選者発表の瞬間: 驚き + 爆笑（confetti + 大きな表示で表現）
- 奢り決定後: 共有したくなる（WinnerCard + シェアボタンで表現）

## 11.4 ISSUEをストックする

「思いついたらすぐISSUE化する」習慣が、長期的な品質向上の源泉だ。

アプリを使っていて「ここが気になる」と思った瞬間に、その場でISSUEを書く。シャワーを浴びていて「あの問題はこうすれば直るかも」と思ったら、すぐメモしてISSUEにする。

ISSUEはストックするほど価値が上がる。「次に何を改善すべきか」に迷う時間がなくなる。ISSUEリストを見れば、常に「次の1手」が明確だ。

OgoRouletteでは、常に20〜30個のISSUEがBacklogにある状態を維持していた。これが「手が止まる」という状況を完全になくした。

# 第12章: バズるアプリ設計

## 12.1 体験設計

バズるアプリは「最高の瞬間」が設計されている。ユーザーが「わー！」と思う瞬間を、意図的に設計する。

```
体験設計の手順:

① アプリを使う「一連のストーリー」を書く
   例: グループで集まる → ルーレットを起動 → 参加者が入る → 回す → 決まる → シェア

② ストーリーの中で「最高の瞬間」を特定する
   例: 「ルーレットが止まって当選者が発表される瞬間」

③ その瞬間を最大化する演出を設計する
   例: 減速 → 停止音 → confetti → 大きな名前表示 → ファンファーレ

④ 「シェアしたくなる瞬間」を作る
   例: WinnerCard のスクショ → OG画像生成 → シェアボタン
```

## 12.2 感情設計

ドキドキ・驚き・笑いを仕込む設計だ。

**ドキドキ**: 「どうなるかわからない」不確実性を演出する。OgoRouletteでは、ルーレットが止まる直前に減速を段階的にすることで「もしかして自分か...!?」という緊張感を作った。

**驚き**: 「まさかこうなるとは」という意外性を設計する。WinnerCardの紙吹雪は「アプリがこんな反応をするとは思わなかった」という驚きを生む。

**笑い**: グループで使うアプリは「笑い」が最強の口コミエンジンだ。OgoRouletteでは当選者のリアクションがSNSに投稿されると、それ自体が宣伝になる。

## 12.3 SNS設計

「シェアしたくなる瞬間」を設計してからアプリを作る。後からSNS機能を追加しても遅い。

OgoRouletteのSNS設計:

1. **WinnerCard**: 当選者名が大きく表示されたカード。スクリーンショットがそのままSNS映えする
2. **動的OG画像**: Next.jsの `ImageResponse` で当選者名入りのOG画像を生成。URLをSNSにシェアすると自動で表示される
3. **ハッシュタグ**: `#OgoRoulette #奢り決め` を投稿文に自動で含める
4. **ルームURL共有**: 「このURLから入れば一緒にできるよ」と友達を招待できる

**プロダクトがマーケティングになる**設計にすることで、広告費ゼロで拡散できる。

## 12.4 OgoRouletteの体験設計図

```
[ユーザーがURLを開く]
    ↓ (LP: 「飲み会の奢り、これで決めよう」)
[使い方がわかる]
    ↓ (3ステップで説明。30秒で理解できる)
[ルームを作る / 参加する]
    ↓ (ルームコード4文字。タイポしない長さ)
[参加者が集まる]
    ↓ (リアルタイムで参加者リストが更新される。「おー来た来た」感)
[ルーレットを回す]
    ↓ (回転音 + 速いスピン。目が離せない)
[徐々に減速]
    ↓ (「誰だ...誰だ...」という緊張感)
★ 当選者発表 ★  ← 最高の瞬間
    ↓ (confetti + 大きな名前 + ファンファーレ)
[WinnerCardが表示される]
    ↓ (「シェアする」ボタンが自然に目に入る)
[SNSにシェア]
    ↓ (OG画像付きでURLがシェアされる)
[友達がURLをタップ]
    ↓ (次のユーザーがLPに来る)
[ループ]
```

この設計図を最初に書いてから開発した。「どこが最高の瞬間か」を知っているから、その瞬間に全力を投資できた。

# 第13章: まとめ

## 13.1 AI駆動開発の本質

**1行で言うなら**: 「思考を言語化して、AIと協働して、人間が判断する」

**3行で言うなら**:
1. AIは「速く書けるツール」ではなく「自分が言語化できていないことを実装に落としてくれるパートナー」だ
2. ボトルネックはコーディング速度ではなく「思考の言語化速度」だ
3. アーキテクチャ・状態管理・UX・ビジネス判断は常に人間が握る

この3行を体に入れてからAIを使うと、使い方が根本から変わる。

## 13.2 再現方法

OgoRouletteと同じことを再現するための5ステップだ。

**ステップ1: CLAUDE.mdを書く**
プロジェクトの「開発憲法」を最初に作る。技術スタック・禁止事項・命名規則・AIへの指示ルールを書く。このファイルがAIとのセッションをまたいでも一貫性を保つ。

**ステップ2: T-01でMVPを絞る**
`templates/t01-requirements.md` を使って、MVPの範囲を3機能以内に絞る。「なければアプリが成立しないか？」という問いで篩にかける。

**ステップ3: 最初のISSUEを書く**
MVP最初の機能のISSUEを書く。第3章のフォーマットを使って、背景・問題・解決策・受け入れ条件を書く。

**ステップ4: ループを1周回す**
ISSUEを書く → テンプレートにはめてAIに渡す → 実装する → gstack評価する → 次のISSUEへ。

**ステップ5: ループを続ける**
1日1〜3個のISSUEを消化する習慣をつける。4ヶ月続ければ100 ISSUEを超える。100 ISSUEは「100回の改善」の証拠だ。

## 13.3 今日からのアクション

明日から始められる具体的な3つのことだ。

**アクション1: 手元のプロジェクトにCLAUDE.mdを作る**（30分）
CLAUDE.mdがなければ今すぐ作る。技術スタック・禁止事項・AIへの基本指示を書く。これだけでAIとのセッションの品質が上がる。

**アクション2: 気になっていることをISSUEにする**（15分）
「なんか気になっていること」を1個、第3章のフォーマットでISSUEに書く。受け入れ条件を必ず書く。「書けた」と思ったらAIに渡してフィードバックをもらう。

**アクション3: T-05でアプリを評価する**（30分）
今作っているアプリに `templates/t05-gstack-review.md` を使って評価を走らせる。数値が出たら改善ISSUEを1個作る。これが品質向上ループの第一歩だ。

## 13.4 最後に

OgoRouletteは「飲み会の奢り決め」という小さなアイデアから始まった。技術的な新しさは何もない。ReactもSupabaseもFramer Motionも、世の中にある技術の組み合わせだ。

でも4ヶ月間、毎日ISSUEを書いてAIと協働し続けた結果、使いたいと思えるアプリができた。150 ISSUEは150回の「よりよくなった」の積み重ねだ。

AI駆動開発は「AIがいれば誰でも良いものが作れる」ではない。「AIと適切に協働できる人が、圧倒的なスピードで高品質なものを作れる」だ。

適切な協働には「思考の言語化」「文脈の転送」「人間の判断」が必要で、これらはスキルだ。スキルは練習で身につく。この教科書は、そのスキルを身につけるための地図だ。

地図を持って、歩き始めてほしい。
