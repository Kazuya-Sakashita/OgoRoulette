# ブログ記事生成テンプレート (t07)

## 使うタイミング
開発を通じて得た知見・体験・気づきを記事にしたいとき。「AIに全部書かせる」のではなく、**自分の実際の体験と数値をAIに渡し、文章化してもらう**使い方をする。体験のない記事はAIが書いても薄くなる。

## 重要度
`★★★☆☆ MEDIUM` — アウトプットの習慣がエンジニアとしての信頼を積み上げる。AIを使えば書く時間を大幅に短縮できる。

## テンプレート

```
以下の情報をもとに、技術ブログ記事を執筆してください。

---

## 記事タイプ（1つ選ぶ）

- [ ] 技術解説 — 特定の技術・ツール・概念を説明する記事
- [ ] 体験記 — 開発・学習・プロジェクトを通じた経験談
- [ ] 比較考察 — AとBを比較して、自分なりの結論を示す記事

選んだタイプ: [上から1つ]

## 核心メッセージ（1文で）
[この記事を読んだ人に最も伝えたいこと。例: AIに曖昧な依頼をすると既存コードが壊れる。テンプレートで「依頼の型」を作ることが品質を守る鍵だった]

## 実際に起きた出来事（体験の素材）
[自分が実際に経験したこと。箇条書きでOK]
- [例: ルーレットのスピン後に結果が出ないバグが発生]
- [例: エラーログを貼らずにAIに聞いたら的外れな修正案が3回来た]
- [例: バグ調査テンプレートを使ったら10分で原因が特定できた]

## コード・数値・スクリーンショット
[記事に含めるべき具体的な数値やコード。なければ「なし」]
- [例: Lighthouse スコア: Before 72 → After 96]
- [例: 修正前後のコード差分]
- [例: バンドルサイズ: 280KB → 140KB（50%削減）]

## 対象読者
[例: Next.js をある程度触ったことがある個人開発者。AI開発ツールに興味はあるが使い方に迷っている人]

## 文字数の目安
[例: 2000〜3000字]

## 掲載先
[例: Zenn / Qiita / 個人ブログ]

---

## 執筆スタイルの指定

- 一人称は「自分」を使う（「私」「僕」「俺」は使わない）
- 段落は短く（3〜5行を目安）。長い段落は分割する
- **体験先行** — まず何が起きたかを書き、その後に解説や考察を続ける
- 見出しは記事全体のストーリーとして独立して読めるように書く
  - NG: 「バグの修正」
  - OK: 「エラーログなしでAIに聞くと3回ハズレを引く」
- 「〜と思います」「〜かもしれません」などの過度な断定回避は使わない
- 技術用語は初出時のみ説明し、以降はそのまま使う

---

## 構成案（変更可）
1. 導入（hook） — 読者が「あるある」と感じる失敗や疑問から始める
2. 背景 — 何を作っていたか・なぜその問題に直面したか
3. 問題の詳細 — 何が起きたか（コード・数値があれば含める）
4. 解決策 — どう対処したか（手順・テンプレートがあれば示す）
5. 結果 — 数値・スコア・体験として何が変わったか
6. まとめ・学び — 読者への持ち帰り可能な教訓1〜3点
```

## 使用例（OgoRouletteでの実績）

ISSUE-147〜150の対応を終えたあと、gstack品質レビューの体験を記事化するために使用。「体験の素材」に実際のISSUE番号と修正前後のLighthouseスコアを入れたことで、AIが具体的な数値入りの記事を生成した。「核心メッセージ」を1文で先に書いたことで、記事全体のトーンがぶれなかった。

## ポイント
- 「核心メッセージ」は最初に1文で書く。これがないとAIが「何について書く記事か」を迷い、薄い総論になる
- 「実際に起きた出来事」には失敗も必ず入れる。失敗談のある記事は読まれやすく、信頼を得やすい
- 数値（スコア・サイズ・時間）は必ず含める。「速くなった」より「50%削減」の方が読者に伝わる
- 見出しのNGパターン（「バグの修正」）とOKパターンを覚えておく。見出しは記事の顔
- 生成後は必ず自分でひと通り読み、体験していない内容が混入していないかチェックする
