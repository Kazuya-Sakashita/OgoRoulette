# リファクタリングテンプレート (t04)

## 使うタイミング
機能は動いているが、コードの質を改善したいとき。**目的を1つに絞る**のが最重要。「全部良くして」という依頼はAIが全コードを書き直す暴走を引き起こす。

## 重要度
`★★★★☆ HIGH` — 目的を複数にするとAIが範囲を拡大解釈し、動いていた機能が壊れるリスクが急増する。

## テンプレート

```
以下のコードをリファクタリングしてください。

---

## リファクタリングの目的（1つだけ選んでチェック）

- [ ] 可読性向上 — 命名・コメント・構造を整理して理解しやすくする
- [ ] パフォーマンス改善 — 不要な再レンダリング・計算の削減
- [ ] 型安全性向上 — any/unknown を排除し、型定義を厳密化する
- [ ] テスタビリティ向上 — 副作用を分離し、ユニットテストを書けるようにする
- [ ] 重複排除 — コピペコードを共通化する

選んだ目的: [上のリストから1つ]

## 対象コード
```tsx
// ファイルパス: [例: src/components/RouletteWheel.tsx]
// 対象範囲: [例: ファイル全体 / 50〜80行目の useEffect]
[コードをここにペースト]
```

## 制約（必ず守ること）
- 外部から見た動作は変えない（既存のPropsインターフェースを維持する）
- 使用ライブラリは追加・削除しない（既存の依存関係を保つ）
- テストがある場合、テストをパスする状態を維持する
- スタイル（Tailwind クラス名）は変更しない

## 変更してよい範囲
[例: 上記ファイルの内部実装のみ。他ファイルへの変更は禁止]

---

【重要な禁止事項】
- 依頼していない箇所を変更することは禁止
- 動作に影響する変更が避けられない場合は、実装前に必ず警告してから確認を取ること
- 「ついでに」「合わせて」という形でのスコープ拡大は禁止
```

## 使用例（OgoRouletteでの実績）

`RouletteWheel` コンポーネントの動的 import 対応（ISSUE-150）で使用。目的を「パフォーマンス改善（バンドルサイズ削減）」に絞り、「動的importへの切り替えのみ」と範囲を明示した。その結果、コンポーネントの描画ロジックには一切手が入らず、意図した変更だけが適用された。

## ポイント
- チェックボックスのうち**必ず1つだけ**にチェックを入れる。2つ以上選ぶと目的が曖昧になる
- 「制約」セクションは削らない。これがないとAIが「改善」の名目で意図しない変更を入れる
- リファクタ後は `git diff` で変更差分を必ず確認する
- 動作確認（手動テストまたは自動テスト）をリファクタ前後で必ず行う
- 大きなファイルは一度に全体を対象にせず、関数・フック単位に分割して依頼する
