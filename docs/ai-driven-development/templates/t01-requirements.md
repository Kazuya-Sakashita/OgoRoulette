# 要件定義テンプレート (t01)

## 使うタイミング
アイデアはあるが「何を作るか」がまだぼんやりしているとき。開発を始める前にAIと一緒にMVPの範囲を固めたいときに使う。

## 重要度
`★★★★★ CRITICAL` — このテンプレートをスキップすると、作りながら仕様が揺れ続けて手戻りが多発する。

## テンプレート

```
あなたはプロダクトマネージャー兼シニアエンジニアです。
以下の情報をもとに、MVP（最小限の価値あるプロダクト）の要件を定義してください。

---

## アプリ概要
[アプリの一行説明。例: グループで使えるルーレットWebアプリ]

## 解決する課題
[誰のどんな痛みを解決するか。例: 飲み会や会議で「誰が担当するか」を決める際、じゃんけんや声の大きい人に偏りがちで不公平感がある]

## 想定ユーザー
- メインターゲット: [例: 社会人のグループ（3〜10人）]
- サブターゲット: [例: 学校のクラス担任、ゲームグループ]
- 技術レベル: [例: スマホは使えるがITリテラシーは普通]

## 必須機能（思いついたもの全部）
- [機能1]
- [機能2]
- [機能3]
- ...

## 技術スタック（決まっていれば）
- フロントエンド: [例: Next.js 15 + TypeScript + Tailwind CSS]
- バックエンド: [例: Vercel（サーバーレス）]
- DB: [例: 不要（ローカルstate管理）]
- デプロイ: [例: Vercel]

---

上記をもとに、以下の形式で出力してください。

### 1. 優先度付きコア機能リスト
| 優先度 | 機能名 | 説明 | 実装難易度 |
|--------|--------|------|-----------|
| P0     |        |      |           |
| P1     |        |      |           |
| P2     |        |      |           |

### 2. MVP定義
MVP（初回リリース）に含める機能のみを箇条書きで。

### 3. 除外機能（今回やらないこと）
理由とともに列挙する。

### 4. リスク・懸念点
技術的・UX的リスクを3点以上。

### 5. フェーズ別ロードマップ
- Phase 1 (MVP):
- Phase 2:
- Phase 3:
```

## 使用例（OgoRouletteでの実績）

OgoRoulette開発初期に「ルーレットアプリを作りたい」という一言から出発し、このテンプレートを使って要件を整理した。その結果、「名前入力」「スピン」「結果表示」の3機能をMVPと定義し、「履歴保存」「チーム保存」はPhase 2に後回しにできた。最初の実装を2日で完了できたのはMVP範囲を絞り込んだから。

## ポイント
- 「必須機能」は遠慮なく全部書く。AIが優先度をつけてくれるので絞る必要はない
- 技術スタックが未決定でも問題なし。「提案してほしい」と書けばAIが推薦する
- P0（なければアプリとして成立しない）とP1（あると便利）の区別を必ずAIに判断させる
- 出てきたリスクは次の設計フェーズのインプットとして保存しておく
- このテンプレートで出力した内容を `docs/plan/` 以下に保存すると後から参照しやすい
