## アプリ概要
グループで「誰が奢るか」をルーレットで決めるWebアプリ（OgoRoulette）

## 解決する課題
飲み会や会議で「誰が担当するか」を決める際、じゃんけんや声の大きい人に偏りがちで不公平感がある。
また、奢りになった人の支払い金額や割り勘額を即座に知りたいが、都度計算するのが面倒。

## 想定ユーザー
- メインターゲット: 社会人グループ（3〜10人）の飲み会・ランチシーン
- サブターゲット: オンラインゲームグループ、学校のクラス
- 技術レベル: スマホは使えるがITリテラシーは普通

## 思いついた機能（全部）
- 参加者名を入力してルーレットを回す
- 当選者をアニメーション付きで表示
- 割り勘・奢り金額の計算
- ルーム機能（招待コードでメンバーが参加）
- QRコードによる招待
- リアルタイム同期（全員の画面が同時に動く）
- ログイン・会員登録
- 履歴機能
- SNSシェア
- PWAプッシュ通知
- グループ機能（メンバープリセット保存）

## 技術スタック（仮）
- フロントエンド: Next.js 16 + TypeScript + Tailwind CSS
- 認証: Supabase Auth
- DB: PostgreSQL (Supabase)
- デプロイ: Vercel

## プロジェクト情報

### 概要
- アプリ名: OgoRoulette
- 説明: グループで「誰が奢るか」をルーレットで決めるWebアプリ
- URL: https://ogo-roulette.vercel.app/

### 技術スタック
- フレームワーク: Next.js 16 (App Router)
- 言語: TypeScript 5.7（strict mode）
- スタイリング: Tailwind CSS v4
- UIコンポーネント: shadcn/ui + Radix UI
- アニメーション: Framer Motion
- 認証: Supabase Auth（Google OAuth、X OAuth）
- DB: PostgreSQL（Supabase）+ Prisma 6
- バリデーション: Zod
- テスト: Vitest
- デプロイ: Vercel
- パッケージマネージャー: pnpm

### コーディング規則（決めていること）
- コンポーネントファイル名はケバブケース（例: roulette-wheel.tsx）
- 型はできるだけ `interface` より `type` を使う
- APIルートは `app/api/` 以下に Route Handler で実装
- クライアントコンポーネントには `'use client'` を必ずファイル先頭に記載
- Supabase クライアントは `lib/supabase/` 以下の関数を通して使う

### やってはいけないこと
- `any` 型の使用
- `console.log` を本番コードに残す（`console.error` は許可）
- DB直接アクセス（必ずAPIルート経由）
- 環境変数のハードコード

### ディレクトリ構成
- `app/` — Next.js App Router のページ・APIルート
- `components/` — 再利用可能なReactコンポーネント
- `lib/` — ユーティリティ・Supabaseクライアント・Prismaクライアント
- `hooks/` — カスタムReact Hooks
- `prisma/` — Prismaスキーマ・マイグレーション
