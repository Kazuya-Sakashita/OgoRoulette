### SLIDE 006 — 定義（1行）

AI駆動開発とは

**本文（大きく1行）**

### SLIDE 008 — AIは「速く書けるツール」ではない

**スライドタイトル**
AIの本当の価値は「速さ」ではない

**本文**
よくある誤解:
「AIを使えばコードを速く書ける」

正しい理解:
「AIは言語化できていないことを実装に落とすパートナー」

**具体例（OgoRouletteより）**
- 私がやりたかったこと: 「ルーレットが止まった瞬間に盛り上がる演出を入れたい」
- AIへの入力: この感情的要求 + 現在のコード
- AIの出力: `canvas-confetti` ライブラリの実装案 + WinnerCardの強化案
- 私が判断: 「Phase 1から始めて段階的に実装する」

**話者ノート**
「速さ」を否定することで、観客の認識を更新する。多くの人が「AIを使えば早く作れる」と思っているが、それは副次効果。本質は「自分が言語化できていない要求を実装に変換するプロセスが短縮される」こと。OgoRoulette ISSUE-060（confetti実装）は、この価値の好例として使う。

### SLIDE 009 — 人間とAIの役割分担

**スライドタイトル**
人間とAIの役割分担

**本文（表）**

| 領域 | AIに任せる | 人間が握る |
|------|-----------|-----------|
| コード生成 | ほぼすべて | アーキテクチャ判断 |
| バグ診断 | 原因分析・仮説生成 | 再現確認・根本原因の確定 |
| UX設計 | 改善案のリストアップ | ユーザーの感情への共感 |
| 技術選定 | 選択肢の提示 | トレードオフの最終判断 |
| テスト | ケースの列挙 | 受け入れ条件の定義 |
| 命名・コピー | 案の生成 | ブランドトーンへの合致判断 |

**絶対に人間が手放してはいけないもの**
「このアプリで誰を幸せにしたいか」という問い

**話者ノート**
表を使って「AIに任せられること」と「人間が持つべきこと」を明確に分ける。これにより「AIに任せすぎて失敗した経験」がある観客が「そこが問題だったんだ」と気づく。最後の一行「誰を幸せにしたいか」はプロダクト開発の本質なので、力を込めて言う。

### SLIDE 010 — ISSUEとは何か

**スライドタイトル**
ISSUEとは「思考の結晶」

**本文**

タスク（NG）:
- 「スピンボタンのバグを直す」
- 「UXを改善する」
- 「パフォーマンスを上げる」

ISSUE（OK）:
- 背景: いつ・どこで・誰に起きるか
- 問題: 何が起きているか（症状）
- 原因: なぜ起きているか（根本）
- 解決策: どう直すか（具体的な実装方針）
- 受け入れ条件: 何ができたら完了か

**なぜISSUEが重要か**
AIはあなたが書いた分だけ賢くなる。曖昧な入力 → 曖昧な出力。構造化された入力 → 的確な出力。

**話者ノート**
「タスクとISSUEの違い」は今日一番伝えたいことの一つ。タスクは「何をするか」、ISSUEは「なぜするか・どうすれば完了か」まで含む。これがあるとAIへの入力品質が劇的に上がる。「ISSUEを書くのは時間がかかる」という反論には、「書かないと実装時間が3倍かかる」と返す。

### SLIDE 011 — 良いISSUEの構造

**スライドタイトル**
良いISSUEのテンプレート

**本文**

```markdown
# [ISSUE-XXX] タイトル（何が・どうなる）

## 概要（1〜3行）
[症状の要約]

## 背景 / なぜ問題か
[再現条件・影響ユーザー・なぜ放置できないか]

## 目的
[このISSUEを解決したら何が実現するか]

## 影響範囲
[対象ファイル・対象コンポーネント・対象ユーザー]

## 修正方針
[具体的な実装アプローチ・コード例]

## リスク / 副作用
[意図しない影響が出る可能性]

## 受け入れ条件
- [ ] 条件1
- [ ] 条件2
```

**話者ノート**
テンプレートを丸ごと見せる。「これをコピーして使ってください」と言う。ポイントは「受け入れ条件」。ここが明確でないとAIもレビューも「完了」の判断ができない。OgoRouletteのISSUEはすべてこの形式で書かれており、150+件が蓄積している。

### SLIDE 012 — ISSUE-001の実例

**スライドタイトル**
実例: ISSUE-001 — SPINボタン永続無効化バグ

**本文**

**症状**: ゲストホストがページをリロードすると、SPINボタンが押せなくなる

**原因（AIが診断）**:
```
isGuestHost が localStorage から非同期で読まれるため、
最初のレンダリングで isOwner === false になる。
この隙間にメンバー用エフェクトが誤発火し、
phase = "preparing" にセットされる。
その後 isOwner = true になっても、phase が "waiting" でないため
SPINボタンは disabled のまま。
```

**解決（AIが提案 → 人間が判断）**:
`isGuestHostResolved` フラグを追加し、loading guardに含める

**受け入れ条件**: ゲストホストがリロード後、SPINボタンが有効な状態で表示される

**話者ノート**
このスライドは実際のISSUEを見せることで「こういうバグをAIが診断できる」という実感を作る。原因の文章はISSUEドキュメントそのものから引用している。「これを私がゼロから考えていたら何時間かかったか」と問いかけると、AIの価値が実感できる。

### SLIDE 015 — 実際のループ: Race Conditionとの戦い

**スライドタイトル**
実際のループ: ISSUE-001〜006 Race Conditionとの戦い

**本文**

```
ISSUE-001: isOwner フリッカー → SPINボタン永続無効化
  ↓ 解決したら新たな問題が出た
ISSUE-002: spinScheduledRef 競合 → phase "preparing" で停止
  ↓ 解決したら次の問題
ISSUE-003: phase タイムアウト機構がない → 自動回復不可
  ↓
ISSUE-004: clock skew → scheduleSpin の delay が何分にもなる
  ↓
ISSUE-005: ルーム IN_SESSION で stuck → 次のスピンができない
  ↓
ISSUE-006: handleRespin のポーリング競合
```

**AIの活用方法**
各ISSUEをAIに投げる形式:
1. ISSUEのフルテキストを貼る
2. 関連コード（該当行周辺）を貼る
3. エラーログ・再現手順を貼る
→ AIが「根本原因」と「修正案」を提示

**話者ノート**
このスライドは「ISSUEが連鎖する」という現実を見せる。一つ直したら次が出る。これは失敗ではなく、システムへの理解が深まっている証拠。重要なのは「次のISSUEを恐れない」こと。ISSUEが出ることは「発見できた」ということ。ISSUE-001からISSUEを作るたびにアプリの品質が上がっていった実感を話す。

| バグ修正 | 約40件 | ISSUE-001〜020, 141〜143 |

**ISSUEが150件あることの意味**
- 150回、問題を発見して言語化した
- 150回、AIと対話して解決策を見つけた
- 150回、アプリの品質が上がった

**話者ノート**
「150件は多すぎる」という印象を持たれないように注意。「150件の問題に気づけた」という視点で伝える。問題に気づけない開発者のプロジェクトには隠れたバグが山積している。ISSUEが多いほど「観察力が高い」証拠。この逆説を伝えることで、「ISSUEを書く文化」の価値が伝わる。

### SLIDE 018 — 最重要原則TOP5

**スライドタイトル**
AI駆動開発 最重要原則 TOP5

**本文**

**原則1: 問いの質がアウトプットの質を決める**
「バグを直して」ではなく「この症状の根本原因を診断して修正案を提示して」

**原則2: AIの出力を鵜呑みにしない**
AIは「それらしい答え」を出す。正しいかどうかの判断は人間の責任。

**原則3: 小さく試して、すぐ計測する**
大きな変更を一気にするより、ISSUEを小さく分割して一つずつ解決する。

**原則4: 頼んだ箇所以外を変更させない**
AIは「関係ない部分」も直したがる。スコープを厳格に定義する。

**原則5: 継続する仕組みを作る**
ISSUEという形式があることで、「次に何をするか」が常に明確になる。

**話者ノート**
5原則をビジュアルに大きく見せる。各原則を一言ずつ解説。最後に「この5つ、どれか一つでも今日から実践できそうですか？」と問いかけて観客に行動を促す。原則4「頼んだ箇所以外を変更させない」は実際にやりがちなミスで、AIに全リファクタを任せてコードが壊れた経験を話すと刺さる。

### SLIDE 019 — アンチパターン

**スライドタイトル**
やりがちなミス3選とその処方箋

**本文**

**ミス1: 「全部AIに任せる」**
症状: AIに「このアプリを作って」と投げる。出力されたコードがどう動くか分からない。
処方箋: まず自分でISSUEを書く。AIはISSUEを実装するパートナー。設計は人間が持つ。

**ミス2: 「エラーをそのままAIに貼る」**
症状: エラーメッセージだけを貼ってAIに「直して」と頼む。的外れな解決策が来る。
処方箋: エラー + 再現手順 + 関連コード + 期待する動作を全部貼る。文脈が答えの質を決める。

**ミス3: 「実装して終わり」**
症状: AIが直してくれたから確認しない。後で予想外のバグが出る。
処方箋: Lighthouse・gstack・手動確認を必ずセットで行う。検証なき実装は負債。

**話者ノート**
ミス1は「AIに任せすぎる」問題。ミス2は「入力品質」の問題。ミス3は「検証をサボる」問題。3つとも実際に自分がやってしまったミスとして話すと信頼感が上がる。「私も最初はやってました」という自己開示が観客の共感を作る。

### SLIDE 020 — AIとの対話で大切なこと

**スライドタイトル**
プロンプトの質 = アウトプットの質

**本文**

**良いプロンプトの条件**
1. 文脈を与える（どのファイルの・どの関数の・何行目の）
2. 期待する出力を明示する（コード / 解説 / 手順書 / レビュー）
3. 制約を伝える（変更してはいけない箇所・使っている技術スタック）
4. 段階を踏む（一気に全部頼まない）

**実際のプロンプト例（OgoRoulette ISSUE-001）**
```
以下のISSUEを解決するコードを提案してください。

[ISSUE-001の全文]

対象ファイル: app/room/[code]/play/page.tsx
変更してはいけない箇所: handleSpin 関数の外部インターフェース
期待する出力: 修正コードとその理由の説明
```

**話者ノート**
「プロンプトの質」を高める具体的な方法を伝える。ここでは実際のプロンプトを見せることで「こう書けばいいのか」という発見を作る。プロンプトは「仕様書の縮小版」だと伝えると、ISSUEをそのままプロンプトに使う発想が生まれる。

## 補足: Q&Aで想定される質問と回答

**Q: AIを使うと自分のスキルが下がらないか？**
A: 逆です。AIの出力を評価するには、あなた自身の技術理解が必要。AIは「問いを立てる力」を鍛えてくれる。問いが上手くなるほどアウトプットが上がり、その評価力がスキルとして身につく。

**Q: Claude Code以外のAIでも同じことができるか？**
A: できます。重要なのはISSUE駆動という「入力の形式」です。どのAIに渡しても、構造化されたISSUEは高品質な出力を引き出します。

**Q: ISSUE 150件は多すぎないか？管理が大変では？**
A: ISSUEが多いことは「問題発見力が高い」証拠です。管理については、GitHubのISSUEやMarkdownファイルで十分。ステータス（対応中/完了）を付けるだけで管理できます。

**Q: 個人開発ではなくチーム開発でも使えるか？**
A: むしろチームでこそ力を発揮します。ISSUEがあることで「なぜこの変更をするか」の文脈がチーム全員に共有される。AIへの入力もISSUEを渡せば統一できる。

**Q: 初心者でも始められるか？**
A: ぜひ始めてください。ISSUE-001を書いた時点で、あなたの開発プロセスは変わります。完璧なISSUEでなくても大丈夫。書く習慣が大事。
