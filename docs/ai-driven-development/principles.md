# AI駆動開発 実践原則30

## ― OgoRoulette開発から学んだプロの原則 ―

> OgoRoulette（飲み会の奢り決めアプリ: Next.js 16 + Supabase + Framer Motion）を  
> AI（Claude Code）駆動で実際に開発した経験をベースにした、再現性ある30の実践原則。

---

## 原則01: ISSUEは思考単位で切れ

### 概要
1つのISSUEは1つの意思決定で解決できる粒度にする。大きなISSUEは問題の本質が混在し、AIへの指示も曖昧になり、解決できないまま放置される。

### なぜ重要か
ISSUEが大きいと、AIは「どこから手をつけるか」を自分で判断してしまう。複数の関心事が混在したISSUEをAIに渡すと、一部だけ解決して残りが未解決のまま「完了」と判断されるリスクがある。また、差分が大きくなりすぎてレビューが困難になる。

### 実例（OgoRoulette）
OgoRoulette ではISSUEを148個に分割して管理した。例えば「ルーレット同期バグ」を1つのISSUEにせず、ISSUE-143（スピン同期ズレ）・ISSUE-002（spinScheduledRef競合）・ISSUE-001（isOwnerフリッカー）と原因単位で切った。ISSUE-001を解決するとISSUE-002も解消されることが分析でわかり、無駄な実装を防げた。

### NGパターン
```markdown
# NG: 曖昧で大きいISSUE
「ルーレットのバグを全部直して、UIも改善して、テストも追加して」

# OK: 思考単位で切ったISSUE
「ISSUE-001: isOwnerフリッカーによるSPINボタン永続無効化」
「ISSUE-015: playページのphaseステートマシン自動テスト追加」
```

### 実践方法
ISSUEを書くとき「このISSUEの解決策は1つの実装で表現できるか？」を自問する。答えがNoなら分割する。ISSUEタイトルは「現象: 原因の一言説明」形式にすると自然に粒度が揃う。

## 原則02: AIに設計を任せるな

### 概要
アーキテクチャ・データモデル・状態管理の設計は人間が決定する。AIは実装の速度を上げるが、設計の善し悪しを判断する「製品の意図」は人間にしかない。

### なぜ重要か
AIは与えられた制約の中で最適解を出すが、「そもそもこの設計で良いのか」という上位の問いには答えられない。AIが設計を提案するとき、それは過去のパターンからの帰納であり、あなたのプロダクトの本質的な制約（コスト・UX・将来の拡張性）を考慮していない。

### 実例（OgoRoulette）
ポーリング vs WebSocketの選択（ISSUE-009）はAIに任せず人間が決定した。Supabase Realtimeへの移行はスケール時のコスト削減のために必要だったが、「Free Tier内で動作させる」「フォールバックとしてポーリングを残す」という制約はプロダクトの事情を知る人間が判断した。AIはその制約の中で最適な実装コードを書いた。

### NGパターン
```
「このアプリのアーキテクチャはどうすればいいですか？」→ AIの提案をそのまま採用

→ 結果: AIは汎用的なベストプラクティスを提案するが、
  あなたのアプリの規模・コスト・チームサイズに合っていない設計になる
```

### 実践方法
AIに設計を聞くときは「決定済みの制約」を先に列挙する。例：「Supabase Free Tier、1人開発、3ヶ月でリリース、という制約のもとで〇〇の実装方針を教えて」。制約を渡さずに聞くと、AIはエンタープライズ向けの過剰設計を提案する。

## 原則03: 状態管理は人間が握れ

### 概要
Reactのstate設計・フェーズ管理・stateマシンの遷移図は、AIに任せず人間が設計する。これはアプリの「神経系」であり、ここを誤ると全体が崩壊する。

### なぜ重要か
stateマシンが複雑になるほど、AIは個々の遷移を正しく実装できても全体の整合性を見失う。特に非同期処理・タイミング依存の状態遷移は、AIが局所最適なコードを書いた結果、別のフローで競合するバグが生まれやすい。

### 実例（OgoRoulette）
`play/page.tsx` の `phase` ステートマシン（waiting → preparing → spinning → result → waiting）はすべて人間が設計した。ISSUE-001〜004はすべてこのphase管理の競合から生じたバグだった。AIが個別修正を重ねるほど「一つ直すと別が壊れる」状態になり、最終的にISSUE-015でphaseステートマシン全体の自動テストを追加して安全網を作った。

### NGパターン
```typescript
// NG: AIに「いい感じに状態管理して」と任せた結果
const [isSpinning, setIsSpinning] = useState(false)
const [isPreparing, setIsPreparing] = useState(false)  // ← 競合する
const [isResult, setIsResult] = useState(false)         // ← 同時にtrueになり得る

// OK: 人間が設計したphaseステートマシン
type Phase = "waiting" | "preparing" | "spinning" | "result"
const [phase, setPhase] = useState<Phase>("waiting")
```

### 実践方法
stateを設計するとき、まず紙に状態遷移図を書く。遷移の条件・トリガー・副作用を明記してからAIに実装を依頼する。AIへの指示文には「この状態遷移図の通りに実装してください」と図を添付する。

## 原則04: バグは再現性から潰せ

### 概要
「なんか動かない」ではAIはデバッグできない。バグを再現する手順・発生条件・環境を明示して渡すと、解決速度が10倍変わる。再現できないバグは修正できない。

### なぜ重要か
AIがデバッグに失敗するとき、ほぼ必ず「再現条件が不明瞭」か「ログが渡されていない」のどちらかが原因だ。AIは手元でコードを動かせないため、人間が「このログが出たときにこのバグが起きる」という情報を渡す必要がある。

### 実例（OgoRoulette）
ISSUE-001の再現条件は以下のように明文化された：「ゲストホストとしてルームを作成しSPINを一度実行する → 結果表示中でページをリロード → プレイページを再度開く」。この3ステップを渡すことでAIは`useEffect`の発火タイミングを特定し、`isGuestHost`の非同期解決を待つガードを実装できた。再現条件なしに渡していたら、AIは無関係な箇所を修正していた可能性が高い。

### NGパターン
```
「SPINボタンが押せなくなることがある。直して」
→ AIは推測で複数の可能性をコードに追加し、問題が複雑化する

正しいアプローチ:
「以下の手順で100%再現する: 1. ゲストホストでルーム作成 2. SPIN実行 
 3. 結果表示中にリロード → SPINボタンがdisabledになる」
```

### 実践方法
バグ報告テンプレートを作る：「再現手順（番号付き）」「期待値」「実際の結果」「ログ（コンソール出力）」「環境（ブラウザ・OS）」。このテンプレートを埋めてからAIに渡す。埋められない項目があるなら、まず再現を試みる。

## 原則05: UXは最初に設計しろ

### 概要
ユーザー体験（UX）は機能実装後に「付け足す」ことができない。どう感じてほしいか・どんな感情を生み出すかを最初に決め、実装はその実現手段として選択する。

### なぜ重要か
後からUXを改善しようとすると、コンポーネントのデータフロー・stateの持ち方・アニメーションのタイミングまで変更が必要になり、リファクタリングコストが跳ね上がる。「機能は動く、でも体験は微妙」という状態は、根本から設計し直さないと解消できない。

### 実例（OgoRoulette）
ISSUE-144でLPのデモを「テキストバブル」から「実際のRouletteWheelコンポーネント」に置き換えた。これは機能的には同じ「デモ表示」だが、体験が根本的に異なる。最初にUX（「登録前に本物の体験を感じてほしい」）を設計していれば、初期実装から`RouletteWheel`を使っていた。後付けで変更したため、`app/page.tsx`のデータフロー全体を修正する必要が生じた。

### NGパターン
```
開発順序のNG例:
1. 機能を実装する（ルーレットが回る）
2. デザインを当てる（見た目を整える）
3. UXを考える（「やっぱり演出が必要だった」と気づく）
4. 全部リファクタ（コスト爆発）

正しい順序:
1. UXを言語化する（「当選者が出た瞬間、全員がどよめく体験」）
2. そのUXに必要な演出を設計する（Framer Motion + 紙吹雪 + 効果音）
3. 実装する
```

### 実践方法
実装前に「このページを使ったユーザーが感じる感情を3つ書く」演習をする。感情が書けたら、その感情を生むために必要な演出・フィードバック・タイミングを列挙し、それを要件として実装依頼に含める。

## 原則06: AIは提案者であって決定者ではない

### 概要
AIが「〇〇の方がよいと思います」と言っても、それはAIの判断ではなく提案だ。採用するかどうかは常に人間が決める。AIの提案を批判的に評価する習慣を持つ。

### なぜ重要か
AIは人間の承認を得やすい答えを返す傾向がある。「この実装で大丈夫ですか？」と聞けば「大丈夫です」と答える可能性が高い。また、AIは自分の出力を振り返るコンテキストが限られているため、矛盾した提案を複数回にわたって出すことがある。

### 実例（OgoRoulette）
CLAUDE.mdに「実装前に『何を・なぜ・どう実装するか』を一言確認する」というルールを設けた。これにより、AIが実装を開始する前に人間がレビューするチェックポイントが生まれた。`router.back()`禁止のルールも、AIが何度も`router.back()`を使おうとしたため、明示的に禁止をCLAUDE.mdに記載した。

### NGパターン
```
AIが提案: 「useCallbackでメモ化するとパフォーマンスが改善されます」
人間が承認: 「わかりました、お願いします」
→ 結果: アプリの規模ではuseCallbackのコストがメリットを上回る場合も

正しい対応:
「なぜこのケースでuseCallbackが必要か説明してください。
 このコンポーネントの再レンダリング頻度はどれくらいですか？」
```

### 実践方法
AIから実装提案を受けたら、必ず「なぜその方法を選んだか」を聞く。理由が「ベストプラクティスだから」「一般的にこうするから」という場合は、あなたのプロジェクトの具体的な状況に照らして本当に必要か検証する。

## 原則07: 小さく作って小さく壊せ

### 概要
MVP（最小動作品）の単位を小さく定義し、小さな失敗から素早く学ぶ。大きく作ると大きく壊れ、どこが問題かわからなくなる。

### なぜ重要か
AIが1つのプロンプトで大量のコードを生成すると、動かない箇所のデバッグが難しくなる。小さい単位で「動くことを確認してから次へ進む」サイクルを回すと、バグの発生源が明確になり、AIへのデバッグ依頼も的確にできる。

### 実例（OgoRoulette）
OgoRoulette はポーリング実装（3秒間隔）から始め、動作を確認してからSupabase Realtime（ISSUE-009）に移行した。最初からRealtimeで実装しようとしていたら、ポーリングのバグとRealtimeの設定問題が混在して、どこで詰まっているか判別できなかった。「まず動く、次に正しく、最後に速く」という原則を守った。

### NGパターン
```
「WebSocket接続・認証・状態同期・エラーハンドリング・再接続ロジックを
 全部まとめて実装してください」
→ 動かない。どこが問題かわからない。AIも「どれが原因か」の特定に苦労する

正しいアプローチ:
Step1: Supabaseチャンネルに接続してメッセージを受け取るだけ
Step2: 受け取ったデータでstateを更新する
Step3: エラーハンドリングを追加
Step4: 再接続ロジックを追加
```

### 実践方法
実装依頼を出す前に「これを何ステップに分けられるか」を考える。最低3ステップに分解できるなら、それぞれを独立したタスクとして実装・確認してから次に進む。「まとめてやった方が早い」は幻想で、デバッグコストで必ず取り返される。

## 原則08: 感情設計を入れろ

### 概要
アプリはデータを処理するツールではなく、人間に感情を与える体験だ。ドキドキ・驚き・笑い・解放感などの感情を意図的に設計する。

### なぜ重要か
機能的に同じアプリでも、感情設計があるとSNSでシェアされ、記憶に残り、リピートされる。「ルーレットが回る」という機能は誰でも作れるが、「奢りが決まった瞬間の緊張と爆笑」という感情はデザインしないと生まれない。

### 実例（OgoRoulette）
OgoRoulette では以下の感情設計を実装した：
- **ドキドキ**: ルーレットの回転速度が徐々に落ちる（`lib/constants.ts`のアニメーション定数）
- **驚き**: 当選者確定時の紙吹雪（`app/confetti.tsx`）
- **笑い**: 当選者カードの「奢らされちゃった！」という自虐的なコピー
- **シェア衝動**: `WinnerCard`のビジュアルを画像として保存・シェアできる機能（`components/winner-card.tsx`）

Framer Motionを使ったアニメーションはすべて感情設計の一部として実装された。

### NGパターン
```
感情設計のないアプリ:
「ルーレットが止まりました。当選者: 山田太郎」（テキスト表示のみ）

vs

感情設計のあるアプリ:
- 回転が遅くなる演出 → ドキドキ
- ピタッと止まる瞬間 → 緊張の解放
- 紙吹雪 + 効果音 → 驚きと喜び
- シェア可能なビジュアル → 体験の共有
```

### 実践方法
UIの各インタラクションに「この瞬間ユーザーはどんな感情になるか？」を書き込む。感情が「なし（普通）」の箇所は意図的に感情を設計し直す。Framer Motionやハプティクス（`lib/haptic.ts`）を使って、視覚・聴覚・触覚で感情を増幅する。

## 原則09: シェア導線を設計しろ

### 概要
バズは偶然ではなくシェアされやすい仕組みを設計した結果だ。「シェアしたくなる瞬間」と「シェアするための手段」を両方設計する。

### なぜ重要か
SNSでのシェアはアプリの最も強力なマーケティングチャネルだが、「シェアボタンを置けば自然にシェアされる」は幻想だ。シェアされるのは感情が高まった瞬間に、シェアの手間が最小限で、シェアしたくなるコンテンツ（ビジュアル・ストーリー）がある場合だけだ。

### 実例（OgoRoulette）
ISSUE-144でバイラルCTAを強化した。当選者確定後の`WinnerCard`に「LINEでシェア」「Twitterでシェア」「画像として保存」ボタンを追加。特に「画像として保存」機能（`lib/share-service.ts`）は、ルーレットの結果画像をSNSにそのまま投稿できる形式で生成する。これにより「誰が奢りになったか」という感情的な出来事が、視覚的なコンテンツとしてシェアされやすくなった。

### NGパターン
```
NGシェア設計:
- シェアボタンを目立たない場所に配置
- シェアすると「このアプリをチェック！ https://...」というURL共有
- シェアのタイミングが感情の高まりとずれている（後からシェアボタンに気づく）

正しいシェア設計:
- 感情が最高潮になった瞬間（当選者確定直後）にシェアボタンを表示
- シェアコンテンツ自体が面白い（誰が奢りかわかるビジュアル）
- OGP画像でシェアしたとき見栄えがする（ISSUE-012）
```

### 実践方法
「このアプリの一番盛り上がる瞬間はどこか？」を特定し、その直後にシェア導線を配置する。シェアされるコンテンツは「コンテキストなしに見ても意味がわかる」ものにする。URLだけ共有されても何も伝わらない。

## 原則10: ISSUE駆動で開発しろ

### 概要
頭の中にある「あれを直したい」「これを追加したい」をすべてISSUEとして文字に落とす。文字になっていない仕事は存在しないも同然で、AIにも自分にも伝えられない。

### なぜ重要か
開発中に「あ、これ直したい」と思ったことを実装中に混ぜると、差分が混濁し、コミットの意図が不明になり、レビューできなくなる。ISSUEとして切り出すことで「今やること」と「後でやること」を分離でき、AIへの指示も1つのISSUEに集中できる。

### 実例（OgoRoulette）
OgoRoulette では`docs/issues/`ディレクトリに148個のISSUEファイルを作成した。各ISSUEには概要・背景・目的・影響範囲・解決策・完了条件を記載した。コミットメッセージは`fix: ISSUE-143 — ルーム内ルーレット回転同期ズレを修正`形式で常にISSUE番号と紐付けた。これにより「いつ・なぜ・何を変えたか」が完全に追跡可能になった。

### NGパターン
```
頭の中だけで管理したとき:
- 「あのバグ直したっけ？」→ 忘れる
- 「なんでこのコード変えたんだっけ？」→ 3日後には思い出せない
- AIに「さっき話してたバグを直して」→ AIはセッションをまたいで記憶しない
- 複数の修正が1コミットに混在 → git bisectが使えない
```

### 実践方法
「やりたいこと」を思いついたら30秒でISSUEファイルを作る。タイトルだけでもいい。詳細は後で埋める。AIとの会話を始める前に「今日はこのISSUEを解決する」と宣言し、そのISSUEファイルをAIに渡す。

## 原則11: CLAUDE.mdを書け

### 概要
プロジェクトのルール・禁止事項・文脈をCLAUDE.mdに書いて、AIが毎回参照できるようにする。これはコストゼロで品質を上げる最強の施策だ。

### なぜ重要か
AIはセッションをまたいで記憶しない。「前に話したこと」は次のセッションで完全に忘れられる。CLAUDE.mdはAIへの「永続的な記憶」であり、「禁止事項」「プロジェクトの方針」「技術スタック」を一度書けば永続的に効く。

### 実例（OgoRoulette）
OgoRoulette のCLAUDE.mdには以下を記載した：
- `router.back()禁止（Link hrefを使う）`
- `頼んだ箇所以外は変更しない`
- `実装前に何を・なぜ・どう実装するかを一言確認する`
- `型定義はlib/types.tsに集約する`
- Supabaseのスキーマ・テーブル構造

これにより、何度セッションをまたいでも同じルールでAIが動作した。`router.back()`のバグは一度修正した後、CLAUDE.mdに記載することで再発ゼロになった。

### NGパターン
```
CLAUDE.mdなし:
セッション1: 「router.back()は使わないで」→ OK
セッション2: 「新しいページを作って」→ router.back()が使われる → また指摘
セッション3: 「また使ってるじゃないですか」→ 同じ会話の繰り返し

CLAUDE.mdあり:
毎回のセッションで自動的にルールが適用される
```

### 実践方法
「AIに同じことを2回言ったら」それはCLAUDE.mdに書くべき内容だ。セッションの最後に「今日AIに注意したことは何か」を振り返り、CLAUDE.mdを更新する。1週間後には、AIへの補足指示がほぼゼロになる。

## 原則12: ポーリングを軽視するな

### 概要
リアルタイム機能の実装では、WebSocketが理想だが、フォールバックとしてポーリングを正しく設計する。ポーリングの粗雑な実装がUXを破壊する。

### なぜ重要か
WebSocket接続は切断・再接続・ブラウザのバックグラウンド遷移などで不安定になる。その場合にポーリングにフォールバックする設計がないと、リアルタイム機能が無音で壊れる。また、ポーリング間隔・キャンセル処理・メモリリーク対策を怠ると、ユーザーが増えるにつれてDBが死ぬ。

### 実例（OgoRoulette）
ISSUE-009でポーリングをSupabase Realtimeに移行する際、`play/page.tsx`のポーリング実装を精査した。当初の実装は`setInterval`ではなく`setTimeout`の再帰呼び出し（cancelled フラグでキャンセル可能）だった。これは正しいパターンだが、ISSUE-143で判明したようにメンバーの受信遅延が10秒に達するケースがあり、タイミング計算で`Math.max(0, 負の値) = 0`となってフルアニメーションが遅れて実行されるバグが生じた。

### NGパターン
```typescript
// NG: メモリリークするポーリング
useEffect(() => {
  setInterval(fetchRoom, 3000)  // ← clearIntervalしていない
}, [])

// NG: キャンセルできないポーリング
const poll = async () => {
  await fetchRoom()
  setTimeout(poll, 3000)  // ← cleanup関数でキャンセルできない
}

// OK: クリーンアップ付きポーリング
useEffect(() => {
  let cancelled = false
  let timeoutId: ReturnType<typeof setTimeout> | null = null
  const poll = async () => {
    await fetchRoom()
    if (!cancelled) timeoutId = setTimeout(poll, 3000)
  }
  poll()
  return () => { cancelled = true; if (timeoutId) clearTimeout(timeoutId) }
}, [code])
```

### 実践方法
ポーリングを実装するとき、必ず「キャンセル」「メモリリーク防止」「エラー時の挙動」の3点を確認する。間隔は「ユーザーが体感できる最長の遅延」から逆算する（会話のリアルタイム性が必要なら1秒、ステータス確認なら5〜10秒）。

## 原則13: 本番でしか発生しないバグがある

### 概要
ローカルで動くことは本番で動くことを保証しない。環境変数・CDN・ビルド最適化・地理的な遅延・実際のユーザー端末の差異が、本番専用のバグを生む。

### なぜ重要か

```
「ローカルで動いてるから大丈夫」→ 本番デプロイ後に障害
```

## 原則14: レース・コンディションを最初に想定しろ

### 概要
非同期処理が複数走る箇所では、「2つの処理が同時に実行されたらどうなるか」を必ず考える。後から気づいたときには、stateが複雑に絡まって修正困難になっている。

### なぜ重要か
Reactの`useEffect`は依存配列の変化ごとに発火する。複数のuseEffectが同じstateを変更する構造は、発火タイミング次第で結果が変わる「レース・コンディション」を引き起こす。これはユニットテストでは再現しにくく、実際のユーザー操作のタイミングでのみ発生することが多い。

### 実例（OgoRoulette）
ISSUE-002は典型的なレース・コンディションだった。`isOwner`の解決（非同期localStorage読み込み）と`phase`変更の`useEffect`が同時に実行されるタイミングで、メンバー用エフェクトがオーナーとして動作すべきユーザーで発火し、`spinScheduledRef.current = true`が誤ってセットされた。この状態は「`handleSpin`を実行しない限りリセットされない」ため、ユーザーはSPINボタンを永遠に押せなくなった。

### NGパターン
```typescript
// NG: 複数のuseEffectが同じstateに依存
useEffect(() => {
  if (!isOwner) scheduleSpin()  // メンバー用
}, [isOwner, room])

useEffect(() => {
  if (isOwner) loadOwnerState()  // オーナー用
}, [isOwner])

// isOwnerが false→true に変化する瞬間、両方のeffectが発火する可能性がある

// OK: isOwnerの解決を待ってから実行
useEffect(() => {
  if (!isOwnerResolved) return  // isOwnerが確定するまでスキップ
  if (!isOwner) scheduleSpin()
}, [isOwnerResolved, isOwner, room])
```

### 実践方法
useEffectを書くとき「この処理が2回同時に実行されたら？」「実行順序が逆になったら？」を自問する。答えが「壊れる」なら、`ref`でフラグを持つか、`isOwnerResolved`のような「解決済みフラグ」で実行を制御する。

## 原則16: ログを渡せ

### 概要
AIにデバッグを依頼するときは、コンソールログ・エラースタック・ネットワークレスポンスを必ず添付する。ログなしのデバッグ依頼は推測ゲームになり、解決速度が10倍落ちる。

### なぜ重要か
AIはコードを読んで「このバグが起きそう」と推測できるが、実際に何が起きているかは実行時のログを見ないとわからない。特に非同期処理・タイミング依存のバグは、コードだけを見ても再現条件を特定できない。ログがあれば「この時点でこの値だった」という事実から原因に辿り着ける。

### 実例（OgoRoulette）
ISSUE-143（スピン同期ズレ）のデバッグでは、`spinStartedAt`のタイムスタンプ・クライアントの現在時刻・`delay = Math.max(0, startedAt - Date.now())`の計算結果をconsole.logで出力してAIに渡した。ログにより「T+6000にメンバーが受信した時点で`spinStartedAt`は3000ms前」という事実が判明し、「`delay = 0`になってフルアニメーションが遅れる」という原因特定が1分以内に完了した。

### NGパターン
```
ログなしのデバッグ依頼:
「ルーレットの同期がズレます。このコードを見て直してください」
→ AIは「おそらくタイムスタンプの計算に問題があると思います」（推測）
→ 10パターンの修正を試す → どれも当たらない

ログありのデバッグ依頼:
「このログを見てください:
 spinStartedAt: 1712345678000
 Date.now(): 1712345681000  （3秒後に受信）
 delay: Math.max(0, -3000) = 0
 → 即座にアニメーション開始」
→ AIは即座に「delay=0のためフルアニメーションが遅れて実行されます」と特定
```

### 実践方法
バグ調査を始めるとき、まず`console.log`を戦略的に配置する。ログに含めるべき情報：タイムスタンプ・関数の入力値・stateの値・分岐条件の真偽値。ログを収集してからAIに渡す。「ログを取ってから依頼する」を習慣化すると、デバッグセッションの平均時間が劇的に短くなる。

## 原則17: 型は先に決めろ

### 概要
TypeScriptの型定義は実装の最初に設計する。後から型を変えると、変更が波及して全コンポーネントに影響する。型は「設計書」であり、コードより先に存在すべきものだ。

### なぜ重要か
型定義が曖昧なまま実装を進めると、`any`や`as unknown as`で誤魔化したコードが積み上がる。後から「このフィールドを追加したい」となると、Supabaseのスキーマ変更・APIの戻り値の型変更・コンポーネントのpropsの型変更が連鎖し、修正コストが爆発する。

### 実例（OgoRoulette）
OgoRoulette では型定義を`lib/`ディレクトリに集約した（`bill-calculator.ts`・`room-spin.ts`など）。Supabaseのテーブル型は`supabase/`ディレクトリから自動生成した。これにより、データベースのスキーマ変更がTypeScriptのコンパイルエラーとして即座に検出され、型が一致しないコンポーネントが自動的に判明した。型を先に決めた箇所は修正コストが低く、型を後から決めた箇所は修正に時間がかかった。

### NGパターン
```typescript
// NG: 型を後から決めた結果
const [room, setRoom] = useState<any>(null)  // ← anyで誤魔化し
// ... 100行後
room.members.map(m => m.name)  // ← membersが存在するか不明
// 本番でTypeErrorが発生

// OK: 型を先に決めた設計
type Room = {
  id: string
  code: string
  members: Member[]
  status: "WAITING" | "IN_SESSION" | "DONE"
}
const [room, setRoom] = useState<Room | null>(null)
```

### 実践方法
実装を始める前に「このAPIの入力と出力の型」を定義する。型定義をAIに依頼するときは「このSupabaseテーブルのスキーマ定義からTypeScriptの型を生成して」と渡す。型が決まったら先にコンパイルエラーを解消し、それからロジックを実装する。

## 原則18: モバイルファーストを徹底しろ

## 原則19: エラーハンドリングを後回しにするな

### 概要
「正常系が動いてから後でエラー処理を追加する」は永遠に実現しない。エラーハンドリングは機能の一部として最初から実装する。

### なぜ重要か
エラーが「なぜ起きているかわからない」状態のUXは、ユーザーをアプリから離脱させる。エラーメッセージが「Something went wrong」だと、ユーザーはリトライすべきかどうかもわからない。特にリアルタイム機能やAPI呼び出しは、失敗を前提とした設計が必須だ。

### 実例（OgoRoulette）
ISSUE-151（ルーレット予期せぬエラー）では、`handleSpin`の例外処理が不十分で、Supabase APIのエラー時にphaseが`"spinning"`のまま固まるバグがあった。エラーハンドリングを最初から実装していれば、エラー発生時に`setPhase("waiting")`にリセットし、ユーザーに明確なエラーメッセージを表示できた。後から追加したため、phaseのリセット処理が他のコードと競合するリスクが生まれた。

### NGパターン
```typescript
// NG: エラー処理を後回しにした実装
const handleSpin = async () => {
  setPhase("preparing")
  const result = await spinAPI()  // エラー時にphaseが"preparing"で固まる
  setPhase("spinning")
}

// OK: エラー処理込みの実装
const handleSpin = async () => {
  setPhase("preparing")
  try {
    const result = await spinAPI()
    setPhase("spinning")
  } catch (error) {
    setPhase("waiting")  // 失敗時は初期状態に戻す
    toast({ title: "スピンに失敗しました。もう一度お試しください。" })
  }
}
```

### 実践方法
関数を書くたびに「この関数が失敗したとき、ユーザーはどうなるか？」を考える。答えが「わからない」か「画面が固まる」なら、try-catchを追加してphaseをリセットし、ユーザーに通知するコードを先に書く。

## 原則20: テストは動く仕様書だ

### 概要
テストを書くことで「この機能がどう動くべきか」が明文化される。テストは品質担保のためだけでなく、仕様の明確化・リグレッション防止・AIへの「期待値の伝達」のために書く。

### なぜ重要か
テストがない状態でAIにリファクタリングを依頼すると、「動いていたものが動かなくなった」というリグレッションが検出できない。テストがあれば「このテストが全部通ること」を条件にリファクタリングを依頼でき、AIが勝手に振る舞いを変えることを防げる。

### 実例（OgoRoulette）

### NGパターン
```
「テストは後で書く」→ 永遠に書かない
「AIがコード生成したから大丈夫」→ AIは仕様を保証しない

テストを書く正しいタイミング:
- バグを修正したとき: そのバグの再現テストを必ず書く（回帰テスト）
- 複雑なビジネスロジックを実装したとき: lib/bill-calculator.test.ts のように
- 重要な状態遷移があるとき: phaseステートマシンのように
```

### 実践方法
バグを1つ修正するたびに、そのバグが「再発したら検出できるテスト」を1つ書く。テストを書く時間の目安は「修正時間の30%」。`lib/bill-calculator.test.ts`・`lib/room-spin.test.ts`のように、ビジネスロジックから始めると書きやすい。

## 原則21: コミットメッセージはISSUE番号と紐付けろ

### 概要
コミットメッセージには必ずISSUE番号を含める。「なぜこの変更をしたか」が後から追跡できる状態を維持する。コミット履歴はプロジェクトの意思決定の記録だ。

### なぜ重要か
3ヶ月後に「このコードはなぜこうなっているのか」を調べるとき、コミットメッセージにISSUE番号があれば即座に背景がわかる。AIに「過去の変更の意図を踏まえて修正して」と依頼するとき、git logとISSUEファイルを渡せば文脈が完全に伝わる。


### NGパターン
```
NG例:
git commit -m "fix bug"
git commit -m "update"
git commit -m "WIP"
git commit -m "いろいろ直した"

OK例:

git commit -m "test: ISSUE-015 — playページphaseステートマシンのE2Eテスト追加"
```

### 実践方法
コミット前に「このコミットで解決したISSUE番号は何か？」を確認する。ISSUEなしに変更している場合、それはISSUEとして先に切り出すべき変更だ。コミットメッセージのフォーマットを`<type>: ISSUE-XXX — <説明>`に統一する。

## 原則22: 一つのコミット一つの変更

### 概要
1つのコミットには1つの意図・1つのISSUEの変更だけを含める。複数の変更を混ぜると差分が読めなくなり、バグの原因特定ができなくなる。

### なぜ重要か
「このデプロイで何かが壊れた」という場合、コミット単位で原因を特定できるかどうかが重要だ。1コミット1変更なら`git bisect`で問題のコミットを特定でき、問題のコミットだけをrevertできる。複数の変更が混ざっていると、部分的なrevertができない。

### 実例（OgoRoulette）
OgoRoulette ではISSUE番号に対応したコミットを1対1で対応させた。ISSUE-143（スピン同期）の修正は1コミット、ISSUE-144（UX改善）は別コミットにした。これにより、ISSUE-144の変更がUIの別の部分を壊した場合、ISSUE-143の修正はそのままに ISSUE-144だけをrevertできる構造になった。

### NGパターン
```
NG: 複数の変更が混在したコミット
「バグ修正 + 新機能 + リファクタ + スタイル変更をまとめてcommit」
→ バグが再発したとき、どの変更が原因かわからない
→ revertすると全部消える
→ git bisectが使えない

OK: 変更の意図ごとに分割
commit 1: fix: ISSUE-001 — isOwnerフリッカー修正
commit 2: test: ISSUE-015 — phaseステートマシンテスト追加  
commit 3: feat: ISSUE-009 — Supabase Realtime移行
```

### 実践方法
コミット前に`git diff --staged`でステージされた変更を確認する。複数のファイルが「異なる目的」で変更されている場合は、`git add -p`で部分的にステージして別々にコミットする。「このコミットを1行で説明できるか？」という問いに答えられないなら分割する。

## 原則23: AIのコードをそのままコピペするな

### 概要
AIが生成したコードは「提案」であり「正解」ではない。必ず読んで意図を理解し、プロジェクトの文脈に合っているかを確認してから使う。

### なぜ重要か
AIは文脈をすべて把握していない。既存のコードと命名規則が違う・既存のutility関数があるのに新しく実装する・セキュリティ上の問題がある実装を提案する、などが起きる。コピペした瞬間に「なぜこうなっているか」を理解できないコードがコードベースに入る。

### 実例（OgoRoulette）
AIが`supabaseClient`を直接コンポーネントで生成するコードを提案したことがあった。しかし OgoRoulette では`lib/supabase/`にクライアント生成のutility関数を集約していた。そのまま使うと「クライアント生成が2パターン存在する」という状態になった。理解してから使うことで、AIの提案を「既存のutilityを使う形」に修正して一貫性を保った。

### NGパターン
```
AIが生成したコード（1000行）→ 動いた！→ そのままコミット
→ 3日後: 「このコードはなぜこうなっているのか？」→ わからない
→ バグが出たとき修正できない
→ AIに「このコードを直して」→ AIも意図がわからない（コンテキストが失われている）

正しい使い方:
AIが生成したコード → 読む → 意図を確認 → プロジェクトの規約に合わせて修正
→ コミット（自分の言葉で説明できる状態で）
```

### 実践方法
AIが生成したコードをコピペする前に「このコードが何をしているか、3行で説明できるか？」を試す。説明できないなら、AIに「このコードを説明してください」と聞く。説明を聞いてから使う。「なぜこのアプローチを選んだか」も確認する。

## 原則24: 依存関係は最小限に保て

### 概要
ライブラリを追加するたびにバンドルサイズが増え、脆弱性のリスクが増え、メンテナンスコストが増える。「自分で書けるものは自分で書く」姿勢で依存を最小限に保つ。

### なぜ重要か
Lighthouseのパフォーマンススコアはバンドルサイズに直結する。初回ロード時間が1秒長くなるたびにコンバージョン率が下がる。特にNext.jsアプリはクライアントサイドJSのサイズが体験に直結する。また、依存ライブラリのセキュリティ脆弱性は自分では制御できない。

### 実例（OgoRoulette）
ISSUE-150（Tailwind CSSバンドル最適化）では、未使用のTailwindクラスがバンドルに含まれていた問題を修正した。また`package.json`を精査し、実際に使われていないライブラリを削除した。`lib/haptic.ts`は外部ライブラリを使わず`navigator.vibrate`をラップした独自実装にし、`lib/spin-sound.ts`も`Howler.js`を使わず`AudioContext` APIで直接実装した。

### NGパターン
```
「日付のフォーマットをしたい」→ date-fnsをインストール（12KB）
→ 実際に使うのは format() だけ
→ 代替: dayjs（2KB）または Intl.DateTimeFormat（ネイティブAPI）

「アニメーションをしたい」→ anime.js, GSAP, Lottie をまとめてインストール
→ 実際に必要なのはフェードインだけ
→ 代替: Framer Motion（すでに使っている場合）またはCSS transitions
```

### 実践方法
ライブラリを追加する前に「これをネイティブAPIで実現できないか？」を確認する。追加する場合は`bundlephobia.com`でサイズを確認し、tree-shakingが効くか調べる。月1回、`npm ls --depth=0`で依存関係を見直し、使っていないものを削除する。

## 原則25: ユーザーフィードバックを定期的に取れ

### 概要
自分の思い込みを定期的にリセットするため、実際のユーザーにアプリを使ってもらいフィードバックを収集する。「自分が使いやすいと思う」と「ユーザーが使いやすい」は全く別物だ。

### なぜ重要か
開発者はアプリを「作った人」として見るため、ユーザーの視点を失いやすい。特にAI駆動開発は高速に機能が増えるため、「本当に必要な機能」と「作れるから作った機能」の区別が難しくなる。ユーザーフィードバックは「何を作るか」の優先順位を正しくリセットする。

### 実例（OgoRoulette）
gstackによるスコア計測は「定量的フィードバック」として機能した。しかしgstackが拾えない「感情的な体験」についてはユーザーインタビューやSNSの反応が重要だった。「奢り決めアプリ」というコンセプト自体が「飲み会でその場で使える」という実際のユースケースから生まれた。開発者の机上の計画ではなく、実際の飲み会の課題から出発したため、初期リリースから実際のニーズを満たせた。

### NGパターン
```
「ユーザーは〇〇が好きなはずだ」（根拠なし）
「自分が使いやすいから大丈夫」（開発者バイアス）
「SNSで1000いいねついたから成功」（バニティメトリクス）

本当に重要な指標:
- リピート率（2回目に使ってくれているか）
- 完了率（ルーレットを最後まで回したか）
- シェア率（結果をSNSに投稿したか）
- 「また使いたい」という感情
```

### 実践方法
リリース後2週間以内に5人のユーザーにアプリを使ってもらい、使用中の反応を観察する（「考えながら声に出してください」という指示を出す）。観察した内容をISSUEとして記録し、次スプリントの優先度に反映する。月1回のサイクルで繰り返す。

## 原則26: ロールバックを考えてデプロイしろ

### 概要
デプロイするときは「このデプロイが本番で壊れたとき、5分以内に戻せるか？」を確認する。ロールバック手順が頭に入っていない状態でデプロイしない。

### なぜ重要か
特に深夜や週末のデプロイで問題が発生すると、ロールバック手順が曖昧な状態でのデバッグは最悪のユーザー体験を生む。Vercelのデプロイはワンクリックでロールバックできるが、データベースのスキーマ変更・Supabaseのマイグレーションは一方向のため、「戻せない変更」を意識する必要がある。

### 実例（OgoRoulette）
Supabaseのスキーマ変更（新しいカラムの追加）は「後方互換性」を意識して実装した。新しいカラムはNULL許容にし、既存のコードが新しいカラムに依存しない設計にしてから、アプリ側のコードを更新した。これにより、アプリをロールバックしてもDBスキーマとの不整合が起きない状態を保った。

### NGパターン
```
危険なデプロイパターン:
1. Supabaseで既存カラムの型を変更（NOT NULL → 別の型）
2. Vercelにデプロイ
3. 本番でエラー発生
4. Vercelをロールバック
5. でもSupabaseのカラム型は変わったまま → 旧コードと不整合 → 完全に詰む

安全なデプロイパターン:
1. 新カラムをNULL許容で追加（既存コードは無影響）
2. アプリコードを更新してデプロイ（新旧両方のカラムに対応）
3. データ移行
4. 旧カラムを削除（不要になってから）
```

### 実践方法
デプロイ前に「このデプロイに含まれる変更で、ロールバックできないものは何か？」をリスト化する。DBマイグレーション・外部APIの設定変更・環境変数の削除は「ロールバックできない変更」として特に慎重に扱う。VercelのPreview URLで事前に動作確認してから本番に反映する。

## 原則27: APIの設計は早めに固めろ

### 概要
フロントエンドとバックエンドを並行開発するとき、APIのインターフェース（エンドポイント・リクエスト・レスポンスの型）を最初に固める。これがないと、両者が別々の前提で実装してインテグレーション時に詰まる。

### なぜ重要か
API設計が後回しになると、フロントエンドは「おそらくこういう形で来る」と仮定してコードを書き、バックエンドは「こういう形で返せばいい」と別の仮定をする。インテグレーション時に両者の型が合わず、どちらかを全面改修する羽目になる。

### 実例（OgoRoulette）
OgoRoulette の`app/api/`ディレクトリには`/api/rooms/[code]/spin`・`/api/rooms/[code]/result`などのエンドポイントがある。これらのAPIのリクエスト・レスポンスの型をTypeScriptで定義してからフロントとバックを実装した。型定義が共通の「契約書」として機能し、インテグレーション時の型の不一致がコンパイルエラーとして検出された。

### NGパターン
```
「APIは後で考える。とりあえずフロントから作ろう」
→ フロントがmockデータで動く
→ バックエンドのAPI設計時に「フロントの期待値」がわからない
→ インテグレーション時: フロントのtypesとバックのレスポンスが全部合わない
→ どちらかを全修正

正しいアプローチ:
1. API仕様書（型定義）を作る（30分）
2. フロントはその型に合わせたmockで開発
3. バックはその型を返すよう実装
4. インテグレーション: 型が保証されているため問題なし
```

### 実践方法
新しい機能を実装するとき、まずAPIの`Request`型と`Response`型をTypeScriptで書く。フロントはその型を満たすmockデータで先行開発し、バックはその型を返すよう実装する。Next.jsの`Route Handlers`は型付けが容易なので積極的に活用する。

## 原則28: セキュリティは後付けできない

### 概要
認証・認可・入力バリデーション・Rate Limitingは最初から設計に組み込む。「後からセキュリティを付ける」は、家を建ててから鍵を付けることと同じで、構造的に脆弱な部分が残る。

### なぜ重要か
特にSupabaseのRow Level Security（RLS）は後から有効にするとアプリが全く動かなくなることがある。最初からRLSを前提にしたクエリを書いていないと、「RLSを有効にしたら全データが取得できなくなった」という問題が発生する。Rate Limitingも最初から実装しないと、本番でDDoS相当の攻撃を受けたときに対処できない。

### 実例（OgoRoulette）
`lib/rate-limit.ts`と`lib/rate-limit.test.ts`がある。APIエンドポイントへのリクエストにRate Limitingを最初から実装した。`middleware.ts`で認証チェックを行い、認証が必要なルートへの未認証アクセスをブロックする設計を最初に作った。Supabaseのゲストトークン認証（`lib/guest-token.ts`）も、「誰でもAPIを叩ける」状態を防ぐため初期設計に含めた。

### NGパターン
```
「まず動くものを作る、セキュリティは後で」
→ 本番リリース後: 「/api/rooms は認証なしで誰でもアクセスできる」
→ 対策: RLSを追加 → 既存コードが全て動かなくなる
→ 全APIを修正（工数: 開発期間の30%）

最初から設計すれば:
- Supabase RLSのポリシーを最初に定義
- APIエンドポイントは全てauthチェックから始まる
- Rate Limitingは middleware.ts で共通処理
- 追加工数: 最初の1日だけ
```

### 実践方法
新しいAPIエンドポイントを作るとき、テンプレートとして「認証チェック → Rate Limit → バリデーション → ビジネスロジック」の順番を守る。このテンプレートをCLAUDE.mdに記載し、AIが自動的にこの順番で実装するようにする。

## 原則29: パフォーマンスは体験だ

### 概要
Lighthouseスコア60点のアプリは「遅い」という体験としてユーザーに伝わる。パフォーマンスは測定可能な体験であり、機能と同等の優先度で扱う。

### なぜ重要か
First Contentful Paint（FCP）が3秒を超えると離脱率が倍増する。特にモバイルの3G回線では、バンドルサイズの違いが体感速度に直結する。OgoRoulette のようなグループ使用アプリは「その場の盛り上がり」が重要なため、ローディングで数秒待たされると場の空気が壊れる。

### 実例（OgoRoulette）
ISSUE-148（Google Fontsパフォーマンス）・ISSUE-150（Tailwind CSSバンドル最適化）はパフォーマンス改善ISSUEだ。`next/font`を使ってGoogle Fontsをセルフホストに切り替え、フォントの初回リクエストをゼロにした。Tailwindの未使用クラスをpurgeすることでCSSバンドルサイズを削減した。これらの改善後、Lighthouseのパフォーマンススコアが改善し、FCPが短縮された。

### NGパターン
```
パフォーマンスを無視した実装:
- Google Fontsを<head>に直接<link>タグで読み込む（外部リクエスト発生）
- 全ページで同じ重いコンポーネントをimport（tree-shakingが効かない）
- 画像をそのままimg要素で表示（Next.js Imageコンポーネントを使わない）
- unused CSSが残ったまま（Tailwindのpurgeを設定しない）

Lighthouseで測定して発見できるが、測定しなければ気づかない
```

### 実践方法
月1回Lighthouseでスコアを計測し、各スコアをスプレッドシートに記録する。Performance・Accessibility・Best Practices・SEOの4指標を追う。スコアが落ちた月は「何が変わったか」をコミット履歴と照合して原因を特定する。パフォーマンス改善はISSUEとして切り出してから実装する。

## 原則30: リリースより改善を怖れるな

### 概要
完璧を待ってリリースしないより、動くものを早く出して改善を続ける方が価値を生む。「完成」は幻想であり、リリース後の実際のフィードバックこそが最も価値ある情報だ。

### なぜ重要か
未リリースのアプリはゼロユーザーだ。どんなに磨いても、実際のユーザーが使うまでは仮説に過ぎない。リリースしてからわかることの方が、リリース前に想像できることより圧倒的に多い。「もう少し完璧にしてから」と思い続けると、競合が先に市場を取る。

### 実例（OgoRoulette）
OgoRoulette は「ルーレットが回る」最小機能でリリースし、その後148個のISSUEを順次修正・改善していった。リリース前から「ポーリングの最適化」「履歴機能」「プッシュ通知」を実装しようとしていたら、リリースまでに何ヶ月もかかっていた。実際のユーザーの使用状況を見て「同期ズレが一番体験を壊す」とわかり、ISSUE-143が高優先度になった。完璧主義ではなく、観察と改善のサイクルがプロダクトを育てた。

### NGパターン
```
完璧主義の罠:
「プッシュ通知ができるまでリリースしない」
「モバイルアプリ（React Native）版ができてから」
「UIデザインをプロに頼んでから」
→ 6ヶ月後: まだリリースできていない
→ モチベーション低下 → 開発停止

正しいアプローチ:
MVP（最小動作品）でリリース → ユーザーの反応を見る
→ 「これが必要」「これは不要」を実際のデータで判断
→ 優先度順に改善を続ける
→ 6ヶ月後: 148個の改善が積み上がった成熟したプロダクト
```

### 実践方法
「これがないとリリースできない」リストを作り、各項目に「なぜ必須なのか？」を書く。「あると良い」は必須ではない。必須リストが5個以下になったらリリースする。リリース後に残りを改善する。「完成」という概念を「今日の最善」に置き換える。

## まとめ

この30原則は「AI駆動開発を失敗から学んだ人間が書いたルール」だ。

**核心は3つ:**

1. **人間が考える領域を守る** — 設計・意思決定・品質判断は人間の仕事
2. **AIを最大限に使う** — 実装速度・デバッグ支援・文字起こしはAIに任せる  
3. **サイクルを回し続ける** — リリース → 観察 → 改善の繰り返しが価値を生む

> AIは「速く書けるツール」ではなく、  
> **「自分が言語化できていないことを実装に落としてくれるパートナー」**だ。  
> 何を作るか・なぜ作るか・品質を判断するのは常に人間の仕事。
