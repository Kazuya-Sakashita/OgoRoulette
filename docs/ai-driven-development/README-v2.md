# 第1章: はじめに

## 1.1 AI駆動開発とは何か

AI駆動開発とは、**AIをペアプログラマーとして中心に据えた開発手法**だ。「AIにコードを書いてもらう」のとは根本的に違う。

AIは「速く書けるツール」ではなく、「自分が言語化できていないことを実装に落としてくれるパートナー」だ。AIは文脈を理解し、候補を出し、実装を提案する。しかし、**何を作るか・なぜ作るか・品質を判断するのは常に人間の仕事**だ。

AI駆動開発の本質は4つある。

1. **思考の言語化**: 頭の中の「なんかおかしい」をISSUEとして言語化する
2. **AIへの文脈転送**: 言語化した文脈をAIに正確に渡す
3. **人間による判断**: AIの提案を評価して採用・却下を決める
4. **フレームワークで数値化**: 評価フレームワークを使ってプロダクトの現状を客観的に数値化し、次の一手を決める

OgoRouletteでは、この4ステップを200回以上繰り返した。毎回のISSUEがドキュメントになり、そのドキュメントが次のISSUEを生む。**品質は積み上がる**のだ。

## 1.2 なぜ今必要か

2026年現在、個人開発の民主化が加速している。かつては「フルスタックエンジニア+デザイナー+PM」の3人チームで数ヶ月かかっていたアプリが、AIを使えば1人で数週間で作れる時代になった。

しかし、多くの個人開発者が「AIを使っているのに品質が上がらない」「AIに丸投げしたら意図しないものが出てきた」という壁に当たっている。

この壁の正体は **「AIの使い方の問題」ではなく「思考の言語化の問題」** だ。AIは賢い。しかし、曖昧な指示には曖昧な実装を返す。「なんかいい感じにして」では何も生まれない。

スピードが重要な時代において、この「思考の言語化→AIへの転送→判断」のサイクルを速く正確に回せる人が、圧倒的に有利だ。

## 1.3 従来開発との違い

| 観点 | 従来開発 | AI駆動開発 | AI駆動開発 ver.2 |
|---|---|---|---|
| 設計 | 人間が全部設計してから実装 | ISSUEを書いてAIと一緒に設計 | 評価フレームワークで設計の正しさを確認 |
| 実装速度 | 人間が一行ずつ書く | AIがドラフトを生成、人間がレビュー | 同左 |
| デバッグ | スタックトレースを読んで手動調査 | ログ+コードをAIに渡して原因仮説を得る | 同左 |
| ドキュメント | 後回しになりがち | ISSUEがドキュメントになる | 評価レポートも資産になる |
| 品質管理 | テスト・レビューのみ | gstack評価→数値化→改善ループ | G-STACK/HEART/AARRR/Kanoで多角的評価 |
| ボトルネック | コーディング速度 | 思考の言語化速度 | 「何を評価すべきか」の判断速度 |
| チームサイズ | 複数人前提 | 1人でフルスタック対応可能 | 同左 |
| 技術的負債 | 気づかず積み上がる | ISSUEで可視化・計画的に返済 | スコア低下で早期検知できる |

従来開発では「コードを書く速さ」がボトルネックだった。AI駆動開発では「思考を言語化する速さ」がボトルネックになる。だからこそ、ISSUEの書き方が最も重要なスキルになる。

## 1.4 この教科書でできること

この教科書を読み終えたとき、あなたには以下が手に入る。

- **AI駆動開発の全体像**: 何をどの順序でやるか、なぜその順序なのかが理解できる
- **ISSUEの書き方**: 「思考の言語化」の技術。一生使えるスキルだ
- **テンプレート8種**: コピペで即使える。要件定義からLP生成・評価フレームワークまで
- **プロンプト20+種**: 場面別に選べる。試行錯誤なしで使える
- **ケーススタディ**: 実際のバグ・設計・最適化の事例。同じ壁にぶつかったときの地図になる
- **失敗パターン**: よくある罠を事前に知れる。回り道を省ける
- **バズ設計**: 作った後に広まる設計を最初から組み込む方法
- **評価フレームワーク**: G-STACK / HEART / AARRR / Kano の4フレームワークで品質を多角的に測る方法

# 第2章: 全体フロー（最重要）

## 2.1 AI駆動開発エンジン

AI駆動開発の核心は、以下のループだ。このループを理解せずにテンプレートやプロンプトだけ使っても、効果は半減する。

```
┌─────────────────────────────────────────────────────┐
│               AI駆動開発エンジン                    │
│                                                     │
│  ① 課題発見                                        │
│      「なんかおかしい」「ここが気になる」           │
│         ↓                                           │
│  ② ISSUE化（思考の言語化）                         │
│      背景・問題・原因・解決策・受け入れ条件を書く  │
│         ↓                                           │
│  ③ AI解決案生成                                    │
│      テンプレート + プロンプトでAIに渡す           │
│         ↓                                           │
│  ④ 実装（AI + 人間の判断）                         │
│      AIの提案を評価・修正・採用する                │
│         ↓                                           │
│  ⑤ フェーズ化・優先度付け                          │
│      評価フレームワーク（T-08）で現状を数値化する  │
│      スコアが低い軸を次フェーズの優先課題にする    │
│         ↓                                           │
│  ⑥ 検証（gstack）                                  │
│      数値で品質を確認する                          │
│         ↓                                           │
│  ⑦ 再ISSUE化                                       │
│      検証で見つかった新課題をISSUEにする           │
│         ↓                                           │
│  ① に戻る（品質が積み上がる）                     │
└─────────────────────────────────────────────────────┘
```

OgoRouletteでは、このループを1日に複数回回した。1ループあたり30分〜2時間。200回回した結果が、現在のアプリだ。

## 2.2 各ステップの詳細解説

### ① 課題発見

「なんかおかしい」「ユーザーが戸惑いそう」「パフォーマンスが悪い気がする」という直感を大切にする。この直感を**その場でメモ**することが重要だ。後回しにすると消える。

OgoRouletteでは、Slackのメモチャンネルにひとこと投げておき、あとでISSUEにまとめた。「ルーレットが止まる瞬間の演出が弱い」「iOSで音が出ない」「ルーム参加時にローディングが長い」。これらがすべてISSUEになった。

### ② ISSUE化（思考の言語化）

課題を発見したら、すぐISSUEに落とす。ISSUEは「思考の結晶」だ。

良いISSUEには5つの要素がある。背景（なぜ今これを解決するのか）、問題（何が起きているか）、原因（なぜそうなっているか）、解決策（どう直すか）、受け入れ条件（どうなれば完了か）。この5要素がそろったISSUEは、AIへの指示書として完璧に機能する。

### ③ AI解決案生成

ISSUEが書けたら、テンプレートにはめてAIに渡す。テンプレートは「AIへの文脈転送の型」だ。適切な型で渡すと、AIの回答の質が劇的に上がる。

例えばデバッグのISSUEなら T-03（バグ調査テンプレート）を使う。UI改善なら T-02（UI生成テンプレート）を使う。「どのテンプレートを使うか」の判断が人間の重要な仕事だ。

### ④ 実装（AI + 人間の判断）

AIの提案は「叩き台」だ。そのまま使うのではなく、評価して採用・却下・修正を判断する。

**AIに任せる**: コードのドラフト生成、テスト作成、コメント追加、リファクタ候補の提案。
**人間が判断する**: アーキテクチャの最終決定、状態管理の設計、ユーザー体験の評価、採用・却下の決定。

この分担を崩すと、AIへの丸投げになる。丸投げは必ず失敗する。

### ⑤ フェーズ化・優先度付け（ver.2追加）

実装が一段落したら、T-08（評価フレームワークテンプレート）を使ってプロダクト全体を評価する。G-STACK / HEART / AARRR の3軸でスコアを出し、最も低い軸を次フェーズの優先課題にする。

「動いている」だけでなく「どの方向に改善すべきか」を数値で判断することで、次のISSUEの優先度が客観的に決まる。

### ⑥ 検証（gstack）

実装したら必ず検証する。gstackはQAテストを実行し、品質を数値化するツールだ。「なんとなく動いている」では品質は上がらない。数値で見ることで、改善の優先度が明確になる。

OgoRouletteでは、Lighthouseスコア・UX観点・視覚的一貫性・アクセシビリティを毎回チェックした。

### ⑦ 再ISSUE化

検証で見つかった問題は、すぐにISSUEにする。「あとで直す」はほぼ実現しない。ISSUEにすれば次のループで確実に処理される。

## 2.3 ループを速く回すコツ

**1ループ = 1〜2個の改善に絞ること**。これが最も重要なルールだ。

「一気にたくさん改善しよう」とすると、何が原因で何が改善したかわからなくなる。デバッグが複雑になり、AIへの指示も曖昧になる。小さく回す方が、結果として速い。

OgoRouletteで学んだ粒度感:

- 良い粒度: 「スピン後のアニメーション演出を追加する」
- 大きすぎる粒度: 「UXを全体的に改善する」
- 小さすぎる粒度: 「ボタンの色を#FF6B6Bから#FF5A5Aに変える」

「大きすぎる」は分割する。「小さすぎる」は関連するものとまとめる。1ISSUEが1セッションで完結するくらいの粒度が理想だ。

# 第3章: ISSUE駆動開発

## 3.1 ISSUEの役割

ISSUEは単なるTODOリストではない。**思考の結晶**だ。

「ルーレットのスピン中にルームを離れると状態がおかしくなる」という問題があったとき、ISSUEを書く過程で「なぜおかしくなるか」を考え、「どう直すか」を設計し、「どうなれば完了か」を定義する。この過程でAIへの指示書が完成する。

ISSUEには以下の4つの機能がある。

1. **思考の整理**: 書くことで問題が明確になる
2. **AIへの文脈転送**: 書いたISSUEをそのままAIに渡せる
3. **開発ログ**: 「なぜこの実装にしたか」が残る
4. **品質の証拠**: 200 ISSUEを消化した事実が品質保証になる

## 3.2 タスクとISSUEの違い

| 観点 | タスク | ISSUE |
|---|---|---|
| 粒度 | 大きい・あいまい | 一つの問題・一つの改善 |
| 背景 | 書かない | 「なぜ今やるか」を必ず書く |
| 原因分析 | しない | 可能な限り原因を書く |
| 受け入れ条件 | 「完了したら消す」 | 「どうなれば完了か」を明記 |
| AIへの使いやすさ | そのままでは使えない | AIへの指示書として完成している |
| 残り方 | 消える | ログとして残る |
| 評価軸との紐付け | なし | G-STACK / HEART / AARRR のどの軸に影響するか |

タスク管理ツールの「ログイン機能を作る」という1行は、AIへの指示書にはならない。ISSUEは、AIが迷わずに動けるだけの情報を含む。

## 3.3 良いISSUEの条件

良いISSUEには6つの条件がある。

**条件1: 背景が書かれている**
「なぜ今これを解決するのか」が書かれている。「ユーザーから報告があった」「gstackでスコアが低かった」「本番でしか発生しない」など。

**条件2: 問題が具体的だ**
「おかしい」「遅い」ではなく、「iOS Safari 17.2でルーレットのスピンアニメーションが途中で止まる」のように具体的だ。

**条件3: 原因の仮説がある**
原因がわからなくても「おそらく〜だと思う」という仮説を書く。AIはこの仮説を検証してくれる。

**条件4: 解決策の方向性がある**
「〜の方向で修正したい」「〜の実装を試したい」という方向性を書く。完璧な解決策でなくていい。

**条件5: 受け入れ条件が明確だ**
「〜の状態になればこのISSUEは完了」が書かれている。これがないと実装が終わらない。

**条件6: 評価軸との紐付けがある**（ver.2追加）
このISSUEを解決するとどの評価軸のスコアが改善するかを書く。「G-STACKのArchitecture軸改善」「HEARTのHappiness軸改善」のように紐付けると、優先度判断の根拠になる。

## 3.4 ISSUE記述フォーマット

```markdown
## ISSUE-XXX: [タイトル]

### 背景
[なぜ今これを解決するのか。ユーザーへの影響・発見のきっかけ・優先度の根拠]

### 問題
[何が起きているか。できるだけ具体的に。再現手順があれば書く]

### 原因
[なぜそうなっているか。わからない場合は仮説を書く]
- 仮説1: [〜ではないか]
- 仮説2: [〜の可能性もある]

### 解決策
[どう直すか。方向性だけでも書く]

### 受け入れ条件
- [ ] [条件1: 〜の状態になっている]
- [ ] [条件2: 〜が正常に動作する]
- [ ] [条件3: 回帰テストを通過している]

### 評価軸との紐付け（ver.2追加）
- 影響するフレームワーク: [G-STACK / HEART / AARRR / Kano]
- 改善が期待される軸: [例: HEART - Happiness, G-STACK - Architecture]

### 参考
- 関連ISSUE: #XXX
- 関連ファイル: `src/components/XXX.tsx`
```

## 3.5 NG例と改善例

OgoRouletteの実際のISSUEをベースにした比較だ。

**NG例（雑なISSUE）**

```
## ISSUE: ルーレット直す

ルーレットがバグる。早めに対応。
```

これをAIに渡しても、AIは何をすればいいかわからない。「どのバグか」「どんな状況で起きるか」「何が正しい状態か」が何もない。

**改善例（良いISSUE）**

```markdown
## ISSUE-143: ルーム内ルーレット同期ズレを修正

### 背景
本番環境（Vercel）でルームを使ったとき、ホストがルーレットを止めた後に
他の参加者の画面でルーレットが止まるタイミングがズレる問題が発生した。
ローカルでは再現しない。

### 問題
- ホストが停止操作をすると、ホスト画面では即座に当選者が表示される
- 他の参加者の画面では2〜5秒遅れて停止する
- まれに停止しないまま永遠に回り続けるケースがある

### 原因（仮説）
- Supabase Realtimeのブロードキャストに遅延が発生している可能性
- phase管理（spinning → stopping → stopped）のタイミングで競合状態が起きている可能性
- waitingForWinnerフラグが適切にリセットされていない可能性

### 解決策の方向性
- phase遷移に明示的なウェイト処理を追加する
- waitingForWinnerフラグをリセットするタイミングを見直す
- Supabaseのbroadcastイベントのハンドリングをデバウンスする

### 受け入れ条件
- [ ] 3人以上のルームで、全員の画面が同時に停止する
- [ ] 停止後5秒以内に全員の当選者表示が一致する
- [ ] 本番環境で10回連続テストして問題が起きない

### 評価軸との紐付け
- 影響するフレームワーク: G-STACK, HEART
- 改善が期待される軸: G-STACK - Risk, HEART - Task Success
```

差は明白だ。改善例のISSUEはそのままAIへの指示書になる。

## 3.6 ISSUEの粒度管理

**思考の単位で切る**のが基本原則だ。

「一つの問題を解決する」「一つの機能を追加する」「一つの最適化を行う」が理想の粒度だ。

大きすぎるISSUEの分割方法:

```
悪い例（大きすぎる）:
ISSUE-50: モバイル対応を全体的に改善する

分割後（良い粒度）:
ISSUE-51: スマホでのルーレット表示サイズを最適化
ISSUE-52: iOS Safariでの音声再生バグを修正
ISSUE-53: タッチ操作でのスワイプ誤反応を防止
ISSUE-54: 縦向き・横向きの切り替え時のレイアウト崩れを修正
```

ISSUEを分割する基準は「1セッションで完了できるか」だ。複数セッションまたがるISSUEは、途中で文脈が失われてAIへの引き継ぎが難しくなる。

# 第4章: OgoRoulette開発ストーリー

## 4.1 Day 0: アイデア誕生

「飲み会で誰が奢るか決めるのって、毎回揉めるよな」

1. 参加者のリストを作る
2. ルーレットを回す

CLAUDE.mdを最初に書いた。これはAIとのセッションをまたいでも、一貫した判断基準を保つための「開発憲法」だ。

## 4.3 Week 3-4: ルーム機能

「全員が同じルーレットを見る」にはリアルタイム同期が必要だった。Supabase Realtimeのbroadcastを使うことは決まっていたが、**どの状態をどのタイミングで同期するか**の設計が難しかった。

ここでAIとの長い設計セッションが始まった。ISSUE-020「リアルタイムルーム設計」を書いて、AIに「Supabase Realtimeを使って、複数ユーザーが同じルーレットを操作する際の状態管理設計を提案してほしい」と渡した。

## 4.4 Month 2: バグとの格闘

Month 2は**バグの月**だった。機能は動いているが、エッジケースで壊れる。特に3つが深刻だった。

**Race Condition（ISSUE-001）**: 複数ユーザーが同時にボタンを押すと、状態が競合してルーレットが変な動きをする。これはロックトークンパターンで解決した。詳細はケーススタディ6.1を参照。

**iOS Safari問題（ISSUE-035）**: iOSでルーレットのアニメーションが途中で止まる。原因はSafariのCSS `transform` の扱いの差異。`will-change: transform` を追加して解決した。

**Phase管理の複雑化（ISSUE-055〜070）**: ルーレットの状態が `idle → spinning → stopping → stopped → winner_display` と増えるにつれて、遷移条件が複雑になりバグが多発した。状態機械パターンへのリファクタで解決した。

**バグデバッグの鉄則**:

```
① ログを取る（console.log地獄ではなく構造化ログ）
② 再現手順を明確にする
③ T-03（バグ調査テンプレート）でAIに渡す
④ 原因の仮説を3つ以上出してもらう
⑤ 仮説を検証する（最も可能性の高いものから）
⑥ 修正してISSUEを閉じる
```

# 第5章: 評価フレームワーク四重奏

## 5.0 なぜ評価フレームワークは「進化」するのか

### 正しいのに弱いプロダクト

G-STACKで高評価を得たプロダクトは、技術的に正しい。でも「技術的に正しい」と「ユーザーが使いたいと思う」は、まったく別の話だ。

たとえば、以下のことはG-STACKでは測れない。
- ルーレットが回ったとき、ドキドキするか
- 当選者が出た瞬間、笑顔になるか
- 「もう一回やりたい」と思うか
- 友達に「これ面白いよ」と送りたくなるか

これらはすべて、**感情の問題**だ。G-STACKは「構造の正しさ」を測るが、「感情の豊かさ」は測らない。構造が正しくても、感情が動かなければ、バズらない。使われない。忘れられる。**「正しいのに弱いプロダクト」は、こうして生まれる。**

### フェーズ別評価戦略の全体像

| フェーズ | 問い | 主なフレームワーク | 卒業条件 |
|---|---|---|---|
| フェーズ1: 動かす | このアプリは成立しているか？ | G-STACK | 主要機能動作、致命的バグなし |
| フェーズ2: 安定させる | このアプリは壊れないか？ | G-STACK + 動作検証 | 実環境で安定動作 |
| フェーズ3: 体験を作る | このアプリは使って気持ちいいか？ | HEART + 感情評価 | ユーザーが「楽しい」「また使いたい」と言う |
| フェーズ4: バズらせる | このアプリは広がるか？ | Kano + AARRR | 自発的シェア・リピートが起きる |

## 5.1 なぜ評価フレームワークが必要か

AI駆動開発で「何を改善すべきか」を決める際、人間は無意識にひいき目で判断する。自分が作ったものを客観的に評価するのは難しい。

フレームワークは「感情的なバイアスなしに、プロダクトの現状を数値化する言語」だ。

- フレームワークがない → 「なんとなく良くなった気がする」
- フレームワークがある → 「HEARTのHappinessが45→87に上がった」

数値で見ることで、AIへの指示が具体的になる。「UXを改善して」ではなく「HEARTのHappiness軸のスコアを上げる施策を提案して」と言えるようになる。

## 5.2 G-STACK フレームワーク

**G-STACK = Goal / Strategy / Tactics / Architecture / Risk**

| 軸 | 意味 | 配点 | OgoRoulette評価例 |
|---|---|---|---|
| **G**oal | 目標が明確か | 20点 | 「飲み会の奢り決め」という単一ゴール: 18点 |
| **S**trategy | 戦略に一貫性があるか | 20点 | リアルタイム同期による公平性担保: 16点 |
| **T**actics | 実装がその戦略に沿っているか | 20点 | Supabase Realtime採用: 17点 |
| **A**rchitecture | 技術構造が健全か | 20点 | コンポーネント設計・DB設計: 14点 |
| **R**isk（SK） | リスクが識別されているか | 20点 | Race condition対策・エラーハンドリング: 12点 |

**OgoRoulette G-STACKスコア: 77/100（月初時点）**

### AIへの渡し方

```
以下の評価基準でOgoRouletteを評価してください。

【G-STACKフレームワーク】
G (Goal): アプリの目標が明確で達成可能か (0-20点)
S (Strategy): 差別化戦略が一貫しているか (0-20点)
T (Tactics): 実装が戦略に沿っているか (0-20点)
A (Architecture): 技術構造が健全か (0-20点)
R (Risk): リスクが識別・管理されているか (0-20点)

【アプリ概要】
[アプリの説明と現在の実装状況を貼り付ける]

各軸を採点し、最も低い軸の改善ISSUEを3つ提案してください。
```

## 5.3 HEART フレームワーク

**HEART = Happiness / Engagement / Adoption / Retention / Task Success**

| 軸 | 意味 | OgoRouletteでの問い | 評価指標例 |
|---|---|---|---|
| **H**appiness | ユーザーは満足しているか | ルーレットを回して楽しかったか | NPS・「また使いたい」発言 |
| **E**ngagement | 積極的に使っているか | 一度のセッションで何回スピンするか | 1セッションあたりのスピン数 |
| **A**doption | 新機能を試しているか | ルーム機能を使ってみようとするか | ルーム機能利用率 |
| **R**etention | 継続して使っているか | 翌週も使いに来るか | 7日後再訪率 |
| **T**ask Success | 目的を達成できているか | 「奢る人を決める」目標を達成できたか | 決定完了率 |

**HEARTは「感情の言語」だ。** G-STACKが「構造の正しさ」を測るのに対し、HEARTは「使った体験の豊かさ」を測る。

OgoRouletteでの実例:
- 月初評価: Happiness 45点（「誰も笑わなかった夜」の前）
- confetti/カウントダウン追加後: Happiness 87点

## 5.4 AARRR フレームワーク

**AARRR = Acquisition / Activation / Retention / Revenue / Referral**（Pirate Metrics）

| 段階 | 問い | OgoRouletteの指標例 | 弱い場合の施策 |
|---|---|---|---|
| **A**cquisition | どこから来たか | SNSからの流入数 | SEO・SNS・紹介施策 |
| **A**ctivation | 最初の体験は良かったか | 初回スピンまでの離脱率 | オンボーディング・初回UX改善 |
| **R**etention | また来るか | 7日後再訪率 | 通知・リマインダー・コンテンツ更新 |
| **R**evenue | お金を生むか | Premium機能の転換率 | 課金設計・プレミアム機能 |
| **R**eferral | 広めるか | SNSシェア数・招待数 | シェア機能・インセンティブ設計 |

AARRRを見ることで、「今どこが一番弱いか」が数字でわかる。感覚ではなく、データで優先順位を決められる。

## 5.5 3フレームワークの加重スコアリング

OgoRouletteでは、以下の加重スコア式を採用した。

```
総合スコア = G-STACK × 0.5 + HEART × 0.3 + AARRR × 0.2
```

| フレームワーク | 重み | 理由 |
|---|---|---|
| G-STACK | 50% | 構造・設計・リスクはすべての基盤。どのフェーズでも重要 |
| HEART | 30% | UXが悪ければ使われない。体験の質が継続利用を決める |
| AARRR | 20% | 成長はまだ途上段階のため重みを低く設定 |

このウェイトは固定ではない。フェーズが進むにつれて変化させる。

| フェーズ | G-STACK | HEART | AARRR |
|---|---|---|---|
| フェーズ1 | 100% | 0% | 0% |
| フェーズ2 | 80% | 20% | 0% |
| フェーズ3 | 50% | 50% | 0% |
| フェーズ4 | 40% | 30% | 30% |

## 5.7 評価駆動改善サイクル：フェーズ化戦略

### フェーズA: 技術基盤強化（G-STACK < 65点）

G-STACKスコアが65点未満の状態では、技術的な基盤が不安定だ。まずここを固める。

- Goal/Strategy軸が低い → ビジョン・ターゲットを再言語化する
- Tactics/Architecture軸が低い → リファクタISSUEを優先する
- Risk軸が低い → エラーハンドリング・セキュリティISSUEを優先する

### フェーズB: 体験品質改善（G-STACK ≥ 65点、HEART < 70点）

技術基盤が整ったら、体験品質を上げる。G-STACKは維持しながら、HEARTを優先する。

- Happiness軸が低い → 演出・アニメーション・感情設計を改善
- Engagement軸が低い → 「もう一回やりたい」と思わせる要素を追加
- Task Success軸が低い → オンボーディング・ナビゲーションを改善

### フェーズC: 拡散・バイラル設計（HEART ≥ 70点、AARRR Referral < 60点）

体験が良くなったら、それが広がる仕組みを作る。

- Referral軸が低い → シェア機能・OG画像・口コミ設計を強化
- Retention軸が低い → 通知・再訪のトリガーを設計
- Acquisition軸が低い → LP・SEO・SNS施策を実施

## 5.9 Kanoモデル

Kanoモデルは、機能を3種類に分類するフレームワークだ。「次に何を作るか」の優先度判断に使う。

| 分類 | 説明 | OgoRoulette例 | 開発優先度 |
|---|---|---|---|
| **当たり前品質** | ないと不満。あっても感動しない | 参加者の追加・削除 | バグ修正は必須。追加しても感動なし |
| **一元的品質** | あればあるほど満足度が上がる | スピン速度の調整 | ユーザーの声を聞きながら改善 |
| **魅力品質** | あると感動。ないことは期待していない | みんなの画面が同時に動く | 差別化・バズりのコアに据える |

OgoRouletteの最大の魅力は「全員の画面でルーレットが同時に動く」という体験だ。これは典型的な魅力品質だ。

**Kanoを使った機能の取捨選択**:
- 当たり前品質を磨いても感動は生まれない
- 一元的品質は「十分」なところで止める
- 魅力品質に開発リソースを集中させる

**魅力品質の発見方法**:
1. 友達に使ってもらって「一番驚いたところは？」を聞く
2. SNSでのコメントで「すごい」「初めて見た」が多い機能を探す
3. AIに「このアプリの機能をKanoモデルで分類して」と聞く

## 5.10 フェーズ別評価戦略（詳細）

### フェーズ1: 動かす段階

**問い：「このアプリは成立しているか？」**

この段階でHEARTやKanoを使おうとしても意味がない。土台がなければ、感情評価も成長評価もできない。G-STACKで基盤を固めることに集中する。

| 評価軸 | 確認内容 |
|---|---|
| Goal | 誰のための何のアプリか明確か |
| Strategy | 差別化の方向性が決まっているか |
| Tactics | 実装がその方向に沿っているか |
| Architecture | DB設計・API設計・コンポーネント設計が健全か |
| Risk | バグ・セキュリティ・パフォーマンスのリスクが管理されているか |

**卒業タイミング**: 主要機能が動作し、致命的バグがない状態

---

### フェーズ2: 安定させる段階

**問い：「このアプリは壊れないか？」**

動いてはいる。でも本番で使えるかどうかは別問題だ。iOS Safariで動くか。ネットワークが不安定でも壊れないか。複数人が同時に使ったとき、競合が起きないか。G-STACKに「動作検証」を加えた評価が必要になる。

| 追加評価軸 | 確認内容 |
|---|---|
| クロスブラウザ | Safari/Chrome/Firefoxで同一挙動か |

OgoRouletteでは、この段階でSupabase Realtimeの接続断問題や、iOS SafariでのWebM非対応問題が発覚した。G-STACKだけでは見つけられなかった問題だ。

**卒業タイミング**: 実環境（スマホ・複数人・低速回線）で安定動作する状態

---

### フェーズ3: 体験を作る段階

**問い：「このアプリは使って気持ちいいか？」**

技術的な正しさではなく、**感情的な豊かさ**を問う段階だ。

- 「ルーレットを回して楽しかったか？」
- 「当選者が出た瞬間、その場が盛り上がったか？」
- 「スマホを置く前に、もう一回やりたくなったか？」

これらはアンケートでも、自己評価でも測ることができる。重要なのは「この問いを開発者が持つこと」だ。

**卒業タイミング**: ユーザーが自発的に「楽しかった」「また使いたい」と言う状態

---

### フェーズ4: バズらせる段階

**問い：「このアプリは広がるか？ハマるか？」**

体験が良くなった。次は、それが広がるかどうかだ。KanoとAARRRを組み合わせて使う。

- Kanoで「どの機能が魅力品質か」を特定する
- AARRRで「どのファネルが最もボトルネックか」を特定する
- 魅力品質 × Referral軸の改善に集中する

**卒業タイミング**: ユーザーが自発的にシェアし、リピートし、紹介してくれる状態

## 5.11 フレームワーク切り替えタイミングの判断基準

### フレームワーク切り替えフローチャート

```
アプリは動いているか？
    │
    ├─ NO  → G-STACKで基盤を固める（フェーズ1）
    │
    └─ YES
         │
         安定しているか（本番で使えるか）？
             │
             ├─ NO  → G-STACK + 動作検証（フェーズ2）
             │
             └─ YES
                  │
                  使って楽しいか？（HEARTスコア ≥ 70？）
                      │
                      ├─ NO  → HEART + 感情評価（フェーズ3）
                      │
                      └─ YES
                           │
                           広がっているか？（Referral ≥ 60？）
                               │
                               ├─ NO  → Kano + AARRR（フェーズ4）
                               │
                               └─ YES → すべてのフレームワークで継続改善
```

### G-STACKだけでよいとき

- アプリが完成していない（主要機能が動いていない）
- 致命的なバグや設計上の問題がある
- 技術的な基盤をまだ固めている段階
- ユーザーに見せる前の内部品質向上フェーズ

### HEARTを入れるべきとき

- ユーザーに実際に使ってもらえる状態になった
- 「動く」のに「使われない」問題が起きている
- 機能を追加しても満足度が上がらない感覚がある
- 「なんか楽しくない」という感想が出始めた

### Kanoを使うべきとき

- 「次に何を作るべきか」の優先順位が見えなくなってきた
- 機能が増えているのに、ユーザーの反応が薄い
- 「どの機能が本当に喜ばれているのか」が分からない
- 競合との差別化を明確にしたい

### AARRRを見るべきとき

- ユーザーが増えない原因を特定したい
- SNSでのシェアが起きない
- ユーザーが来るが続かない
- 「どこが一番ボトルネックか」をデータで知りたい

### なぜAI駆動開発と相性が良いのか

人間だけで複数のフレームワークを使い分けるのは難しい。「今、自分はフェーズ3にいる」という客観的な認識を持ち続けることは、開発に没頭しているときほど難しくなる。

AIはここで真価を発揮する。

```
私：「このアプリの現状を評価してください」

AI：「現状のフェーズはフェーズ3に移行していると思います。
     G-STACKでの基盤評価は十分できています。
     次はHEARTフレームワークで体験品質を評価すべきタイミングです。
     特に Happiness の軸から始めましょう。」
```

AIはバイアスを持たない。「感情的な愛着なしに、フレームワークに沿って評価してください」とAIに頼めば、客観的な分析が返ってくる。**AIは冷静な評価者として機能する。人間は感情的な意思決定者として機能する。** この組み合わせが、AI駆動開発の本質だ。

## 6.1 ISSUE-001: Race Condition × AI診断

### 背景

Week 3でリアルタイムルームを実装した直後、「2人が同時にスタートボタンを押すとルーレットが変な動きをする」という問題が発見された。ローカルのシングルユーザーテストでは再現しなかった。

### 問題

ホストと参加者が同時に操作可能な状態で、以下の症状が出た。
- ルーレットが2回分回って止まらなくなる
- phase が `spinning` から `idle` に直接飛ぶ
- 当選者が表示されない

### 原因

Supabase Realtimeのブロードキャストが複数発火した際に、phase遷移のロジックが2回実行されていた。「1回目のstop」と「2回目のstop」が競合し、状態が矛盾した。

### 解決策: ロックトークンパターン

```typescript
// スピン開始時にlockTokenを生成
const lockToken = crypto.randomUUID();
await broadcastSpinStart({ lockToken });

// 停止処理はlockTokenが一致する場合のみ受け付ける
const handleSpinStop = (event: BroadcastEvent) => {
  if (event.lockToken !== currentLockToken) return; // 無視
  processSpinStop(event);
};
```

AIが提案したパターンの1つが「ロックトークンによる冪等性担保」だった。このパターンを採用して、race conditionを完全に解消した。

### AI活用のポイント

「Race conditionが起きている」という仮説をISSUEに書いて渡したところ、AIは「楽観的ロック」「悲観的ロック」「ロックトークン」の3パターンを提案した。それぞれのトレードオフを説明してもらい、リアルタイムアプリに最適なロックトークンを選択した。

### 学び

分散システムのrace conditionはローカルでは再現しない。「本番でしか起きないバグ」は、同期の問題を疑え。ロックトークンパターンを最初から設計に組み込んでおくべきだった。

## 6.3 ISSUE-143: ルーム同期ズレ修正

### 背景と問題

本番環境（Vercel）でのみ発生する同期ズレ。ホストが停止操作をすると、他の参加者の画面で停止が2〜5秒遅れる。最悪のケースでは停止しないまま永遠に回り続ける。

### 原因特定のプロセス

ローカルでは再現しないため、本番ログを仕込んで調査した。

```typescript
// デバッグ用ログ追加
console.log(`[SYNC] phase: ${phase}, lockToken: ${lockToken}, timestamp: ${Date.now()}`);
```

ログを分析すると、`stopping` フェーズのbroadcastイベントが参加者側で2回受信されていた。1回目でphaseを `stopped` にしようとするが、2回目のイベントが `stopping` に戻してしまう。

### 解決策

`waitingForWinner` フラグを導入して、停止処理の冪等性を担保した。

```typescript
// 停止処理中フラグ
const [waitingForWinner, setWaitingForWinner] = useState(false);

const handleStopEvent = (event: BroadcastEvent) => {
  if (waitingForWinner) return; // 処理中は無視
  setWaitingForWinner(true);
  
  // 停止処理...
  
  // 完了後にフラグをリセット
  setTimeout(() => setWaitingForWinner(false), 1000);
};
```

### タイミング問題の教訓

本番環境でのみ発生するバグは、ネットワーク遅延を仮定してコードを書く必要がある。「ローカルで動く」は「本番で動く」ではない。Supabase Realtimeのbroadcastは最低1回だが、最大複数回配信される可能性があることをコードに織り込む。

## 6.6 ISSUE-201: Vercel KV × レートリミット

### 背景

本番環境で特定のユーザーが短時間に大量のルーム作成リクエストを送り、Supabaseの書き込みクォータが圧迫される問題が発生した。

### 問題

1分間に20回以上のルーム作成リクエストが来るケースがあった。悪意のある攻撃ではなく、クライアント側のリトライロジックのバグが原因だった。

### 解決策: Upstash Redis によるレートリミット

`@vercel/kv` は非推奨のため、`@upstash/redis` を使用する（`vercel integration add upstash` でワンクリック導入可能）。

```typescript
import { Redis } from '@upstash/redis';

const redis = Redis.fromEnv();

export async function rateLimit(userId: string): Promise<boolean> {
  const key = `rate_limit:room_create:${userId}`;
  const current = await redis.incr(key);
  
  if (current === 1) {
    // 初回アクセス時に60秒のTTLを設定
    await redis.expire(key, 60);
  }
  
  return current <= 5; // 1分間に5回まで許可
}
```

必要な環境変数（Vercel Dashboard または `vercel env pull` で自動設定される）:
```
UPSTASH_REDIS_REST_URL=...
UPSTASH_REDIS_REST_TOKEN=...
```

### AI活用のポイント

「Upstash Redisを使ってレートリミットを実装したい。ルーム作成を1分間に5回まで制限する」というISSUEをAIに渡した。AIは `@upstash/redis` の `incr` + `expire` を使ったシンプルなパターンを提案した。

### 学び

レートリミットはセキュリティではなく「インフラの安定性」のためにも必要だ。クライアント側のバグがサーバー側を圧迫するケースは想定外だったが、KVによる実装は10分で完了した。

## 6.7 ISSUE-202: NTP時刻同期

### 背景

ルーレットの回転タイミングをホストと参加者で同期する際、デバイスのシステム時刻のズレが問題になった。一部の参加者のデバイスが数秒ずれており、「カウントダウン」が参加者ごとに異なるタイミングで終わる問題が発生した。

### 原因

`Date.now()` を使った時刻比較では、デバイス間のシステム時刻差（最大3〜5秒）を吸収できなかった。

### 解決策: サーバー時刻を基準にしたオフセット計算

```typescript
// クライアント起動時にサーバー時刻とのオフセットを計算
const serverTime = await fetch('/api/time').then(r => r.json());
const clockOffset = serverTime.timestamp - Date.now();

// 時刻使用時はオフセットを補正
const syncedNow = () => Date.now() + clockOffset;
```

### 学び

マルチデバイスのリアルタイムアプリでは、クライアントの時刻に頼ってはいけない。サーバー時刻を基準にしたオフセット補正をアーキテクチャ設計時に組み込むべきだった。

## 6.8 ISSUE-152: ChunkLoadError無限ループ

### 背景

一部のユーザーから「アプリが白い画面のまま動かない」という報告があった。コンソールに `ChunkLoadError: Loading chunk X failed` というエラーが繰り返し出力されていた。

### 原因

Next.jsのdynamic importで読み込まれるチャンクが、CDNキャッシュの更新タイミングと競合していた。古いHTMLが新しいチャンクURLを参照しようとして404になり、クライアント側でリトライが無限ループしていた。

### 解決策

```typescript
// ChunkLoadErrorを検出して強制リロードする
window.addEventListener('error', (event) => {
  if (event.message?.includes('ChunkLoadError') || 
      event.message?.includes('Loading chunk')) {
    // リロード済みフラグで無限ループを防止
    const reloaded = sessionStorage.getItem('chunk_reload');
    if (!reloaded) {
      sessionStorage.setItem('chunk_reload', 'true');
      window.location.reload();
    }
  }
});
```

### 学び

ChunkLoadErrorはデプロイ直後のユーザーに起きやすい。解決策は「一度だけ強制リロード」だ。`sessionStorage` でリロード済みフラグを持つことで、無限リロードを防止できる。

## 7.1 テンプレート選択ガイド

どのテンプレートを使うかを決める判断フロー:

```
何をしようとしているか？
│
├─ 新機能を作る前 ──────────────→ T-01（要件定義）
│
├─ UIコンポーネントを作る ────────→ T-02（UI生成）
│
├─ バグが発生している ────────────→ T-03（バグ調査）
│
├─ コードが汚くなってきた ────────→ T-04（リファクタ）
│
├─ 品質を数値で確認したい ────────→ T-05（gstackレビュー）
│
├─ LPやマーケティングページを作る → T-06（LP生成）
│
├─ 開発体験を記事にしたい ────────→ T-07（ブログ生成）
│
└─ プロダクトを多角的に評価したい → T-08（評価フレームワーク）
```

## T-01: 要件定義テンプレート

**重要度**: ★★★★★ CRITICAL

使うタイミング: アイデアはあるが「何を作るか」がぼんやりしているとき。開発を始める前に必ず使う。

このテンプレートをスキップすると、作りながら仕様が揺れ続けて手戻りが多発する。OgoRouletteでも最初にこれを使ったことで、「ルーム機能はMVP後」という判断ができた。

→ [`templates/t01-requirements.md`](templates/t01-requirements.md)

## T-02: UI生成テンプレート

**重要度**: ★★★★☆

使うタイミング: デザインや実装のビジョンはあるが、コンポーネントのコードを書いてほしいとき。

禁止事項を必ず入れること。「既存の他コンポーネントには触れない」「Tailwindのカスタムクラスは使わない」など。禁止事項がないと、AIが意図しない変更をしてしまう。

→ [`templates/t02-ui-generation.md`](templates/t02-ui-generation.md)

## T-03: バグ調査テンプレート

**重要度**: ★★★★★ CRITICAL

使うタイミング: バグが発生しているが原因がわからないとき。

ログ + 関連コード + 再現手順を渡すだけで、調査時間が1/3になる。AIは「このエラーはおそらく〜が原因で、以下の3つを確認してほしい」と具体的に教えてくれる。

→ [`templates/t03-bug-investigation.md`](templates/t03-bug-investigation.md)

## T-04: リファクタテンプレート

**重要度**: ★★★☆☆

使うタイミング: コードが汚くなってきた・複雑度が高くなってきた・次の機能追加の前に整理したいとき。

「何が汚いか」を言語化してから渡すのがコツ。「全体的にリファクタして」は禁止。「この関数の責任を分割したい」のように具体的に渡す。

→ [`templates/t04-refactor.md`](templates/t04-refactor.md)

## T-05: gstackレビューテンプレート

**重要度**: ★★★★★ CRITICAL

使うタイミング: 機能追加・デザイン変更のあと、品質を数値で確認したいとき。

gstackはQAを数値化するツールだ。このテンプレートを使って定期的に評価することで、品質が目に見える形で積み上がる。OgoRouletteでは62点から95点まで改善した。

→ [`templates/t05-gstack-review.md`](templates/t05-gstack-review.md)

## T-06: LP生成テンプレート

**重要度**: ★★★★☆

使うタイミング: ランディングページやマーケティングページを作るとき。

感情設計（ユーザーがLPを見た瞬間〜離脱までの感情曲線）を事前に設計してから使うと効果が倍増する。コンテンツの90%はAIが生成できる。

→ [`templates/t06-lp-creation.md`](templates/t06-lp-creation.md)

## T-07: ブログ生成テンプレート

**重要度**: ★★★☆☆

使うタイミング: 開発体験・技術的な学びを記事にしてコミュニティに発信したいとき。

開発ログ（ISSUE・コミットメッセージ・決定の背景）をAIに渡すと、記事の骨格を生成してくれる。「体験の言語化」はAIが得意な領域だ。

→ [`templates/t07-blog-writing.md`](templates/t07-blog-writing.md)

# 第8章: プロンプト集

## 8.1 プロンプト設計の原則

良いプロンプトには5つの要素がある。

**要素1: 役割の設定**
「あなたはシニアフロントエンドエンジニアです」のように、AIに演じてほしい役割を最初に設定する。役割によって回答のトーン・深さ・専門性が変わる。

**要素2: 文脈の提供**
「このアプリは飲み会の奢り決めに使う。技術スタックはNext.js + Supabase + Framer Motion」のように、必要な文脈を渡す。文脈がないと一般的な回答しか返ってこない。

**要素3: タスクの明確化**
「〜してください」「〜の原因を特定してください」「〜の候補を3つ提案してください」のように、何を期待するかを明確にする。

**要素4: 制約・禁止事項**
「既存のコンポーネントには触れない」「TypeScriptの型を省略しない」のような制約を入れる。AIは制約なしだと最も一般的な解を出す。意図した解を出すには制約が必要だ。

**要素5: 出力フォーマットの指定**
「マークダウンのコードブロックで出力してください」「箇条書きで3点まとめてください」のように、出力の形式を指定する。

# 第9章: AI活用戦略

## 9.1 AIに任せる領域

以下の作業はAIへの全面委任が可能だ。人間がやる意味がほぼない。

**ボイラープレート生成**: フォームのバリデーション・APIルートの骨格・型定義のドラフト。これらを手書きしている時間は純粋な損失だ。

**テストの初期実装**: 「このコンポーネントのテストを書いて」でJestのテストが即座に生成される。カバレッジ100%近くのドラフトが5分で完成する。

**ドキュメント生成**: コードのコメント・JSDoc・READMEの初稿。AIは「コードを読んでドキュメントを書く」のが極めて得意だ。

**コンテンツ生成**: LPコピー・ブログ記事・SNS投稿文。「感情設計」と「事実」を渡せば、90%以上のコンテンツをAIが書ける。

**候補の列挙**: 「技術選定の候補を5つ出して」「このバグの原因仮説を3つ出して」。候補出しはAIが速い。選択は人間がやる。

**評価レポート作成**: T-08（評価フレームワークテンプレート）をAIに渡してプロダクトを評価させる。スコアリングと改善候補の列挙はAIが速い。

## 9.2 人間が握る領域

以下は絶対に人間が主導する。AIに丸投げすると必ず後悔する。

**アーキテクチャの最終決定**: 「どの設計パターンを採用するか」はAIが提案しても、決定は人間がする。アーキテクチャの失敗は後工程で何倍も痛い。

**状態管理の設計**: 複雑な状態管理（特にリアルタイムアプリのphase管理）はAIが得意ではない。AIは局所的に正しいコードを書くが、全体の整合性を保つのは人間の責任だ。

**ユーザー体験の評価**: 「これは使いやすいか」「この演出は盛り上がるか」はAIには判断できない。人間が使って感じることが唯一の正解だ。

**ビジネス判断**: 「この機能をMVPに入れるか」「この問題の優先度は何か」は、ビジネスコンテキストを理解している人間がやる。

**セキュリティの最終確認**: AIはセキュリティホールを見落とすことがある。認証・認可・入力検証は人間が必ずレビューする。

**評価スコアの解釈・判断**: AIが出した評価スコアを「正しい」と盲信しない。スコアの背景にあるビジネスコンテキストを理解して、重みを判断するのは人間の仕事だ。

## 9.4 AIとの対話スタイル

**指示の出し方**:

悪い: 「ルーレットを改善して」
良い: 「スピン停止後の当選者発表アニメーションに祝祭感を追加したい。Framer Motionを使って、停止時のバウンス + confetti + 当選者名の大きな表示を実装してほしい。既存の Roulette.tsx には触れない」

違いは「何を」「なぜ」「どう」「制約」の4要素が揃っているかどうかだ。

**フィードバックの与え方**:

AIの提案が気に入らなかった場合、「違う」と言うだけでは改善しない。「このアプローチはパフォーマンスに問題があるので別の方法を提案してほしい。具体的には〜の点が気になる」と理由と代替の方向性を伝える。

**確認のタイミング**:

実装が複雑なとき、「まず実装の方針を説明してから、コードを書いて」と先に確認を求める。コードが出てきてから「方向性が違う」と言うのは双方の損失だ。

```
良い確認プロセス:
① 「この問題の解決策として〇〇を考えているが、どう思うか?」
② AIが評価と代替案を出す
③ 「では〇〇の方針で実装して」
④ 実装が出てくる
⑤ 「ここは〜に変更して」
⑥ 完成
```

# 第10章: プロの実践原則

AI駆動開発を続ける中で体得した実践原則がある。詳細は [`principles.md`](principles.md) にまとめているが、ここでは代表的な原則を紹介する。

### 原則1: 「わからない」を「ISSUEになる前に消す」

技術的にわからないことがあったら、「後で調べよう」ではなく「今AIに聞く」。30秒で解決できることを1週間放置するのはプロではない。ただし、答えをコピペするのではなく、「なぜそうなるか」を理解してから進む。

### 原則2: 「動いた」で終わらない

「動いた」はゴールではない。「品質基準を満たして動いた」がゴールだ。動いたら必ずgstack評価を回す。数値で確認して、次のISSUEを作って初めて「このISSUEは完了」だ。

### 原則3: 「AIが書いたコードを自分のコードにする」

AIが生成したコードをコピペしただけでは、そのコードを「理解したコード」にならない。自分でコメントを書く・変数名を変える・テストを書く、のどれかをやると理解が深まる。将来バグが起きたとき、理解していないコードは直せない。

### 原則4: 「文脈を渡せない指示は指示ではない」

AIに「前に話した〇〇の続き」は通じない。セッションをまたぐと文脈がリセットされる。毎回のプロンプトに必要な文脈を含める習慣をつける。CLAUDE.mdはこの問題を解決するためのツールだ。

### 原則5: 「人間の判断をAIに委ねない」

「AIはこう言っているから正しい」という思考停止が最も危険だ。AIは間違える。AIは古い情報を持っている。AIはビジネスコンテキストを知らない。AIの提案は「叩き台」であり、「答え」ではない。最終判断は常に人間がする。

---

### 原則31: 評価は月1回以上やれ（ver.2追加）

感覚での改善には限界がある。T-08（評価フレームワークテンプレート）を使って、月1回以上の定期評価を習慣化する。スコアが上がっていれば改善している証拠。下がっていれば問題が増えている証拠だ。

### 原則32: スコアは変化量で見ろ（ver.2追加）

「G-STACKが70点」ではなく「G-STACKが65→70点になった」が重要だ。変化量が「このフェーズの改善の効果」を示す。絶対値ではなく、前回からのデルタで判断する。

### 原則33: 最低軸を優先的に改善しろ（ver.2追加）

全軸を均等に改善しようとすると、全部中途半端になる。スコアの最低軸を特定して、そこに集中投資する。「90点の軸をさらに上げる」より「50点の軸を70点にする」方が総合スコアへの影響が大きい。

### 原則34: フレームワークの重みはビジネスに合わせろ（ver.2追加）

OgoRouletteでは「G-STACK 50% / HEART 30% / AARRR 20%」を採用した。だが、これはOgoRouletteの現フェーズに合わせた重みだ。BtoBサービス、サブスクリプション、ゲームアプリなど、ビジネスの性質によって最適な重みは変わる。自分のプロダクトに合わせて調整せよ。

### 原則35: 評価レポートはAIへの文脈として使え（ver.2追加）

評価レポートは「スコアを見て終わり」にしてはいけない。「現在のスコアと最低軸はこれです。次のISSUEを提案してください」とAIに渡すことで、評価結果が次の改善アクションに直結する。評価はループの一部だ。

### 原則36: フレームワークはフェーズに合わせて進化させろ（ver.2追加）

フェーズ1でG-STACKだけ使って正解だった。でもフェーズ3になってもG-STACKしか使わないのは誤りだ。フレームワークを固定したまま使い続けると、評価の死角が生まれる。「今自分はどのフェーズにいるか」を月1回確認して、フレームワークの構成を見直す。

# 第11章: よくある失敗

## 11.1 ISSUEが雑

**症状**: 「〇〇を直す」という1行のISSUEが多い。バグが直らない・AIへの指示が通じない・実装が何を目指しているかわからない。

**原因**: 「ISSUEを書く」という行為の価値を理解していない。タスク管理ツールに慣れすぎて、1行で済ませる習慣がついている。

**処方箋**: 第3章のISSUEフォーマットを使う。特に「受け入れ条件」を必ず書く。受け入れ条件がないISSUEは永遠に完了しない。

---

## 11.2 AIに丸投げ

**症状**: AIが出したコードをそのままコミットする。しばらくすると動かないコードが増えて、デバッグが追いつかなくなる。

**原因**: 「AIが速く書いてくれる」の恩恵にフォーカスして、「AIが間違える」というリスクを忘れている。

**処方箋**: AIのコードは必ずレビューする。特に「状態管理」「認証・認可」「非同期処理」の3領域は入念に確認する。自分が理解できないコードはマージしない。

---

## 11.3 検証不足

**症状**: 「実装した→完了」で次に進む。気づいたらバグが積み上がっている。品質が上がっている実感がない。

**原因**: 検証を「別の作業」として捉えている。実装と検証はセットだという認識がない。

**処方箋**: 「ISSUEを閉じる前にgstack評価を必ず走らせる」をルールにする。数値が改善していない場合はISSUEを閉じない。これを徹底するだけで品質が上がる。

---

## 11.4 状態管理の放棄

**症状**: Propsのバケツリレーが深くなる・useEffectが乱立する・どこで状態が変わっているかわからなくなる。

**原因**: 「とりあえず動けばいい」で進め、状態管理の設計を後回しにし続けた結果。AIが局所的に正しいコードを書いても、全体の整合性が崩れる。

**処方箋**: 月1回、状態管理のリファクタISSUEを作る。「この状態はどこで管理するべきか」を設計から見直す。T-04（リファクタテンプレート）を使ってAIと一緒に整理する。

---

## 11.5 MVPを絞れない

**症状**: 作り始めると追加したい機能が次々出てきて、いつまでもリリースできない。完成のゴールが見えない。

**原因**: 「全部必要だ」という思い込み。「これはMVP後でいい」という判断ができていない。

**処方箋**: T-01（要件定義テンプレート）を開発開始前に必ず使う。「なければアプリが成立しないか？」という問いで機能を篩にかける。答えがNoなら全部MVP後だ。OgoRouletteはMVPの機能を3つに絞ったから2週間でリリースできた。

---

## 11.6 評価せずに改善する（ver.2追加）

**症状**: 「感覚的に悪いと思ったので直した」という改善が続く。何が改善したかわからない。スコアが上がっているのか下がっているのかも不明。

**原因**: 評価フレームワークなしに改善を繰り返している。「よくなった気がする」という感覚だけで進んでいる。

**処方箋**: T-08（評価フレームワークテンプレート）を使って月1回評価する。改善前と改善後のスコアを比較することで、「この施策は効いた」「この施策は効かなかった」が数値でわかる。

---

## 11.7 全部同時に改善しようとする（ver.2追加）

**症状**: G-STACK・HEART・AARRRの全軸を一度に改善しようとする。結果として全部中途半端になり、スコアが全然上がらない。

**原因**: 「全部大事だから全部やる」という思考。優先順位をつける勇気がない。

**処方箋**: 最低軸を1つ特定して、そこだけに集中する。「今月はHEARTのHappinessだけ上げる」という決定をする。他の軸は現状維持でいい。集中することでスコアが大きく動く。

---

## 11.8 フレームワークを変えないまま進む（ver.2追加）

**症状**: フェーズが変わったのに、ずっとG-STACKだけで評価している。「動いているか」は確認しているが「楽しいか」「広がるか」は確認していない。

**原因**: 最初に覚えたフレームワークで安定してしまい、切り替えの必要性を感じていない。

**処方箋**: 月1回の評価時に「今のフェーズはどこか」を確認する。フェーズ3以降はHEARTを必ず組み込む。フレームワークを固定することは、評価の死角を増やすことだ。

# 第12章: 成功パターン

## 12.1 小さく回す

1ループ1改善の徹底が、最も重要な成功パターンだ。

```
良いループ（小さい）:
ISSUEを1個書く → AIに渡す → 実装する → 検証する → 次のISSUEへ

悪いループ（大きすぎる）:
10個のISSUEをまとめて実装しようとする
→ AIへの指示が複雑になる
→ 何が原因でバグるかわからなくなる
→ 検証が複雑になる
→ 戻し方がわからなくなる
```

小さく回すことで、「改善した事実」が積み上がる。毎日1〜3個のISSUEを消化する習慣が、4ヶ月で200 ISSUEになった。

## 12.2 検証を必ず入れる

「実装 → 検証」をセットにすること。検証なしの実装は「やった気分」だけで品質は上がらない。

gstackによる数値化が有効な理由は「感覚ではなく数値で議論できる」からだ。「なんかUXが悪い」ではなく「happiness scoreが45点」という具体的な指標があると、改善の優先度が決めやすい。

検証のタイミング:
- 各ISSUEの実装完了後
- 大きな機能追加後
- リリース前
- 月次の品質確認

## 12.3 感情を設計に入れる

「機能が動く」と「使いたくなる」の間には大きな差がある。この差を埋めるのが感情設計だ。

**感情設計のプロセス**:

1. 「ユーザーがアプリを使い始めてから終わるまでの感情曲線」を書く
2. 「最も盛り上がってほしい瞬間」を特定する
3. その瞬間の演出を設計する
4. AIと実装する

OgoRouletteでの感情設計:
- ルーレットが止まる直前: ドキドキ（減速アニメーションで表現）
- 当選者発表の瞬間: 驚き + 爆笑（confetti + 大きな表示で表現）
- 奢り決定後: 共有したくなる（WinnerCard + シェアボタンで表現）

## 12.4 ISSUEをストックする

「思いついたらすぐISSUE化する」習慣が、長期的な品質向上の源泉だ。

アプリを使っていて「ここが気になる」と思った瞬間に、その場でISSUEを書く。シャワーを浴びていて「あの問題はこうすれば直るかも」と思ったら、すぐメモしてISSUEにする。

ISSUEはストックするほど価値が上がる。「次に何を改善すべきか」に迷う時間がなくなる。ISSUEリストを見れば、常に「次の1手」が明確だ。

OgoRouletteでは、常に20〜30個のISSUEがBacklogにある状態を維持していた。これが「手が止まる」という状況を完全になくした。

## 12.5 評価結果をフェーズ化する（ver.2追加）

評価スコアを出して終わりにしない。**評価 → フェーズ特定 → 優先ISSUE決定 → 実装** というサイクルで回す。

```
評価駆動の成功パターン:
① T-08で月1回評価する
② 最低軸を特定する
③ 最低軸を改善するISSUEを3〜5個作る
④ そのISSUEだけに集中する（他は現状維持）
⑤ 翌月また評価する
⑥ スコアの変化量を確認する
```

OgoRouletteでは、この方法でHEARTのHappinessを45点から87点に引き上げた。「感覚的に改善する」より「数値を見て集中投資する」の方が、結果が速く出る。

# 第13章: バズるアプリ設計

## 13.1 体験設計

バズるアプリは「最高の瞬間」が設計されている。ユーザーが「わー！」と思う瞬間を、意図的に設計する。

```
体験設計の手順:

① アプリを使う「一連のストーリー」を書く
   例: グループで集まる → ルーレットを起動 → 参加者が入る → 回す → 決まる → シェア

② ストーリーの中で「最高の瞬間」を特定する
   例: 「ルーレットが止まって当選者が発表される瞬間」

③ その瞬間を最大化する演出を設計する
   例: 減速 → 停止音 → confetti → 大きな名前表示 → ファンファーレ

④ 「シェアしたくなる瞬間」を作る
   例: WinnerCard のスクショ → OG画像生成 → シェアボタン
```

## 13.2 感情設計

ドキドキ・驚き・笑いを仕込む設計だ。

**ドキドキ**: 「どうなるかわからない」不確実性を演出する。OgoRouletteでは、ルーレットが止まる直前に減速を段階的にすることで「もしかして自分か...!?」という緊張感を作った。

**驚き**: 「まさかこうなるとは」という意外性を設計する。WinnerCardの紙吹雪は「アプリがこんな反応をするとは思わなかった」という驚きを生む。

**笑い**: グループで使うアプリは「笑い」が最強の口コミエンジンだ。OgoRouletteでは当選者のリアクションがSNSに投稿されると、それ自体が宣伝になる。

## 13.3 SNS設計

「シェアしたくなる瞬間」を設計してからアプリを作る。後からSNS機能を追加しても遅い。

OgoRouletteのSNS設計:

1. **WinnerCard**: 当選者名が大きく表示されたカード。スクリーンショットがそのままSNS映えする
2. **動的OG画像**: Next.jsの `ImageResponse` で当選者名入りのOG画像を生成。URLをSNSにシェアすると自動で表示される
3. **ハッシュタグ**: `#OgoRoulette #奢り決め` を投稿文に自動で含める
4. **ルームURL共有**: 「このURLから入れば一緒にできるよ」と友達を招待できる

**プロダクトがマーケティングになる**設計にすることで、広告費ゼロで拡散できる。

## 13.4 OgoRouletteの体験設計図

```
[ユーザーがURLを開く]
    ↓ (LP: 「飲み会の奢り、これで決めよう」)
[使い方がわかる]
    ↓ (3ステップで説明。30秒で理解できる)
[ルームを作る / 参加する]
    ↓ (ルームコード4文字。タイポしない長さ)
[参加者が集まる]
    ↓ (リアルタイムで参加者リストが更新される。「おー来た来た」感)
[ルーレットを回す]
    ↓ (回転音 + 速いスピン。目が離せない)
[徐々に減速]
    ↓ (「誰だ...誰だ...」という緊張感)
★ 当選者発表 ★  ← 最高の瞬間
    ↓ (confetti + 大きな名前 + ファンファーレ)
[WinnerCardが表示される]
    ↓ (「シェアする」ボタンが自然に目に入る)
[SNSにシェア]
    ↓ (OG画像付きでURLがシェアされる)
[友達がURLをタップ]
    ↓ (次のユーザーがLPに来る)
[ループ]
```

この設計図を最初に書いてから開発した。「どこが最高の瞬間か」を知っているから、その瞬間に全力を投資できた。

## 13.5 HEART/AARRR/Kanoで体験設計を評価する（ver.2追加）

体験設計の各ステップを、評価フレームワークの軸と対応させることで、「どのステップが評価に影響するか」が明確になる。

| 体験ステップ | 評価軸 | 確認する問い |
|---|---|---|
| LPを開く | AARRR - Acquisition | どこから来たか。LPに来る人が増えているか |
| 使い方を理解する | HEART - Task Success | 30秒で理解できるか。初回スピンまでの離脱率は |
| ルームを作る/参加する | HEART - Adoption | ルーム機能を試してみようとするか |
| ルーレットを回す | HEART - Happiness | 回している間、楽しいか。緊張感があるか |
| 当選者発表 | HEART - Happiness | 「わあ！」という声が上がるか。盛り上がるか |
| SNSにシェア | AARRR - Referral | 自発的にシェアしようとするか |
| また使いに来る | AARRR - Retention | 翌週また使いに来るか |
| 友達を招待する | Kano - 魅力品質 | 「これ面白いよ」と友達に教えたくなるか |

体験設計を評価軸と紐付けることで:
- 「最高の瞬間」を設計する際に「どの軸のスコアを上げたいか」が明確になる
- 演出改善がHEART-Happinessに直結することが事前に予測できる
- シェア機能強化がAARRR-Referralに直結することが設計時からわかる

# 第14章: まとめ

## 14.1 AI駆動開発の本質

**1行で言うなら**: 「思考を言語化して、AIと協働して、人間が判断する」

**4行で言うなら**:
1. AIは「速く書けるツール」ではなく「自分が言語化できていないことを実装に落としてくれるパートナー」だ
2. ボトルネックはコーディング速度ではなく「思考の言語化速度」だ
3. アーキテクチャ・状態管理・UX・ビジネス判断は常に人間が握る
4. 評価フレームワーク（G-STACK/HEART/AARRR/Kano）でプロダクトを数値化し、「何を」「なぜ」改善するかを客観的に決める

この4行を体に入れてからAIを使うと、使い方が根本から変わる。

## 14.2 OgoRouletteから学んだ発見

**発見1: 評価は課題発見エンジン**

G-STACKスコアが上がっても、HEARTを入れたら「楽しくない」という課題が見えた。AARRRを入れたら「拡散していない」という課題が見えた。フレームワークを増やすたびに、見えていなかった課題が次々と発見された。評価は「答え合わせ」ではなく「課題発見エンジン」だ。

**発見2: フレームワークが違えば見えるものが違う**

G-STACKだけを使っていた時期と、HEART/AARRRを加えた時期では、「次に何をすべきか」の見え方が根本的に変わった。同じアプリを同じ目で見ているのに、フレームワークを変えることで死角が消える。これがフレームワークの価値だ。

**発見3: 評価フレームワークは進化させるもの**

フェーズ1でG-STACKだけ使って正解だった。でもフェーズ3でまだG-STACKだけ使っていたら、「誰も笑わなかった夜」の問題は永遠に見えなかった。フレームワークは「一度決めたら固定するもの」ではなく「フェーズに合わせて進化させるもの」だ。

## 14.3 再現方法

OgoRouletteと同じことを再現するための5ステップだ。

**ステップ1: CLAUDE.mdを書く**
プロジェクトの「開発憲法」を最初に作る。技術スタック・禁止事項・命名規則・AIへの指示ルールを書く。このファイルがAIとのセッションをまたいでも一貫性を保つ。

**ステップ2: T-01でMVPを絞る**
`templates/t01-requirements.md` を使って、MVPの範囲を3機能以内に絞る。「なければアプリが成立しないか？」という問いで篩にかける。

**ステップ3: 最初のISSUEを書く**
MVP最初の機能のISSUEを書く。第3章のフォーマットを使って、背景・問題・解決策・受け入れ条件を書く。

**ステップ4: ループを1周回す**
ISSUEを書く → テンプレートにはめてAIに渡す → 実装する → gstack評価する → 次のISSUEへ。

**ステップ5: 月1回T-08で評価してフェーズ化を実装する**
`templates/t08-evaluation.md` を使って月1回の定期評価を行う。スコアの最低軸を特定し、そこを改善するISSUEをフェーズの優先課題にする。4ヶ月続ければ200 ISSUEを超える。200 ISSUEは「200回の改善」の証拠だ。

## 14.4 この教科書がver.3になるとき

ver.2を書いている今、OgoRouletteはフェーズCへの移行期にある。

フェーズC（Referral軸の強化・バイラル設計）の実装が進んだとき、ver.3で追加されるであろう内容がある。

- **フェーズC実装記録**: AARRRのReferral軸を60点→80点に引き上げた施策の詳細
- **Kano × AARRR の組み合わせ実践**: 魅力品質を特定してReferral軸に直結させるプロセス
- **スコア推移の完全記録**: フェーズ1〜C全期間のスコアグラフ
- **コミュニティとの共創**: 外部フィードバックをISSUEに変換するプロセス

**ver.3が出るとき、それはOgoRouletteがさらに良くなった証拠だ。**

# 付録

## フレームワーク切り替え早見表

| 状況 | 使うフレームワーク | 優先する評価軸 |
|---|---|---|
| アプリが動いていない・基盤が弱い | G-STACK | Architecture, Risk |
| 動いているが本番で壊れる | G-STACK + 動作検証 | Risk + クロスブラウザ |
| 動いているが「楽しくない」 | HEART | Happiness, Engagement |
| 機能があるのに使われない | HEART | Adoption, Task Success |
| 来るが続かない | HEART + AARRR | Retention |
| 広がらない | AARRR + Kano | Referral + 魅力品質 |
| 「次に何を作るか」がわからない | Kano | 魅力品質の発見 |

## G-STACK / HEART / Kano / AARRR チェックリスト

### G-STACK チェックリスト

```
□ Goal：誰のための何のアプリか言語化できているか
□ Strategy：差別化の方向が決まっているか
□ Tactics：実装がその方向に沿っているか
□ Architecture：DB・API・コンポーネントが健全か
□ Risk：バグ・セキュリティ・パフォーマンスのリスクが管理されているか
```

### HEART チェックリスト

```
□ Happiness：使った後、満足感があるか
□ Engagement：セッション中に何度も操作したくなるか
□ Adoption：新機能を試してみようとするか
□ Retention：また来てくれるか
□ Task Success：ユーザーの目的が達成できているか
```

### Kano 分類ガイド

```
当たり前品質：「これがないと使えない」機能
  → バグ修正・基本機能の完成に注力

一元的品質：「多いほど嬉しい」機能
  → ユーザーの声を聞きながら改善

魅力品質：「あったら最高だった！」機能
  → 差別化・バズりのコアに据える
```

### AARRR 診断フロー

```
Acquisition（流入）が弱い → SEO・SNS・紹介施策
Activation（初体験）が弱い → オンボーディング・初回UX改善
Retention（継続）が弱い → 通知・リマインダー・コンテンツ更新
Revenue（収益）が弱い → 課金設計・プレミアム機能
Referral（紹介）が弱い → シェア機能・インセンティブ設計
```

## 加重スコア計算式

```
フェーズ1: 総合スコア = G-STACK × 1.0
フェーズ2: 総合スコア = G-STACK × 0.8 + 動作検証スコア × 0.2
フェーズ3: 総合スコア = G-STACK × 0.5 + HEART × 0.5
フェーズ4: 総合スコア = G-STACK × 0.4 + HEART × 0.3 + AARRR × 0.3

OgoRouletteで使用した式（現在フェーズB-C移行期）:
総合スコア = G-STACK × 0.5 + HEART × 0.3 + AARRR × 0.2
```

## フレームワーク用途サマリ表

| フレームワーク | 何を測るか | 使うタイミング | 代表的な問い |
|---|---|---|---|
| G-STACK | 構造の正しさ | 常に（全フェーズ） | このアプリは設計上成立しているか |
| HEART | 感情の豊かさ | フェーズ3以降 | このアプリは使って気持ちいいか |
| AARRR | 成長の健全さ | フェーズ4以降 | このアプリは広がっているか |
| Kano | 機能の価値分類 | フェーズ4 | どの機能が魅力品質か |
