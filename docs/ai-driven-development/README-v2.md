# 第1章: はじめに

## 1.1 AI駆動開発とは何か

AI駆動開発とは、**AIをペアプログラマーとして中心に据えた開発手法**だ。「AIにコードを書いてもらう」のとは根本的に違う。

AIは「速く書けるツール」ではなく、「自分が言語化できていないことを実装に落としてくれるパートナー」だ。AIは文脈を理解し、候補を出し、実装を提案する。しかし、**何を作るか・なぜ作るか・品質を判断するのは常に人間の仕事**だ。

AI駆動開発の本質は4つある。

1. **思考の言語化**: 頭の中の「なんかおかしい」をISSUEとして言語化する
2. **AIへの文脈転送**: 言語化した文脈をAIに正確に渡す
3. **人間による判断**: AIの提案を評価して採用・却下を決める
4. **フレームワークで数値化**: 品質を感覚ではなく数値で把握する（ver.2 追加）

OgoRouletteでは、この4ステップを200回以上繰り返した。毎回のISSUEがドキュメントになり、そのドキュメントが次のISSUEを生む。**品質は積み上がる**のだ。

## 1.2 なぜ今必要か

2026年現在、個人開発の民主化が加速している。かつては「フルスタックエンジニア+デザイナー+PM」の3人チームで数ヶ月かかっていたアプリが、AIを使えば1人で数週間で作れる時代になった。

しかし、多くの個人開発者が「AIを使っているのに品質が上がらない」「AIに丸投げしたら意図しないものが出てきた」「どこを改善すればいいかわからない」という壁に当たっている。

この壁の正体は2層ある。

**第1の壁**: 「思考の言語化の問題」。AIは賢い。しかし、曖昧な指示には曖昧な実装を返す。「なんかいい感じにして」では何も生まれない。

**第2の壁（ver.2 で重点解説）**: 「評価の不在の問題」。実装したが品質が上がっているかわからない。どこを優先して直せばいいかわからない。この壁は「評価フレームワーク」で突破できる。

## 1.3 従来開発との違い

| 観点 | 従来開発 | AI駆動開発 ver.1 | AI駆動開発 ver.2（本書） |
|---|---|---|---|
| 設計 | 人間が全部設計してから実装 | ISSUEを書いてAIと一緒に設計 | 同左 |
| 実装速度 | 人間が一行ずつ書く | AIがドラフトを生成、人間がレビュー | 同左 |
| 品質管理 | テスト・レビューのみ | gstack評価→数値化→改善ループ | **3フレームワーク評価→加重スコア→フェーズ化** |
| 改善の優先度 | 経験と直感 | gstackの指摘 | **G-STACK/HEART/AARRRの軸別スコアで明確化** |
| ボトルネック | コーディング速度 | 思考の言語化速度 | 思考の言語化速度 + **評価の精度** |

## 1.4 この教科書でできること

この教科書を読み終えたとき、あなたには以下が手に入る。

- **AI駆動開発の全体像**: 何をどの順序でやるか、なぜその順序なのかが理解できる
- **ISSUEの書き方**: 「思考の言語化」の技術。一生使えるスキルだ
- **評価フレームワークの使い方**: G-STACK / HEART / AARRRで品質を数値化する方法
- **フェーズ化戦略**: 評価結果からフェーズA/B/Cを設計して体系的に改善する方法
- **テンプレート8種**: コピペで即使える。要件定義から評価フレームワークまで
- **プロンプト20+種**: 場面別に選べる。試行錯誤なしで使える
- **ケーススタディ**: 実際のバグ・設計・最適化・本番障害の事例

# 第2章: 全体フロー（最重要）

## 2.1 AI駆動開発エンジン ver.2

AI駆動開発の核心は、以下のループだ。ver.2では評価ループが強化されている。

```
┌───────────────────────────────────────────────────────────┐
│               AI駆動開発エンジン ver.2                     │
│                                                           │
│  ① 課題発見                                              │
│      「なんかおかしい」「スコアが低い」「バグ報告」      │
│         ↓                                                 │
│  ② ISSUE化（思考の言語化）                               │
│      背景・問題・原因・解決策・受け入れ条件を書く        │
│         ↓                                                 │
│  ③ AI解決案生成                                          │
│      テンプレート + プロンプトでAIに渡す                 │
│         ↓                                                 │
│  ④ 実装（AI + 人間の判断）                               │
│      AIの提案を評価・修正・採用する                      │
│         ↓                                                 │
│  ⑤ 評価（G-STACK / HEART / AARRR）★ ver.2 強化 ★       │
│      3フレームワークで品質を多角的に数値化する           │
│         ↓                                                 │
│  ⑥ フェーズ化・優先度付け ★ ver.2 追加 ★               │
│      スコアを元にフェーズA/B/Cを設計する                 │
│         ↓                                                 │
│  ⑦ 再ISSUE化                                             │
│      評価で見つかった新課題をISSUEにする                 │
│         ↓                                                 │
│  ① に戻る（品質が積み上がる）                           │
└───────────────────────────────────────────────────────────┘
```

OgoRouletteでは4回の評価ラウンドを実施した。1ラウンドで数十のISSUEが生まれ、それを消化してスコアが上がる。スコアが上がると次の評価で新しい課題が見つかる。**評価は課題発見エンジン**だ。

## 2.2 各ステップの詳細解説

### ① 課題発見

「なんかおかしい」「ユーザーが戸惑いそう」「パフォーマンスが悪い気がする」という直感を大切にする。この直感を**その場でメモ**することが重要だ。

ver.2での追加: 評価スコアが課題を自動的に発見してくれる。HEART Retentionが12/20なら「再訪設計が弱い」という課題が数値で明示される。直感に加えて、**数値が課題を教えてくれる**ようになる。

### ② ISSUE化（思考の言語化）

課題を発見したら、すぐISSUEに落とす。ISSUEは「思考の結晶」だ。良いISSUEには5要素がある（詳細は第3章参照）。

### ③ AI解決案生成

ISSUEが書けたら、テンプレートにはめてAIに渡す。評価レポートそのものをAIに渡して「このスコアを改善する施策を優先度付きで提案してほしい」という使い方が特に効果的だ。

### ④ 実装（AI + 人間の判断）

AIの提案は「叩き台」だ。評価スコアを念頭に置きながら、「これはどの軸のスコアに効くか」を考えながら実装判断する。

### ⑤ 評価（G-STACK / HEART / AARRR）

実装したら3フレームワークで評価する。「感覚では良くなった気がする」では品質は上がらない。数値で見ることで、「HEARTは上がったがAARRRが伸びていない」など、改善の方向性が見える。

**第5章で詳しく解説する。**

### ⑥ フェーズ化・優先度付け

評価結果から「フェーズA（バグ修正）」「フェーズB（UX完成度）」「フェーズC（成長設計）」のように、改善をフェーズ分けして優先度を設計する。全部一度にやろうとすると混乱するが、フェーズ化すると「今はフェーズAだけ集中する」という判断が明確になる。

## 2.3 ループを速く回すコツ

**1ループ = 1〜2個の改善に絞ること**。これが最も重要なルールだ。

OgoRouletteで学んだ粒度感:

- 良い粒度: 「スピン後のアニメーション演出を追加する」
- 大きすぎる粒度: 「UXを全体的に改善する」
- 小さすぎる粒度: 「ボタンの色を#FF6B6Bから#FF5A5Aに変える」

「大きすぎる」は分割する。「小さすぎる」は関連するものとまとめる。1ISSUEが1セッションで完結するくらいの粒度が理想だ。

# 第3章: ISSUE駆動開発

## 3.1 ISSUEの役割

ISSUEは単なるTODOリストではない。**思考の結晶**だ。

ISSUEには4つの機能がある。

1. **思考の整理**: 書くことで問題が明確になる
2. **AIへの文脈転送**: 書いたISSUEをそのままAIに渡せる
3. **開発ログ**: 「なぜこの実装にしたか」が残る
4. **品質の証拠**: 200 ISSUEを消化した事実が品質保証になる

## 3.2 タスクとISSUEの違い

| 観点 | タスク | ISSUE |
|---|---|---|
| 粒度 | 大きい・あいまい | 一つの問題・一つの改善 |
| 背景 | 書かない | 「なぜ今やるか」を必ず書く |
| 原因分析 | しない | 可能な限り原因を書く |
| 受け入れ条件 | 「完了したら消す」 | 「どうなれば完了か」を明記 |
| AIへの使いやすさ | そのままでは使えない | AIへの指示書として完成している |
| 残り方 | 消える | ログとして残る |
| **評価との紐付け** | なし | **「このISSUEはHEART Retentionを改善する」と紐付く** |

## 3.3 良いISSUEの条件

良いISSUEには5つの条件がある。

**条件1: 背景が書かれている**
「なぜ今これを解決するのか」が書かれている。

**条件2: 問題が具体的だ**
「おかしい」「遅い」ではなく、「iOS Safari 17.2でルーレットのスピンアニメーションが途中で止まる」のように具体的だ。

**条件3: 原因の仮説がある**
原因がわからなくても「おそらく〜だと思う」という仮説を書く。AIはこの仮説を検証してくれる。

**条件4: 解決策の方向性がある**
「〜の方向で修正したい」という方向性を書く。完璧な解決策でなくていい。

**条件5: 受け入れ条件が明確だ**
「〜の状態になればこのISSUEは完了」が書かれている。

**条件6（ver.2 追加）: 評価軸との紐付けがある**
「このISSUEはG-STACK の Risk スコアを改善する」「HEART Happiness に効く」のように、どの評価軸を改善するかを書く。優先度判断が劇的に速くなる。

## 3.4 ISSUE記述フォーマット（ver.2 更新版）

```markdown
## ISSUE-XXX: [タイトル]

### ステータス
🔴 未着手 / 🟡 進行中 / ✅ 完了

### 優先度
Critical / Major / Minor

### 評価軸（ver.2 追加）
G-STACK: [Risk / Architecture / etc.] +N点見込み
HEART:   [Retention / Happiness / etc.] +N点見込み
AARRR:   [Referral / Revenue / etc.] +N点見込み

### 背景
[なぜ今これを解決するのか]

### 問題
[何が起きているか。再現手順があれば書く]

### 原因
[なぜそうなっているか。仮説でもよい]

### 解決策
[どう直すか。方向性だけでも書く]

### 受け入れ条件
- [ ] [条件1]
- [ ] [条件2]
- [ ] [テスト通過・ビルド成功]
```

## 3.5 ISSUEの粒度管理

**思考の単位で切る**のが基本原則だ。1セッションで完了できるかを判断基準にする。

```
悪い例（大きすぎる）:
ISSUE-50: モバイル対応を全体的に改善する

分割後（良い粒度）:
ISSUE-51: スマホでのルーレット表示サイズを最適化
ISSUE-52: iOS Safariでの音声再生バグを修正
ISSUE-53: タッチ操作でのスワイプ誤反応を防止
ISSUE-54: 縦向き・横向きの切り替え時のレイアウト崩れを修正
```

# 第4章: OgoRoulette開発ストーリー

## 4.1 Day 0: アイデア誕生

「飲み会で誰が奢るか決めるのって、毎回揉めるよな」

このひとことが始まりだった。MVPとして絞った機能は3つだけだった。

1. 参加者のリストを作る
2. ルーレットを回す
3. 当選者を表示する

リアルタイムルーム、割り勘計算、WinnerCard、LPはすべてMVP後の機能として後回しにした。**アイデアが生まれたら即T-01（要件定義テンプレート）**を開く。これがAI駆動開発の初動だ。

## 4.2 Week 1-2: MVP構築

CLAUDE.mdを最初に書いた。これはAIとのセッションをまたいでも、一貫した判断基準を保つための「開発憲法」だ。

```markdown
# OgoRoulette — 開発ルール（CLAUDE.md）

## 絶対に守ること
- 頼んだ箇所以外は変更しない
- 実装前に「何を・なぜ・どう実装するか」を一言確認する
- 理解できないコードは書かない

## 禁止事項
- router.back() 禁止（Link href を使う）
- useEffect の依存配列を空にするだけのhack禁止
- 状態管理を props drilling で解決しない
```

最初のルーレットは2日で動いた。AIがコンポーネントのドラフトを生成し、人間がロジックと状態管理を設計した。

## 4.3 Month 1: リアルタイムルーム

「全員が同じルーレットを見る」にはリアルタイム同期が必要だった。AIが出した3つの設計案（① 全状態をDBに持つ、② ブロードキャストのみ、③ ハイブリッド）を評価して、③を採用した。

AIは候補を出した。選択は人間がした。**この分担こそがAI駆動開発の本質だ。**

## 4.4 Month 2: バグとの格闘

Month 2は**バグの月**だった。特に3つが深刻だった。

**Race Condition（ISSUE-001）**: 複数ユーザーが同時にボタンを押すと状態が競合。ロックトークンパターンで解決。

**iOS Safari問題（ISSUE-035）**: iOSでアニメーションが途中で止まる。`will-change: transform` を追加して解決。

**Phase管理の複雑化（ISSUE-055〜070）**: `idle → spinning → stopping → stopped → winner_display` と状態が増えるにつれバグが多発。状態機械パターンへのリファクタで解決。

## 4.5 Month 3: UX強化フェーズ

gstack評価を走らせたところ、スコアが62点だった。具体的な指摘:
- ルーレットが止まる瞬間の演出が弱い
- 勝者表示が地味すぎて「盛り上がらない」
- LCP: 4.2s と遅い

WinnerCardの設計がこの中で最も重要だった。「奢りが決まった瞬間の祝祭感を最大化する」というゴールを設定して、AIと感情設計を共創した。紙吹雪・ファンファーレ・震えるアニメーション・シェアボタンが一体になったWinnerCardはこのフェーズで生まれた。Lighthouseスコアは最終的に95点まで上がった。

## 4.6 Month 4: 評価フレームワーク導入（ver.2 新規）

Month 4の転換点は「評価フレームワークを3つに増やした」ことだ。

それまでは「gstack」という単一フレームワークで評価していた。しかし「何が良くて何が悪いかが曖昧」という問題があった。UXは改善されたが、ビジネス観点や技術品質観点が抜け落ちていたのだ。

第4回評価から **G-STACK / HEART / AARRR の3フレームワーク**を導入した。

```
総合スコア = G-STACK(50%) + HEART(30%) + AARRR(20%)
```

この加重計算式を使うことで「G-STACKは良いがAARRRが低い」という多次元的な判断ができるようになった。評価レポートを作成してAIに渡すと「このスコアを改善するための施策」を優先度付きで提案してもらえる。

**スコア推移**:

| 評価ラウンド | G-STACK | HEART | AARRR | 総合 |
|------------|---------|-------|-------|------|
| 第1回（Month 3）| 64 | — | — | 64 |
| 第2回（Month 4初） | 68 | 69 | 50 | 65 |

## 5.1 なぜ評価フレームワークが必要か

個人開発者が「品質を上げたい」と思うとき、何を基準にするか。

- Lighthouseスコア？ → パフォーマンスしか測れない
- ユーザーの感覚？ → 主観で再現性がない
- 機能数？ → 量が品質ではない

評価フレームワークが必要な理由は3つだ。

**理由1: 改善の優先度が数値で決まる**
「HEARTのRetentionが12/20で最低軸」という事実があれば、「次はRetention改善に集中する」という判断ができる。数値がない状態では「なんとなく全体を改善する」になり、焦点が散る。

**理由2: AIへの指示精度が上がる**
「アプリを改善してください」より「AARRRのReferralスコアが12/20です。シェアループを強化する施策を3つ提案してください」の方が、AIは10倍質の高い提案を返す。評価スコアはAIへの「文脈の地図」になる。

**理由3: 改善の事実が蓄積できる**
「第2回評価からHEART Happinessが72→79に改善した」という具体的な事実が残る。なぜ上がったかも追跡できる。これが「データ駆動改善」だ。

## 5.3 HEART フレームワーク（Google UX評価）

## 5.4 AARRR フレームワーク（Pirate Metrics）

AARRRは「海賊指標（Pirate Metrics）」とも呼ばれる、スタートアップの成長を5段階で測るフレームワークだ。

```
AARRR = Acquisition + Activation + Retention + Revenue + Referral
```

### 各軸の意味

| 軸 | 測定対象 | 問い |
|----|---------|------|
| **Acquisition** | ユーザーの獲得 | 初めてのユーザーがどこから来るか |
| **Activation** | 価値体験の実現 | 初訪問で「これは良い」と思えるか |
| **Retention** | 継続利用 | 翌日・翌週・翌月も戻ってくるか |
| **Revenue** | 収益化 | どこでお金が発生するか |
| **Referral** | 紹介・口コミ | ユーザーが他の人を連れてくるか |

### AARRRのポイント: HEARTとの違い

HEARTは「ユーザーの体験品質」を測る。AARRRは「ビジネスの成長動力」を測る。

- HEART Retention = 「長期的に使い続けるか（体験の質）」
- AARRR Retention = 「実際に戻ってきているか（行動の事実）」

同じ「Retention」でも視点が違う。これが3フレームワークを使う理由だ。

### OgoRoulette AARRR実例（第4回評価）

| 軸 | スコア | 主な根拠と改善ポイント |
|----|--------|---------------------|
| Acquisition | 12/20 | SEO keywords 拡充。Google Search Console 未申請が継続 |
| Activation | 15/20 | ゲストモード即スタートは良好。ルーム機能への誘導が弱い |
| Retention | 12/20 | 履歴表示で+2。プッシュ通知・リマインダーなし |
| Revenue | 5/20 | 完全無料。マネタイズ設計なし |
| Referral | 12/20 | シェア機能は基本実装済み。OGP静止画シェアカードが未完成 |

AARRRの Revenue が5/20と突出して低い。これはビジネスとして持続不可能なことを示す。「個人の趣味プロジェクトとして許容するか」「収益化を目指すか」という方針決定が必要だ。このような「ビジネス判断」をAIへの指示文の前提として渡すと、施策の提案が現実的になる。

## 6.3 ISSUE-201: Vercel KV × レートリミット（ver.2 新規）

### 背景

G-STACK Risk スコアが12/20と低い主因がこのバグだった。`lib/rate-limit.ts` が `globalThis` でメモリ内Mapを管理しているが、Vercel Serverless Functions は並行リクエストを異なるインスタンスで処理するため、インスタンス間でMapが共有されない。実質的にレートリミットが機能していなかった。

```typescript
// 問題のコード
const store: Map<string, RateLimitEntry> =
  globalForRateLimit.rateLimitStore ?? new Map()
// ↑ 複数インスタンスでは共有されない
```

### 解決策: Vercel KV + メモリフォールバック

```typescript
// KVが設定されている環境ではKVを使う
if (KV_CONFIGURED) {
  const count = await kv.incr(windowKey)
  if (count === 1) await kv.expire(windowKey, Math.ceil(windowMs / 1000))
  return { allowed: count <= limit, remaining: Math.max(0, limit - count), ... }
}
// ローカル開発ではメモリフォールバック
return checkRateLimitMemory(ip, endpoint, limit, windowMs)
```

### AI活用のポイント

G-STACKレポートをAIに渡して「Risk: 12/20の主因はrate-limitが無効であることです。Vercel Serverless環境で機能するrate-limit実装を提案してください」と指示したところ、Vercel KV採用が即座に提案された。評価スコアを渡すことで、AIの提案が「症状への対処」から「根本原因への対処」に変わった。

### 学び

クラウド環境のサーバーレス関数では「インスタンス非共有」が前提だ。メモリ内ステートに依存した設計は本番環境で機能しない。KVやRedisなど外部ストアを使う設計を最初から検討する。

## 6.4 ISSUE-202: NTP時刻同期 × マルチプレイ（ver.2 新規）

### 背景

HEART Task Success が17/20に留まった主因。ホストと参加者の端末間でシステム時刻が1〜3秒ずれている場合、演出開始タイミングが同様にズレる。ホストが `/api/rooms/[code]/spin` をPOSTした際にのみ `clockOffsetMsRef` がセットされ、メンバークライアントでは常に0だった。

### 解決策: NTP方式のクロック同期

```typescript
// app/api/time/route.ts（新規エンドポイント）
export const runtime = "edge"
export function GET() {
  return new Response(null, {
    status: 204,
    headers: {
      "X-Server-Time": Date.now().toString(),
      "Cache-Control": "no-store",
    },
  })
}

// use-spin.ts（メンバー用時刻同期）
useEffect(() => {
  if (isOwner) return
  const t0 = performance.now()
  const t0Date = Date.now()
  fetch("/api/time").then((res) => {
    const serverTime = Number(res.headers.get("X-Server-Time") ?? 0)
    const rtt = performance.now() - t0
    // RTTの半分を補正値として使う（NTP方式）
    const offset = serverTime - t0Date + Math.round(rtt / 2)
    clockOffsetMsRef.current = offset
  })
}, [isOwner])
```

### 学び

マルチプレイアプリでの「タイミング同期」は難しい問題だ。NTP（Network Time Protocol）の考え方（往復時間の半分をオフセットとして使う）を応用することで、±500ms 程度の精度でクロックを合わせられる。この問題はローカルでは発生しにくく「本番でのみ体感できるバグ」だ。HEARTのTask Successスコアが低いときは「基本タスクが確実に完了するか」をマルチプレイ観点で再確認する。

## 6.5 ISSUE-152: ChunkLoadError無限ループ（ver.2 新規）

### 背景

`RouletteWheel` を `next/dynamic` で動的インポートに変えたところ、デプロイ後のユーザーで以下の無限ループが発生した。

```
古い HTML (キャッシュ済み) → chunk 404 → ChunkLoadError
→ window.location.reload() → 古い HTML をキャッシュから取得
→ chunk 404 → ... （無限ループ）
```

### 解決策: 二段構え

**修正①**: `next/dynamic` を廃止して静的 import に戻す（根本解決）

```typescript
// 廃止（動的import）
const RouletteWheel = dynamic(() => import("@/components/roulette-wheel"))

// 元に戻す（静的import）
import { RouletteWheel } from "@/components/roulette-wheel"
```

**修正②**: error boundary の reload をキャッシュバイパス版に変更

```typescript
// 悪い: キャッシュをバイパスしない
window.location.reload()

// 良い: ランダムクエリでCDN/ブラウザキャッシュをバイパス
const RELOAD_KEY = "chunk-error-reload-count"
const data = JSON.parse(sessionStorage.getItem(RELOAD_KEY) ?? '{"count":0}')

if (data.count < 3) {
  sessionStorage.setItem(RELOAD_KEY, JSON.stringify({ count: data.count + 1 }))
  window.location.href = window.location.pathname + "?_r=" + Date.now()
}
// 3回超えたら通常エラー画面を表示（無限ループ防止）
```

### 学び

`next/dynamic` でのコード分割は Lighthouse が推奨するが、「デプロイごとにchunkハッシュが変わる」という性質がstale cache問題を引き起こす。Lighthouseスコア改善のための最適化が、本番での致命的バグにつながりうる。G-STACK Risk スコアの低下につながるトレードオフとして評価する。

## 6.6 パフォーマンス最適化: Lighthouse 62→95

| 指標 | Before | After |
|---|---|---|
| LCP (Largest Contentful Paint) | 4.2s | 1.1s |
| CLS (Cumulative Layout Shift) | 0.28 | 0.02 |
| Performance Score | 62 | 95 |

改善の主な施策:
1. `next/image` への移行と WebP 変換（LCP改善の主因）
2. `next/font` でフォントの先読みと CLS 防止
3. Realtimeの接続最適化（ルームに入ったときだけ接続・離脱時に切断）

# 第10章: プロの実践原則

実践原則の詳細は [`principles.md`](principles.md) を参照。ここではver.2で追加・更新された原則を抜粋する。

## 原則01〜07（ver.1から継続）

- **01**: ISSUEは思考単位で切れ
- **02**: AIに設計を任せるな
- **03**: 状態管理は人間が握れ
- **04**: バグは再現性から潰せ
- **05**: UXは最初に設計しろ
- **06**: AIは提案者であって決定者ではない
- **07**: 小さく作って小さく壊せ

## 原則31〜35（ver.2 新規追加）

### 原則31: 評価は月1回以上やれ

実装を続けるだけでは「何が良くなったか」が見えない。月1回は3フレームワーク評価を実施して現在地を把握する。OgoRouletteでは4回の評価で「何が足りないか」が毎回明確になった。

評価をサボると「なんとなく良くなっている気がする」という感覚だけが残る。数値がなければ改善の優先度が決まらない。

### 原則32: スコアは変化量で見ろ

「70点」という絶対値より「前回65点→今回70点（+5点）」という変化量の方が重要だ。何の施策が何点の改善をもたらしたかを追跡する。変化量を記録することで「次に5点上げるには何が必要か」の判断精度が上がる。

### 原則33: 最低軸を優先的に改善しろ

G-STACK/HEART/AARRRのすべての軸を同時に上げようとしてはいけない。最低スコアの軸に集中する。OgoRouletteでは AARRR Revenue 5/20 が突出して低かった。ここを放置したまま Happiness を上げても総合スコアへの貢献は小さい。

### 原則34: フレームワークの重みはビジネスに合わせろ

`G-STACK(50%) + HEART(30%) + AARRR(20%)` の重み付けはデフォルト値だ。マネタイズが重要なフェーズなら Revenue の重みを上げる。ユーザー獲得フェーズなら Acquisition を上げる。重み付けの変更はビジネス判断だ。AIに任せてはいけない。

### 原則35: 評価レポートはAIへの文脈として使え

評価レポートをAIに渡すと、「何を改善すべきか」の提案精度が10倍になる。「アプリを改善してください」という指示と「Risk: 12/20の主因がrate-limit無効です」という指示では、AIの回答の質が天と地ほど違う。評価レポートはAIへの「文脈の地図」だ。

# 第11章: よくある失敗

## 11.1 ISSUEが雑

**症状**: 「〇〇を直す」という1行のISSUEが多い。

**処方箋**: 第3章のISSUEフォーマットを使う。特に「受け入れ条件」と「評価軸との紐付け」を必ず書く。

## 11.2 AIに丸投げ

**症状**: AIが出したコードをそのままコミットする。

**処方箋**: AIのコードは必ずレビューする。特に「状態管理」「認証・認可」「非同期処理」の3領域は入念に確認する。

## 11.3 検証不足

**症状**: 「実装した→完了」で次に進む。

**処方箋**: 「ISSUEを閉じる前に評価を必ず走らせる」をルールにする。

## 11.4 評価せずに改善する（ver.2 追加）

**症状**: 「なんとなく良くなった気がする」が積み重なる。品質が上がっているか数値で確認していない。

**原因**: 評価を「別の作業」として捉えている。実装・評価・改善はセットだという認識がない。

**処方箋**: T-08（評価フレームワークテンプレート）を月1回使う。評価なき改善は「方向感のない努力」だ。

## 11.5 全部同時に改善しようとする（ver.2 追加）

**症状**: 評価スコアが出たら全軸を一気に改善しようとする。何も完了しないまま次の評価になる。

**処方箋**: フェーズ化する。「今週はフェーズAのバグ修正だけ」と絞ることで、完了の事実が積み上がる。OgoRouletteではフェーズA（バグ修正）→フェーズB（UX完成度）→フェーズC（成長設計）の順で実施した。

## 11.6 状態管理の放棄

**症状**: propsのバケツリレーが深くなる・useEffectが乱立する。

**処方箋**: 月1回、状態管理のリファクタISSUEを作る。T-04（リファクタテンプレート）を使ってAIと一緒に整理する。

## 11.7 MVPを絞れない

**症状**: 作り始めると追加したい機能が次々出てきて、いつまでもリリースできない。

**処方箋**: T-01（要件定義テンプレート）を開発開始前に必ず使う。「なければアプリが成立しないか？」という問いで機能を篩にかける。

# 第14章: まとめ

## 14.1 AI駆動開発 ver.2 の本質

**1行で言うなら**: 「思考を言語化して、AIと協働して、フレームワークで評価して、人間が判断する」

**4行で言うなら**:
1. AIは「速く書けるツール」ではなく「自分が言語化できていないことを実装に落としてくれるパートナー」だ
2. ボトルネックはコーディング速度ではなく「思考の言語化速度」だ
3. 評価フレームワーク（G-STACK/HEART/AARRR）は「改善の地図」だ。地図なき改善は方向感のない努力だ
4. アーキテクチャ・状態管理・UX・ビジネス判断・評価の解釈は常に人間が握る

## 14.2 OgoRouletteから学んだ最重要な発見

**発見1: 評価は課題発見エンジンだ**

評価をやるまで「なんとなく良くなっている」という感覚しかなかった。第4回評価でBUG-01〜06が発見されたとき、「評価なしではこれらは見えなかった」と気づいた。評価は「改善の確認」だけじゃない。「次の課題の発見」が評価の最大の価値だ。

**発見2: フレームワークが違えば見えるものが違う**

G-STACKだけでは「ユーザーが楽しいか」がわからない。HEARTだけでは「収益化の道筋」がわからない。AARRRだけでは「エンジニアリング品質」がわからない。3つを組み合わせることで、初めてプロダクトの全体像が見える。

**発見3: ISSUEと評価軸を紐付けると優先度が自明になる**

「このISSUEはHEART Task Successを+2点改善する。あのISSUEはAARRR Revenue+3点見込み。」という紐付けができると、「何から手をつけるか」に迷わない。ISSUEの評価軸への紐付けは、最初は面倒に感じるが、一度習慣化すると優先度判断にかかる時間が劇的に減る。

## 14.3 再現方法: 5ステップ

**ステップ1: CLAUDE.mdを書く**
プロジェクトの「開発憲法」を最初に作る。技術スタック・禁止事項・命名規則を書く。

**ステップ2: T-01で要件定義する**
MVPの機能を3つに絞る。「なければアプリが成立しないか？」で篩にかける。

**ステップ3: ISSUEを書いてAIに渡す**
第3章のフォーマットで書く。「評価軸との紐付け」を必ず含める。

**ステップ4: 月1回T-08で評価する**
G-STACK/HEART/AARRRで採点する。スコアをAIに渡して改善施策を提案させる。

**ステップ5: フェーズ化して実装する**
評価結果からフェーズA/B/Cを設計して、一つずつ完了させる。

## 付録: フレームワーク早見表

### G-STACK チェックリスト

```
□ Goal: 「誰の・どんな問題を・どう解くか」が1文で言えるか
□ Strategy: 「どうやって広まるか」の設計があるか
□ Tactics: 戦略を実現する具体的な機能・施策があるか
□ Architecture: スケールできる構造か / 技術負債がないか
□ Risk: セキュリティ・可用性・コスト面の致命的リスクがないか
```

### HEART チェックリスト

```
□ Happiness: 使って「楽しい・気持ちいい」と感じるか
□ Engagement: 1回の訪問で深く使っているか
□ Adoption: 新機能が自然に使われているか
□ Retention: 翌週も戻ってくるか / 再訪の動機があるか
□ Task Success: 主要タスクが100%の確率で完了するか
```

### AARRR チェックリスト

```
□ Acquisition: 新規ユーザーの流入経路が設計されているか
□ Activation: 初訪問で「これは良い」と思える体験があるか
□ Retention: 翌日・翌週・翌月も戻ってくるか
□ Revenue: 収益化の経路が設計されているか
□ Referral: ユーザーが自発的に他の人を連れてくるか
```

### 加重スコア計算式

```
総合スコア = G-STACK × 0.5 + HEART × 0.3 + AARRR × 0.2

例）G-STACK:71, HEART:79, AARRR:56 の場合
  = 71×0.5 + 79×0.3 + 56×0.2
  = 35.5 + 23.7 + 11.2
  = 70.4 → 70点
```
