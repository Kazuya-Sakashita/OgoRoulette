1.1 AI駆動開発とは何か

AI駆動開発とは、AIをペアプログラマーとして中心に据えた開発手法だ。「AIにコードを書いてもらう」のとは根本的に違う。AIは「速く書けるツール」ではなく、「自分が言語化できていないことを実装に落としてくれるパートナー」だ。

本質は3つある。① 思考の言語化（頭の中の「なんかおかしい」をISSUEとして言語化する）、② AIへの文脈転送（言語化した文脈をAIに正確に渡す）、③ 人間による判断（AIの提案を評価して採用・却下を決める）。この3つがそろって初めて機能する。

🎯
AI駆動開発の核心公式
AI駆動開発 ＝ ISSUE駆動開発 × AI × プロの判断基準
どれか一つ欠けると「AIにコードを書いてもらってるだけ」になる。OgoRouletteではこの3ステップを230回以上繰り返した。毎回のISSUEがドキュメントになり、品質は積み上がる。

1.2 なぜ今必要か

2026年現在、個人開発の民主化が加速している。かつては「フルスタックエンジニア＋デザイナー＋PM」の3人チームで数ヶ月かかっていたアプリが、AIを使えば1人で約3週間で作れる時代だ。しかし多くの開発者が「AIを使っているのに品質が上がらない」「意図しないものが出てきた」という壁に当たっている。

この壁の正体は 「AIの使い方の問題」ではなく「思考の言語化の問題」だ。思考の言語化→AIへの転送→判断のサイクルを速く正確に回せる人が、圧倒的に有利な時代になった。

2.2 各ステップの詳細（①②③）

①

課題発見

「なんかおかしい」「ユーザーが戸惑いそう」という直感を大切にする。この直感をその場でメモすることが重要だ。後回しにすると消える。OgoRouletteでは「ルーレットが止まる瞬間の演出が弱い」「iOSで音が出ない」「ルーム参加時にローディングが長い」という直感がすべてISSUEになった。

②

ISSUE化（思考の言語化）

課題を発見したら、すぐISSUEに落とす。ISSUEは「思考の結晶」だ。良いISSUEには5つの要素がある：背景（なぜ今これを解決するのか）・問題・原因仮説・解決策・受け入れ条件。この5要素がそろったISSUEは、AIへの指示書として完璧に機能する。

③

AI解決案生成

ISSUEが書けたら、テンプレートにはめてAIに渡す。テンプレートは「AIへの文脈転送の型」だ。デバッグのISSUEなら T-C（評価テンプレート）を使う。実装指示なら T-D を使う。「どのテンプレートを使うか」の判断が人間の重要な仕事だ。

2.3 各ステップの詳細（④⑤⑥）

④

実装（AI + 人間の判断）

AIの提案は「叩き台」だ。そのまま使うのではなく、評価して採用・却下・修正を判断する。AIに任せる：コードのドラフト生成、テスト作成、コメント追加、リファクタ候補の提案。人間が判断する：アーキテクチャの最終決定、状態管理の設計、ユーザー体験の評価、採用・却下の決定。この分担を崩すと丸投げになる。丸投げは必ず失敗する。

⑤

検証（6軸評価）

実装したら必ず検証する。6軸評価はG-STACK/HEART/AARRR/Kano/感情/技術の6軸で品質を数値化するフレームワークだ。「なんとなく動いている」では品質は上がらない。数値で見ることで、改善の優先度が明確になる。OgoRouletteでは毎回スコアを記録し、CLAUDE.mdの現スコア欄を更新した。

⑥

再ISSUE化

検証で見つかった問題は、すぐにISSUEにする。「あとで直す」はほぼ実現しない。ISSUEにすれば次のループで確実に処理される。OgoRouletteでは常に20〜30個のISSUEがBacklogにある状態を維持した。これが「手が止まる」状況を完全になくした。

2.4 ループを速く回すコツ

⚡
最重要ルール：1ループ = 1〜2個の改善に絞ること
「一気にたくさん改善しよう」とすると、何が原因で何が改善したかわからなくなる。デバッグが複雑になり、AIへの指示も曖昧になる。小さく回す方が、結果として速い。1ループの目標は30〜40分、60分を超えたらISSUEを分割するサインだ。

2.5 粒度の基準

✅ OK — 1セッションで完結する粒度
「スピン後のアニメーション演出を追加する」
「iOS Safariでの音声バグを修正する」
「WinnerCardにシェアボタンを追加する」
「next/fontをCDNからself-hostに移行する」
❌ NG — 分割が必要な粒度
「UXを全体的に改善する」（大きすぎ→分割する）
「ボタンの色を#FF6B6Bから変える」（小さすぎ→まとめる）
「モバイル対応を全部やる」（大きすぎ→分割する）
「バグを全部直す」（大きすぎ→優先度をつける）

「大きすぎる」は分割する。「小さすぎる」は関連するものとまとめる。1ISSUEが1セッションで完結するくらいの粒度が理想だ。OgoRouletteでは1日10〜30本のISSUEを集中的に消化するサイクルを回し続け、3週間で230+ ISSUEになった。

3.1 ISSUEの役割 — タスクとは何が違うか

ISSUEは単なるTODOリストではない。課題を構造化して言語化する思考プロセスだ。ISSUEを書く過程で「なぜおかしくなるか」を考え、「どう直すか」を設計し、「どうなれば完了か」を定義する。この過程でAIへの指示書が完成する。

🧠

思考の整理ツール

「なんかおかしい」が「〇〇のときに△△が起きる」という構造化された問題になる。

📤

AIへの文脈転送

5要素がそろったISSUEは、そのまま完璧なAI指示書になる。余分な説明が不要。

📚

開発ログ

「なぜこの実装にしたか」が残る。3ヶ月後も意図が分かる。コードレビューが不要になる。

🏆

品質の証拠

230+ ISSUEを消化した事実が品質保証になる。「このアプリは230回改善した」は最強の説明だ。

観点タスク（NG）ISSUE（OK）
粒度大きい・あいまい一つの問題・一つの改善
背景書かない「なぜ今やるか」を必ず書く

3.2 良いISSUEの5条件

1

背景が書かれている

「なぜ今これを解決するのか」が書かれている。「感情スコアが低かった」「本番でしか発生しない」など。背景のないISSUEは後で「なぜやったのか」がわからなくなる。

2

問題が具体的だ

「おかしい」ではなく「iOS Safari 17.2でルーレットが途中で止まる」のように具体的だ。AIは具体的な問題記述から正確な原因を推論できる。

3

原因の仮説がある

「おそらく〜だと思う」という仮説を書く。仮説がないとAIは全方位から調査を始めて時間を浪費する。間違った仮説でも価値がある。

4

解決策の方向性がある

「コンポーネントを分割する」「useRefに変更する」など方向性がある。完全な答えでなくていい。方向性があるとAIが具体的な実装提案を出せる。

5

受け入れ条件がある

「どうなれば完了か」がチェックボックス形式で書かれている。受け入れ条件のないISSUEは永遠に完了しない。「iPhoneで問題なく動く」のように検証可能な形で書く。

3.5 ISSUE管理の実践ルール

バックログ管理

常に20〜30個のISSUEをBacklogに維持する。「次に何をやるか迷わない」状態を保つ。ISSUEが0になったら6軸評価を回して新しいISSUEを生成する。

保留ISSUEの扱い

「重要だが今じゃない」ISSUEは削除しない。保留理由と再検討タイミングを明記して残す。「スコア80点を超えたら再検討」のように条件を書く。

🎯

発見

症状の特定

ロックトークンパターンで解決

サイクル B: ISSUE-214 — 動的OGPシェアURL（所要時間: 45分）

SNSシェアに当選者名入りOGP画像を表示してAAR RR-Referralを改善した事例

2

原因調査（10分）

lib/share-service.ts を読むと、roomCodeがある場合は /join?room=...（静的OGP）を使っていた。roomCodeなしの場合は /result?winner=...（動的OGP）。条件分岐が問題だった。

3

発見（気づき）

/resultページはすでにroomパラメータを読んで「このグループに参加する」CTAを表示する機能があった（ISSUE-094）。両立できる。

Before: roomCodeあり → /join（静的OGP）
if (payload.roomCode) {
return `/join?room=${code}&ref=share`
}
// → SNSに「アプリロゴ」が表示される
After: 常に /result（動的OGP + バイラルCTA）
// roomCodeも room=CODE として result に渡す
if (payload.roomCode) params.set("room", payload.roomCode)
return `/result?winner=...&room=CODE`
// → SNSに「田中さんが奢ります！」が表示
4

テストで保護（10分）

「/joinに戻っていないこと」を明示的にテストに追加。expect(url).not.toContain('/join')。リグレッション防止。

5

再評価（5分）

AARRR-Referral: 3.5/4 → 4/4。総合スコア: AARRR 17.5 → 18.5（+1）。SNSシェア時のOGP画像が当選者名入りに変わった。
