1.1 AI駆動開発とは何か

AI駆動開発とは、AIをペアプログラマーとして中心に据えた開発手法だ。「AIにコードを書いてもらう」のとは根本的に違う。AIは「速く書けるツール」ではなく、「自分が言語化できていないことを実装に落としてくれるパートナー」だ。

🎯
AI駆動開発の核心公式 AI駆動開発 = ISSUE駆動開発 × AI × プロの判断基準
この3要素がそろって初めて機能する。どれか一つ欠けると「AIにコードを書いてもらってるだけ」になる。

今日から始める30分クイックスタート
この3ステップを終えたとき、あなたはAI駆動開発の最初のループを完走している
⏱ Step 1 — 10分

CLAUDE.md（プロジェクト指示書）を作る

プロジェクトルートに CLAUDE.md を作成する。AIはこのファイルを自動で読み込む。これがないとAIは毎回文脈ゼロで答える。以下をコピーして、{}の部分を書き換えるだけでいい。

# {アプリ名} ## 概要 {アプリの1行説明} **本番URL:** {URL} **スタック:** {技術スタック} ## 設計思想 - {設計の重要な考え方1} - {設計の重要な考え方2} ## 絶対ルール **AI指示** - 破壊的操作の前は必ず git status を確認する - 実装後は必ず npx tsc --noEmit で型チェックする **評価** - 評価は6軸: G-STACK / HEART / AARRR / Kano / 感情 / 技術 - 現スコア: **{点数}/100**
⏱ Step 2 — 10分

最初のISSUEを書く

アプリを実際に使ってみて「なんかおかしい」と思った1つのことをISSUEにする。以下のテンプレートを使う。

# ISSUE-001: {動詞+対象+状態}のタイトル（例：スピン後の演出がなくて盛り上がらない） **背景（なぜ今やるか）:** {gstackで感情スコアが低かった / ユーザーから指摘があった / など} **問題（何が起きているか）:** {具体的に何が起きているか} **原因仮説（なぜ起きているか）:** {おそらく〜だと思う} **解決方向（どう直すか）:** {具体的な修正方針} **受け入れ条件:** - [ ] {完了と判断できる具体的な状態1} - [ ] {完了と判断できる具体的な状態2} **対象スコア:** {軸名}+{点}
⏱ Step 3 — 10分

AIに最初の指示を出す

ISSUE をそのままAIに渡す。以下のひと言を冒頭に追加する。

以下のISSUEを実装してください。 CLAUDE.mdのルールを守り、関連ファイルを読んでから実装してください。 実装後は npx tsc --noEmit で型チェックを行い、エラーがないことを確認してください。 {上のISSUEをそのまま貼る}
🎉
これで最初のループが完走した 課題発見（Step 1で発見）→ ISSUE化（Step 2）→ AI解決案生成（Step 3）→ 実装（AIが実行）→ 次のループへ。この3ステップを習慣にするだけで、1週間後には5〜10個のISSUEが消化されている。

Chapter 03
ISSUE駆動開発
ISSUEは「思考の結晶」だ。書く過程でAIへの指示書が完成する

3.1 良いISSUEの5条件

1

背景が書かれている

「なぜ今これを解決するのか」が書かれている。「gstackでスコアが低かった」「本番でしか発生しない」など。背景がないISSUEは後で「なぜやったのか」がわからなくなる。

2

問題が具体的だ

「おかしい」ではなく「iOS Safari 17.2でルーレットが途中で止まる」のように具体的だ。AIは具体的な問題記述から正確な原因を推論できる。

3

原因の仮説がある

「おそらく〜だと思う」という仮説を書く。仮説がないとAIは全方位から調査を始めて時間を浪費する。

4

解決策の方向性がある

「コンポーネントを分割する」「useRefに変更する」など方向性がある。完全な答えでなくていい。方向性があるとAIが具体的な実装提案を出せる。

5

受け入れ条件がある

「どうなれば完了か」がチェックボックス形式で書かれている。受け入れ条件がないISSUEは永遠に完了しない。

3.2 ISSUE NG/OK 比較

❌ NG — 意味がないISSUE
「ルーレットを直す」
背景なし / 問題が曖昧 / 受け入れ条件なし。
AIに渡しても「何を直しますか？」と聞き返される。
✅ OK — AIへの指示書になるISSUE
「iOS Safariでルーレットが途中で止まるバグ修正」
背景: 本番でユーザー報告あり
問題: iOS Safari 17.2 でsetTimeout遅延
受け入れ: [ ] iPhone 15でスムーズに動く
📌
保留ISSUEの正しい管理 「重要だが今じゃない」ISSUEは削除せず 保留理由と再検討タイミングを明記して残す。OgoRouletteでは常に20〜30個のバックログを維持した。これが「次に何をやるか迷わない」状態を保証した。

HEART (/20)

体験品質

Happiness（満足）・Engagement（継続使用）・Adoption（新規獲得）・Retention（定着）・Task Success（目標達成）。UXの全側面を網羅。

Before/After形式で処方箋付き。これを知るだけで同じ轍を踏まずに済む
❌ 失敗01: ISSUEが雑
症状: 「ルーレット直す」1行だけ。
結果: AIが「何を直しますか？」と聞き返す。永遠に完了しない。
処方箋: T-Bを使う。5分でISSUEフォーマットを埋める。
✅ After
背景・問題・原因仮説・解決方向・受け入れ条件の5要素が揃ったISSUE。AIへの指示書として即使える。所要時間は5〜10分。

症状: AIのコードをそのまま本番デプロイ。
結果: 認証バイパス・SQLインジェクション・状態競合が発生。

✅ After
AIが生成 → 人間がレビュー → 採用・修正・却下を判断 → 理解した上でマージ。AIは「叩き台製造機」。最終判断は常に人間。
❌ 失敗03: 検証なし
症状: 実装 → 完了で次のISSUEへ。
結果: スコアが全然上がらない。何が改善したかわからない。

再現可能な成功の型。4パターンをOKとNGで対比する
✅ 成功01: 小さく回す
1ループ = 1〜2改善。毎日1〜3 ISSUE。

❌ 失敗: 大きく回す
「週1回大きな変更」をしようとすると、何が原因で何が改善したかわからなくなる。バグ発生時の特定も困難。
✅ 成功02: 検証を必ず入れる

❌ 失敗: 実装で終わる
実装が完了したら次のISSUEへ。スコアを確認しないため、改善の方向性が正しいか分からない。同じ失敗を繰り返す。
✅ 成功03: 感情を設計に入れる

❌ 失敗: 機能だけを考える
感情設計がないと、機能は動くが誰も盛り上がらない。後から演出を入れようとすると、設計の複雑化を招く。
✅ 成功04: ISSUEをストックする
常に20〜30個のISSUEをバックログに持つ。「次に何をやるか迷う」という状態がゼロになる。毎日の最初の判断コストがなくなる。

サイクル B: ISSUE-214 — 動的OGPシェアURL（所要時間: 45分）

SNSシェアに当選者名入りOGP画像を表示してAAR RR-Referralを改善した事例

1

発見（10分）

XでシェアするとOGP画像がアプリロゴ（毎回同じ）。「田中さんが奢ります！」という情報が全くSNSプレビューに出ない。AARRR-Referralのボトルネック。

2

原因調査（10分）

lib/share-service.ts を読むと、roomCodeがある場合は /join?room=...（静的OGP）を使っていた。roomCodeなしの場合は /result?winner=...（動的OGP）。条件分岐が問題だった。

3

発見（気づき）

/resultページはすでにroomパラメータを読んで「このグループに参加する」CTAを表示する機能があった（ISSUE-094）。両立できる。

Before: roomCodeあり → /join（静的OGP）
if (payload.roomCode) {
return `/join?room=${code}&ref=share`
}
// → SNSに「アプリロゴ」が表示される
After: 常に /result（動的OGP + バイラルCTA）
// roomCodeも room=CODE として result に渡す
if (payload.roomCode) params.set("room", payload.roomCode)
return `/result?winner=...&room=CODE`
// → SNSに「田中さんが奢ります！」が表示
4

テストで保護（10分）

「/joinに戻っていないこと」を明示的にテストに追加。expect(url).not.toContain('/join')。リグレッション防止。

5

再評価（5分）

AARRR-Referral: 3.5/4 → 4/4。総合スコア: AARRR 17.5 → 18.5（+1）。SNSシェア時のOGP画像が当選者名入りに変わった。

思考の言語化

AIとの協働

🧠

人間が判断

採用・却下・修正は人間が決める。AIに丸投げしない。
