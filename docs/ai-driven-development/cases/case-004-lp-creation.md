## 背景

### なぜLPが必要だったか

OgoRouletteはアプリ自体は完成していたが、「初めて来た人が10秒でアプリの価値を理解する」ための入口がなかった。

- トップページ（`/`）はアプリの操作画面そのもの
- 「何ができるか」「なぜ使うべきか」「安全か」を説明する場所がない
- SNSでシェアされたURLを踏んで来た人が「これ何？」となる

LP（ランディングページ）の目的:

1. **価値の即時伝達**: 「飲み会の奢りをルーレットで決める」を3秒で理解させる
2. **信頼の構築**: 無料・登録不要・個人情報収集なし、を明示する
3. **CTAへの誘導**: 「今すぐ試す」ボタンをクリックさせる

### AIとのLP共創を選んだ理由

従来のLP制作は:
1. デザイナーにワイヤーフレームを依頼
2. コピーライターにテキストを依頼
3. コーダーにHTML/CSSを依頼

この3工程を、AIとの対話で1人でやる。AI駆動開発の「実験」として位置づけた。

## LP設計のプロセス

### Phase 1: ユーザーストーリーの定義

LPを作る前に、「誰が・どういう文脈で・何を求めてLPに来るか」をAIと一緒に整理した。

**プロンプト:**
```
OgoRouletteという飲み会の奢り決めアプリのLPを作ります。

想定ユーザー:
1. SNSで見かけて興味を持ったユーザー
2. 友人から「これで奢り決めよう」と送られてきたユーザー
3. 「飲み会 奢り 決め方」でGoogle検索したユーザー

各ユーザーがLPに来たとき:
- 最初に何を疑問に思うか
- 何を確認したいか
- 何がボトルネックになってCTAをクリックしないか

分析して、LPに必ず入れるべきコンテンツ要素をリストアップしてください。
```

**AIの分析（抜粋）:**

> **SNS経由ユーザーの疑問:**
> - 「これって本当にルーレットで決めるの？」（デモが欲しい）
> - 「アプリのインストールが必要？」（Webアプリであることを明示）
> - 「登録とかいる？」（登録不要を強調）
>
> **友人送付経由ユーザーの疑問:**
> - 「これ安全？個人情報取られない？」（プライバシー説明必須）
> - 「誰でも使える？スマホでも？」（対応環境の明示）
>
> **検索経由ユーザーの疑問:**
> - 「他のアプリと何が違うの？」（差別化ポイント）
> - 「無料？途中で課金される？」（無料であることの強調）
>
> **必ず入れるべき要素:**
> 1. ヒーローセクション（アプリ名・キャッチコピー・CTAボタン）
> 2. デモ動画またはGIF（30秒で操作を見せる）
> 3. 特徴の3点説明（アイコン + テキスト）
> 4. 使い方フロー（3ステップ）
> 5. プライバシー・無料の明示
> 6. 社会的証明（ユーザー数・口コミ・メディア掲載）
> 7. 最下部のCTA（もう一度「今すぐ試す」）

この分析をベースに、LPのセクション構成を決定した。

### Phase 2: コピーライティングの共創

**キャッチコピーの生成:**

```
OgoRouletteのLP用キャッチコピーを10案生成してください。

アプリの本質: 飲み会で奢りを決める際の揉め事をなくす
ターゲット: 20〜40代の社会人（飲み会・ランチ・打ち上げに参加する人）
トーン: 軽い・楽しい・ちょっとユーモア。重たくない。

避けること: ビジネス系の堅いコピー・過度な誇張

フォーマット:
メインコピー（10文字以内）+ サブコピー（30文字以内）
```

**AIが生成した10案（抜粋）:**

```
案1:
メイン: 今日の奢り、運命に任せろ。
サブ: 揉めない。謝らない。ルーレットが決める。

案2:
メイン: 奢りは、公平に。
サブ: 飲み会の不公平感を、ルーレットで解決。

案3:
メイン: 全員平等の奢り決め。
サブ: 誰が奢るか、みんなで回して決めよう。

案4:
メイン: 最後は運命に任せろ。
サブ: ランチ・飲み会・打ち上げ。奢り決めはOgoRouletteへ。

案5:
メイン: 奢る側も、奢られる側も、笑える。
サブ: ルーレットが決める透明な奢り決め。
```

**人間による選定:**
- 案1「今日の奢り、運命に任せろ。」を採用
- 理由: 「運命」という言葉がゲーム感・エンタメ感を出している。「揉めない」「謝らない」というネガティブな言葉を避けている

**ヒーローセクションのサブコピー:**

```
採用したキャッチコピー「今日の奢り、運命に任せろ。」に合わせて、
ヒーローセクションの説明文（30〜50文字）を5案生成してください。

条件:
- アプリの価値（公平・楽しい・簡単）を3秒で伝える
- CTAへの橋渡しになる
- 「登録不要・無料」を含める
```

採用案:
> 「飲み会・ランチ・打ち上げ。誰が奢るか、ルーレットで一発決定。公平で盛り上がる、無料のWebアプリ。」

### Phase 3: HTML/CSS の実装

#### デザイントークンの決定

まずブランドカラーとデザインシステムをAIと設計した。

**プロンプト:**
```
OgoRouletteのLPのデザイントークンを設計してください。

アプリのイメージ:
- カジノ・ゲーム・エンタメ感
- 暗い背景（夜の飲み会のイメージ）
- オレンジ・ピンクのグラデーション（情熱・楽しさ）

設計してほしいもの:
1. カラーパレット（CSS変数として）
2. タイポグラフィスケール
3. スペーシングシステム
4. ボタンスタイル（primary / secondary）

モバイルファースト（480px幅基準）で設計してください。
```

**AIが設計したデザイントークン（採用したもの）:**

```css
:root {
  /* Colors */
  --c-bg:       #080F1A;   /* 深夜の海の暗さ */
  --c-surface:  #0F1E30;
  --c-card:     #111D2E;
  --c-border:   rgba(255,255,255,0.08);
  --c-orange:   #F97316;   /* メインアクセント */
  --c-pink:     #EC4899;   /* サブアクセント */
  --c-gold:     #FBBF24;   /* 当選・強調 */
  --c-text:     #F5F5F7;
  --c-muted:    #94A3B8;

  /* Gradients */
  --grad:       linear-gradient(135deg, #F97316 0%, #EC4899 100%);
  --grad-r:     linear-gradient(135deg, #EC4899 0%, #F97316 100%);
  --grad-glow:  radial-gradient(ellipse at center, rgba(249,115,22,0.3) 0%, transparent 70%);
}
```

AIのコメント:
> `#080F1A` は「深夜の海」をイメージした背景色。ネイビーとブラックの中間で、カジノの照明を暗い部屋で楽しむ雰囲気を作ります。オレンジ（#F97316）とピンク（#EC4899）のグラデーションは、情熱と楽しさを同時に表現します。

この説明があることで、後に色を変更する際の「なぜこの色か」が分かった。

#### セクション別の実装

**ヒーローセクション:**

```
以下の要件でヒーローセクションのHTMLとCSSを実装してください。

コンテンツ:
- タイトル: 「今日の奢り、運命に任せろ。」
- サブタイトル: 「飲み会・ランチ・打ち上げ...」
- CTAボタン: 「今すぐルーレットを回す →」（href: https://ogo-roulette.vercel.app/）
- バッジ: 「無料」「登録不要」「スマホ対応」

デザイン制約:
- CSS変数 [上記トークン] を使う
- モバイル（480px）最適化
- CTAボタンはグラデーション（--grad）
- グロー効果（--grad-glow）を背景に使う
```

このパターンをセクションごとに繰り返した。

**社会的証明セクション（ISSUE-137）:**

```
社会的証明セクションを設計してください。

現状:
- OgoRouletteはユーザー数を公開していない（まだ少ない）
- メディア掲載実績もない
- 口コミも少ない

それでも「信頼感」を作るためのコンテンツを提案してください。
過度な誇張はNG。正直な範囲で。
```

**AIの提案:**

> ユーザー数や媒体掲載がない段階での社会的証明は、以下で代替できます:
>
> 1. **プロセスの透明性**: 「ルーレットはアルゴリズムで公平に決定」などの説明
> 2. **プライバシーの明示**: 「個人情報を収集しない」を機能として見せる（ISSUE-140）
> 3. **技術的信頼性**: 「Firebase / Vercel を使用」などのインフラ説明
> 4. **「使ってみた」の誘導**: 「まず1回試してみてください」という低コミットのCTA
>
> 実際のユーザー数・口コミが集まったら差し替え可能な構造で設計することを推奨します。

この提案を受けて「まず1回試してみよう」という低コミットのセカンダリCTAを追加した。

**プライバシー安心設計（ISSUE-140）:**

```
プライバシー不安を解消するセクションを設計してください。

OgoRouletteの実際のデータ扱い:
- 認証なしで使える（メールアドレス等不要）
- 収集するデータ: ルーム情報・メンバー名（任意のニックネーム可）
- Firebase Firestore に保存（24時間で自動削除）
- 広告なし・トラッキングなし

これをユーザーが「安心できる」形で見せるHTML/CSSを実装してください。
シールド・錠前などのアイコンを絵文字で使ってください。
```

### Phase 4: design-review による品質確認

実装完了後、gstack の design-review で視覚的な品質を確認した。

**レビュー結果（主要な指摘）:**

1. **ヒーローセクションのCTAボタンが小さい**: モバイルで44px以下のタッチターゲット
2. **フォントサイズのヒエラルキーが不明確**: H2とH3のサイズ差が小さい
3. **カード間のスペースが不均一**: セクションによって padding が 48px と 72px が混在
4. **CTAのグロー効果が過剰**: ボタンの box-shadow が大きすぎてビジー感がある

各指摘をISSUEに変換して修正した。

## コンテンツ共創の実践例

### シチュエーション説明の生成

LPの「使い方」セクションで、どんな場面で使えるかを説明するコンテンツをAIに生成させた。

```
OgoRouletteをどんな場面で使えるか、具体的なシナリオを10個生成してください。

条件:
- 実際に起きそうなリアルな場面
- 「揉める」→「ルーレットで解決」の流れが分かる
- 日本の文化・習慣に合っている（居酒屋・ランチ・打ち上げ）
- 1シナリオ = 2〜3文

NG: 作り話感が強いもの・誇張しすぎたもの
```

**採用したシナリオ（AIが生成）:**

> シナリオ1: 金曜の夜、同僚5人で飲みに行った帰り。誰が払うかで毎回モメる。今回はOgoRouletteで一発決定。「〇〇さん今日はありがとう！」でスッキリ解散。

> シナリオ2: ランチで新しい店に行くたびに「俺が出す」「いや私が」の不毛なやり取りをしていた。OgoRouletteを使い始めてから、公平に決まるし全員が笑顔で食べられるようになった。

> シナリオ3: 部活の打ち上げ。OBも含めて10人。誰が多く払うかで気まずくなるのが嫌だった。ルーレットで決めたら全員「仕方ない！」と笑い飛ばせた。

OgoRouletteのLPにFAQセクションを追加します。

想定される質問を10個生成し、回答を書いてください。

## 人間の判断が重要だった場面

### 1. コピーの選択

AIは10案生成するが、どれを選ぶかは人間の判断。判断の基準:
- 「自分がユーザーだったら、このコピーを見てアプリを試したいと思うか」
- 「このトーンはOgoRouletteのブランドに合っているか」
- 「誇張・誇大広告になっていないか（実際の体験と乖離していないか）」

### 2. 「社会的証明なし」を正直に伝える判断

ユーザー数や口コミがない状態で、「架空の数字」や「作り物の口コミ」を入れる誘惑があった。AIに「ユーザー数が少ないように見せないためのコピーを書いて」と頼んだが、AIは「誇大広告のリスクがある」と指摘し、正直なアプローチを提案した。

この提案を受けて、「ユーザー数を示さない代わりにプロダクトの透明性を見せる」設計にした。これは長期的なブランド信頼の構築において正しい判断だったと思う。

### 3. アニメーションの量

AIは「スクロールアニメーション・フェードイン・パーティクル」などを提案したが、「LP は情報を伝えることが目的。アニメーションは主役ではない」という判断で、CSSトランジションのみに絞った。派手さより、ページロード速度と情報の明確さを優先した。

## 学び

### LP共創の一般的な教訓

**教訓1: 「誰に向けて書くか」を先に定義する**

コピーライティングの前に「3種類のユーザー」を定義したことで、全セクションのトーンと内容が一貫した。この定義なしにコピーを生成しても、バラバラなメッセージになる。

**教訓2: AIが生成したHTMLは必ず動作確認する**

AIは「意味的に正しいHTML」を書くが、「ブラウザで見たときに美しいかどうか」は実際に確認しないと分からない。10回に3回は微調整が必要だった。

**教訓3: コンテンツと実装を分けて考える**

「何を書くか（コンテンツ）」と「どう見せるか（デザイン）」は別のプロンプトで依頼する方が品質が高い。同時に依頼すると、どちらも中途半端になる。

**教訓4: LP は「完成」ではなく「改善の起点」**

LP を公開した後もISSUE-137・140・144として改善が続いた。完璧なLPを最初から作ろうとするより、「まず出して、計測して、改善する」サイクルを回す方が速い。
