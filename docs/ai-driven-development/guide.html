01Chapter / Introduction

はじめに — AI駆動開発とは

開発の常識が変わりつつある。その本質と従来手法との違いを整理する。

🤖

AI駆動開発とは

AIをコード補完ツールではなく、設計・実装・改善のパートナーとして使う開発スタイル。「指示 → 実装 → 評価 → 改善」のループを高速で回す。

⚡

なぜ今必要か

知らない技術でも「まず作ってみる」ができる時代に。学習コストが激減し、個人開発の限界が大幅に拡大している。

🔄

従来開発との違い

「勉強してから実装」→「対話しながら並行学習」。AIはコードを書くが、何を作るかは人間が決める。オーナーシップは変わらない。

観点従来の個人開発AI駆動開発
知らない技術勉強してから実装（数日〜週単位）AIと並行学習しながら実装（時間単位）
バグ調査ひたすら検索・試行錯誤ログ+コードをAIに渡して仮説を得る
コードレビュー1人で読み返す or チームに依頼AIに複数視点でレビューさせる
LP・文章ゼロから時間をかけて書く構造を渡してAIにたたき台を生成させる
スピード感週単位日〜時間単位
⚠️
重要な前提AI駆動開発は「AIに丸投げ」ではない。何を作るか・なぜ作るか・品質を判断するのは常に人間側の仕事。AIはその実行速度を上げるためにある。

📊 gstack とは

Claude Code に組み込まれた評価フレームワーク。CEO・エンジニア・デザイナーなど複数の視点でプロダクトを自動評価する。評価軸：Lighthouseスコア・UI/UX品質・保守性・セキュリティ・アクセシビリティ。

🧠 HEART フレームワーク

Google提唱のUX評価5軸。
Happiness（満足度）
Engagement（関与度）
Adoption（採用率）
Retention（継続率）
Task Success（完了率）

03Chapter / Development Story

OgoRoulette 開発ストーリー

アイデア誕生からリリースまで。感情と意思決定も含めた開発の全記録。

Act 1 — アイデアの誕生

飲み会の帰り道、友人から言われた。「今日の奢り決め、あんまり盛り上がらなかったな。」

考えてみると原因はシンプルだった。「決め方」に個性がない。コイントスは一瞬で終わる。アプリのルーレットはスマホ1人でぽちぽちするだけ。周りは何が起きているかすら分からない。

そのころ Claude Code の話をよく聞くようになっていた。「AIと一緒にアプリを完成まで持っていく」が本当にできるのか、確かめたくなった。

🎯 解決する問題

「盛り上がらない奢り決め」

疑われない・全員で見られる・結果が分かりやすい。この3つを同時に満たす必要があった。

⚡ 技術スタック（未経験）

Next.js ✓ 少し経験あり
Supabase ✗ 名前だけ知っている
Prisma ✗ ほぼ初見
Framer Motion ✗ 完全に未知

🚀 AI駆動開発の判断

「知らなくても始めてみる」ができると確信した。Claude Code があれば未知の技術でも動けると判断。

Act 2 — 最初の3日間（快調期）

「このAPIのルートを追加して」と言えば動くコードが来る。「このコンポーネント作って」と言えばすぐ出てくる。「これは革命だ」と思った。

違和感が生まれたのは3日目くらいだった。前日のセッションで決めた設計が、次のセッションでは消えている。「戻るボタンは /home に向ける。router.back() は絶対に使わない」と決めたのに、翌朝 Claude が出してきたのは router.back() を使うコードだった。

💡
気づき：セッションをまたぐと記憶はリセットされるClaude はセッションが変わると前の会話を覚えていない。決めたルールを毎回説明し直す必要があった。解決策：CLAUDE.md にルール・禁止事項・設計方針を書いておく。

03 / Story Act 2-3

ストーリー続き — 驚き・失敗・そして楽しさへ

Act 3 — 初めての「すごい」

CLAUDE.md を書いた翌日から、違和感の密度が下がり始めた。Supabase Realtime の実装を頼んだとき、頼んでいないフォールバック処理まで実装してきた。

「リアルタイム接続が切れた場合、ポーリングにフォールバックする」 — これは頼んでいなかったが、必要だった。スマホのバックグラウンドでは接続が頻繁に切れる。フォールバックがなければ、接続が切れた瞬間に全員が取り残される。

「自分が言語化できていないことを、実装に落としてくれる」が最初の「すごい」の正体だった。

Act 4 — 壊れた日

本番だけルーレット結果が表示されない。ローカルでは動いている。ログを見ると pendingMemberWinner: null。

原因はアニメーション完了 vs サーバー結果受信の競合状態（race condition）だった。ネットワーク遅延が大きいとき、アニメーションが先に終わり、当選者データがまだ届いていない状態が発生した。

「自分がこのコードのオーナーではないかもしれない」という不安が、この日に明確になった。理解できないコードはマージしないというルールを設けた。

Act 5 — 楽しくなってきた瞬間

会社の飲み会でルーレットを実際に動かした日、友人5人でひとつの画面を囲んで、ルーレットが止まる瞬間を待った。「うわー！」というリアクションが出た。

「自分だけでは辿り着けなかった場所に辿り着く楽しさ」が、「速く作れる楽しさ」を超えた。これがAI駆動開発の本当の価値だった。

📌 Key Lesson 1

CLAUDE.mdでコンテキストを永続化する。セッションをまたいでも一貫した開発ができる。

📌 Key Lesson 2

「なぜこう書いた？」を必ず聞く。理解できないコードは絶対にマージしない。

📌 Key Lesson 3

「作って」から「考えて」へ。AIを実装ツールではなく思考パートナーとして使う。

04Chapter / Each Process

各工程の詳細 — AIをどう使ったか

7つの工程それぞれで、AIをどのように活用したかを具体的に解説する。

Process 01
要件定義
「飲み会で盛り上がりながら奢り手を決めたい」という課題から出発。AIに機能リストを優先度付きで出させた。

出力：①参加者登録、②ルーレット演出、③サーバー側当選決定（不正防止）の3機能に絞り込み。MVPを最初に確定させることで、スコープが広がり続けることを防いだ。
💬 AIへの依頼：「MVP機能リストを優先度HIGH/MED/LOW付きで出して」
Process 02
UI設計
「スマホで全員が同時に見られる」という制約から、設計を逆算。ダークテーマ・大きなルーレット・全画面アニメーションの方向性をAIと対話しながら決めた。

コンポーネントは1つずつ依頼。ページ全体ではなく部品単位で依頼することでレビューコストを低く保った。
💬 AIへの依頼：「RouletteWheelコンポーネントの設計を提案して」
Process 03
実装
Supabase + Next.js + Framer Motion。すべて未経験または浅い経験の技術スタック。

実装前に「何を・なぜ・どう実装するか」を一言確認するルールで進めた。Supabase Realtimeのフォールバック実装は頼んでいなかったが、AIが自律的に提案してきた。
💬 AIへの依頼：「まず方針を確認してから実装して」
Process 04
デバッグ
本番のみルーレット結果が表示されないバグ。ローカルでは再現しない。

エラーログ・再現条件・試したこと・関連コードをセットで渡し、「原因仮説を3つ挙げてから修正案を提示して」と依頼。race conditionを特定し、waitingForWinnerフラグで解決。
💬 AIへの依頼：「ログを渡す。原因仮説3つ挙げてから修正案を提示して」

04 / Each Process (continued)
Process 05
リファクタ
「コードを整理して」は曖昧すぎる。リファクタの目的を1つ選んで依頼するルールにした。

可読性・パフォーマンス・型安全性・テスタビリティ・重複削除の5つから1つを選び、「外部から見た動作を変えない」「使用ライブラリを変えない」制約を明示した。
💬 AIへの依頼：「目的：可読性向上。動作は変えないこと」
Process 06
UX改善
gstack評価でPerformance 62、Accessibility 86を検出。改善内容：
・viewport zoom制限の削除（WCAG違反）
・Google Fonts 466KB → 90KBに削減
・テキストコントラスト 3.5:1 → 4.6:1（WCAG AA）

結果：Accessibility 86 → 92
💬 AIへの依頼：「gstack評価を実行して、ISSUEを優先度付きで出して」
Process 07
バズ設計
「誰が奢るか決まった瞬間」を中心に感情設計を行った。

全員の画面でルーレットが同時に止まる → 紙吹雪アニメーション → 当選者ハイライト。操作 → 結果ではなく、期待 → 緊張 → 演出 → 結果 → 笑い → シェアというフローを設計。
💬 AIへの依頼：「当選者が決まる瞬間の演出パターンを3つ提案して」
Process 08
LP作成
Lighthouse 95+を目標にLP HTMLをAIで生成。Before/After構造・機能紹介・CTAの順に設計。

外部CDN不使用・単一HTMLファイル・モバイルファーストの制約をテンプレートに組み込んだ。Google Fontsは最小ウェイトのみ使用。
💬 AIへの依頼：「ターゲット・課題・価値提案を整理してLP構成を提案して」
✅
各工程共通のルール「頼んだ箇所以外は変更しない。実装前に何を・なぜ・どう実装するか一言確認する。」この2文を毎回入れることで、意図しない変更と確認なしの実装が激減した。

失敗パターン症状根本原因対策
AIに任せすぎ「自分のコードでない」感。バグ原因が特定できない理解なしにマージ「なぜこう書いた？」を必ず聞く。理解できないコードはマージしない

09Chapter / Going Viral

バズるアプリを作る考え方

機能が揃っているだけでは拡散しない。感情と体験を設計することが必要。

❤️

体験設計

「動く」だけでなく「使いたい」と思わせる。演出・アニメーション・声が出るポイントを意識的に設計する。

🎭

感情設計

体験する感情を先に決める。ドキドキ → 安堵 → 笑い → シェアという感情フローを意図して設計する。

📱

SNS設計

「スクショしたくなる」「シェアしたくなる」瞬間を作る。結果画面はSNS投稿を前提にデザインする。

要素バズらない設計バズる設計
結果の出し方テキストがポップアップする全員の画面でルーレットが同時に止まる
演出なし紙吹雪・効果音・ハイライトアニメーション
透明性1人の画面で操作全員がリアルタイムで同じ画面を見る
UXフロー操作 → 結果期待 → 緊張 → 演出 → 結果 → 笑い → シェア
信頼性誰でも結果を操作できるサーバーが当選者を決定（不正不可能）
🔑
「疑われない」が最重要抽選アプリで最も大切なのは透明性。当選者をサーバーが決定し、クライアントが操作できない設計にすることで「信頼」が生まれる。技術的な正しさが体験の信頼を作る。これはAIとの対話で設計した最重要の判断だった。
