1.1 AI駆動開発とは何か

AI駆動開発とは、AIをペアプログラマーとして中心に据えた開発手法だ。「AIにコードを書いてもらう」のとは根本的に違う。AIは「速く書けるツール」ではなく、「自分が言語化できていないことを実装に落としてくれるパートナー」だ。

AI駆動開発の本質は3つある。① 思考の言語化（頭の中の「なんかおかしい」をISSUEとして言語化する）、② AIへの文脈転送（言語化した文脈をAIに正確に渡す）、③ 人間による判断（AIの提案を評価して採用・却下を決める）。

🎯
AI駆動開発の核心公式 AI駆動開発 ＝ ISSUE駆動開発 × AI × プロの判断基準。この式を体に叩き込むのが、この教科書の目的だ。OgoRouletteではこの3ステップを200回以上繰り返した。毎回のISSUEがドキュメントになり、品質は積み上がる。

1.2 なぜ今必要か

2026年現在、個人開発の民主化が加速している。かつては「フルスタックエンジニア+デザイナー+PM」の3人チームで数ヶ月かかっていたアプリが、AIを使えば1人で数週間で作れる時代になった。しかし多くの個人開発者が「AIを使っているのに品質が上がらない」「AIに丸投げしたら意図しないものが出てきた」という壁に当たっている。

この壁の正体は 「AIの使い方の問題」ではなく「思考の言語化の問題」だ。スピードが重要な時代において、「思考の言語化→AIへの転送→判断」のサイクルを速く正確に回せる人が、圧倒的に有利だ。

2.2 各ステップの詳細解説

①

課題発見

「なんかおかしい」「ユーザーが戸惑いそう」という直感を大切にする。この直感をその場でメモすることが重要だ。後回しにすると消える。OgoRouletteでは「ルーレットが止まる瞬間の演出が弱い」「iOSで音が出ない」「ルーム参加時にローディングが長い」という直感がすべてISSUEになった。

②

ISSUE化（思考の言語化）

課題を発見したら、すぐISSUEに落とす。ISSUEは「思考の結晶」だ。良いISSUEには5つの要素がある：背景（なぜ今これを解決するのか）・問題・原因・解決策・受け入れ条件。この5要素がそろったISSUEは、AIへの指示書として完璧に機能する。

③

AI解決案生成

ISSUEが書けたら、テンプレートにはめてAIに渡す。テンプレートは「AIへの文脈転送の型」だ。デバッグのISSUEなら T-03（バグ調査テンプレート）を使う。UI改善なら T-02（UI生成テンプレート）を使う。「どのテンプレートを使うか」の判断が人間の重要な仕事だ。

④

実装（AI + 人間の判断）

AIの提案は「叩き台」だ。そのまま使うのではなく、評価して採用・却下・修正を判断する。AIに任せる：コードのドラフト生成、テスト作成、コメント追加、リファクタ候補の提案。人間が判断する：アーキテクチャの最終決定、状態管理の設計、ユーザー体験の評価、採用・却下の決定。この分担を崩すと、AIへの丸投げになる。丸投げは必ず失敗する。

⑤

検証（gstack）

実装したら必ず検証する。gstackはQAテストを実行し、品質を数値化するツールだ。「なんとなく動いている」では品質は上がらない。数値で見ることで、改善の優先度が明確になる。OgoRouletteでは、Lighthouseスコア・UX観点・視覚的一貫性・アクセシビリティを毎回チェックした。

⑥

再ISSUE化

検証で見つかった問題は、すぐにISSUEにする。「あとで直す」はほぼ実現しない。ISSUEにすれば次のループで確実に処理される。OgoRouletteでは常に20〜30個のISSUEがBacklogにある状態を維持した。これが「手が止まる」という状況を完全になくした。

2.3 ループを速く回すコツ

⚡
最重要ルール：1ループ = 1〜2個の改善に絞ること 「一気にたくさん改善しよう」とすると、何が原因で何が改善したかわからなくなる。デバッグが複雑になり、AIへの指示も曖昧になる。小さく回す方が、結果として速い。

OgoRouletteで学んだ粒度感：

OK — 良い粒度
「スピン後のアニメーション演出を追加する」
「iOS Safariでの音声バグを修正する」
「WinnerCardにシェアボタンを追加する」
NG — 避けるべき粒度
「UXを全体的に改善する」（大きすぎ）
「ボタンの色を#FF6B6Bから変える」（小さすぎ）
「モバイル対応を全部やる」（大きすぎ）

「大きすぎる」は分割する。「小さすぎる」は関連するものとまとめる。1ISSUEが1セッションで完結するくらいの粒度が理想だ。1日1〜3個のISSUEを消化する習慣が、継続的な改善で200+ ISSUEになった。

3.1 ISSUEの役割

ISSUEは単なるTODOリストではない。思考の結晶だ。ISSUEを書く過程で「なぜおかしくなるか」を考え、「どう直すか」を設計し、「どうなれば完了か」を定義する。この過程でAIへの指示書が完成する。

🧠

思考の整理

「なんかおかしい」が「〇〇のときに△△が起きる」に変わる。

📤

AIへの文脈転送

良いISSUEはそのまま完璧なAI指示書になる。

📚

開発ログ

「なぜこの実装にしたか」が残る。3ヶ月後も意図が分かる。

🏆

品質の証拠

200+ ISSUEを消化した事実が品質保証になる。

3.2 タスクとISSUEの違い

観点タスクISSUE
粒度大きい・あいまい一つの問題・一つの改善
背景書かない「なぜ今やるか」を必ず書く
受け入れ条件「完了したら消す」「どうなれば完了か」を明記
AIへの使いやすさそのままでは使えないAIへの指示書として完成

3.3 良いISSUEの5条件

1

背景が書かれている

「なぜ今これを解決するのか」が書かれている。「gstackでスコアが低かった」「本番でしか発生しない」など。

2

問題が具体的だ

「おかしい」ではなく「iOS Safari 17.2でルーレットが途中で止まる」のように具体的だ。

3

原因の仮説がある

「おそらく〜だと思う」という仮説を書く。AIはこの仮説を検証してくれる。

4

解決策の方向性がある

「〜の方向で修正したい」という方向性を書く。完璧な解決策でなくていい。

5

受け入れ条件が明確だ

「〜の状態になれば完了」が書かれている。これがないと実装が終わらない。

3.6 ISSUEの粒度管理

ISSUEを分割する基準は「1セッションで完了できるか」だ。大きすぎるISSUEは意思決定が混在して解決できない。

【悪い例】ISSUE-50: モバイル対応を全体的に改善する 【分割後・OgoRoulette実際の例】 ISSUE-51: スマホでのルーレット表示サイズを最適化 ISSUE-52: iOS Safariでの音声再生バグを修正 ISSUE-53: タッチ操作でのスワイプ誤反応を防止 ISSUE-54: 縦向き・横向き切替時のレイアウト崩れ修正
💡
OgoRoutletteのISSUE管理実績 200個以上をdocs/issues/で管理。「ルーレット同期バグ」を1つにせず、ISSUE-143・002・001と原因単位で分割。ISSUE-001の解決でISSUE-002も解消されることが判明し、無駄な実装を防げた。保留ISSUEも明示的に管理：実装リスクが高い・優先度が低い・外部依存のISSUEはステータスを「⏸️ 保留」とし保留理由を記録して積み残しを可視化している。

Case 1: ISSUE-001 — Race Condition × AI診断

分散システムのrace conditionをロックトークンパターンで解決した事例

背景：Week 3でリアルタイムルームを実装した直後、「2人が同時にスタートボタンを押すとルーレットが変な動きをする」という問題が発見された。ローカルのシングルユーザーテストでは再現しなかった。

症状：ルーレットが2回分回って止まらなくなる・phaseが spinning から idle に直接飛ぶ・当選者が表示されない

原因：Supabase Realtimeのブロードキャストが複数発火した際に、phase遷移のロジックが2回実行されていた。「1回目のstop」と「2回目のstop」が競合し、状態が矛盾した。

解決策：ロックトークンパターン

// スピン開始時にlockTokenを生成 const lockToken = crypto.randomUUID(); await broadcastSpinStart({ lockToken }); // 停止処理はlockTokenが一致する場合のみ受け付ける const handleSpinStop = (event: BroadcastEvent) => { if (event.lockToken !== currentLockToken) return; // 無視（冪等性担保） processSpinStop(event); };

AI活用のポイント：「Race conditionが起きている」という仮説をISSUEに書いて渡したところ、AIは「楽観的ロック」「悲観的ロック」「ロックトークン」の3パターンを提案した。それぞれのトレードオフを説明してもらい、リアルタイムアプリに最適なロックトークンを選択した。AIは候補を出し、人間がトレードオフを評価して選択した。

💡
学び 分散システムのrace conditionはローカルでは再現しない。「本番でしか起きないバグ」は同期の問題を疑え。ロックトークンパターンを最初から設計に組み込んでおくべきだった。

原則01〜05 — ISSUE・設計・状態管理・バグ・UXの基礎原則
Principle 01
ISSUEは思考単位で切れ
1つのISSUEは1つの意思決定で解決できる粒度にする。大きなISSUEは問題の本質が混在し、AIへの指示も曖昧になり、解決できないまま放置される。ISSUEが大きいと、AIは「どこから手をつけるか」を自分で判断してしまう。
NG: 「ルーレットのバグを全部直して、UIも改善して、テストも追加して」
OK: 「ISSUE-001: isOwnerフリッカーによるSPINボタン永続無効化」と原因単位で切る
Principle 02
AIに設計を任せるな
アーキテクチャ・データモデル・状態管理の設計は人間が決定する。AIは実装の速度を上げるが、設計の善し悪しを判断する「製品の意図」は人間にしかない。AIの設計提案は過去のパターンからの帰納であり、コスト・UX・拡張性の制約を考慮していない。
NG: 「このアプリのアーキテクチャはどうすればいいですか？」→ AIの提案をそのまま採用
OK: 「Supabase Free Tier、1人開発、3ヶ月でリリースという制約のもとで〇〇の実装方針を教えて」
Principle 03
状態管理は人間が握れ
Reactのstate設計・フェーズ管理・stateマシンの遷移図は人間が設計する。AIは個々の遷移を正しく実装できても全体の整合性を見失う。特に非同期処理・タイミング依存の状態遷移は競合バグが生まれやすい。
NG: 複数のisSpinning/isPreparing/isResultフラグを並列管理 → 同時にtrueになり得る競合状態
OK: type Phase = "waiting" | "preparing" | "spinning" | "result" の単一phaseステートマシンを人間が設計
Principle 04
バグは再現性から潰せ
「なんか動かない」ではAIはデバッグできない。バグを再現する手順・発生条件・環境を明示して渡すと、解決速度が10倍変わる。AIがデバッグに失敗するとき、ほぼ必ず「再現条件が不明瞭」か「ログが渡されていない」が原因だ。
NG: 「SPINボタンが押せなくなることがある。直して」→ AIは推測で複数の可能性をコードに追加し複雑化
OK: 「以下の手順で100%再現する: 1. ゲストホストでルーム作成 2. SPIN実行 3. 結果表示中にリロード」
Principle 05
UXは最初に設計しろ
ユーザー体験は機能実装後に「付け足す」ことができない。どう感じてほしいか・どんな感情を生み出すかを最初に決め、実装はその実現手段として選択する。後からUXを改善しようとすると、コンポーネント・state・アニメーションの変更が必要になりコストが跳ね上がる。
NG順序: 機能実装→デザイン当て→UX考える→全部リファクタ（コスト爆発）
OK順序: UXを言語化→演出を設計→実装する（感情が書けたら必要な演出を列挙してから実装依頼）

原則06〜10 — AI判断・スモールスタート・感情・シェア・ISSUE駆動
Principle 06
AIは提案者であって決定者ではない
AIが「〇〇の方がよいと思います」と言っても、それはAIの判断ではなく提案だ。採用するかどうかは常に人間が決める。AIは人間の承認を得やすい答えを返す傾向があり、「この実装で大丈夫ですか？」と聞けば「大丈夫です」と答える可能性が高い。

Principle 07
小さく作って小さく壊せ
MVP（最小動作品）の単位を小さく定義し、小さな失敗から素早く学ぶ。大きく作ると大きく壊れ、どこが問題かわからなくなる。AIが1つのプロンプトで大量のコードを生成すると、動かない箇所のデバッグが難しくなる。

Principle 08
感情設計を入れろ — スコア15未満は演出を優先せよ
アプリはデータを処理するツールではなく、人間に感情を与える体験だ。ドキドキ・驚き・笑い・解放感を意図的に設計する。機能的に同じアプリでも、感情設計があるとSNSでシェアされ、記憶に残り、リピートされる。6軸評価で感情スコアが15/20未満になったとき、機能追加・技術改善より演出改善を必ず優先すること。 OgoRouletteでは「誰も笑わなかった夜」の翌日からこのルールを設けた。
NG: 感情スコアが11/20なのに新機能を追加し続ける → 「動くが盛り上がらない」アプリのまま
OK: 感情スコア11/20 → 演出改善ISSUEを最優先（スロービルドアップ・ニアミス・WinnerCard reveal）→ 14/20に回復してから機能追加へ
Principle 09
シェア導線を設計しろ
バズは偶然ではなくシェアされやすい仕組みを設計した結果だ。「シェアしたくなる瞬間」と「シェアするための手段」を両方設計する。シェアされるのは感情が高まった瞬間に、手間が最小限で、シェアしたくなるコンテンツがある場合だけだ。
NG: シェアボタンを目立たない場所に配置・URLだけ共有・感情の高まりとずれたタイミング
OK: 感情が最高潮になった瞬間（当選者確定直後）にシェアボタンを表示。シェアコンテンツ自体が面白い（誰が奢りかわかるビジュアル）
Principle 10
ISSUE駆動で開発しろ
頭の中にある「あれを直したい」「これを追加したい」をすべてISSUEとして文字に落とす。文字になっていない仕事は存在しないも同然で、AIにも自分にも伝えられない。実装中に「あ、これ直したい」と思ったことを混ぜると差分が混濁し、コミットの意図が不明になる。
NG: 「あのバグ直したっけ？」→ 忘れる。AIに「さっき話してたバグを直して」→ AIはセッションをまたいで記憶しない
OK: 「やりたいこと」を思いついたら30秒でISSUEファイルを作る。タイトルだけでもいい。詳細は後で埋める。

原則11〜15 — CLAUDE.md・ポーリング・本番バグ・Race Condition・数値化
Principle 11
CLAUDE.mdを書け
プロジェクトのルール・禁止事項・文脈をCLAUDE.mdに書いて、AIが毎回参照できるようにする。AIはセッションをまたいで記憶しない。CLAUDE.mdはAIへの「永続的な記憶」だ。一度書けば永続的に効く。コストゼロで品質を上げる最強の施策。
NG: セッション1「router.back()は使わないで」→ セッション2: また使われる → 同じ会話の繰り返し
OK: CLAUDE.mdに「router.back()禁止（Link hrefを使う）」を記載 → 毎回のセッションで自動適用
Principle 12
ポーリングを軽視するな
リアルタイム機能の実装では、WebSocketが理想だが、フォールバックとしてポーリングを正しく設計する。ポーリングの粗雑な実装がUXを破壊する。キャンセル処理・メモリリーク対策を怠ると、ユーザーが増えるにつれてDBが死ぬ。
NG: setInterval(fetchRoom, 3000) → clearIntervalなし → メモリリーク確定
OK: cancelled フラグ + setTimeout再帰呼び出し。cleanup関数でキャンセル可能な設計にする。
Principle 13
本番でしか発生しないバグがある
ローカルで動くことは本番で動くことを保証しない。環境変数・CDN・ビルド最適化・地理的な遅延・実際のユーザー端末の差異が、本番専用のバグを生む。ISSUE-019のハイドレーションエラーはローカルでは再現しない本番バグだった。
NG: 「ローカルで動いてるから大丈夫」→ 本番デプロイ後に障害（localStorageのSSR中アクセス・RLSポリシーが本番のみ有効など）
OK: 新機能デプロイ後は必ず実機（iPhone + Android）でVercel本番URLをテストする
Principle 14
レース・コンディションを最初に想定しろ
非同期処理が複数走る箇所では、「2つの処理が同時に実行されたらどうなるか」を必ず考える。ReactのuseEffectは依存配列の変化ごとに発火する。複数のuseEffectが同じstateを変更する構造は、タイミング次第で結果が変わるレース・コンディションを引き起こす。
NG: 複数のuseEffectが同じstateに依存 → isOwnerがfalse→trueに変化する瞬間、両方のeffectが発火する可能性
OK: isOwnerResolved フラグで実行を制御。「解決済みフラグ」でisOwnerが確定するまで処理をスキップする。

gstackで数値化しろ
「なんとなく良い」「なんとなく悪い」という主観的評価を卒業し、gstackによるスコア（0〜100点）で品質を数値化する。数値があると改善の優先順位が決まり、AIへの依頼も具体的になる。週1回スコアを計測して変化を記録する。
NG: 「なんかLPが微妙」→「どう微妙？」→「全体的に」→ 議論が終わらない
OK: 「Adoption: 58/100 — 理由: デモが静的でアプリの価値が伝わらない」→ 具体的な改善タスクに直結する

原則16〜18 — ログ・型設計・理解度検証
Principle 16
ログを渡せ
AIにデバッグを依頼するときは、コンソールログ・エラースタック・ネットワークレスポンスを必ず添付する。ログなしのデバッグ依頼は推測ゲームになり、解決速度が10倍落ちる。ISSUE-143（スピン同期ズレ）のデバッグでは、タイムスタンプを含むログを渡すことで原因特定が1分以内に完了した。
NG: 「ルーレットの同期がズレます。このコードを見て直してください」→ AIは推測で10パターンの修正を試す→ どれも当たらない
OK: ログに「spinStartedAt: 1712345678000 / Date.now(): 1712345681000 / delay = 0」を含めて渡す → 即座に原因特定
Principle 17
型は先に決めろ
TypeScriptの型定義は実装の最初に設計する。後から型を変えると、変更が波及して全コンポーネントに影響する。型定義が曖昧なまま実装を進めると、anyやas unknown asで誤魔化したコードが積み上がる。型を「設計書」として捉え、コードより先に存在させる。
NG: 実装を進めながら型を後付けする → スキーマ変更のたびに全コンポーネントが壊れる
OK: OgoRouletteでは型定義をlib/ディレクトリに集約。DB型はSupabaseから自動生成。スキーマ変更がTypeScriptエラーとして即検出された。
Principle 18（ボーナス）
AIが書いたコードを自分のコードにする
AIが生成したコードをコピペしただけでは、そのコードを「理解したコード」にならない。自分でコメントを書く・変数名を変える・テストを書く、のどれかをやると理解が深まる。将来バグが起きたとき、理解していないコードは直せない。
NG: AIのコードをそのままコミット → 3ヶ月後に「なんでこうなってるの？」と自分でも理解できなくなる
OK: AIのコードに自分でコメントを付ける。「このロジックはXXのときにYYが起きるのを防ぐため」と理解を言語化する。

よくある失敗
ISSUE雑・AI丸投げ・検証不足・状態管理放棄・MVP未絞りの5大失敗パターン
⚠️
失敗パターンを知ることで回り道を省ける OgoRouletteで実際に踏んだ失敗、または踏みかけた失敗を正直に公開する。これを読んだあなたは同じ轍を踏まずに済む。
失敗01

ISSUEが雑

症状: 「〇〇を直す」という1行のISSUEが多い。バグが直らない・AIへの指示が通じない・実装が何を目指しているかわからない。

原因: ISSUEを書く行為の価値を理解していない。タスク管理ツールに慣れすぎて、1行で済ませる習慣がついている。

処方箋: 第3章のISSUEフォーマットを使う。特に「受け入れ条件」を必ず書く。受け入れ条件がないISSUEは永遠に完了しない。

失敗02

AIに丸投げ

症状: AIが出したコードをそのままコミットする。しばらくすると動かないコードが増えて、デバッグが追いつかなくなる。

原因: 「AIが速く書いてくれる」の恩恵にフォーカスして、「AIが間違える」というリスクを忘れている。

処方箋: AIのコードは必ずレビューする。特に「状態管理」「認証・認可」「非同期処理」の3領域は入念に確認する。自分が理解できないコードはマージしない。

失敗03

検証不足

症状: 「実装した→完了」で次に進む。気づいたらバグが積み上がっている。品質が上がっている実感がない。

原因: 検証を「別の作業」として捉えている。実装と検証はセットだという認識がない。

処方箋: 「ISSUEを閉じる前にgstack評価を必ず走らせる」をルールにする。数値が改善していない場合はISSUEを閉じない。

失敗04

状態管理の放棄

症状: Propsのバケツリレーが深くなる・useEffectが乱立する・どこで状態が変わっているかわからなくなる。

原因: 「とりあえず動けばいい」で進め、状態管理の設計を後回しにし続けた結果。AIが局所的に正しいコードを書いても、全体の整合性が崩れる。

処方箋: 月1回、状態管理のリファクタISSUEを作る。T-04（リファクタテンプレート）を使ってAIと一緒に整理する。

🎯
失敗05: MVPを絞れない 症状: 作り始めると追加したい機能が次々出てきて、いつまでもリリースできない。処方箋: T-01（要件定義テンプレート）を開発開始前に必ず使う。「なければアプリが成立しないか？」という問いで機能を篩にかける。答えがNoなら全部MVP後だ。OgoRouletteはMVPの機能を3つに絞ったから2週間でリリースできた。

✅
成功パターンは再現できる OgoRouletteで機能した成功パターンを、あなたのプロジェクトに転用するための具体的な方法を示す。パターンを知れば、同じ結果を出せる。

11.1 小さく回す（最重要成功パターン）

1ループ1改善の徹底が、最も重要な成功パターンだ。毎日1〜3個のISSUEを消化する習慣が、継続的な改善で200+ ISSUEになった。小さく回すことで、「改善した事実」が積み上がる。

良いループ（小さい）
ISSUEを1個書く → AIに渡す → 実装する → 検証する → 次のISSUEへ。何が改善したか明確。バグの発生源が特定しやすい。
悪いループ（大きすぎる）
10個のISSUEをまとめて実装しようとする → AIへの指示が複雑になる → 何が原因でバグるかわからなくなる → 戻し方がわからなくなる。

11.2 検証を必ず入れる

「実装 → 検証」をセットにすること。検証なしの実装は「やった気分」だけで品質は上がらない。gstackによる数値化が有効な理由は「感覚ではなく数値で議論できる」からだ。検証のタイミング：各ISSUEの実装完了後・大きな機能追加後・リリース前・月次の品質確認。

11.3 感情を設計に入れる

「機能が動く」と「使いたくなる」の間には大きな差がある。この差を埋めるのが感情設計だ。

😰

ドキドキ

不確実性を演出する。OgoRoulette: ルーレットが止まる直前に減速を段階的にして「もしかして自分か...!?」という緊張感を作った。

🎉

驚き

意外性を設計する。WinnerCardの紙吹雪は「アプリがこんな反応をするとは思わなかった」という驚きを生む。

😂

笑い

グループで使うアプリは「笑い」が最強の口コミエンジンだ。当選者のリアクションがSNSに投稿されるとそれ自体が宣伝になる。

11.4 ISSUEをストックする

「思いついたらすぐISSUE化する」習慣が、長期的な品質向上の源泉だ。アプリを使っていて「ここが気になる」と思った瞬間に、その場でISSUEを書く。OgoRouletteでは常に20〜30個のISSUEがBacklogにある状態を維持した。これが「手が止まる」という状況を完全になくした。ISSUEリストを見れば、常に「次の1手」が明確だ。

13.1 AI駆動開発の本質

AI駆動開発 ＝ 思考を言語化して、AIと協働して、人間が判断する
— この教科書で最も重要な1行

① 思考の言語化

AIは「速く書けるツール」ではなく「自分が言語化できていないことを実装に落としてくれるパートナー」だ。ボトルネックはコーディング速度ではなく思考の言語化速度だ。

② AIとの協働

AIは候補を出す。人間が選択する。この分担を崩すとAIへの丸投げになる。丸投げは必ず失敗する。適切な協働には正確な文脈転送が必要だ。

③ 人間が判断

アーキテクチャ・状態管理・UX・ビジネス判断は常に人間が握る。「AIはこう言っているから正しい」という思考停止が最も危険だ。

13.2 再現5ステップ

OgoRouletteと同じことを再現するための5ステップだ。

1

CLAUDE.mdを書く（30分）

プロジェクトの「開発憲法」を最初に作る。技術スタック・禁止事項・命名規則・AIへの指示ルールを書く。このファイルがAIとのセッションをまたいでも一貫性を保つ。

2

T-01でMVPを絞る（1時間）

templates/t01-requirements.mdを使って、MVPの範囲を3機能以内に絞る。「なければアプリが成立しないか？」という問いで篩にかける。

3

最初のISSUEを書く（30分）

MVP最初の機能のISSUEを書く。第3章のフォーマットを使って、背景・問題・解決策・受け入れ条件を書く。

4

ループを1周回す

ISSUEを書く → テンプレートにはめてAIに渡す → 実装する → gstack評価する → 次のISSUEへ。

5

ループを続ける

1日1〜3個のISSUEを消化する習慣をつける。続ければ100 ISSUEを超える。100 ISSUEは「100回の改善」の証拠だ。

13.3 今日からのアクション（明日から始められる3つ）

1

手元のプロジェクトにCLAUDE.mdを作る

目安時間: 30分

CLAUDE.mdがなければ今すぐ作る。技術スタック・禁止事項・AIへの基本指示を書く。これだけでAIとのセッションの品質が上がる。「AIに同じことを2回言ったら」それはCLAUDE.mdに書くべき内容だ。1週間後には、AIへの補足指示がほぼゼロになる。

2

気になっていることをISSUEにする

目安時間: 15分

「なんか気になっていること」を1個、第3章のフォーマットでISSUEに書く。受け入れ条件を必ず書く。「書けた」と思ったらAIに渡してフィードバックをもらう。ISSUEを書く過程で問題が明確になり、解決策が見えてくる体験をしてほしい。

3

T-05でアプリを評価する

目安時間: 30分

今作っているアプリに templates/t05-gstack-review.md を使って評価を走らせる。数値が出たら改善ISSUEを1個作る。これが品質向上ループの第一歩だ。「なんとなく」から「数値で見る」への転換がAI駆動開発の最初の変化だ。

13.4 最後に

OgoRouletteは「飲み会の奢り決め」という小さなアイデアから始まった。技術的な新しさは何もない。でも約3週間でリリースし、その後も毎日ISSUEを書いてAIと協働し続けた結果、使いたいと思えるアプリができた。200 ISSUEは200回の「よりよくなった」の積み重ねだ。
— AI駆動開発は「AIがいれば誰でも良いものが作れる」ではない。「AIと適切に協働できる人が、圧倒的なスピードで高品質なものを作れる」だ。

適切な協働には「思考の言語化」「文脈の転送」「人間の判断」が必要で、これらはスキルだ。スキルは練習で身につく。この教科書は、そのスキルを身につけるための地図だ。

地図を持って、歩き始めてほしい。
